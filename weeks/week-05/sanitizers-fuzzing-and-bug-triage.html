Lesson 5.4: Sanitizers, Fuzzing, and Bug Triage

This lesson explains how security teams use sanitizers, fuzzing, and structured bug triage to find memory and parser bugs safely before release.

Estimated reading time300–430 minutes
Difficulty levelBeginner → MS foundation
Required backgroundLessons 5.1–5.3 recommended; no prior fuzzing experience required
Practice time5–8 hours

1. Introduction

A program can look correct during normal testing and still fail when it receives unusual input. Sanitizers, fuzzing, and bug triage help security teams find and understand those hidden failures before attackers, customers, or production systems find them.

A sanitizer is a compiler/runtime testing tool that adds extra checks to a program. It watches for errors such as out-of-bounds memory access, use-after-free, undefined behavior, data races, and memory leaks. A fuzzer repeatedly gives a program many generated inputs to discover crashes, hangs, memory errors, and unexpected behavior. Bug triage is the disciplined process of turning a raw crash into a useful engineering decision: reproduce it, classify it, assess impact, assign severity, fix root cause, and prevent regression.

Why the topic exists

Human reviewers cannot manually test every possible file, packet, image, string, length field, encoding, and edge case. Modern software receives enormous amounts of untrusted input. Automated testing tools extend human review by exploring many more execution paths than a person can cover manually.

Why it matters

  • Many serious vulnerabilities start as ordinary-looking parser bugs, boundary mistakes, integer mistakes, or lifetime mistakes.
  • Sanitizers can expose memory errors at the exact failing line during local testing.
  • Fuzzing can discover inputs that developers did not anticipate.
  • Triage prevents panic, exaggeration, and dismissal by separating evidence, root cause, exploitability, severity, and organizational risk.

Where it is used

This topic is used in secure software engineering, browser security, operating-system development, cryptographic libraries, cloud services, firmware testing, incident response, vulnerability research, continuous integration, and software supply-chain assurance.

Who uses it

  • Developers use sanitizers and fuzzers during testing.
  • Security engineers use them to discover and reproduce vulnerability classes safely.
  • Build engineers integrate them into continuous integration pipelines.
  • Incident responders use triage evidence to determine whether crashes indicate attack activity or reliability defects.
  • Researchers design better fuzzers, sanitizers, symbolic execution systems, and automated root-cause tools.
Safety boundary: all fuzzing in this lesson is local-only against toy programs that you own. Do not fuzz, scan, crash, or stress real services, third-party applications, university systems, company systems, APIs, or public websites without explicit written authorization.

2. Learning Outcomes

  • Define sanitizer, instrumentation, AddressSanitizer, UndefinedBehaviorSanitizer, MemorySanitizer, ThreadSanitizer, LeakSanitizer, fuzzer, fuzz target, harness, corpus, seed input, mutation, coverage guidance, crash, hang, minimization, deduplication, reproduction, root cause, severity, priority, and regression test.
  • Explain why normal unit tests are necessary but insufficient for input-heavy software.
  • Describe how sanitizers add runtime checks and why sanitized builds are usually testing builds rather than production builds.
  • Explain how mutation-based and coverage-guided fuzzing work at a high level.
  • Write a safe toy fuzz harness for a local parser.

  • Explain how fuzzing fits into CI/CD and release integrity without treating it as a complete security guarantee.

3. Prerequisites

Program input

Input is data given to a program. It may come from a user, a file, a packet, a database record, an API request, an environment variable, or another process.

Parser

A parser converts raw input into structured meaning. Example: an image parser reads bytes and interprets them as width, height, pixels, metadata, and compression fields.

Crash

A crash is an abnormal program stop. A crash is evidence of a defect. It is not automatically proof of remote code execution.

Stack trace

A stack trace is a list of active function calls at the time of an error. It helps identify where the program failed and how it reached that point.

Compiler

A compiler translates source code into a runnable program. Sanitizers usually require compiler support because the compiler inserts extra checking code.

Build configuration

A build configuration controls compiler options. Debug, release, sanitizer, and fuzzing builds often behave differently because they use different flags and runtime libraries.

4. Core Concepts

4.1 Testing layers

Software testing is not one technique. A strong program uses multiple layers:

LayerQuestion answeredSecurity value
Unit testsDoes this small function behave as expected?Catches known edge cases and prevents regression.
Integration testsDo components work together?Catches boundary mismatches between modules.
Security testsDo known abuse cases fail safely?Checks authentication, authorization, validation, and error handling.
Sanitizer testsDoes the program perform invalid memory or undefined operations?Exposes hidden implementation errors.
Fuzz testsWhat happens with many unexpected inputs?Discovers unknown edge cases and crash paths.

4.2 Sanitizer

A sanitizer is a testing aid that instruments a program. Instrumentation means the compiler adds extra code that checks operations at runtime. The program runs slower, but it can report defects that would otherwise remain silent.

4.3 AddressSanitizer

AddressSanitizer, often called ASan, detects many memory-access errors, including heap buffer overflow, stack buffer overflow, global buffer overflow, and use-after-free. It is widely used during testing of C and C++ programs.

4.4 UndefinedBehaviorSanitizer

UndefinedBehaviorSanitizer, often called UBSan, detects operations where the C or C++ language does not define what should happen. Examples include signed integer overflow, certain invalid shifts, and invalid assumptions about object types.

4.5 MemorySanitizer

MemorySanitizer, often called MSan, detects use of uninitialized memory. It is more demanding to deploy because the program and its libraries often need compatible instrumentation.

4.6 ThreadSanitizer

ThreadSanitizer, often called TSan, detects data races in multithreaded programs. A data race occurs when threads access the same memory concurrently and at least one access writes, without proper synchronization.

4.7 LeakSanitizer

LeakSanitizer, often called LSan, detects memory that was allocated but not freed. Memory leaks are often reliability defects; in long-running services, they can become availability risks.

4.8 Fuzzer

A fuzzer is a testing tool that generates many inputs and feeds them to a program. The goal is to find crashes, hangs, sanitizer findings, logic failures, or unexpected behavior.

4.9 Fuzz target and harness

A fuzz target is the function or component being tested. A harness is a small wrapper that accepts arbitrary bytes and passes them safely into that target. A good harness is stable, deterministic, fast, and isolated.

4.10 Corpus and seed inputs

A corpus is a collection of input files used by a fuzzer. Seed inputs are starting examples. For an image parser, seed inputs might include tiny valid images and intentionally simple invalid files.

4.11 Mutation

Mutation means changing an existing input. A fuzzer may flip bits, insert bytes, delete bytes, duplicate ranges, change numbers, or combine parts of inputs.

4.12 Coverage-guided fuzzing

Coverage describes which parts of code executed. A coverage-guided fuzzer keeps inputs that reach new code paths. The intuition is simple: if an input reaches a new branch, it may be closer to hidden bugs.

4.13 Crash, hang, and timeout

  • Crash: the program stops abnormally.
  • Hang: the program gets stuck and stops making progress.
  • Timeout: the fuzzer stops a run because it exceeded a configured time limit.

4.14 Reproduction

Reproduction means running the same program version with the same input and observing the same bug. Without reproduction, triage is weak because the evidence may be incomplete or environment-specific.

4.15 Minimization

Minimization reduces a crashing input to the smallest or simplest input that still triggers the same bug. A small reproducer is easier to understand, safer to store, and better for regression tests.

4.16 Deduplication

Deduplication groups multiple crash inputs that have the same root cause. This prevents a team from counting one bug as many separate issues.

4.17 Root cause

The root cause is the actual design or coding error, not just the line where the program crashed. Example: the crash might occur during a copy, but the root cause may be trusting an unvalidated length field read earlier.

4.18 Severity vs priority

Severity describes technical impact. Priority describes how soon the organization should act. A severe bug in unused code may have lower priority than a medium-severity bug in an internet-facing service.

4.19 Regression test

A regression test proves that a fixed bug does not return. The minimized crashing input often becomes a regression test.

4.20 Limitations

Sanitizers and fuzzers are powerful but incomplete. They do not prove absence of bugs. They depend on harness quality, input corpus, coverage, build flags, runtime limits, and whether the relevant code is reachable.

7. Mathematical Foundations

7.1 Input-space explosion

If a program accepts n bytes, and each byte has 256 possible values, the number of possible inputs is:

256^n

Worked example: a 4-byte input has 256^4 = 4,294,967,296 possible values. Exhaustively testing all inputs quickly becomes impossible as n grows.

7.2 Coverage percentage

A simple line-coverage percentage can be written as:

coverage = executed_lines / total_lines × 100%

Worked example: if 840 lines execute out of 1,200 lines, coverage is 840 / 1200 × 100% = 70%. High line coverage is useful evidence but does not prove security.

7.3 Fuzzing throughput

Fuzzing throughput can be estimated as:

executions_per_second = total_executions / elapsed_seconds

Worked example: if a fuzzer runs 600,000 tests in 300 seconds, throughput is 2,000 executions/second. Faster is useful only if the harness is testing meaningful behavior.

7.4 Bug density caution

A naive bug density metric is:

bugs_per_KLOC = bugs_found / thousands_of_lines_of_code

This is incomplete because bugs are not evenly distributed. A parser handling untrusted input may deserve more attention than a larger internal formatting module.

7.5 Probability intuition

Random fuzzing becomes inefficient when only very specific input structures reach deep code. Coverage-guided fuzzing improves the search by keeping inputs that move execution into new areas.

8. Practical Examples

Linux examples

# Build a local C program with AddressSanitizer and UndefinedBehaviorSanitizer.
clang -fsanitize=address,undefined -g -O1 -Wall -Wextra parser.c -o parser_san

# Run the local test binary with a local test input.
./parser_san tests/input.txt

# Save output for triage.
./parser_san tests/input.txt > triage-output.txt 2> triage-error.txt

# Never point fuzzers at systems you do not own or have written authorization to test.

Windows examples

# PowerShell: keep a local triage folder.
New-Item -ItemType Directory -Force triage

# Save build information for reproducibility.
cl 2>&1 | Select-Object -First 5 | Out-File triage\compiler-version.txt

# MSVC supports AddressSanitizer for supported C/C++ configurations.
# Project settings and compiler versions matter, so always document the exact build.

Python examples

Python is memory-safe for ordinary Python lists and strings, but it can still help with fuzz-style testing of parsers and validation logic.

import random
import string

# Toy function: accepts only course codes such as CYB101.
def parse_course_code(text: str) -> bool:
    return (
        len(text) == 6
        and text[:3].isalpha()
        and text[:3].isupper()
        and text[3:].isdigit()
    )

alphabet = string.ascii_letters + string.digits + "-_!"

for _ in range(1000):
    candidate = "".join(random.choice(alphabet) for _ in range(random.randint(0, 12)))
    result = parse_course_code(candidate)
    # This local loop checks that the function returns a Boolean for many inputs.
    assert isinstance(result, bool)

print("local fuzz-style validation check finished")

Network examples

Network services should not be fuzzed over the internet. In a safe lab, fuzz the parser function directly instead of sending traffic to a live service. This is faster, safer, and easier to reproduce.

Web examples

A web upload feature may call a native image parser. A safe test targets the local parser library with saved sample files, not a public upload endpoint.

Cloud examples

Cloud-native teams often run sanitizer and fuzzing jobs in isolated CI runners with resource limits, no production credentials, no access to customer data, and artifact retention policies for crash reproducers.

9. Code Examples

9.1 Toy parser with a bug

This program parses a tiny local format: the first byte says how many bytes to copy. The bug is that the length byte is trusted without checking the destination buffer size.

#include <stdint.h>
#include <stddef.h>
#include <string.h>
#include <stdio.h>

int parse_record_bad(const uint8_t *data, size_t size) {
    uint8_t name[8];

    if (size < 1) {
        return 0;
    }

    uint8_t length = data[0];

    // BUG: length comes from input and may be larger than sizeof(name).
    // BUG: the code also does not verify that size contains length bytes.
    memcpy(name, data + 1, length);

    printf("parsed length: %u\n", length);
    return 1;
}

int main(void) {
    uint8_t input[] = {20, 'A', 'A', 'A', 'A', 'A'};
    return parse_record_bad(input, sizeof(input));
}

Line-by-line explanation

  • uint8_t name[8] creates an 8-byte local buffer.
  • length = data[0] reads a length value from untrusted input.
  • memcpy(name, data + 1, length) copies length bytes into the buffer.
  • If length is greater than 8, the copy writes outside name.
  • If the input does not actually contain length bytes after the first byte, the copy may also read outside the input array.

9.2 Safer parser

#include <stdint.h>
#include <stddef.h>
#include <string.h>
#include <stdio.h>
#include <stdbool.h>

bool parse_record_safe(const uint8_t *data, size_t size) {
    uint8_t name[8];

    if (data == NULL) {
        return false;
    }

    if (size < 1) {
        return false;
    }

    size_t length = data[0];

    // Check destination capacity.
    if (length > sizeof(name)) {
        return false;
    }

    // Check source availability.
    if (length > size - 1) {
        return false;
    }

    memcpy(name, data + 1, length);
    printf("parsed safe length: %zu\n", length);
    return true;
}

int main(void) {
    uint8_t input[] = {5, 'A', 'l', 'i', 'c', 'e'};
    return parse_record_safe(input, sizeof(input)) ? 0 : 1;
}

Why the safer version is better

  • It rejects null pointers.
  • It checks that at least one length byte exists.
  • It converts the length to size_t for size comparisons.
  • It checks destination capacity before copying.
  • It checks source availability before copying.
  • It returns success/failure instead of pretending every input is valid.

9.3 A minimal libFuzzer-style harness

This harness is for local learning. It accepts arbitrary bytes and passes them into the parser. It does not open network sockets, scan hosts, or contact external systems.

#include <stdint.h>
#include <stddef.h>
#include <stdbool.h>

bool parse_record_safe(const uint8_t *data, size_t size);

int LLVMFuzzerTestOneInput(const uint8_t *data, size_t size) {
    // The fuzzer supplies data and size.
    // The harness calls the local parser and ignores ordinary reject/accept results.
    (void)parse_record_safe(data, size);

    // Returning 0 tells the fuzzer that the run completed normally.
    return 0;
}

9.4 Common mistakes

  • Fuzzing a whole command-line program when a smaller parser function would be faster and more precise.
  • Leaving network calls, file deletion, cloud credentials, or production dependencies reachable from the fuzz harness.
  • Ignoring timeouts and memory limits.
  • Keeping huge crash files when a minimized input would be easier to understand.
  • Fixing the crash location but not the root cause.
  • Removing an assertion without understanding why it failed.

10. Security Perspective

Common attacks and failures

  • Malformed file input: a crafted document, image, archive, or media file reaches a parser bug.
  • Malformed network message: unusual protocol fields reach parser code in a service.
  • Denial of service: a crash or hang makes a service unavailable.
  • Information disclosure: an out-of-bounds read exposes memory contents.
  • Privilege-boundary failure: a bug in high-privilege code has greater impact than the same bug in isolated low-privilege code.

Common vulnerabilities

  • Out-of-bounds read or write.
  • Use-after-free.
  • Double free.
  • Signed integer overflow.
  • Uninitialized memory use.
  • Race condition.
  • Parser state-machine confusion.
  • Resource-exhaustion loops.

Defensive strategies

  • Use memory-safe languages where practical.
  • Keep unsafe code small, reviewed, and isolated.
  • Compile test builds with sanitizers.
  • Fuzz input parsers and protocol decoders.
  • Use regression tests from minimized reproducers.
  • Run fuzzing in isolated environments with resource limits.
  • Track coverage and known gaps.
  • Document severity, priority, and residual risk.

Detection methods

  • Sanitizer reports.
  • Crash dumps and core files.
  • Application logs showing repeated abnormal termination.
  • CI fuzzing results.
  • Production telemetry showing crash loops or unusual input patterns.
  • EDR or server monitoring showing repeated process failure.

Mitigation

  • Patch root cause, not just symptoms.
  • Reject malformed input early.
  • Add bounds and lifetime checks.
  • Reduce parser privilege.
  • Sandbox risky parsers.
  • Deploy compiler and OS mitigations.
  • Move high-risk parsing code toward memory-safe implementations where feasible.

Best practices

  • Make fuzzing part of secure SDLC, not a one-time exercise.
  • Keep sanitized builds reproducible.
  • Use small deterministic harnesses.
  • Save minimized reproducers securely.
  • Assign ownership for triage queues.
  • Measure fix time and recurrence rate.

12. Research Perspective

Current research tries to find deeper bugs faster, explain findings better, and reduce the cost of continuous testing.

Research directions

  • Coverage-guided fuzzing: improving how fuzzers choose inputs that explore new code.
  • Grammar-aware fuzzing: generating inputs that follow file or protocol structure so deeper parser logic is reached.
  • Hybrid fuzzing: combining fuzzing with symbolic execution or constraint solving.
  • Directed fuzzing: steering input generation toward recent code changes or high-risk functions.
  • Continuous fuzzing: running fuzzers automatically as projects change.
  • Crash deduplication: grouping findings by root cause rather than superficial stack similarity.
  • Automated root-cause analysis: helping developers understand why a bug happened.
  • Fuzzing AI/ML systems: exploring model inputs, prompt interfaces, data pipelines, and format parsers.

Open problems

  • How can teams measure security improvement rather than only counting crashes?
  • How can fuzzers generate semantically valid inputs for complex protocols?
  • How can fuzzing scale to distributed systems without causing unsafe side effects?
  • How can research tools produce developer-friendly reports instead of overwhelming triage queues?
  • How can fuzzing integrate with memory-safe migrations and supply-chain assurance?

Notable research and practice areas

  • Coverage-guided fuzzing engines such as AFL-family approaches and libFuzzer-style approaches.
  • Large-scale continuous fuzzing services such as OSS-Fuzz.
  • Sanitizer design for memory safety, undefined behavior, data races, and leak detection.
  • Empirical studies of how fuzzing bugs are introduced, detected, and fixed.

13. Case Studies

Case Study 1: OSS-Fuzz and continuous fuzzing

What happened: OSS-Fuzz applied continuous fuzzing to open-source projects at scale. Its purpose is to improve the security and stability of common open-source infrastructure through modern fuzzing and scalable execution.

Root cause addressed: many bugs live in parsers and libraries that receive complex inputs. Continuous fuzzing helps find them before they become production failures.

Impact: open-source projects receive automated bug reports with reproducer inputs, sanitizer findings, and build context.

Lessons learned: fuzzing works best when projects maintain good harnesses, fix findings quickly, and keep fuzzing active as the code changes.

Case Study 2: Heartbleed and memory over-read

What happened: Heartbleed was a serious OpenSSL vulnerability involving a bounds-check failure in TLS heartbeat handling, allowing memory over-read under affected conditions.

Root cause: the implementation trusted a length value without sufficiently validating that the corresponding input actually contained that much data.

Impact: sensitive memory could be exposed from affected systems.

Lessons learned: input length fields are untrusted data. Boundary checks and negative tests are essential for protocol parsers.

Case Study 3: Image parser risk

What happened: Image libraries historically receive many vulnerability reports because image formats contain lengths, chunks, compression data, metadata, and transformations.

Root cause pattern: complex file formats create many parser states and boundary conditions.

Impact: a single parser library may be reused by browsers, servers, desktop applications, and mobile apps.

Lessons learned: parsers deserve fuzzing, sanitizer builds, strict validation, and strong regression-test suites.

Case Study 4: CI triage overload

What happened: A team enabled fuzzing and received dozens of crashes in the first week.

Root cause: many crashes were duplicates from the same unchecked length field.

Impact: without deduplication, the queue looked larger than it was.

Lessons learned: triage must reproduce, minimize, and group findings before management decisions are made.

14. Common Misconceptions

MisconceptionWhy it is incorrect
“Fuzzing proves software is secure.”Fuzzing finds bugs; it does not prove absence of bugs.
“A crash always means remote code execution.”A crash is evidence of a defect. Exploitability requires separate analysis.
“Sanitizers should always run in production.”Sanitized builds usually have performance and compatibility costs; they are mainly testing builds.
“More fuzzing time always means better results.”Harness quality, coverage, and corpus quality matter as much as time.
“Random bytes are enough for every parser.”Structured formats often need grammar-aware or seed-based approaches to reach deep code.
“If the fuzzer finds no crashes, the code is safe.”The fuzzer may not have reached vulnerable code paths.
“Fixing the crashing line fixes the bug.”The real root cause may be earlier validation, state handling, or lifetime management.
“All findings deserve the same urgency.”Reachability, exposure, privilege, impact, and exploitability affect priority.

15. Summary

Sanitizers, fuzzing, and bug triage help teams discover and understand defects that ordinary tests often miss. Sanitizers instrument programs and report invalid operations. Fuzzers generate many inputs and explore unexpected paths. Triage turns raw findings into engineering decisions by reproducing, minimizing, deduplicating, classifying, assessing, fixing, and verifying. These tools are not magic and do not prove security, but they are essential parts of a mature secure software lifecycle.

16. Glossary

AddressSanitizer: A sanitizer that detects many memory-access errors such as out-of-bounds access and use-after-free.

Bug triage: The process of evaluating a bug report to determine reproduction, root cause, impact, severity, priority, ownership, and next action.

Corpus: A collection of input samples used by a fuzzer.

Coverage: A measure of which code paths, lines, blocks, or edges executed during testing.

Coverage-guided fuzzing: Fuzzing that keeps inputs that cause execution to reach new code areas.

Crash: An abnormal program termination.

Deduplication: Grouping multiple findings that have the same root cause.

Fuzz harness: A wrapper that feeds arbitrary test input into the component being fuzzed.

Fuzz target: The function, parser, or component under fuzz testing.

Fuzzer: A tool that generates many inputs to test software behavior.

Hang: A condition where a program stops making progress.

Instrumentation: Extra code inserted into a program to observe or check behavior.

LeakSanitizer: A sanitizer that detects memory leaks.

Minimization: Reducing a failing input to a smaller input that still triggers the same failure.

Mutation: A change made to an input to produce a new test case.

Regression test: A test that ensures a previously fixed bug does not return.

Reproducer: An input or procedure that reliably triggers a bug.

Sanitizer: A testing tool that adds runtime checks to detect invalid behavior.

Seed input: A starting input supplied to a fuzzer.

ThreadSanitizer: A sanitizer that detects data races in multithreaded programs.

UndefinedBehaviorSanitizer: A sanitizer that detects certain undefined behavior in C/C++ programs.

18. Practice Lab

Title

Safe Local Sanitizer and Fuzz-Style Triage Lab

Prerequisites

  • Linux VM, macOS terminal, or Windows with WSL.
  • GCC or Clang installed. Clang is recommended for sanitizer/fuzzer examples.
  • No network targets. No public systems. No real customer data.

Objectives

  • Compile a toy parser with sanitizers.
  • Trigger a local sanitizer finding.
  • Patch the root cause.
  • Create a minimized reproducer.
  • Write a triage note.

Step 1: Create lab folder

mkdir -p sanitizer-fuzz-triage-lab
cd sanitizer-fuzz-triage-lab

Step 2: Create the toy parser

cat > parser_bad.c <<'C'
#include <stdint.h>
#include <stddef.h>
#include <string.h>
#include <stdio.h>

int parse_record(const uint8_t *data, size_t size) {
    uint8_t small[8];

    if (size < 1) {
        return 0;
    }

    uint8_t length = data[0];
    memcpy(small, data + 1, length); // intentional local lab bug
    printf("length=%u\n", length);
    return 1;
}

int main(void) {
    uint8_t input[] = {20, 'A', 'B', 'C'};
    return parse_record(input, sizeof(input));
}
C

Step 3: Compile with sanitizers

clang -fsanitize=address,undefined -g -O1 -Wall -Wextra parser_bad.c -o parser_bad
./parser_bad

Expected result: the program should produce an AddressSanitizer report or related sanitizer finding. Exact output varies by compiler and platform.

Step 4: Patch the parser

cat > parser_safe.c <<'C'
#include <stdint.h>
#include <stddef.h>
#include <string.h>
#include <stdio.h>
#include <stdbool.h>

bool parse_record(const uint8_t *data, size_t size) {
    uint8_t small[8];

    if (data == NULL || size < 1) {
        return false;
    }

    size_t length = data[0];

    if (length > sizeof(small)) {
        return false;
    }

    if (length > size - 1) {
        return false;
    }

    memcpy(small, data + 1, length);
    printf("length=%zu\n", length);
    return true;
}

int main(void) {
    uint8_t input[] = {3, 'A', 'B', 'C'};
    return parse_record(input, sizeof(input)) ? 0 : 1;
}
C

clang -fsanitize=address,undefined -g -O1 -Wall -Wextra parser_safe.c -o parser_safe
./parser_safe

Step 5: Optional local fuzz-style loop in Python

This does not use libFuzzer. It simply generates local inputs for a Python version of the parser rule.

cat > parser_rule.py <<'PY'
import random

MAX_LEN = 8

def parse_record(data: bytes) -> bool:
    if len(data) < 1:
        return False
    length = data[0]
    if length > MAX_LEN:
        return False
    if length > len(data) - 1:
        return False
    return True

for i in range(5000):
    size = random.randint(0, 32)
    data = bytes(random.randint(0, 255) for _ in range(size))
    result = parse_record(data)
    assert isinstance(result, bool)

print("safe local fuzz-style loop completed")
PY
python3 parser_rule.py
