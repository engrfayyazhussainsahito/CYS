Lesson 5.2: Stack, Heap, and Process Memory

This lesson explains how a running program is arranged in memory: stack frames for function calls, heap allocations for dynamic objects, executable code, global data, shared libraries, virtual memory mappings, and permissions.

Estimated reading time300–420 minutes
Difficulty levelBeginner → MS foundation
Required backgroundLesson 5.1 recommended; no assembly background required
Practice time5–8 hours

1. Introduction

A running program is not just a file on disk. When you double-click an application or run a command, the operating system creates a process. That process receives a private-looking memory space where it can keep instructions, variables, temporary data, dynamically allocated objects, shared libraries, environment variables, and stack frames.

This lesson explains that memory space from first principles. You will learn the difference between the stack, the heap, code sections, global data, memory mappings, virtual memory, pages, and permissions. These concepts are needed before studying buffer overflows, use-after-free bugs, memory forensics, malware behavior, and operating-system mitigations.

Why the topic exists

Programs need places to store data while they run. Some data lives only during a function call, some lives until the program frees it, some is loaded from the executable file, and some is mapped from shared libraries or files. Operating systems organize these regions so programs can run efficiently and with isolation.

Why it matters

  • Many vulnerabilities are described in terms of stack buffers, heap objects, memory mappings, or address-space layout.
  • Defenders need to interpret crash reports, sanitizer logs, process maps, core dumps, and vulnerability advisories.
  • Security controls such as address space layout randomization, no-execute memory, guard pages, and sandboxing depend on the process-memory model.

Where it is used

This topic appears in secure coding, malware analysis, exploit mitigation, digital forensics, reverse engineering, incident response, performance debugging, compiler security, and operating-system hardening.

Who uses it

  • Developers use it to reason about object lifetime, recursion, allocation, and crashes.
  • Security engineers use it to triage memory-corruption advisories and validate mitigations.
  • Forensic analysts use it to interpret memory snapshots and process maps.
  • Researchers use it to design safer runtimes, allocators, sanitizers, and hardware-assisted protections.
Safety boundary: all examples inspect your own local processes or toy programs. Do not inspect, dump, attach to, fuzz, or manipulate processes that you do not own or administer.

2. Learning Outcomes

  • Define process, address space, virtual memory, page, stack, stack frame, call stack, heap, allocation, deallocation, memory mapping, segment, section, shared library, guard page, and memory permission.
  • Explain how the operating system creates an isolated memory view for each process.
  • Distinguish stack memory from heap memory using lifetime, ownership, size, and common failure modes.
  • Read a simple process-memory map and identify code, heap, stack, shared libraries, and permissions.
  • Explain why local variables usually live on the stack and dynamically allocated objects usually live on the heap.
  • Use safe local commands to inspect your own process memory layout.
  • Explain stack overflow, heap corruption, memory leak, dangling pointer, and double free at a root-cause level.
  • Describe defensive controls such as ASLR, NX/DEP, guard pages, sanitizers, allocator hardening, and least privilege.
  • Write a short technical report comparing stack and heap risks in a toy program.

3. Prerequisites

Process

A process is a running instance of a program. The same program can have many processes running at once, and each process has its own process identifier, memory mappings, open files, and execution state.

Memory address

A memory address is a number used to locate bytes. In modern operating systems, a process usually sees virtual addresses, not raw physical memory addresses.

Pointer

A pointer is a variable whose value is an address. A pointer can point to stack memory, heap memory, global data, mapped files, or invalid memory.

Function call

A function call transfers control to another block of code. The program must remember where to return and must store function-local data somewhere. That temporary function data is commonly kept in a stack frame.

Hexadecimal notation

Process memory maps often show addresses in hexadecimal because memory addresses are large binary values. For example, 0x1000 is 4096 in decimal.

4. Core Concepts

4.1 Process memory is a controlled illusion

Each process appears to have its own large continuous memory space. This is mostly an illusion created by the operating system and hardware memory-management unit. The process uses virtual addresses. The operating system maps those virtual addresses to physical memory, files, shared libraries, or nothing at all.

Analogy: a hotel guest sees room numbers and hallways. The guest does not know the building’s wiring, plumbing, or structural layout. Virtual memory is similar: the process sees organized addresses, while the operating system manages the real hardware underneath.

4.2 Address space

An address space is the set of addresses a process can use. If a process tries to use an address that is not mapped or lacks the needed permission, the operating system can stop it with a memory fault.

4.3 Pages

Memory is commonly managed in fixed-size chunks called pages. A common page size is 4096 bytes, although systems can use other sizes. Permissions are often tracked per page or per mapping.

4.4 Memory permissions

Memory regions have permissions such as read, write, and execute. A code page may be readable and executable, while a stack page is usually readable and writable but not executable. This separation supports no-execute defenses.

4.5 Program sections

A compiled program file contains sections. Common examples include:

  • .text: executable instructions.
  • .rodata: read-only constants and string literals.
  • .data: initialized global/static variables.
  • .bss: zero-initialized global/static variables.

4.6 Stack

The stack stores short-lived data related to function calls. Each call gets a stack frame, which can contain local variables, saved state, and return information. When the function returns, its frame is no longer valid.

Intuition: a stack is like a stack of trays. You add a tray when a function starts and remove it when the function finishes. The last tray added is the first removed.

4.7 Call stack

The call stack is the current chain of active function calls. If main() calls read_file(), and read_file() calls parse_line(), those calls form a stack.

4.8 Stack overflow

A stack overflow can occur when the stack grows beyond its allowed size, often because of uncontrolled recursion or very large local variables. In security, the term is also used for stack-based buffer overflows, where a local stack buffer is overwritten. This lesson focuses on root cause, not exploitation.

4.9 Heap

The heap stores data requested dynamically while the program runs. In C, a program can request heap memory using malloc and release it using free. In other languages, the runtime or garbage collector may manage heap objects automatically.

4.10 Heap lifetime

Heap memory can outlive the function that allocated it if a pointer is returned or stored elsewhere. That flexibility is useful, but it creates lifetime risk: memory can be leaked, freed too early, freed twice, or used after being freed.

4.11 Memory leak

A memory leak happens when a program allocates memory and loses the ability to free it. Long-running services can become unavailable if leaks slowly consume memory.

4.12 Dangling pointer

A dangling pointer points to memory that is no longer valid. Using it may crash the program, corrupt data, or expose security risk.

4.13 Memory mapping

A memory mapping connects part of a process address space to something: anonymous memory, a file, a shared library, the heap, the stack, or a special kernel-provided region.

4.14 Shared libraries

A shared library is code loaded into a process at runtime so multiple programs can reuse common functionality. Shared libraries appear as mappings in process memory.

4.15 Guard pages

A guard page is an intentionally inaccessible page placed near sensitive regions such as stacks. If a program crosses into the guard page, it triggers a fault instead of silently corrupting nearby memory.

4.16 ASLR

Address Space Layout Randomization changes where memory regions are placed each time a process runs. It does not fix bugs. It makes some classes of attack less reliable by reducing address predictability.

4.17 NX/DEP

No-execute or Data Execution Prevention marks some memory as non-executable. For example, stack and heap pages should usually not execute code.

7. Mathematical Foundations

7.1 Page size and page count

If a system uses 4096-byte pages, then memory size divided by 4096 gives the number of pages needed, rounded up.

pages_needed = ceil(bytes_needed / page_size)

Worked example: 10,000 bytes with 4096-byte pages needs ceil(10000 / 4096) = 3 pages, because two pages hold only 8192 bytes.

7.2 Address range size

If a mapping starts at address S and ends at address E, its size is:

size = E - S

Worked example: a mapping from 0x1000 to 0x3000 has size 0x2000, which is 8192 bytes.

7.3 Stack-frame growth model

If a program uses approximately F bytes per recursive call and the allowed stack size is L, then a rough upper bound on recursive depth is:

max_depth ≈ floor(L / F)

This is only a model. Real systems need additional space for saved registers, alignment, runtime metadata, guard pages, and library calls.

7.4 Pointer arithmetic reminder

For an array starting at base address B, element index i, and element size S:

address = B + i × S

This formula explains why an off-by-one index can move a pointer outside the intended object.

7.5 Entropy intuition for ASLR

If a randomized placement has n unpredictable bits, there are 2^n possible placements. More entropy means more possible layouts, but real protection depends on implementation, information leaks, architecture, and attack model.

8. Practical Examples

Linux examples

# Show memory mappings for your current shell process.
cat /proc/$$/maps | head

# Show your shell process ID.
echo $$

# Show stack-size limit for your shell session.
ulimit -s

# Show rough memory use for your shell process.
ps -o pid,comm,vsz,rss -p $$

The file /proc/PID/maps lists mapped memory regions and their permissions for a process. Access is controlled by the operating system; only inspect processes you own or administer.

Windows examples

# PowerShell: show the current PowerShell process.
Get-Process -Id $PID | Select-Object Id,ProcessName,VirtualMemorySize64,WorkingSet64

# Show loaded modules for the current PowerShell process.
(Get-Process -Id $PID).Modules | Select-Object -First 5 ModuleName,FileName

Python example: stack-like recursion

import sys

print("Python recursion limit:", sys.getrecursionlimit())

def countdown(n):
    if n == 0:
        return "done"
    return countdown(n - 1)

print(countdown(5))

# Do not intentionally set huge recursion values on shared systems.

Network example

A network daemon may allocate a heap buffer for each connection. If it forgets to release buffers when clients disconnect, memory use can grow until availability is affected.

Web example

A web server written in C or C++ may parse request headers into stack or heap buffers. Even if the web application is written in a safer language, native libraries underneath may still parse images, compression formats, TLS records, or HTTP data.

Cloud example

Containers do not remove process-memory risks. A containerized service still has stack, heap, mappings, loaded libraries, and memory limits. Cloud teams monitor memory usage, crashes, restart loops, and out-of-memory events as reliability and security signals.

9. Code Examples

9.1 C program: observe stack, heap, and global addresses

#include <stdio.h>   // printf
#include <stdlib.h>  // malloc, free

int global_counter = 7; // global variable, usually in .data

void show_addresses(void) {
    int local_number = 42;             // local variable, usually on stack
    int *heap_number = malloc(sizeof(int)); // heap allocation

    if (heap_number == NULL) {         // always check allocation result
        printf("malloc failed\n");
        return;
    }

    *heap_number = 99;                 // write to allocated heap memory

    printf("address of function      : %p\n", (void *)&show_addresses);
    printf("address of global_counter: %p\n", (void *)&global_counter);
    printf("address of local_number  : %p\n", (void *)&local_number);
    printf("address stored in pointer: %p\n", (void *)heap_number);

    free(heap_number);                 // release heap memory once
}

int main(void) {
    show_addresses();
    return 0;
}

Line-by-line explanation:

  • global_counter exists for the lifetime of the process.
  • local_number exists while show_addresses is active.
  • malloc requests heap memory.
  • heap_number == NULL checks whether allocation failed.
  • free releases the heap memory exactly once.
  • The printed addresses may change between runs because of ASLR.

9.2 Compile and run

gcc -Wall -Wextra -Wpedantic -O0 address_demo.c -o address_demo
./address_demo
./address_demo

Run it more than once. If address randomization is enabled, some addresses will likely change.

9.3 Toy lifetime mistake: returning address of a local variable

#include <stdio.h>

int *bad_pointer(void) {
    int local_value = 123;
    return &local_value; // WRONG: local_value stops being valid after return
}

int main(void) {
    int *p = bad_pointer();
    // Do not dereference p. It is a dangling pointer.
    printf("dangling pointer value: %p\n", (void *)p);
    return 0;
}

Why this is wrong: the local variable belongs to a stack frame. When the function returns, that frame is gone. The pointer remains as a number, but it no longer points to a valid object.

9.4 Safer alternative: caller owns the storage

#include <stdio.h>

void set_value(int *output) {
    if (output == NULL) {
        return;
    }
    *output = 123;
}

int main(void) {
    int value = 0;
    set_value(&value);
    printf("value: %d\n", value);
    return 0;
}

Why this is safer: the caller provides storage that remains valid after set_value returns.

9.5 Heap allocation with cleanup path

#include <stdio.h>
#include <stdlib.h>

int main(void) {
    size_t count = 10;
    int *items = calloc(count, sizeof(int));

    if (items == NULL) {
        fprintf(stderr, "allocation failed\n");
        return 1;
    }

    for (size_t i = 0; i < count; i++) {
        items[i] = (int)i;
    }

    printf("last item: %d\n", items[count - 1]);

    free(items);
    items = NULL; // reduces accidental reuse in this scope
    return 0;
}

Common mistakes

  • Returning a pointer to a local stack variable.
  • Freeing heap memory twice.
  • Forgetting to free memory in long-running services.
  • Assuming printed addresses will be stable across runs.
  • Assuming stack and heap always grow in the same direction on every platform.
  • Ignoring memory permissions in process maps.

10. Security Perspective

Common attacks

  • Triggering stack or heap boundary errors through malformed local input in vulnerable programs.
  • Causing excessive recursion or allocation to exhaust memory.
  • Abusing use-after-free conditions when stale pointers reference memory that has been reused.
  • Using information leaks to learn memory layout and weaken randomization.

Common vulnerabilities

  • Stack-based overflow: a local stack buffer receives more data than it can hold.
  • Heap-based overflow: a dynamically allocated heap buffer is overwritten beyond its bounds.
  • Use-after-free: code uses heap memory after it is released.
  • Double free: code releases the same heap object twice.
  • Memory leak: code loses references to allocated memory and cannot release it.
  • Stack exhaustion: uncontrolled recursion or large frames exceed the stack limit.

Defensive strategies

  • Prefer memory-safe languages for new high-risk components where feasible.
  • Use clear ownership rules for heap objects.
  • Keep local stack buffers small and bounds-checked.
  • Check allocation failures and integer-size calculations.
  • Compile with warnings, hardening flags, and sanitizers in test builds.
  • Use fuzzing and regression tests for parsers and input-heavy components.
  • Run exposed services with least privilege and sandboxing.

Detection methods

  • AddressSanitizer and related runtime sanitizers.
  • Static analyzers that find lifetime and allocation errors.
  • Crash telemetry, core dumps, minidumps, and debugger stack traces.
  • Out-of-memory alerts and restart-loop detection.
  • Code review focused on ownership, bounds, and cleanup paths.

Mitigation

  • Patch root cause rather than only catching the crash.
  • Add regression tests that reproduce the old failure safely.
  • Enable ASLR, NX/DEP, stack canaries, hardened allocators, and compiler hardening where supported.
  • Limit blast radius with process isolation, containers, seccomp-style filtering, and privilege reduction.

Best practices

  • Document who owns each heap allocation.
  • Use one cleanup path for complex C functions when appropriate.
  • Set pointers to NULL after freeing only when it reduces accidental reuse in the same scope.
  • Do not treat ASLR or NX as substitutes for memory-safe code.
  • Do not collect process memory dumps casually; they may contain credentials, tokens, personal data, or cryptographic material.

11. Industry Perspective

Banks

Banks run low-latency systems, payment platforms, fraud engines, and legacy C/C++ components. They care about memory safety because crashes affect availability and memory disclosure can expose customer or transaction data.

Governments

Government systems often include long-lived legacy code, classified workloads, and critical services. Memory-safety roadmaps help prioritize risky components without claiming all legacy code can be replaced immediately.

Cloud providers

Cloud providers operate kernels, hypervisors, storage engines, proxies, databases, and container runtimes. Process isolation and memory protection are central to tenant separation and service reliability.

Healthcare

Healthcare devices and systems may include embedded software and native libraries. Memory failures can affect privacy, availability, and in some cases patient safety.

Defense

Defense environments treat memory corruption as a high-risk class because it can affect mission systems, embedded platforms, sensors, and communications software.

Universities

Universities teach process memory because it connects programming, operating systems, compilers, reverse engineering, and security research.

Large enterprises

Large enterprises use memory concepts to triage vendor advisories, interpret crash telemetry, manage endpoint stability, and choose where memory-safe migration is worth the cost.

12. Research Perspective

Current research does not focus only on finding more memory bugs. It also asks how to prevent, contain, or reason about entire classes of memory errors.

Research directions

  • Memory-safe language migration: how to replace or wrap risky C/C++ components without breaking existing systems.
  • Safe interoperability: how Rust, Java, Go, Swift, or other safer languages call legacy C libraries without losing safety guarantees.
  • Hardware memory tagging: how processors can tag memory and pointers to detect invalid use.
  • Capability machines: how hardware-enforced capabilities can restrict pointer authority.
  • Allocator hardening: how heap allocators can detect or reduce impact of corruption.
  • Formal verification: how to mathematically prove memory properties for critical code.
  • Sanitizer and fuzzer scalability: how to detect bugs earlier with acceptable performance and false-positive trade-offs.

Open problems

  • How can organizations measure memory-safety risk across millions of lines of legacy code?
  • How can high-performance systems gain strong memory protection with low overhead?
  • How can teams verify safety across language boundaries?
  • How should vulnerability severity account for mitigations, memory layout, exposure, and exploit maturity?

Notable reading areas

Look for research on AddressSanitizer, memory tagging, CHERI capabilities, Rust adoption in systems code, WebAssembly memory safety, C/C++ hardening, and formal methods for memory safety. Do not reproduce papers; read abstracts, threat models, evaluation methods, and limitations.

13. Case Studies

Heartbleed

What happened: a flaw in OpenSSL’s heartbeat handling allowed extra memory to be returned to a requester under certain conditions.

Root cause: insufficient validation of a length value before reading memory.

Impact: sensitive process memory could be exposed, including data that should have remained private.

Lessons learned: validate length fields, test protocol parsers aggressively, minimize sensitive data exposure in memory, and treat out-of-bounds reads as serious.

Stack Clash class of issues

What happened: researchers showed that stack growth behavior and guard-page assumptions could be abused in certain circumstances.

Root cause: stack memory could grow into other memory regions without sufficient probing or guard enforcement in affected configurations.

Impact: privilege and process-boundary risk depended on platform and vulnerable software.

Lessons learned: stack layout, guard pages, compiler behavior, and operating-system policies all matter.

Use-after-free in browsers

What happened: browsers have repeatedly patched use-after-free bugs in complex components such as rendering engines and JavaScript runtimes.

Root cause: object lifetimes in large asynchronous systems are difficult to manage manually.

Impact: browser memory bugs can be serious because browsers process untrusted web content.

Lessons learned: use safer ownership models, sandbox risky components, fuzz heavily, and deploy rapid patch mechanisms.

Cloud service memory exhaustion

What happened: services can become unavailable when memory leaks or unbounded allocation consume available memory.

Root cause: missing cleanup, unbounded queues, per-request allocation without limits, or failure to release resources.

Impact: availability degradation, restarts, increased cost, and possible incident response workload.

Lessons learned: monitor memory growth, set resource limits, test long-running behavior, and build graceful failure paths.

14. Common Misconceptions

  • Misconception: stack means insecure and heap means secure. Correction: both can be safe or unsafe depending on code and controls.
  • Misconception: ASLR fixes memory bugs. Correction: ASLR makes some attacks harder; it does not remove the bug.
  • Misconception: a program that does not crash is memory-safe. Correction: undefined behavior can silently corrupt data or appear stable during testing.
  • Misconception: setting a pointer to NULL after free solves all use-after-free risk. Correction: other aliases may still point to the freed object.
  • Misconception: garbage-collected languages have no memory problems. Correction: they reduce many manual memory bugs but can still have leaks, resource exhaustion, unsafe native extensions, and logic flaws.
  • Misconception: process memory maps are secret attack instructions. Correction: they are normal diagnostic information, but they must be handled responsibly and only for authorized systems.
  • Misconception: small local buffers are always safe. Correction: safety depends on bounds checks, input size, encoding, and use context.

15. Summary

A running program is a process with an address space. That address space contains mappings such as executable code, global data, shared libraries, heap, and stack. The stack supports function calls and short-lived local data. The heap supports dynamic allocation and flexible object lifetimes. Security failures happen when code crosses bounds, mismanages object lifetimes, leaks memory, or assumes layout details that are not guaranteed. Defenders reduce risk through careful design, ownership rules, safe APIs, compiler warnings, sanitizers, fuzzing, runtime mitigations, sandboxing, monitoring, and migration to memory-safe languages where feasible.

16. Glossary

The set of memory addresses visible to a process.
ASLR
Address Space Layout Randomization; a mitigation that randomizes memory-region locations.
Call Stack
The chain of currently active function calls.
Dangling Pointer
A pointer that refers to memory that is no longer valid.
DEP
Data Execution Prevention; a control that prevents execution from memory regions marked as data.
Double Free
Freeing the same heap allocation more than once.
Guard Page
An inaccessible memory page used to catch unintended memory growth or boundary crossing.
Heap
A process memory region used for dynamic allocation.
Memory Leak
Allocated memory that is no longer reachable and cannot be released.
Memory Mapping
A region of process address space connected to memory, a file, a library, or a special system region.
NX
No-execute; a memory permission preventing code execution from a region.
Page
A fixed-size unit used by the operating system to manage virtual memory.
Process
A running instance of a program.
Shared Library
Reusable code loaded into a process at runtime.
Stack
A memory region used for function calls and short-lived local data.
Stack Frame
The portion of the stack used by one active function call.
Use-After-Free
Using memory after it has been released.
Virtual Memory
An abstraction that gives processes virtual addresses mapped by the operating system and hardware.

18. Practice Lab

Title

Safe Local Stack, Heap, and Process Map Lab

Prerequisites

  • Local Linux VM, macOS terminal, or Windows with WSL.
  • GCC or Clang installed.
  • No external targets. No unauthorized process inspection.

Objectives

  • Observe stack, heap, and global addresses in a toy program.
  • Inspect your own process mapping safely.
  • Compare address changes between runs.
  • Write a short defensive analysis.

Step 1: Create the lab folder

mkdir -p process-memory-lab
cd process-memory-lab

Step 2: Create the address demo

cat > address_demo.c <<'C'
#include <stdio.h>
#include <stdlib.h>

int global_value = 7;

void show_addresses(void) {
    int local_value = 42;
    int *heap_value = malloc(sizeof(int));
    if (heap_value == NULL) {
        puts("allocation failed");
        return;
    }
    *heap_value = 99;

    printf("function address : %p\n", (void *)&show_addresses);
    printf("global address   : %p\n", (void *)&global_value);
    printf("local address    : %p\n", (void *)&local_value);
    printf("heap address     : %p\n", (void *)heap_value);

    free(heap_value);
}

int main(void) {
    show_addresses();
    puts("Press Enter to exit.");
    getchar();
    return 0;
}
C

Step 3: Compile and run

gcc -Wall -Wextra -Wpedantic -O0 address_demo.c -o address_demo
./address_demo

Leave the program waiting at Press Enter to exit while you inspect it from another terminal.

Step 4: Inspect the process map

# In a second terminal inside the same lab folder:
pidof address_demo
cat /proc/$(pidof address_demo)/maps | head -30

Identify lines that correspond to the program, heap, stack, shared libraries, and permissions. Do not inspect unrelated processes.

Step 5: Run twice and compare

./address_demo
./address_demo

Record whether the printed addresses change. Explain whether ASLR may be involved. Do not disable system protections for this course lab.

Step 6: Optional sanitizer check

gcc -fsanitize=address -g -O1 address_demo.c -o address_demo_asan
./address_demo_asan

Expected outcomes

  • You can identify stack-like, heap-like, global, and code addresses in a toy program.
  • You can read basic process-map permissions.
  • You can explain that printed addresses are diagnostic evidence, not stable design assumptions.
  • You can write a short report without creating exploit instructions.

Report template

  • System used.
  • Compiler used.
  • Observed addresses from two runs.
  • Process-map regions identified.
  • Explanation of stack vs heap lifetime.
  • Security controls observed or inferred.
  • Safety statement confirming only local authorized processes were inspected.
