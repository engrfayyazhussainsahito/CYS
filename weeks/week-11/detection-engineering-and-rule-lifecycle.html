Week 11 · Lesson 11.3 · Detection Engineering

Detection Engineering and Rule Lifecycle

Detection engineering turns behavior hypotheses into tested, documented, operational signals that analysts can trust.

Estimated reading time70 minutes
Difficulty levelIntermediate
Required backgroundLessons 11.1–11.2
Practice time90–120 minutes

1. Introduction

Lesson 11.2 showed that detection quality depends on telemetry quality. Lesson 11.3 moves one layer upward: how to turn useful telemetry into detection logic that can be explained, tested, deployed, tuned, and retired.

A weak detection program often fails because it treats rules as isolated snippets of query syntax. A strong detection program treats each rule as an operational product. It has a hypothesis, required data, expected signal, known false positives, severity logic, test cases, owner, version history, response guidance, and a review schedule.

The core lesson is this: detection engineering is not guessing suspicious strings. It is the disciplined conversion of attacker behavior, policy-relevant risk, and environmental knowledge into reliable decision support for analysts.

2. Learning Outcomes

  • Define detection engineering, detection hypothesis, detection rule, coverage, tuning, suppression, severity, confidence, false positive, and false negative.
  • Describe the lifecycle of a detection from idea to retirement.
  • Translate a behavior into required telemetry, rule logic, test data, and analyst guidance.
  • Differentiate rule logic from severity logic and response guidance.
  • Explain why every detection needs known false-positive expectations and test cases.
  • Build a small local fictional rule set in Python and produce structured alerts.
  • Identify common detection failures such as missing telemetry, brittle matching, alert floods, stale rules, and undocumented exceptions.

3. Prerequisites

You should understand the SOC workflow from Lesson 11.1 and the telemetry pipeline concepts from Lesson 11.2. You should also be comfortable reading JSONL and basic Python dictionaries, lists, and functions.

Authorization boundary: This lesson uses fictional local data only. Do not test detection logic by provoking alerts in systems you do not own or administer. Do not collect, export, or share real user logs unless your organization has approved the purpose, scope, retention, and privacy handling process.

4. Core Concepts

Detection engineering

Detection engineering is the practice of designing, testing, deploying, measuring, tuning, and retiring security detections. It connects threat knowledge, telemetry, analytics, and SOC workflows.

Detection hypothesis

A detection hypothesis is a precise statement about behavior that should be visible in data. Example: “A high-risk administrator who denies multiple MFA prompts and then approves one may deserve review.”

Detection rule

A detection rule is executable logic that evaluates events or event sequences and produces a signal when conditions are met.

Signal

A signal is the meaningful observation produced by a detection. It may become an alert, case enrichment, dashboard indicator, hunt result, or risk score.

Severity and confidence

Severity estimates potential impact if the signal is meaningful. Confidence estimates how strongly the evidence supports the detection hypothesis. They should be separate fields because a low-confidence signal can still be high-impact.

Tuning

Tuning changes detection logic, thresholds, exclusions, severity, or enrichment to improve usefulness. Tuning should be documented so the SOC understands what was changed and why.

Suppression

Suppression hides or lowers repeated alerts under defined conditions. Suppression is useful for known benign activity, but dangerous when it becomes a silent way to ignore unresolved risk.

Rule lifecycle

The rule lifecycle is the path from idea to retirement: hypothesis, data validation, logic, test, review, deploy, monitor, tune, document, and eventually deprecate.

5. Visual Learning

A detection should move through a controlled lifecycle. Skipping validation or testing creates rules that look impressive but fail under real SOC pressure.

flowchart LR
  A[Behavior or risk idea] --> B[Detection hypothesis]
  B --> C[Required telemetry check]
  C --> D[Rule logic draft]
  D --> E[Test data and expected results]
  E --> F[Peer review]
  F --> G[Deploy in monitor mode]
  G --> H[Measure alerts and analyst feedback]
  H --> I[Tune severity, thresholds, exclusions]
  I --> J[Document and operationalize]
  J --> K[Periodic review]
  K --> L{Still useful?}
  L -- yes --> H
  L -- no --> M[Retire rule]

The important detail is that the lifecycle loops. A detection is not “finished” merely because it was deployed.

7. From Behavior to Signal

Detection work should begin with behavior, not query syntax. A behavior is something an actor does: authenticate, create a key, modify a group, run a process, access a repository, disable logging, or change a policy. A signal is what the detection produces after evaluating evidence of that behavior.

StepQuestionExample
BehaviorWhat action or sequence matters?Multiple MFA denials followed by approval.
HypothesisWhy might it matter?The user may have approved an unwanted push after fatigue or confusion.
TelemetryWhich records prove or disprove it?Identity provider MFA prompt outcome events.
ContextWhat changes priority?Admin role, risky user, external source, privileged follow-up action.
SignalWhat should the SOC receive?Structured alert with actor, source IP, event count, time range, severity, and rationale.

This sequence keeps the engineer honest. If no data can prove the hypothesis, the correct next step is telemetry improvement, not a cleverer query.

8. Anatomy of a Good Detection Rule

A useful rule contains more than matching logic. At minimum, it should include:

  • Rule ID: stable identifier used in documentation, cases, and change history.
  • Name: clear behavior-focused title.
  • Hypothesis: what behavior the rule is intended to surface.
  • Required data: sources, fields, and freshness requirements.
  • Logic: filters, joins, thresholds, windows, and exclusions.
  • Severity: expected impact when true.
  • Confidence: expected evidentiary strength.
  • Known false positives: expected benign patterns.
  • Response guidance: first questions an analyst should answer.
  • Tests: examples that should alert and examples that should not.
  • Owner and review date: who maintains the rule and when it must be rechecked.

Missing metadata is not cosmetic. It causes analyst confusion, inconsistent triage, and stale rules that nobody wants to remove.

9. Data Requirements and Coverage

Every detection depends on data assumptions. Before writing logic, document the minimum data needed.

RequirementWhy it mattersFailure mode
Source availabilityThe rule cannot detect behavior that is not collected.Blind spot disguised as “no alerts.”
Field completenessActor, action, target, result, and timestamp are often required.Rule matches too broadly or not at all.
Timestamp reliabilitySequence and time-window rules depend on ordering.False sequences or missed chains.
NormalizationCross-source logic requires consistent field names.Rule works for one source but misses another.
EnrichmentAsset criticality and identity role affect severity.Low-value and high-value alerts look identical.
RetentionLonger behaviors may require historical context.Investigation cannot confirm the pattern.

A detection’s coverage should be stated plainly: which systems, users, platforms, and behaviors it can see, and which it cannot.

10. Severity and Confidence

Severity and confidence are often mixed together, which creates poor prioritization. Keep them separate.

ScenarioSeverityConfidenceReason
Privileged group change after suspicious MFA patternCriticalHighPotential impact is large and sequence is specific.
Two failed logins followed by one successMediumMediumCould be suspicious, but also common user error.
Blocked encoded PowerShell on one laptopMediumLow to mediumMay be malicious, admin script, or security tool behavior.
Cloud key creation by approved deployment botLowHigh benign confidencePattern is expected if documented and scoped.

Good severity logic is contextual. The same action can be low priority on a test system and high priority on a production identity or finance system.

11. False Positives, False Negatives, and Tuning

A false positive is not just an annoyance. Enough false positives teach analysts to distrust the detection system. A false negative is worse in a different way: it creates a quiet blind spot that may only be discovered after an incident.

Tuning should be evidence-based. Do not add broad exclusions because an alert is inconvenient. First ask: Is the behavior benign? Is it authorized? Is it documented? Does it happen regularly? Is there a safer narrower condition?

Tuning optionGood useRisk
ThresholdRequire enough repeated behavior to reduce noise.Attackers below threshold may be missed.
AllowlistKnown service account or maintenance host.Compromise of allowlisted identity may hide activity.
Time windowDetect meaningful sequences close together.Slow activity may evade the rule.
Context filterRaise severity for privileged identities or critical assets.Poor context data can misprioritize alerts.
SuppressionReduce duplicate alerts for the same case.Can hide ongoing escalation if suppressions are too broad.

12. Rule Types

Different detections require different logic patterns.

  • Single-event rule: one event is sufficient, such as logging disabled on a critical system.
  • Threshold rule: repeated events exceed a count, such as many failures in a short window.
  • Sequence rule: events occur in a meaningful order, such as denial, approval, privileged change.
  • Join rule: combine different sources, such as identity login plus cloud audit action.
  • Baseline rule: compare behavior to historical norms. This requires careful statistics and drift management.
  • Correlation rule: combine multiple lower-confidence signals into a stronger alert.
  • Hunt query: exploratory logic used by humans; it may not be ready for continuous alerting.

A common failure is turning every hunt query into an alert. Hunt queries tolerate ambiguity. Alert rules need repeatability and operational guidance.

13. Mapping and Coverage Frameworks

Framework mapping helps organize coverage, but mapping is not detection quality by itself. A rule mapped to a technique still needs valid telemetry, useful logic, tests, tuning, and response guidance.

Use mapping to answer coverage questions:

  • Which behaviors are visible in current telemetry?
  • Which critical assets or identities are covered?
  • Which detections are only single-source and which correlate multiple sources?
  • Which rules have not fired because behavior is absent, telemetry is missing, or logic is broken?
  • Which rules are stale because systems, fields, or attacker behavior changed?

The dangerous mistake is “coverage theater”: producing a colorful matrix while the underlying rules are untested, noisy, or unsupported by data.

14. Testing and Validation

Detection testing should include positive tests, negative tests, parser checks, historical searches, and analyst review.

Test typePurposeExample
Positive testConfirm the rule fires when expected.Two MFA denials followed by approval for a high-risk admin.
Negative testConfirm benign activity does not fire.Same MFA pattern for a low-risk non-admin when policy accepts that risk.
Boundary testCheck thresholds and windows.One denial should not fire if the threshold is two.
Parser testConfirm required fields exist.Actor, source IP, action, outcome, and timestamp are populated.
Historical testEstimate volume and false positives.Run on past data before enabling alerting.
Analyst testCheck usefulness of the generated alert.Can an analyst answer what happened and what to do next?

Validation should produce artifacts: test cases, observed alert counts, tuning notes, and a decision to deploy, revise, or reject.

15. Deployment and Change Control

Detection deployment should be controlled like software release, even in a small SOC.

  1. Draft the rule and metadata.
  2. Confirm data source health and field availability.
  3. Run positive and negative tests.
  4. Peer review logic and analyst guidance.
  5. Deploy in monitor mode where possible.
  6. Measure alert volume and false-positive causes.
  7. Enable alerting with ownership and escalation path.
  8. Review after a defined period.

Change control matters because a small rule edit can flood a queue, suppress a critical signal, or break a downstream workflow.

16. Detection Documentation

Documentation is part of the detection, not an optional report. At minimum, document:

  • Rule ID and title.
  • Detection hypothesis.
  • Required sources and fields.
  • Logic summary in plain language.
  • Severity and confidence rationale.
  • Known false positives and safe tuning guidance.
  • Initial triage questions.
  • Escalation criteria.
  • Test cases and last test date.
  • Owner, reviewer, and next review date.

An analyst should not need to reverse-engineer a query during a live investigation to understand why the alert exists.

17. Detection-as-Code

Detection-as-code means managing detection logic and metadata using software engineering practices: version control, review, tests, release notes, and rollback. The goal is not complexity. The goal is repeatability and accountability.

PracticeSecurity operations benefit
Version controlShows who changed a rule and why.
Peer reviewCatches weak logic, broad exclusions, and unclear severity.
Unit testsPrevent accidental breakage during tuning.
Staging deploymentEstimates alert volume before production alerting.
RollbackAllows fast recovery from noisy or broken detections.

For learners, the simplest detection-as-code exercise is a local folder with JSONL test events, Python rule functions, and assertions for expected alerts.

18. Common Failure Points

  • Query-first thinking: starting with syntax instead of a behavior hypothesis.
  • Missing telemetry: building rules on fields that are absent, inconsistent, or delayed.
  • Overbroad matching: matching a string or command without context.
  • No negative tests: rule fires in a demo but floods real queues.
  • No owner: nobody tunes or retires stale rules.
  • Silent suppressions: exceptions hide the very behavior the rule was meant to reveal.
  • Severity inflation: everything becomes high priority, so nothing is high priority.
  • No analyst guidance: alerts create work but not decisions.
  • Coverage theater: mapped rules look complete while evidence quality is poor.

19. Worked Example: MFA Pattern to Privilege Change

Consider this detection hypothesis:

Hypothesis: A high-risk administrator who denies multiple MFA prompts, then approves one, and soon performs a privileged group change may require urgent investigation.

The hypothesis requires at least two data sources: identity provider MFA events and directory group-change events. It also requires identity context: whether the actor is an administrator and whether the account is high risk.

ComponentDesign decision
Required fieldstimestamp, actor, source_ip, action, outcome, target, actor_role, actor_risk
SequenceMFA denied more than once, MFA approved, privileged group changed
SeverityCritical when privileged group membership changes
ConfidenceHigher when source IP and actor are consistent across the sequence
Known false positivesAdmin training, device enrollment, repeated user mistakes, approved emergency access
First triage questionsWas the actor expected to make this change? Was there a ticket? Are there other actions from the same source?

The rule should not conclude “compromise.” It should produce a defensible operational signal that tells the analyst what evidence exists and what remains unknown.

Step 3: Create detection rules

Save this as detect_rules.py.

import json
from collections import defaultdict
from pathlib import Path
from datetime import datetime, timezone

EVENTS = Path("normalized_events.jsonl")


def parse_time(value):
    return datetime.fromisoformat(value.replace("Z", "+00:00")).astimezone(timezone.utc)


def load_events():
    events = []
    with EVENTS.open(encoding="utf-8") as f:
        for line in f:
            if line.strip():
                event = json.loads(line)
                event["_time"] = parse_time(event["timestamp"])
                events.append(event)
    return sorted(events, key=lambda e: e["_time"])


def alert(rule_id, title, severity, confidence, events, rationale):
    return {
        "rule_id": rule_id,
        "title": title,
        "severity": severity,
        "confidence": confidence,
        "actors": sorted({e.get("actor", "unknown") for e in events}),
        "sources": sorted({e.get("event_source", "unknown") for e in events}),
        "event_count": len(events),
        "start": min(e["timestamp"] for e in events),
        "end": max(e["timestamp"] for e in events),
        "rationale": rationale,
    }


def rule_failures_then_success(events):
    alerts = []
    by_actor_ip = defaultdict(list)
    for event in events:
        if event.get("action") == "login":
            by_actor_ip[(event.get("actor"), event.get("source_ip"))].append(event)

    for (actor, source_ip), group in by_actor_ip.items():
        failures = [e for e in group if e.get("outcome") == "failure"]
        successes = [e for e in group if e.get("outcome") == "success"]
        if len(failures) >= 2 and successes:
            alerts.append(alert(
                "DET-1103-001",
                "Repeated login failures followed by success",
                "medium",
                "medium",
                failures + successes[:1],
                f"{actor} had {len(failures)} failures followed by a successful login from {source_ip}."
            ))
    return alerts


def rule_mfa_denials_then_approval_for_high_risk_admin(events):
    alerts = []
    by_actor_ip = defaultdict(list)
    for event in events:
        if event.get("action") == "mfa_push":
            by_actor_ip[(event.get("actor"), event.get("source_ip"))].append(event)

    for (actor, source_ip), group in by_actor_ip.items():
        denials = [e for e in group if e.get("outcome") == "denied"]
        approvals = [e for e in group if e.get("outcome") == "approved"]
        high_risk_admin = any(e.get("actor_role") == "admin" and e.get("actor_risk") == "high" for e in group)
        if len(denials) >= 2 and approvals and high_risk_admin:
            alerts.append(alert(
                "DET-1103-002",
                "High-risk admin approved MFA after repeated denials",
                "high",
                "medium",
                denials + approvals[:1],
                f"{actor} denied multiple MFA prompts and then approved one from {source_ip}."
            ))
    return alerts


def rule_privilege_change_after_mfa_pattern(events):
    alerts = []
    mfa_actors = {a["actors"][0] for a in rule_mfa_denials_then_approval_for_high_risk_admin(events)}
    for event in events:
        if event.get("actor") in mfa_actors and event.get("action") == "AddMemberToGroup" and event.get("target") in {"Domain Admins", "Global Admins"}:
            alerts.append(alert(
                "DET-1103-003",
                "Privileged group change after suspicious MFA pattern",
                "critical",
                "high",
                [event],
                f"{event.get('actor')} changed membership of {event.get('target')} after a suspicious MFA sequence."
            ))
    return alerts


def rule_access_key_on_high_criticality_resource(events):
    alerts = []
    allowed_roles = {"service_account", "cloud_admin"}
    for event in events:
        if event.get("action") == "CreateAccessKey" and event.get("asset_criticality") == "high" and event.get("actor_role") not in allowed_roles:
            alerts.append(alert(
                "DET-1103-004",
                "Access key created for high-criticality resource by unexpected role",
                "high",
                "medium",
                [event],
                f"{event.get('actor')} with role {event.get('actor_role')} created a key for {event.get('resource')}."
            ))
    return alerts


def main():
    events = load_events()
    rules = [
        rule_failures_then_success,
        rule_mfa_denials_then_approval_for_high_risk_admin,
        rule_privilege_change_after_mfa_pattern,
        rule_access_key_on_high_criticality_resource,
    ]
    all_alerts = []
    for rule in rules:
        all_alerts.extend(rule(events))

    print(json.dumps(all_alerts, indent=2))


if __name__ == "__main__":
    main()

22. Practical Deliverables

After completing this lesson, create the following artifacts:

  • A one-page detection design for one behavior.
  • A required-telemetry checklist for that detection.
  • At least one positive and one negative test case.
  • A plain-language analyst guidance note.
  • A tuning note that explains one expected false-positive scenario.
  • A retirement condition: when the rule should be removed or redesigned.

These artifacts matter more than tool-specific syntax. Syntax changes by platform; disciplined detection design transfers across platforms.
