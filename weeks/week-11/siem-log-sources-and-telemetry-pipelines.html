Week 11 · Lesson 11.2 · SIEM and Telemetry Pipelines

SIEM, Log Sources, and Telemetry Pipelines

A SIEM is only useful when the organization knows what evidence it needs, where that evidence comes from, how it is parsed, how long it is retained, and what decisions it supports.

Estimated reading time65 minutes
Difficulty levelIntermediate
Required backgroundLesson 11.1 and Weeks 0–10
Practice time75–120 minutes

1. Introduction

Lesson 11.1 introduced security operations as an evidence-processing workflow. Lesson 11.2 focuses on the evidence system itself: log sources, telemetry pipelines, and SIEM platforms. The main lesson is simple but often missed: a SIEM does not create useful evidence by magic. It receives records from other systems, parses them, enriches them, stores them, and allows analysts or detection logic to query them.

A weak SIEM deployment usually fails before an analyst opens a dashboard. Important sources are missing. Field names are inconsistent. Timestamps disagree. Hostnames do not map to asset owners. Usernames appear in different formats. Retention is too short. High-volume low-value logs consume budget while high-value audit trails are absent. Detection engineering becomes guesswork because the data cannot answer basic questions.

This lesson teaches how to design a telemetry pipeline from first principles. The focus is not a specific product such as Splunk, Microsoft Sentinel, Elastic, Chronicle, QRadar, or a cloud-native logging service. The focus is the durable architecture behind all of them: source selection, collection, parsing, normalization, enrichment, storage, query, alerting, retention, access control, and quality measurement.

2. Learning Outcomes

  • Define SIEM, log source, event, telemetry pipeline, parser, normalization, enrichment, retention, and index.
  • Classify log sources by security value, volume, sensitivity, and operational owner.
  • Explain how raw logs become searchable fields and why schema design matters.
  • Describe timestamp, identity, asset, network, and cloud-context problems in telemetry pipelines.
  • Build a small local fictional pipeline that parses JSONL events, normalizes fields, enriches context, and produces a summary.
  • Write a telemetry inventory and a log onboarding checklist for a new source.
  • Identify common SIEM failure modes: missing sources, excessive noise, broken parsing, bad retention, duplicate events, and weak access controls.

3. Prerequisites

You should already understand the difference between an event, alert, case, and incident. You should also be comfortable reading simple JSON, CSV, and Python. The lab does not connect to any real cloud, endpoint, directory, or network source. It uses fictional local files only.

Authorization boundary: Do not export company logs, user activity records, packet captures, endpoint telemetry, email logs, cloud audit records, or SIEM data for practice unless you have explicit written approval and a defined privacy handling process.

4. Core Concepts

SIEM

A Security Information and Event Management system collects, stores, searches, correlates, and alerts on security-relevant events from many sources. A SIEM is a data platform for security decisions, not a substitute for log strategy or analyst judgment.

Log source

A log source is any system that emits records useful for security or operations. Examples include identity providers, operating systems, EDR tools, firewalls, DNS resolvers, VPN gateways, web servers, cloud audit services, Kubernetes control planes, SaaS applications, vulnerability scanners, and CI/CD platforms.

Event

An event is a record that something happened. It usually includes a timestamp, actor, action, target, result, source, and additional attributes. Not every event is security-relevant, and not every security-relevant event becomes an alert.

Telemetry pipeline

A telemetry pipeline is the path from source system to searchable, usable evidence. It includes collection, transport, parsing, normalization, enrichment, deduplication, storage, retention, and access control.

Parsing

Parsing converts raw text or structured data into fields. For example, a raw authentication message becomes fields such as timestamp, user, source_ip, action, and result.

Normalization

Normalization maps different source-specific fields into a consistent schema. For example, src_ip, client_ip, and ipAddress may all become source.ip.

Enrichment

Enrichment adds context that was not present in the original event: asset owner, business criticality, identity department, geolocation, vulnerability status, threat intelligence labels, or known maintenance windows.

Retention

Retention defines how long logs are kept, where they are stored, how searchable they remain, and when they are archived or deleted. Retention should reflect investigation needs, legal requirements, privacy obligations, cost, and risk.

7. Mathematical Foundations

SIEM design uses simple capacity and quality calculations. These calculations do not prove security, but they expose hidden constraints.

  • Events per second: the rate at which a source or pipeline produces events.
  • Daily ingest: average event size multiplied by event count per day.
  • Retention cost: storage and indexing cost over the retention period.
  • Coverage: percentage of critical assets, identities, or behaviors that produce usable logs.
  • Parse success rate: records successfully mapped into required fields divided by total records.
  • Detection dependency coverage: percentage of detection rules whose required log sources are available and healthy.
Example capacity estimate:
Average event size: 900 bytes
Events per second: 250
Seconds per day: 86,400
Daily raw volume = 900 × 250 × 86,400 = 19,440,000,000 bytes
Approximate daily volume = 19.44 GB before compression, indexing overhead, and replicas

Budget pressure can distort security. Keeping every noisy debug log for one year may be less useful than keeping high-value identity, endpoint, DNS, cloud audit, and administrative logs for the period needed to reconstruct an incident.

8. Log Source Taxonomy

A useful telemetry inventory classifies sources by the security questions they answer. The same source may support multiple investigations, but every source should have a reason for being collected.

Endpoint / EDRWhat process, file, command, registry, or network action occurred on a host?hostname, process name, command line, parent process, hash, user

9. Pipeline Design

A telemetry pipeline should be designed backwards from decisions. Start with the incident or investigation questions, then identify the minimum evidence needed to answer them.

  1. Define the question. Example: Can we identify suspicious administrative cloud API activity?
  2. Identify required fields. Principal, action, target resource, result, timestamp, source IP, region, user agent, and request ID.
  3. Choose source systems. Cloud audit logs, identity logs, privileged access management logs, and change management records.
  4. Define collection method. Agent, API pull, streaming export, webhook, syslog, object storage export, or message queue.
  5. Build parsing and schema mapping. Map source-specific names to consistent fields.
  6. Add enrichment. Link principal to department, role, risk tier, asset owner, and approved maintenance windows.
  7. Set retention and access controls. Decide hot, warm, archive, deletion, and permissions.
  8. Test with known events. Generate or locate authorized benign events to prove fields arrive correctly.
  9. Document operational ownership. Record who owns the source, parser, pipeline, detection, and escalation path.

10. Schema and Normalization

Normalization is what lets analysts ask one question across many sources. Without normalization, every investigation becomes a translation exercise. This does not mean all original fields are discarded. Good pipelines preserve raw events while also extracting normalized fields for search and detection.

Actoruser, username, principal, actorEmailactor.name or actor.email
ActioneventName, operation, activity, verbevent.action

Schema debates can become theoretical. The practical test is whether an analyst can answer common questions quickly: who acted, what did they do, where did it happen, what was affected, did it succeed, and what context changes severity?

11. Timestamps, Ordering, and Time Zones

Timeline building is fragile when timestamps are inconsistent. Logs may use local time, UTC, epoch time, different precision, or delayed delivery. Some sources record event time; others record ingest time. Analysts must know which timestamp they are using.

  • Event time: when the activity happened at the source.
  • Ingest time: when the pipeline received the record.
  • Processing time: when parsing, enrichment, or detection ran.
  • Clock skew: disagreement between system clocks.
  • Latency: delay between event occurrence and searchable availability.
Operational rule: Prefer UTC for storage and correlation, preserve the original timestamp field, and document known source latency. During incident response, always state which timestamp you are using.

12. Enrichment and Context

Enrichment changes the value of telemetry. A failed login from a new country may be low priority for a test account and high priority for a privileged administrator. A denied firewall connection may be routine internet noise or a critical sign when it targets a crown-jewel service from a compromised host.

Common enrichment sources include:

  • Asset inventory: owner, environment, criticality, internet exposure, data classification.
  • Identity inventory: department, role, privilege level, MFA requirement, account type.
  • Network inventory: subnet, zone, business application, expected traffic patterns.
  • Vulnerability data: known exposed weaknesses and patch status.
  • Change calendar: deployments, maintenance windows, approved administrative work.
  • Threat intelligence: known malicious infrastructure, malware family indicators, abuse history.

Enrichment should be explainable. If a system automatically labels an asset as critical, analysts should know the source and freshness of that label.

13. Retention, Access Control, and Privacy

Logs often contain sensitive data: usernames, IP addresses, URLs, email subjects, filenames, hostnames, customer IDs, administrator actions, and sometimes secrets accidentally written by applications. SIEM access should be role-based and audited. Analysts should see what they need for authorized work, not unlimited personal or business data by default.

Retention has three practical layers:

  • Hot searchable data: fast query access for recent investigations and detections.
  • Warm or reduced-index data: slower access for longer lookback and incident reconstruction.
  • Archive: low-cost evidence preservation with stricter retrieval process.

Deleting logs too soon can make investigations impossible. Keeping all logs forever can create cost, privacy, and legal exposure. A mature retention plan balances security value, regulatory requirements, business risk, cost, and data minimization.

14. Query Thinking Before Query Syntax

SIEM products use different query languages, but the mental process is similar. A query should express an investigation question, not just search random strings.

Question: Did the same identity fail many logins and then succeed from a new IP?
Required fields: actor, source_ip, event.action, event.outcome, timestamp
Logic:
  1. Find failed login events grouped by actor and source_ip.
  2. Count failures in a short time window.
  3. Look for a successful login by the same actor soon after.
  4. Enrich with MFA status, device trust, geography, and account privilege.
  5. Decide whether this is password guessing, normal user error, or another pattern.

Beginners often start with tool syntax too early. First define the behavior, fields, time window, grouping, thresholds, false positives, and response action. Syntax comes after logic.

15. Practical Examples

Example 1: Missing identity logs

A SOC receives endpoint alerts for suspicious commands but cannot see whether the user recently authenticated from a new location. The endpoint data is useful, but the investigation is incomplete because identity logs are missing or not joined to the case.

Example 2: Broken parser

A firewall format changes after an appliance upgrade. The SIEM still ingests raw records, but event.action and destination.port become empty. Detection rules silently stop matching. Ingest volume looks healthy, but detection coverage is degraded.

Example 3: No asset enrichment

A cloud storage access event appears low severity because it is only a read operation. After enrichment, analysts learn the storage bucket contains regulated data and belongs to a production payment workflow. Context changes priority.

Example 4: Retention too short

A suspected compromise is discovered on day 45, but authentication logs are retained for only 14 days. The team cannot reconstruct initial access. The cost saved on retention becomes a major investigation gap.

16. Tools, Roles, and Ownership

Telemetry work is shared across teams. Security analysts know investigation needs. Detection engineers know rule dependencies. Platform engineers know pipelines and storage. System owners know source behavior. Privacy, legal, and compliance teams define handling constraints. Without explicit ownership, pipelines degrade quietly.

RoleTypical responsibility
Source ownerEnsures the source emits required events and communicates format changes.
Pipeline ownerMaintains collection, transport, parsing, indexing, and pipeline health.
Detection engineerDefines data requirements for rules and validates detection logic against available fields.
SOC analystUses telemetry for triage and reports gaps that block investigations.
Incident responderDefines evidence needed for containment, eradication, recovery, and reporting.
Privacy/security governanceDefines acceptable access, retention, masking, and audit requirements.

17. Research Perspective

Research in SIEM and telemetry pipelines examines log quality, alert overload, automated correlation, streaming analytics, privacy-preserving monitoring, anomaly detection, schema design, and the economics of security data. A recurring theme is that more data does not automatically mean better detection. Useful telemetry is evidence that supports a decision.

Modern research also studies how summarization and machine learning can help analysts navigate large cases. The risk is that automation may produce confident narratives unsupported by evidence. In security operations, generated summaries must point back to source events, timestamps, and fields.

18. Common Misconceptions

  • Misconception: If logs are ingested, visibility is solved. Correction: Ingest does not guarantee parsing, normalization, retention, access, or detection coverage.
  • Misconception: Raw volume is the best measure of SIEM value. Correction: Value depends on whether logs answer security questions.
  • Misconception: A dashboard is the same as detection. Correction: Dashboards visualize data; detection logic defines suspicious behavior and produces reviewable alerts.
  • Misconception: Cloud logs are automatically complete. Correction: Some cloud logs require explicit enabling, routing, retention, permissions, or paid tiers.
  • Misconception: Normalization means losing raw logs. Correction: Mature pipelines preserve raw records and add normalized fields.
  • Misconception: Privacy is separate from SIEM design. Correction: Telemetry design determines who can see sensitive user, customer, and business activity.

19. Summary

A SIEM is a security evidence platform. It becomes useful only when log sources are selected intentionally, parsed correctly, normalized consistently, enriched with context, retained long enough, protected appropriately, and connected to detection and investigation workflows.

The central skill in this lesson is pipeline reasoning. Before writing rules or dashboards, define the security question, the required fields, the source of truth, the expected latency, the enrichment needed for severity, the retention requirement, and the owner responsible for keeping the source healthy.

20. Glossary

  • SIEM: A platform for collecting, storing, searching, correlating, and alerting on security-relevant events.
  • Log source: A system that emits records used for security, operations, compliance, or investigation.
  • Telemetry pipeline: The path that turns source events into searchable, normalized, enriched evidence.
  • Parser: Logic that extracts structured fields from raw records.
  • Normalization: Mapping source-specific fields into a consistent schema.
  • Enrichment: Adding external context such as asset owner, identity role, geolocation, or criticality.
  • Index: A storage structure optimized for search and retrieval.
  • Retention: The policy for how long logs are kept and how accessible they remain.
  • Ingest latency: The delay between event occurrence and availability in the logging platform.
  • Parse failure: A record that is collected but not correctly converted into useful fields.

Step 3: Create the normalizer

Save this as normalize_pipeline.py.

import json
from collections import Counter, defaultdict
from pathlib import Path

RAW_EVENTS = Path("raw_events.jsonl")
CONTEXT = Path("context.json")

FIELD_MAP = {
    "actorEmail": "actor",
    "principal": "actor",
    "user": "actor",
    "ipAddress": "source_ip",
    "src_ip": "source_ip",
    "client_ip": "source_ip",
    "eventName": "action",
    "activity": "action",
    "resultCode": "outcome",
    "status": "outcome",
    "time": "timestamp",
}


def normalize(record):
    normalized = dict(record)
    for old, new in FIELD_MAP.items():
        if old in record and new not in normalized:
            normalized[new] = record[old]
    normalized.setdefault("outcome", record.get("action", "unknown"))
    normalized.setdefault("actor", record.get("hostname", "unknown"))
    normalized.setdefault("source_ip", "unknown")
    normalized.setdefault("timestamp", record.get("time", "unknown"))
    normalized["event_source"] = record.get("source", "unknown")
    return normalized


def enrich(event, context):
    actor = event.get("actor")
    resource = event.get("resource") or event.get("hostname")
    identity_context = context.get("identities", {}).get(actor, {})
    asset_context = context.get("assets", {}).get(resource, {})
    event["actor_role"] = identity_context.get("role", "unknown")
    event["actor_risk"] = identity_context.get("risk", "unknown")
    event["asset_owner"] = asset_context.get("owner", "unknown")
    event["asset_criticality"] = asset_context.get("criticality", "unknown")
    return event


def load_events():
    with RAW_EVENTS.open(encoding="utf-8") as f:
        for line in f:
            if line.strip():
                yield json.loads(line)


def main():
    context = json.loads(CONTEXT.read_text(encoding="utf-8"))
    events = [enrich(normalize(record), context) for record in load_events()]

    print("Normalized events:")
    for event in events:
        print(json.dumps(event, indent=2, sort_keys=True))

    print("\nSource counts:")
    for source, count in Counter(e["event_source"] for e in events).items():
        print(f"- {source}: {count}")

    failures = defaultdict(int)
    successes = []
    for event in events:
        if event.get("action") == "login" and event.get("outcome") == "failure":
            failures[(event.get("actor"), event.get("source_ip"))] += 1
        if event.get("action") == "login" and event.get("outcome") == "success":
            successes.append(event)

    print("\nCandidate signals:")
    for event in successes:
        key = (event.get("actor"), event.get("source_ip"))
        if failures[key] >= 2 and event.get("actor_risk") == "high":
            print(f"- High-risk identity had {failures[key]} failures then success: {key}")

    for event in events:
        if event.get("action") == "CreateAccessKey" and event.get("asset_criticality") == "high":
            print(f"- Access key created for high-criticality resource: {event.get('resource')} by {event.get('actor')}")


if __name__ == "__main__":
    main()
