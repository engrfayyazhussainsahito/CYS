Week 11 · Lesson 11.1 · Security Operations

Security Operations from First Principles

Build the mental model behind SOC work before touching SIEM rules: telemetry, detection, triage, investigation, response, and improvement.

Estimated reading time55 minutes
Difficulty levelIntermediate
Required backgroundWeeks 0–10
Practice time60–90 minutes

1. Introduction

Security operations is the discipline of finding, understanding, and reducing security harm while systems are running. It is not only a room, a tool, or a shift schedule. It is a decision system that receives telemetry, applies security judgment, and produces actions: investigate, contain, escalate, tune, document, or close.

The common beginner mistake is to think a SOC succeeds because it buys a SIEM or receives many alerts. High alert volume can mean the opposite: poor signal quality, weak asset context, duplicate rules, missing ownership, and slow response. A useful operation starts with first principles: assets generate events; events become telemetry; telemetry supports detections; detections create alerts; alerts become cases; cases lead to response decisions.

This lesson builds the operating model before introducing SIEM-specific query languages. The goal is to understand what a security operations team is trying to prove, what evidence it needs, and where failure usually enters the workflow.

2. Learning Outcomes

  • Define security operations, SOC, telemetry, alert, incident, case, runbook, playbook, and escalation.
  • Explain how monitoring differs from detection and how detection differs from investigation.
  • Map a basic SOC workflow from signal collection to closure and improvement.
  • Identify required context for alert triage: asset, identity, time, source, destination, action, result, and business impact.
  • Separate true positive, false positive, benign true positive, and suspicious but unconfirmed activity.
  • Write a safe first triage note using facts, assumptions, unknowns, and recommended next actions.

3. Prerequisites

You should understand the earlier lessons on logs, identity, networks, endpoints, cloud auditing, and incident response. For this lesson, you do not need Splunk, Sentinel, Chronicle, Elastic, QRadar, or a commercial SOC platform. The lab uses local fictional JSON logs and a small Python script so the concepts stay visible.

The safety boundary is strict: do not collect logs from systems you do not own, do not monitor other users without authorization, and do not use production telemetry for class exercises unless privacy, retention, and approval requirements are already satisfied.

4. Core Concepts

Security operations

Security operations is the continuous process of observing systems, detecting suspicious behavior, investigating evidence, responding proportionally, and improving controls. It is both technical and procedural.

SOC

A Security Operations Center, or SOC, is the function responsible for operational security monitoring and response. It may be a physical room, a distributed team, a managed service, or a small internal workflow owned by IT and security staff.

Telemetry

Telemetry is security-relevant data produced by systems. Examples include authentication events, process creation logs, DNS requests, firewall decisions, endpoint alerts, cloud audit records, email security events, vulnerability findings, and SaaS access logs.

Alert

An alert is a notification that detection logic or a tool believes something may require attention. An alert is not automatically an incident. It is a request for judgment.

Incident

An incident is a confirmed or strongly suspected event that threatens confidentiality, integrity, availability, safety, privacy, compliance, or business operations. Confirmation may be partial at first, but the decision to declare an incident should be recorded.

Case

A case is the investigation record. It should contain timeline, evidence, analyst notes, decisions, containment actions, communications, and closure rationale.

Runbook and playbook

A runbook is a procedural guide for a specific task. A playbook is a coordinated workflow for a scenario, such as suspicious login, malware alert, leaked credential, cloud policy change, or ransomware report.

5. Visual Learning

The SOC workflow is an evidence pipeline. Each stage can lose quality if ownership, context, or decision criteria are unclear.

  E --> F[Triage]
  F --> G{Decision}

The important loop is at the end. Security operations improves when investigations produce better detections, better asset context, better logging, and better response procedures.

7. Mathematical Foundations

SOC work uses simple but important measurements. The numbers do not prove security by themselves, but they reveal bottlenecks and bad assumptions.

  • Alert volume: number of alerts over a time window.
  • True positive rate: confirmed useful alerts divided by total reviewed alerts.
  • False positive rate: alerts closed as not suspicious divided by total reviewed alerts.
  • Mean time to detect: average time from malicious or risky activity to detection.
  • Mean time to respond: average time from detection to containment or other agreed response action.
  • Coverage: percentage of important assets, identities, and behaviors that produce usable telemetry and detections.

Weak measurement creates false confidence. For example, closing many alerts quickly is not proof of maturity if analysts are dismissing noisy alerts without evidence, or if critical log sources are missing.

Example:
Reviewed alerts this week: 120
Confirmed useful alerts: 18
False positives: 82
Benign true positives: 20

Useful-alert ratio = 18 / 120 = 15%
Noise requiring tuning = 82 / 120 = 68.3%

8. Practical Examples

Example 1: Suspicious login

An identity provider reports a successful login from a new country followed by MFA enrollment. The first task is not to panic. The analyst checks the user, device, travel history, VPN pattern, previous login locations, MFA method, source IP reputation, and whether any sensitive resources were accessed.

Example 2: Endpoint malware alert

An endpoint tool reports a suspicious PowerShell command. The analyst collects process lineage, command line, user context, file hashes, network connections, parent process, and whether the command was blocked. A blocked command on a test machine is handled differently from the same command on a domain controller.

Example 3: Cloud policy change

A cloud audit log shows a storage bucket policy changed to allow public access. The analyst checks actor identity, change time, change source, affected data classification, object access after the change, and whether the change was part of an approved deployment.

basic_soc_triage.py
import json
from pathlib import Path

LOG_FILE = Path("soc_events.jsonl")
KNOWN_OFFICE_IPS = {"198.51.100.20", "198.51.100.44"}
HIGH_RISK_EVENTS = {
    "mfa_method_added": "New MFA enrollment after unusual login",
    "storage_policy_changed": "Cloud storage exposure risk",
    "malware_blocked": "Endpoint control blocked suspicious activity",
}


def load_events(path: Path) -> list[dict]:
    events = []
    for line_no, line in enumerate(path.read_text().splitlines(), start=1):
        if not line.strip():
            continue
        try:
            events.append(json.loads(line))
        except json.JSONDecodeError as exc:
            print(f"Skipping malformed line {line_no}: {exc}")
    return events


def score(event: dict) -> tuple[int, list[str]]:
    points = 0
    reasons = []

    if event.get("event") in HIGH_RISK_EVENTS:
        points += 4
        reasons.append(HIGH_RISK_EVENTS[event["event"]])

    if event.get("src_ip") not in KNOWN_OFFICE_IPS:
        points += 2
        reasons.append("Source IP is not in the training office allowlist")

    if event.get("result") == "blocked":
        points += 1
        reasons.append("Blocked action still needs review for attempted behavior")

    return points, reasons


def main() -> None:
    events = load_events(LOG_FILE)
    print("Basic SOC triage")
    print("================")

    for event in events:
        points, reasons = score(event)
        if points >= 4:
            print(f"score={points} time={event['time']} user={event['user']} event={event['event']} asset={event['asset']}")
            for reason in reasons:
                print(f"  - {reason}")

    print("\nTriage question: which events belong to the same case?")


if __name__ == "__main__":
    main()

10. Security Perspective

A defender should think in terms of attacker objectives and defender evidence. An attacker wants credentials, execution, persistence, privilege, data access, or disruption. A defender needs evidence that connects an identity, asset, action, time, and impact.

Security operations fails when teams treat every tool alert as equal. A low-severity event on a critical server may deserve more attention than a high-severity event on a disposable lab machine. Context changes priority.

Good triage asks: what changed, who did it, from where, on which asset, was it allowed, was it expected, what happened next, and what evidence is missing?

11. Industry Perspective

In industry, SOC work is usually split across tiers or functions. Tier 1 analysts handle initial review and routing. Tier 2 analysts investigate deeper. Incident responders manage confirmed incidents. Detection engineers maintain rules and analytics. Threat hunters search for suspicious behavior that may not yet generate alerts. Security engineers improve logging, controls, and integrations.

Small organizations may combine all these roles into one or two people. That is acceptable if the workflow is explicit. The dangerous version is informal operation where nobody owns logging, nobody tunes detections, and nobody knows when to escalate.

Managed detection and response providers can help, but they still require asset context, identity context, escalation contacts, and authority boundaries. Outsourcing monitoring does not outsource accountability.

12. Research Perspective

Research in security operations studies detection quality, alert fatigue, analyst cognition, adversary behavior modeling, privacy-preserving telemetry, automated correlation, and human-machine teaming. Modern work also examines how large language models can summarize cases or assist triage without inventing evidence.

A research mindset is useful even in beginner SOC work. Every alert is a hypothesis: this pattern may indicate risky behavior. The analyst tests the hypothesis with evidence. If the hypothesis is weak, the detection must be tuned or retired.

13. Case Studies

Case study A: No logging during compromise

A team discovers suspicious administrator activity but has only seven days of logs and no cloud audit archive. The incident cannot be reconstructed confidently. The lesson is that logging is a security control, not a storage afterthought.

Case study B: Alert fatigue hides the real incident

A SOC receives thousands of repeated low-quality alerts. Analysts close them in bulk. A real suspicious login is missed because it looks like routine noise. The lesson is that tuning and prioritization are operational security requirements.

Case study C: Fast containment breaks production

An analyst disables a shared service account without checking ownership, stopping a critical deployment pipeline. The security concern was real, but the response was not coordinated. The lesson is that containment must be proportional and reversible when possible.

14. Common Misconceptions

  • Misconception: A SIEM automatically creates security. Correction: A SIEM stores and correlates data; people and processes still define useful detection and response.
  • Misconception: More alerts means better visibility. Correction: More alerts can mean more noise. Useful coverage matters more than raw volume.
  • Misconception: A blocked attack needs no investigation. Correction: Blocked behavior can reveal compromised credentials, exposed services, or repeated targeting.
  • Misconception: Closing an alert means nothing happened. Correction: Closure requires a reason, evidence, and sometimes a tuning recommendation.
  • Misconception: Automation can replace analysts. Correction: Automation can enrich, route, and contain known patterns; judgment is still needed for uncertainty and business impact.

15. Summary

Security operations is an evidence-driven workflow for detecting, investigating, responding to, and learning from suspicious activity. The SOC is not defined by a specific tool. It is defined by telemetry, detection logic, triage quality, response authority, documentation, and continuous improvement.

The most important beginner skill is disciplined triage: separate facts from assumptions, record unknowns, preserve evidence, and choose the next action based on risk and authorization.

16. Glossary

  • SOC: The security operations function responsible for monitoring, triage, investigation, and response.
  • Telemetry: Security-relevant data generated by systems, users, networks, endpoints, cloud services, and applications.
  • Alert: A notification that detection logic or a security tool believes activity may require review.
  • Case: The record that tracks an investigation, evidence, decisions, actions, and closure rationale.
  • Incident: A confirmed or strongly suspected event that threatens security, privacy, safety, compliance, or operations.
  • Runbook: A step-by-step procedure for a repeatable operational task.
  • Playbook: A coordinated workflow for a security scenario.
  • Escalation: Moving an alert or case to a person or team with greater authority, expertise, or responsibility.
  • Alert fatigue: Reduced analyst effectiveness caused by excessive, repetitive, or low-value alerts.
