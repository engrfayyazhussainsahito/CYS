Security Operations, Monitoring, and Detection Engineering

Week 11 moves from building and securing systems into operating defense continuously. Students learn how a security operations function turns telemetry into detection, triage, investigation, response, measurement, and improvement.

Estimated reading time45–60 minutes for overview; 6–10 hours per full lesson
Difficulty levelIntermediate MS foundation
Required backgroundWeeks 0–10 recommended
DeliverableSOC workflow map, telemetry inventory, and triage note

Overview

Security operations is the disciplined process of watching systems, interpreting evidence, deciding what matters, and coordinating defensive action. It is not simply a room of analysts looking at dashboards. A useful SOC operates like an evidence-processing system: it collects telemetry, enriches context, runs detections, prioritizes alerts, investigates suspicious behavior, escalates incidents, and feeds lessons back into engineering.

The central problem of Week 11 is signal quality. Modern environments generate huge volumes of logs, endpoint events, cloud audit records, authentication events, network metadata, vulnerability findings, and application traces. Most events are benign. Some are suspicious. A small number require urgent response. The defender's job is to convert noisy data into justified decisions without overreacting, ignoring weak signals, or violating privacy boundaries.

This week connects the earlier course modules. Identity lessons explain why authentication logs matter. Network lessons explain DNS, HTTP, and routing evidence. Endpoint lessons explain process and file telemetry. Cloud lessons explain audit trails, IAM activity, storage access, and deployment events. Week 11 combines those sources into operational defense.

Core Mental Model

A mature security operation repeatedly asks four questions:

  1. What evidence exists? Identify logs, sensors, timestamps, identities, assets, and known gaps.
  2. What behavior matters? Define suspicious patterns in terms of attacker behavior, business risk, and expected baseline activity.
  3. What decision is required? Decide whether the output is an alert, a case, an incident, a hunt lead, or no action.
  4. What action is safe and authorized? Choose proportional containment, escalation, evidence preservation, communication, or tuning.
Key idea: dashboards show data, but operations require decisions. A detection that cannot support a decision creates noise, not security.

Week 11 Learning Objectives

  • Explain what a SOC does and why security operations must be grounded in evidence, authorization, and repeatable workflows.
  • Differentiate telemetry, logs, events, alerts, cases, investigations, incidents, and threat hunts.
  • Describe the role of SIEM platforms, log sources, parsing, normalization, enrichment, retention, and telemetry pipelines.
  • Write detection ideas as behavior, logic, required evidence, expected false positives, severity rationale, and response guidance.
  • Perform basic alert triage using facts, assumptions, uncertainty, impact, scope, and next actions.
  • Use SOC metrics to reason about visibility, alert quality, response time, automation risk, and maturity.

Operational Workflow

The workflow below is intentionally simple. Real SOCs vary by size and tooling, but the same control loop appears repeatedly.

flowchart LR
  A[Telemetry sources] --> B[Collection and parsing]
  B --> C[Normalization and enrichment]
  C --> D[Detection logic]
  D --> E[Alert queue]
  E --> F[Triage]
  F -->|Benign or duplicate| G[Tune, close, document]
  F -->|Suspicious| H[Investigation case]
  H --> I[Scope and impact]
  I --> J[Incident response]
  J --> K[Lessons learned]
  K --> B
  K --> D

The feedback loops matter. If an alert was noisy, tune the detection or improve context. If an incident was missed, improve telemetry or create new logic. If analysts lack authority to act, clarify the escalation path. Operations improve when evidence, decisions, and actions are connected.

Common Failure Points

  • Collecting logs without a question. Storage fills up, costs rise, and analysts still cannot decide what happened.
  • Writing rules before defining behavior. Syntax may work, but the rule does not map cleanly to risk or response.
  • Treating every alert as an incident. This burns analyst time and makes true incidents harder to recognize.
  • Ignoring context. A login failure, process start, or cloud API call may be normal or critical depending on asset, identity, timing, and baseline.
  • Automating containment too early. Blocking accounts, hosts, or workloads without confidence can cause business damage.
  • Forgetting privacy and authorization. Monitoring real users, systems, or network traffic requires a lawful, approved scope.

Week 11 Practical Work

All practice stays local and fictional. The labs are designed to teach reasoning without using real company telemetry or attacking live systems.

  • SOC workflow map: draw the path from raw event to alert, triage, case, incident, and lesson learned.
  • Telemetry inventory: list useful sources such as authentication logs, endpoint events, DNS logs, cloud audit records, web access logs, and vulnerability data.
  • Detection brief: define one suspicious behavior, required evidence, false positives, severity, and response guidance.
  • Triage note: summarize facts, assumptions, unknowns, impact, scope, and next action from fictional alerts.
  • Metrics review: distinguish useful measures from vanity metrics such as raw alert count.

Safety and Ethics Boundary

Do not use real telemetry for class practice unless you have explicit authorization. Do not collect, export, inspect, or share user logs, cloud audit trails, endpoint telemetry, packet captures, email events, production application logs, or company SIEM data outside an approved scope. Use fictional data, personal lab systems, or sanitized training datasets only.

Week 11 is defensive, but defensive monitoring can still expose sensitive data. Logs may contain usernames, IP addresses, tokens, URLs, filenames, emails, customer identifiers, and business records. A good SOC protects evidence and privacy at the same time.

Readiness Checklist

  • You can explain the difference between a log event and an alert.
  • You can name at least five useful telemetry sources for a modern environment.
  • You can describe why identity context and asset criticality change alert priority.
  • You can separate a detection hypothesis from a finished detection rule.
  • You can write a short triage note that separates facts from assumptions.
  • You can explain why alert count alone is not a good SOC performance metric.

Exit Criteria for Week 11

By the end of Week 11, a student should be able to look at a small fictional alert dataset and produce a defensible operational answer: what happened, what evidence supports that view, what remains unknown, how serious it may be, what should happen next, what detection or telemetry improvement would reduce future uncertainty, and which SOC metrics would prove the operation is improving.
