Lesson 1.2: CIA Triad and Security Goals

The CIA triad is the classic starting model for security goals: confidentiality, integrity, and availability. This lesson teaches the triad from zero knowledge and then extends it into modern goals such as privacy, authenticity, accountability, non-repudiation, safety, and resilience.

Estimated reading time140–210 minutes
Difficulty levelBeginner → Graduate foundation
Required backgroundLesson 1.1 recommended
Practice time4–8 hours

Week 1 Context

Overview: Week 1 teaches security thinking. Lesson 1.1 asked what cybersecurity protects. Lesson 1.2 asks what kind of protection is required. The answer is expressed as security goals.

Learning objectives: after this lesson you should be able to define confidentiality, integrity, and availability; identify which goal is at risk in a scenario; explain common controls; understand trade-offs; and extend CIA analysis to privacy, safety, authenticity, accountability, non-repudiation, and resilience.

Required background: no technical background is required. You should understand the idea of an asset, stakeholder, harm, and control from Lesson 1.1.

Estimated reading time: 140–210 minutes. Difficulty: starts at beginner level and finishes with graduate-level reasoning about competing security objectives.

1. Introduction

Plain-English idea: a security goal says what kind of harm you are trying to prevent. If private information is exposed, the violated goal is usually confidentiality. If a grade, balance, medical dose, or software update is changed incorrectly, the violated goal is integrity. If a hospital system, bank app, or emergency service cannot be used when needed, the violated goal is availability.

The CIA triad exists because security professionals needed a simple way to describe different types of protection. Without this model, teams may talk past each other. One person might say “make the system secure” and mean encryption. Another might mean preventing changes. Another might mean keeping the system online during failure. The CIA triad separates these ideas.

The topic matters because almost every security decision depends on it. Passwords, encryption, backups, digital signatures, monitoring, access control, disaster recovery, patching, redundancy, and incident response all support one or more security goals.

The CIA triad is used by security engineers, auditors, software developers, cloud architects, incident responders, compliance teams, privacy officers, risk managers, researchers, and executives. It is also used in formal security categorization. For example, NIST FIPS 199 categorizes information systems by potential impact to confidentiality, integrity, and availability.

Important limitation

The CIA triad is powerful, but incomplete. It does not fully cover privacy, safety, accountability, legal compliance, usability, fairness, human harm, or resilience. Treat it as a foundation, not the entire field.

2. Learning Outcomes

  • Define confidentiality, integrity, availability, privacy, authenticity, accountability, non-repudiation, safety, resilience, and trust.
  • Explain the difference between secrecy, correctness, and usability.
  • Classify incidents by the security goals they violate.
  • Connect security controls to security goals.
  • Explain why one control can support multiple goals.
  • Explain why security goals can conflict with one another.
  • Use simple risk math to compare confidentiality, integrity, and availability scenarios.
  • Calculate basic service availability from uptime and downtime.
  • Explain how cryptographic hashes support integrity at a conceptual level.
  • Build a security-goal matrix for a fictional system.
  • Describe how CIA goals appear in banks, governments, healthcare, defense, universities, cloud providers, and large enterprises.
  • Identify research problems related to integrity, availability, privacy, safety, and resilience.

3. Prerequisites

3.1 Asset

An asset is anything valuable enough to protect. Examples include records, systems, credentials, source code, services, reputation, and human safety.

3.2 Security objective

A security objective is a protection goal. It describes the kind of protection an asset needs. “Only authorized doctors can see patient records” is a confidentiality objective. “Prescriptions must not be changed without authorization” is an integrity objective. “The emergency department system must work during a crisis” is an availability objective.

3.3 Control

A control is a safeguard. A control can prevent, detect, respond to, or recover from a problem. Encryption is a control. Backups are controls. Logs are controls. Code review is a control. Training can also be a control.

3.4 Authorized user

An authorized user has permission to perform an action. Authorization is not the same as identity. A student may be a valid user but still not authorized to edit another student’s grade.

3.5 System state

A state is the condition of a system at a moment in time. A database state includes the current records. A server state includes running processes, configuration, and files. Integrity often asks whether the state is correct.

3.6 Service

A service is something a system provides to users or other systems. Examples include login, email, file storage, payment processing, course registration, and DNS resolution. Availability often asks whether a service can be used when needed.

4. Core Concepts

4.1 Security goal

A security goal tells you what “secure” means for one asset in one context. The word secure is too broad by itself. A public university website may not require confidentiality for public news pages, but it does require integrity because false announcements could mislead people. A public emergency-alert system may intentionally publish information, so confidentiality may be low, but availability and integrity are critical.

4.2 Confidentiality

Confidentiality means preventing unauthorized disclosure. In plain English: the wrong people should not see the information.

Intuition: confidentiality is like sealed envelopes, locked cabinets, private conversations, or curtains on a window. The goal is not to destroy the information. The goal is to control who can observe it.

Examples: salary records, student grades, patient diagnoses, passwords, encryption keys, legal documents, investigation notes, unreleased exam papers, and business plans.

Common confidentiality controls:

  • Authentication: prove identity.
  • Authorization: restrict access to allowed users.
  • Encryption: make data unreadable without a key.
  • Data classification: label information by sensitivity.
  • Network segmentation: reduce who can reach sensitive systems.
  • Logging and monitoring: detect unusual access.
  • Secure disposal: prevent recovery of old data.

Common confidentiality failures: exposed cloud storage, weak permissions, stolen credentials, phishing, eavesdropping, leaked backups, logs containing secrets, screenshots, misplaced laptops, and overbroad database access.

4.3 Integrity

Integrity means preventing unauthorized or incorrect modification. In plain English: information and systems should remain accurate, complete, and trustworthy.

Intuition: integrity is like a tamper-evident seal on medicine, a teacher checking that a grade sheet was not altered, or a bank verifying that every debit and credit balances correctly.

Examples: bank balances, grades, medical prescriptions, election records, source code, software updates, firewall rules, backups, DNS records, audit logs, and sensor readings.

Common integrity controls:

  • Input validation: reject malformed or unsafe data.
  • Access control: restrict who can modify data.
  • Version control: track changes over time.
  • Hashes and checksums: detect accidental or malicious changes.
  • Digital signatures: verify origin and detect modification.
  • Database constraints: enforce valid relationships.
  • Code review and change approval: reduce unauthorized changes.
  • Audit logs: record who changed what and when.

Common integrity failures: unauthorized database updates, modified software packages, log tampering, DNS poisoning, transaction manipulation, corrupted backups, compromised build pipelines, and fake alerts.

4.4 Availability

Availability means authorized users can access systems, data, and services when needed. In plain English: the system should work when legitimate users need it.

Intuition: availability is like a hospital generator, a second bridge when one bridge is closed, a spare key, or a backup classroom if one room floods.

Examples: online banking, emergency services, hospital scheduling, cloud storage, payment systems, DNS, learning management systems, payroll, logistics, and identity services.

Common availability controls:

  • Backups and restore testing.
  • Redundancy: multiple servers, links, or data centers.
  • Load balancing.
  • Rate limiting and DDoS protection.
  • Patch management.
  • Capacity planning.
  • Monitoring and alerting.
  • Incident response and disaster recovery plans.

Common availability failures: ransomware, denial-of-service attacks, hardware failure, power loss, cloud outage, expired certificate, bad deployment, database overload, accidental deletion, and dependency failure.

4.5 Privacy

Privacy concerns appropriate collection, use, sharing, retention, and protection of personal information. Confidentiality helps privacy, but they are not identical.

Example: a company may keep location data confidential from outsiders but still misuse it internally for excessive tracking. That may be a privacy problem even if no external breach occurs.

4.6 Authenticity

Authenticity means confidence that something is genuine. A login page should really belong to the bank. A software update should really come from the vendor. A message should really be from the claimed sender.

Authenticity often depends on authentication, certificates, digital signatures, trusted distribution channels, and user-interface clarity.

4.7 Accountability

Accountability means actions can be traced to responsible identities, systems, or processes. It depends on identity, logging, time synchronization, audit trails, and governance.

Accountability does not mean blaming the first person found in a log. Logs can be incomplete, spoofed, or generated by compromised accounts. A security professional must distinguish evidence from conclusion.

4.8 Non-repudiation

Non-repudiation means a person or entity cannot credibly deny having performed a specific action when there is strong supporting evidence. Digital signatures can support non-repudiation, but policy, key protection, identity proofing, and legal process matter too.

Example: if an executive digitally signs a contract using a protected signing key under a formal policy, the organization has stronger evidence that the action occurred.

4.9 Safety

Safety means protection against harm to people or the physical world. Safety is crucial in hospitals, vehicles, power systems, factories, aircraft, ships, and military systems.

A system can preserve confidentiality but still be unsafe. For example, a medical device that keeps patient data secret but delivers the wrong dose has an integrity and safety failure.

4.10 Resilience

Resilience means the ability to withstand, adapt to, and recover from disruption. Availability asks whether the service is usable. Resilience asks how the organization continues and recovers when things go wrong.

Backups alone are not resilience. Recovery plans must be tested. People must know roles. Dependencies must be understood. Communication must work during the incident.

4.11 Security goals can conflict

Security goals are not always aligned. More logging can improve accountability and detection, but excessive logging can harm privacy if logs contain personal data. Stronger authentication can improve confidentiality, but poor implementation can reduce availability if users are locked out during emergencies. Encryption can protect confidentiality, but lost keys can destroy availability.

DecisionHelpsCan hurtReason
Encrypt all filesConfidentialityAvailabilityLost keys may make recovery impossible.
Collect detailed logsAccountability, detectionPrivacy, confidentialityLogs may contain sensitive data.
Strict account lockoutConfidentialityAvailabilityAttackers may intentionally lock out legitimate users.

4.12 Control mapping

Controls are not tied to only one goal. A digital signature supports integrity, authenticity, and non-repudiation. Backups support availability and sometimes integrity. Access control supports confidentiality and integrity. Monitoring supports detection for all three CIA goals.

7. Mathematical Foundations

7.1 Simple risk score

A common teaching model is:

Risk score ≈ likelihood × impact

This is not a complete actuarial model. It is a beginner-friendly way to compare scenarios. Use the same scale for all comparisons. For example, use 1 to 5 for likelihood and 1 to 5 for impact.

ScenarioGoal affectedLikelihoodImpactScore
Student records exposedConfidentiality3515
Course grade changedIntegrity2510
Portal unavailable during registrationAvailability4416

In this simple model, the availability problem has the highest score. That does not mean confidentiality and integrity are unimportant. It means this scenario deserves urgent planning in this particular comparison.

7.2 Availability percentage

Availability is often expressed as a percentage:

Availability = uptime / total time

If a service is expected to run for 30 days, total time is:

30 days × 24 hours/day = 720 hours

If the service was down for 3 hours:

Uptime = 720 - 3 = 717 hours
Availability = 717 / 720 = 0.995833...
Availability percentage = 99.5833%

Small percentages can hide large downtime. A service with 99% availability can still be down for about 7.2 hours in a 30-day month.

7.3 Expected loss

A simple expected-loss model is:

Expected loss = probability of event × loss if event happens

Example: if there is a 10% chance of a $100,000 incident in one year:

Expected loss = 0.10 × 100,000 = 10,000

This does not prove the organization should spend exactly $10,000. It provides a rational starting point for risk discussion.

7.4 Integrity and hash comparison

A cryptographic hash function maps data to a fixed-size value. The same input should produce the same output. If a file changes, the hash should change unpredictably.

trusted_hash = hash(original_file)
current_hash = hash(current_file)
if trusted_hash == current_hash:
    integrity check passes
else:
    integrity check fails or requires investigation

A hash alone does not prove who created the file. For origin authenticity, use a digital signature or trusted distribution process.

7.5 Reliability of redundant systems

If two independent servers each have a 0.02 probability of failing during a period, the probability that both fail at the same time is approximately:

0.02 × 0.02 = 0.0004 = 0.04%

This independence assumption is often false. A bad software deployment, shared power failure, or shared cloud dependency can break both servers. Graduate-level availability analysis must identify shared failure modes.

8. Practical Examples

8.1 Linux examples

Check file permissions to support confidentiality and integrity:

ls -l private-notes.txt
chmod 600 private-notes.txt
ls -l private-notes.txt

Meaning: chmod 600 lets the owner read and write the file while removing access for group and others. This supports confidentiality by reducing who can read the file and integrity by reducing who can modify it.

Create a SHA-256 hash to support integrity checking:

sha256sum assignment.txt > assignment.txt.sha256
sha256sum -c assignment.txt.sha256

The first command stores a trusted hash. The second command checks whether the current file still matches.

8.2 Windows examples

Use PowerShell to calculate a file hash:

Get-FileHash .\assignment.txt -Algorithm SHA256

Check recent failed logon events on a local Windows system:

Get-WinEvent -FilterHashtable @{LogName='Security'; Id=4625} -MaxEvents 5

This is a defensive local-system example. Do not collect logs from systems you do not own or administer.

8.3 Python examples

The following safe script classifies fictional incidents by security goal.

incidents = [
    {"event": "Student grade changed without approval", "goal": "Integrity"},
    {"event": "Private medical record emailed to wrong person", "goal": "Confidentiality"},
    {"event": "Course portal offline during registration", "goal": "Availability"},
]

for item in incidents:
    print(f"{item['goal']}: {item['event']}")
LineExplanation
incidents = [Creates a list. A list stores multiple items.
{"event": ..., "goal": ...}Creates a dictionary. A dictionary stores named fields.
for item in incidents:Loops through each incident one at a time.
print(...)Displays the security goal and event in readable text.

8.4 Network examples

Safe local-only example: start a local web server and observe availability.

mkdir cia-local-demo
cd cia-local-demo
printf "Hello from local server\n" > index.html
python3 -m http.server 8000

Open http://127.0.0.1:8000 in your own browser. The address 127.0.0.1 means your own machine. This does not scan or attack any external system.

8.5 Web examples

Web applications must protect all three CIA goals:

  • Confidentiality: users should only see their own private records.
  • Integrity: users should not modify prices, grades, balances, or roles without permission.
  • Availability: the service should remain usable during expected traffic and failure.

Beginner-safe design checklist:

□ Use HTTPS in production.
□ Do not store passwords in plain text.
□ Check authorization on every sensitive action.
□ Validate input on the server.
□ Keep audit logs for important changes.
□ Test backup and restore procedures.
□ Avoid logging passwords, tokens, or private messages.

8.6 Cloud examples

A cloud storage bucket can violate different goals depending on configuration:

  • Public read access to private files violates confidentiality.
  • Public write access violates integrity.
  • No backup or lifecycle plan can violate availability and resilience.

Safe pseudo-policy example:

{
  "effect": "allow",
  "principal": "course-admin-role",
  "actions": ["read", "write"],
  "resource": "fictional-course-bucket",
  "condition": "MFA required"
}

This is not a provider-specific policy. It is a teaching example showing how identity, action, resource, and condition support security goals.

9. Code Examples

9.1 Integrity checker in Python

This script calculates the SHA-256 hash of a local file. It is safe because it only reads a file that you provide on your own machine.

from pathlib import Path
import hashlib

file_path = Path("assignment.txt")

if not file_path.exists():
    raise SystemExit("File not found: assignment.txt")

file_bytes = file_path.read_bytes()
digest = hashlib.sha256(file_bytes).hexdigest()

print("File:", file_path)
print("SHA-256:", digest)

9.2 Common coding mistakes

  • Mistake: using a hash as encryption. Correction: hashing detects change; encryption protects confidentiality.
  • Mistake: trusting client-side checks only. Correction: enforce authorization and validation on the server.
  • Mistake: logging secrets. Correction: redact passwords, tokens, keys, and private messages.
  • Mistake: assuming a backup exists because a job is scheduled. Correction: test restoration.
  • Mistake: assuming encryption solves all security. Correction: also protect keys, integrity, availability, and users.

10. Security Perspective

10.1 Common attacks

AttackMain goal attackedExample harm
Credential theftConfidentiality, authenticityAttacker accesses private systems as another user.
Data exfiltrationConfidentialityPrivate records are copied out.
Data tamperingIntegrityRecords, logs, or configuration are changed.
Supply-chain compromiseIntegrity, authenticityTrusted software update carries malicious code.
RansomwareAvailability, integrity, confidentialityData is encrypted, stolen, or disrupted.
DDoSAvailabilityService is overloaded and unavailable.
Log deletionAccountability, integrityInvestigation evidence is weakened.

10.2 Common vulnerabilities

  • Weak or reused passwords.
  • Overly broad permissions.
  • Unencrypted sensitive data.
  • Missing authorization checks.
  • Insecure software update process.
  • No backup testing.
  • Single points of failure.
  • Insufficient logging.
  • Logs that contain secrets.
  • No incident response plan.

10.3 Defensive strategies

  • Classify assets and assign security goals.
  • Use least privilege.
  • Protect secrets and keys.
  • Use encryption where confidentiality is needed.
  • Use integrity checks and signed updates where correctness is critical.
  • Design for redundancy and tested recovery.
  • Monitor important actions and failures.
  • Train users and administrators.
  • Review trade-offs before implementing controls.

10.4 Detection methods

  • Confidentiality detection: unusual data downloads, impossible travel logins, access to unusual records, secret exposure scans.
  • Integrity detection: file integrity monitoring, unexpected configuration changes, unsigned binaries, database audit anomalies.
  • Availability detection: uptime checks, error-rate monitoring, resource saturation alerts, failed dependency health checks.
  • Accountability detection: missing logs, time drift, disabled auditing, shared account usage.

10.5 Mitigation and best practices

  • Use multi-factor authentication for sensitive systems.
  • Apply role-based or attribute-based access control.
  • Encrypt sensitive data in transit and at rest.
  • Use authenticated encryption or digital signatures when integrity matters.
  • Maintain offline or immutable backups for critical data.
  • Test recovery procedures, not only backup creation.
  • Segment networks and limit administrative access.
  • Protect logs from alteration.
  • Document assumptions, ownership, and recovery priorities.

12. Research Perspective

Research uses CIA goals as building blocks but investigates harder questions: how to prove properties, how to measure harm, how to design trustworthy systems, and how to defend under uncertainty.

12.1 Current research directions

  • Confidential computing: protecting data while it is processed, not only while stored or transmitted.
  • Secure multiparty computation: allowing parties to compute jointly without revealing private inputs.
  • Formal verification: mathematically proving that software or protocols satisfy integrity and safety properties.
  • Supply-chain security: proving that software builds, packages, and dependencies have not been tampered with.
  • Resilient distributed systems: maintaining availability despite failure, attack, or untrusted infrastructure.
  • Usable security: designing controls that real users can understand and use correctly.
  • Privacy engineering: minimizing data collection and limiting inference, not merely hiding data.
  • AI security: applying confidentiality, integrity, and availability to models, prompts, training data, outputs, and agents.

12.2 Open problems

  • How should organizations quantify privacy harm, not only financial loss?
  • How can integrity be guaranteed across complex software supply chains?
  • How can cloud customers verify confidentiality and integrity in infrastructure they do not physically control?
  • How can critical services remain available during large-scale dependency failures?
  • How can systems produce enough logs for accountability without creating privacy risks?
  • How should CIA goals be adapted for AI systems that generate text, decisions, and code?

12.3 Notable paper directions

Read research papers by identifying the security goal first. Ask: is the paper protecting secrecy, correctness, service continuity, privacy, identity, accountability, safety, or recovery? Do not copy paper text into your notes. Summarize the problem, method, evidence, limitation, and security goal in your own words.

13. Case Studies

13.1 Equifax data breach: confidentiality failure

What happened: Equifax announced in September 2017 that a breach exposed personal information of about 147 million people, according to the U.S. Federal Trade Commission’s settlement information.

Root cause theme: sensitive personal information was accessible after security weaknesses were exploited. The case is mainly a confidentiality failure, but it also shows governance, patching, detection, and risk-management failures.

Impact: exposed personal information created identity-theft and fraud risk for affected people and led to major legal and reputational consequences.

Lessons learned: confidentiality requires more than encryption. Organizations need asset inventory, patch management, least privilege, monitoring, segmentation, and timely incident response.

13.2 SolarWinds Orion compromise: integrity and authenticity failure

What happened: CISA described the SolarWinds incident as a software supply-chain compromise involving trojanized Orion software updates.

Root cause theme: trusted update mechanisms can become an attack path if the build or distribution process is compromised.

Impact: many organizations had to investigate whether trusted software delivered malicious functionality into their environments.

Lessons learned: integrity of source code, build systems, dependencies, update channels, and signing infrastructure is critical. Trust must be verified continuously, not assumed permanently.

13.3 Mirai/Dyn DDoS: availability failure

What happened: CISA reported that Mirai malware used vulnerable Internet of Things devices to create DDoS attacks. Large DDoS events can disrupt dependent services by overwhelming infrastructure.

Root cause theme: insecure default credentials and internet-exposed devices can be abused at scale.

Impact: users may be unable to reach services even if the services’ data remains secret and unmodified.

Lessons learned: availability requires capacity planning, DDoS defenses, secure device management, redundancy, and dependency analysis.

13.4 Ransomware and destructive malware: multiple goals at once

What happened: ransomware and destructive malware incidents can encrypt data, interrupt operations, steal information, and damage trust.

Root cause theme: weak access control, unpatched systems, exposed services, poor segmentation, and insufficient recovery planning often increase harm.

Impact: confidentiality, integrity, and availability can all be affected in one incident.

Lessons learned: do not classify every incident as only one CIA goal. Real incidents often combine goals, controls, people, and business processes.

14. Common Misconceptions

  • Misconception: cybersecurity means confidentiality only. Reality: integrity and availability can be equally or more important depending on the system.
  • Misconception: encryption solves security. Reality: encryption does not automatically solve access control, integrity, key management, malware, or outages.
  • Misconception: if data is public, security does not matter. Reality: public data still needs integrity and availability.
  • Misconception: backups guarantee availability. Reality: backups help only if they are complete, protected, current, and restorable.
  • Misconception: hashes keep data secret. Reality: hashes detect changes; they do not hide the original content in the same way encryption does.
  • Misconception: logs always prove who did something. Reality: logs are evidence, but accounts can be compromised and logs can be incomplete.
  • Misconception: more security always improves safety. Reality: poorly designed controls can block emergency access or create unsafe workarounds.
  • Misconception: availability is only an IT problem. Reality: availability depends on people, suppliers, power, network dependencies, contracts, and recovery procedures.
  • Misconception: privacy and confidentiality are identical. Reality: privacy also concerns appropriate collection, use, retention, and sharing.
  • Misconception: CIA is outdated. Reality: CIA remains useful, but modern systems require additional goals and context.

15. Summary

The CIA triad gives you three foundational security goals: confidentiality, integrity, and availability. Confidentiality asks who can see information. Integrity asks whether information, code, configuration, or decisions remain correct and authorized. Availability asks whether systems and data are usable when needed.

Modern cybersecurity extends CIA with privacy, authenticity, accountability, non-repudiation, safety, resilience, usability, and governance. Strong security analysis identifies which goal matters for each asset, which harms matter to stakeholders, which controls support the goals, and which trade-offs must be managed.

The next lesson builds on this by separating threats, vulnerabilities, exploits, and risk. That distinction is essential because a security goal tells you what you want to protect, while risk analysis helps you decide what could go wrong and how seriously to treat it.

16. Glossary

Accountability
The ability to trace actions to responsible identities, systems, or roles.
Asset
Anything valuable enough to protect.
Authenticity
Confidence that something or someone is genuine.
Availability
A security goal meaning authorized users can access systems, data, or services when needed.
Backup
A copy of data used for recovery after loss, corruption, or disruption.
CIA Triad
The security model based on confidentiality, integrity, and availability.
Confidentiality
A security goal meaning information should not be disclosed to unauthorized parties.
Control
A safeguard that prevents, detects, responds to, or recovers from risk.
Digital signature
A cryptographic mechanism that can support origin authentication, data integrity, and non-repudiation.
Encryption
A method for transforming readable data into unreadable form without the correct key.
Hash
A fixed-size value calculated from data, commonly used to detect changes.
Integrity
A security goal meaning data, code, configuration, and decisions should remain accurate, complete, and authorized.
Non-repudiation
Protection against a person falsely denying that they performed an action.
Privacy
Appropriate collection, use, sharing, retention, and protection of personal information.
Redundancy
Additional components or paths used to continue operation if one fails.
Resilience
The ability to withstand, adapt to, and recover from disruption.
Risk
The possibility of harm, often considered through likelihood and impact.
Safety
Protection against harm to people or the physical world.
Security objective
A protection goal for an asset.
Uptime
The amount of time a service is operational.

18. Practice Lab

Lab title

Build a CIA Goal Matrix for a Fictional Student Services Platform

Safety scope

  • Use only the fictional scenario below.
  • Do not test, scan, log into, or probe any real university, company, or public system.
  • Do not collect or use real student, patient, employee, or customer data.
  • All commands must run on your own local computer or local VM.

Scenario

A fictional university runs a platform called CampusOne. It has student profiles, grades, course registration, fee invoices, teacher grade upload, administrator approval, email notifications, audit logs, backups, and a public announcement page.

Prerequisites

  • Lesson 1.1: What Cybersecurity Protects.
  • Basic text editor or spreadsheet.
  • Optional Python from Lesson 0.4.

Objectives

  • Assign CIA and extended security goals to assets.
  • Identify controls that support each goal.
  • Detect goal conflicts and trade-offs.
  • Calculate simple availability and risk examples.
  • Produce a short security-goal report.

Step 1: Create a CIA matrix

| Asset | Confidentiality | Integrity | Availability | Extended goals | Reason |
|---|---|---|---|---|---|
| Student grades | High | High | Medium | Accountability, privacy | Grades are private and must be correct. |
| Public announcements | Low | High | High | Authenticity | Public data must be trustworthy and reachable. |
| Audit logs | Medium | High | High | Accountability, non-repudiation | Logs support investigation and evidence. |

Step 2: Add controls

| Asset | Goal | Control | Prevent / Detect / Recover |
|---|---|---|---|
| Student grades | Confidentiality | Role-based access control | Prevent |
| Student grades | Integrity | Approval workflow and audit logs | Prevent + Detect |
| Registration service | Availability | Load testing and backups | Prevent + Recover |

Step 3: Calculate availability

Assume CampusOne should run for 168 hours in one week. It was unavailable for 2 hours.

Total time = 168 hours
Downtime = 2 hours
Uptime = 166 hours
Availability = 166 / 168 = 0.988095
Availability percentage ≈ 98.81%

Step 4: Optional local hash exercise

Create a local file and detect whether it changes.

printf "Approved grade file\n" > grades-demo.txt
sha256sum grades-demo.txt > grades-demo.sha256
printf "Changed grade file\n" > grades-demo.txt
sha256sum -c grades-demo.sha256

Expected result: the check should fail after the file changes. This demonstrates integrity checking on your own local file only.

Step 5: Write a short report

Your report should include:

  • At least 8 assets.
  • CIA ratings for each asset.
  • At least 5 extended goals.
  • At least 10 controls mapped to goals.
  • At least 3 trade-offs.
  • One availability calculation.
  • One local integrity-check result.

Expected outcomes

  • You can classify security goals for real-looking systems without touching real targets.
  • You understand that different assets need different goals.
  • You can explain why controls must be chosen based on harm, not tool popularity.
