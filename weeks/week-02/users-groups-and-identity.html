Lesson 2.2: Users, Groups, and Identity

Cybersecurity starts by answering a deceptively simple question: who or what is making this request? This lesson teaches digital identity from zero: people, accounts, groups, service accounts, identifiers, attributes, account lifecycle, and the security risks created when identity is wrong, stale, overprivileged, or unaudited.

Difficulty levelBeginner → MS foundation
Required backgroundLesson 2.1 recommended; no IAM background required

Week 2 Context

Overview: Week 2 explains how systems decide who may access what. Lesson 2.1 introduced the operating system as the enforcement layer. Lesson 2.2 focuses on identities: users, groups, service accounts, machine accounts, identifiers, attributes, lifecycle events, and auditability.

Learning objectives: after this lesson you should be able to explain how operating systems and organizations represent users and groups, why identifiers are more reliable than display names, how group membership simplifies access control, and why identity lifecycle management is a security control.

Required background: no identity-management background is required. You should understand the terms operating system, process, file, permission, and privilege from Lesson 2.1.

Estimated reading time: 190–280 minutes. Difficulty: starts from zero and ends with graduate-level identity-design and research questions.

1. Introduction

Plain-English idea: an identity is the system’s representation of a person, device, application, or service. A user account represents a person or process that can interact with a system. A group collects accounts so access can be managed together.

Why the topic exists: computers need a way to distinguish Alice from Bob, a student from a teacher, a backup service from a web server, and an ordinary user from an administrator. Without identity, every request would look the same. If every request looks the same, the system cannot enforce meaningful security.

Why it matters: many breaches do not begin with advanced malware. They begin with a valid account used in the wrong way: a stolen password, an old account that was never disabled, a service account with excessive privilege, a shared administrator login, or a group that gives too much access.

Where it is used: local laptops, Linux servers, Windows domains, cloud consoles, university portals, hospital systems, banking systems, government services, source-code platforms, databases, containers, Kubernetes clusters, and software-as-a-service platforms.

Who uses it: students, employees, system administrators, help desks, security analysts, auditors, cloud engineers, developers, identity architects, incident responders, and researchers.

Analogy: a university identity card contains a name, student number, department, role, photo, and expiration date. The card is not the person. It is a representation that systems use to make decisions. Digital identity works the same way: the account is not the human; it is a record that links a subject to rights, attributes, and activity.

Safety boundary

This lesson uses only local observation and fictional examples. Do not create, modify, disable, unlock, reset, or inspect accounts on systems you do not own or administer with explicit written authorization.

2. Learning Outcomes

  • Define identity, subject, user, account, security principal, identifier, attribute, group, role, service account, machine account, local account, directory account, and privileged account.
  • Explain why identity is not the same as a username, password, email address, or real-world person.
  • Explain how Linux represents users and groups with usernames, UIDs, group names, GIDs, /etc/passwd, /etc/group, and /etc/shadow.
  • Explain how Windows represents accounts and groups with security principals, Security Identifiers, local accounts, domain accounts, and Active Directory-style groups.
  • Distinguish authentication from authorization and identity proofing.
  • Explain why groups are used to assign access at scale.
  • Identify common identity risks: shared accounts, stale accounts, orphaned accounts, weak naming, overbroad groups, unmanaged service accounts, privilege creep, and poor logging.
  • Apply least privilege, separation of duties, account review, and joiner-mover-leaver lifecycle thinking to a fictional organization.
  • Use safe Linux and Windows commands to observe your own identity and group membership.
  • Design a small identity model for a fictional university lab system and justify security decisions.

3. Prerequisites

3.1 Resource

A resource is anything a system protects: a file, database table, printer, network share, cloud bucket, API, virtual machine, medical record, exam result, or admin console.

3.2 Request

A request is an attempted action, such as opening a file, reading a database, submitting a form, starting a service, or changing a password.

3.3 Subject

A subject is the active thing making the request. It may be a human user, a process, a service, a script, or a device.

3.4 Authentication

Authentication is the process of verifying a claimed identity. Example: the system verifies that the person claiming to be student1 can present the correct authenticator.

3.5 Authorization

Authorization decides whether an authenticated subject may perform a requested action. Example: even after login, student1 may not be authorized to change another student’s grade.

3.6 Accountability

Accountability means actions can be traced to the responsible identity. Shared accounts weaken accountability because logs show only the shared account, not the actual person.

4. Core Concepts

4.1 Identity is a representation, not the person

A person is a human being. A digital identity is a record about a subject. The record may contain a username, email address, unique number, role, department, manager, group memberships, credentials, and status.

Example: Fatima the student is a person. fatima.s23 is an account. uid=1042 is a Linux identifier. fatima@example.edu is an email address. These should be linked carefully, but they are not the same thing.

4.3 Username versus identifier

A username is a human-friendly label. An identifier is a value the system uses to distinguish identities reliably. Human-friendly labels may change. Identifiers should remain stable.

Linux example: the username may be student, but the user ID, or UID, may be 1000. Files are owned internally by numeric IDs. Tools map those numbers back to names for humans.

Windows example: Windows uses Security Identifiers, or SIDs, to identify accounts and groups. A display name can change while the SID remains the security identity.

4.4 Local accounts

A local account exists on one device. It is managed by that device. A local Linux account on one VM does not automatically exist on another VM. A local Windows account on one laptop does not automatically grant access to another laptop.

Local accounts are useful for single machines and recovery scenarios. They become difficult to manage at enterprise scale because every device has separate account records.

4.5 Directory accounts

A directory service centrally stores identity information. Examples include Active Directory, LDAP directories, Microsoft Entra ID, Google Workspace identity, Okta, and university identity providers.

Centralized identity helps organizations onboard users, remove access, enforce policy, and audit activity across many systems. It also creates high-value infrastructure: if the identity provider is compromised, many connected systems may be affected.

4.6 Users

A user account usually represents a human. It should be tied to one person, not a group of people. This supports accountability and makes it possible to revoke access when that person leaves or changes role.

Bad pattern: a shared account called labadmin used by ten people. Logs cannot show which person acted. Password rotation becomes hard. Former users may still know the password.

Better pattern: each administrator has a named account, and privileged activity requires elevation, approval, or stronger authentication.

4.7 Groups

A group is a named collection of identities. Groups make access control manageable. Instead of adding 300 students individually to a folder, the system grants access to cybersecurity-ms-2026, and students become members of that group.

Intuition: a group is like a class roster. The classroom door rule can say “students enrolled in CS-701 may enter.” The door does not need a separate rule for every student.

Groups can be nested in some systems: one group can contain another group. Nesting is powerful but can create hidden access paths if not documented.

4.8 Roles

A role describes a job function or responsibility, such as student, instructor, finance officer, database administrator, or incident responder. A role may map to one or more groups or permissions.

Role-based access control assigns access based on roles rather than individual preferences. This reduces random permission sprawl.

4.9 Service accounts

A service account is an account used by software, not a person. Examples include a web application connecting to a database, a backup tool reading files, or a monitoring agent collecting logs.

Service accounts are security-sensitive because they often run continuously and may hold credentials. They should have limited permissions, strong secrets management, owner documentation, monitoring, and rotation plans.

4.10 Machine accounts and workload identities

A machine account represents a computer or device. A workload identity represents software running in a cloud, container, serverless function, or CI/CD pipeline.

Modern cloud security increasingly treats services, containers, and automation pipelines as identities. This is important because machines make many more requests than humans.

4.11 Privileged accounts

A privileged account has elevated power. Examples include root, local Administrator, domain administrator, cloud owner, database administrator, and backup administrator.

Privileged accounts need stronger protection because compromise has higher impact. Common controls include MFA, just-in-time access, separate admin accounts, session logging, approval workflows, and frequent review.

4.12 Identity lifecycle: joiner, mover, leaver

Identity has a lifecycle:

  1. Joiner: a new person or service is created.
  2. Mover: an identity changes department, class, project, or job function.
  3. Leaver: an identity no longer needs access and must be disabled or removed.

Many failures happen during mover and leaver events. Access is added when someone joins a project, but old access is not removed when the person moves away.

4.13 Orphaned and stale accounts

An orphaned account no longer has a valid owner. A stale account has not been used recently or has outdated purpose. Both are risky because attackers can use forgotten accounts without immediately disrupting normal work.

4.14 Privilege creep

Privilege creep means an identity accumulates more access over time than it still needs. A student assistant may first receive lab access, then database access, then temporary admin access. If old access is never removed, the account becomes overpowered.

4.15 Separation of duties

Separation of duties means no single identity should control every step of a sensitive process. For example, the person who creates a vendor should not also approve payment to that vendor without review.

4.16 Linux identity files

Linux commonly stores local identity information in text files:

  • /etc/passwd: local account records, readable by users so names can be mapped to IDs.
  • /etc/shadow: password hash and aging information, restricted because it is sensitive.
  • /etc/group: local group records.

Modern Linux systems may also use external identity sources, such as LDAP, Kerberos, Active Directory integration, or cloud-managed identity.

4.17 Linux UID and GID

A UID is a numeric user identifier. A GID is a numeric group identifier. The system often enforces access using these numbers, while humans see names.

Example: if a file is owned by UID 1000 and GID 1000, tools may display student student if those names map to those IDs.

4.18 Windows security principals and SIDs

In Windows, users, groups, computers, and services that can be assigned permissions are security principals. Each is associated with a Security Identifier, or SID. Access Control Lists reference these identities when deciding who may access a resource.

4.19 Naming conventions

Good account naming improves operations and investigation. A naming convention may show account type, department, environment, and owner. For example:

  • a.khan for a human account.
  • svc_backup_prod for a production backup service account.
  • adm.a.khan for a separate admin account.

Naming does not replace security controls. It helps humans manage and audit identity more accurately.

4.20 Identity proofing, authentication, and authorization

These terms are often confused:

TermQuestion answeredExample
Identity proofingWho is this person in the real world?Verify a student using enrollment records.
AuthenticationIs the claimant controlling the account?Password + MFA verifies login.
AuthorizationWhat may this identity do?Only instructors can submit final grades.
AuditingWhat happened and which identity did it?Log shows adm.a.khan changed access.

7. Mathematical Foundations

7.1 Why group management reduces complexity

Suppose an organization has U users and R resources. If every user-resource permission is managed individually, the maximum number of relationships is:

individual_assignments = U × R

If there are 500 users and 80 resources:

500 × 80 = 40,000 possible direct assignments

If users are assigned to 10 groups and resources are assigned to groups, administrators may manage closer to:

user_to_group + group_to_resource
= 500 + (10 × 80)
= 1,300 relationships

This is not a perfect formula because real environments have exceptions, nested groups, and temporary access. But it explains why groups exist: they reduce permission-management complexity.

7.2 Risk score for identity review priority

A simple classroom risk score can prioritize account reviews:

Identity Risk Score = Privilege × Exposure × Staleness × Sensitivity

Use a 1–5 scale for each factor:

  • Privilege: 1 ordinary read-only, 5 full admin.
  • Exposure: 1 internal only, 5 internet-facing login.
  • Staleness: 1 active recently, 5 unused for months.
  • Sensitivity: 1 public data, 5 regulated or critical data.

Worked example:

old_admin = 5 × 4 × 5 × 5 = 500
current_student = 1 × 2 × 1 × 2 = 4

The old admin account should be reviewed first. This score is a teaching model, not a universal standard.

7.3 Measuring excessive access

Define:

Excess Access Ratio = unnecessary_permissions / total_permissions

If an account has 20 permissions and 8 are unnecessary:

8 / 20 = 0.40 = 40%

The goal is not zero access. The goal is enough access for legitimate work and no unnecessary access.

8. Practical Examples

8.1 Linux examples: observing your own identity

# Show your effective user and group IDs
id

# Show the username you are logged in as
whoami

# Show the groups for your account
groups

# Safely query your own passwd entry through the system database
getent passwd "$USER"

# Safely query groups that include your username
getent group | grep "$(whoami)"

Explanation: these commands observe identity information on your own lab system. They do not create, delete, or modify accounts.

8.2 Linux examples: reading test identity records

mkdir -p ~/identity-lab
cd ~/identity-lab

cat > mock-passwd.txt <<'EOF'
root:x:0:0:root:/root:/bin/bash
student:x:1000:1000:Student User:/home/student:/bin/bash
svc_backup:x:1500:1500:Backup Service:/nonexistent:/usr/sbin/nologin
EOF

# Print usernames and UIDs from the mock file only
cut -d: -f1,3 mock-passwd.txt

Why use mock files? beginners can learn the format without modifying real system identity files.

8.3 Windows examples: observing your identity

# Show the current user
whoami

# Show user SID and group memberships
whoami /user
whoami /groups

# List local users if your Windows edition and permissions allow it
Get-LocalUser | Select-Object Name, Enabled, LastLogon

# List local groups
Get-LocalGroup | Select-Object Name

Safety note: observe only your own lab machine. Do not enumerate organizational directories without authorization.

8.4 Python example: detect stale fictional accounts

from datetime import date

accounts = [
    {"name": "aisha", "role": "student", "last_login": date(2026, 6, 20), "privilege": 1},
    {"name": "svc_backup", "role": "service", "last_login": date(2026, 1, 5), "privilege": 4},
    {"name": "old_admin", "role": "admin", "last_login": date(2025, 9, 1), "privilege": 5},
]

today = date(2026, 6, 28)

for account in accounts:
    days_unused = (today - account["last_login"]).days
    if days_unused > 90:
        print(f"Review {account['name']}: unused for {days_unused} days")

This code uses fictional records. It teaches how account review automation works without touching real accounts.

8.5 Network example: identity in remote login

When a user connects to a remote service, the network carries a request. The service still needs identity decisions:

  1. Who is claiming to connect?
  2. How is that claim authenticated?
  3. Which groups or roles does the identity have?
  4. What is the identity authorized to do?
  5. What should be logged?

8.6 Web example: account identity in a portal

A university portal may store:

{
  "user_id": "u-1042",
  "username": "fatima.s26",
  "email": "fatima@example.edu",
  "role": "student",
  "groups": ["ms-cyber-2026", "lab-users"],
  "status": "active"
}

The application should use a stable internal user_id for security decisions, not just a display name.

8.7 Cloud example: human and non-human identities

A cloud environment may include human users, admin roles, virtual-machine identities, serverless-function identities, CI/CD pipeline identities, and storage-access identities. The same principle applies: every identity needs a clear owner, purpose, least privilege, logging, and review.

9. Code Examples

9.1 Secure identity lookup using a stable ID

Problem: using names as security identifiers can break when names change or collide. Use stable IDs internally.

# Fictional user records for a classroom example.
# In a real system, these would come from a trusted database or identity provider.
users = {
    "u-1001": {"username": "aisha", "role": "student", "active": True},
    "u-1002": {"username": "bilal", "role": "instructor", "active": True},
    "u-1003": {"username": "old.temp", "role": "assistant", "active": False},
}

# Permissions are assigned to roles, not directly to display names.
role_permissions = {
    "student": {"read_lessons", "submit_lab"},
    "instructor": {"read_lessons", "submit_lab", "grade_lab"},
    "assistant": {"read_lessons"},
}

def is_allowed(user_id: str, action: str) -> bool:
    """Return True only if an active user has permission for the action."""
    user = users.get(user_id)

    # If the identifier does not exist, deny by default.
    if user is None:
        return False

    # Disabled accounts must not be authorized.
    if not user["active"]:
        return False

    # Look up permissions through the user's role.
    permissions = role_permissions.get(user["role"], set())

    # Allow only if the requested action is present.
    return action in permissions

print(is_allowed("u-1001", "grade_lab"))  # False: student cannot grade.
print(is_allowed("u-1002", "grade_lab"))  # True: instructor can grade.
print(is_allowed("u-1003", "read_lessons"))  # False: account is disabled.

9.2 Line-by-line explanation

  • users stores fictional identity records using stable IDs such as u-1001.
  • username is present for humans, but authorization is not based on it.
  • active blocks disabled accounts.
  • role_permissions maps roles to allowed actions.
  • is_allowed denies unknown users, disabled users, and unauthorized actions.
  • The function uses deny by default, a core secure-design principle.

9.4 Safer audit logging example

from datetime import datetime, timezone

# Store only what is needed for the audit event.
# Do not log passwords, secrets, or unnecessary private data.
def log_access_decision(user_id: str, action: str, allowed: bool) -> None:
    timestamp = datetime.now(timezone.utc).isoformat()
    result = "ALLOW" if allowed else "DENY"
    print(f"{timestamp} user_id={user_id} action={action} result={result}")

allowed = is_allowed("u-1001", "grade_lab")
log_access_decision("u-1001", "grade_lab", allowed)

10. Security Perspective

10.1 Common attacks

  • Credential theft: stealing passwords, tokens, keys, cookies, or certificates.
  • Password spraying: trying a few common passwords across many accounts.
  • Credential stuffing: trying leaked username/password pairs from other services.
  • Phishing: tricking users into giving credentials or approving login prompts.
  • Privilege escalation: turning low-level access into higher-level access.
  • Group abuse: adding an account to a powerful group or exploiting forgotten membership.
  • Service-account abuse: using long-lived credentials embedded in scripts or applications.
  • Account takeover: gaining control of a legitimate account.
  • Insider misuse: a legitimate user misuses granted access.

10.2 Common vulnerabilities

  • Shared accounts.
  • Weak or reused passwords.
  • No multi-factor authentication for important accounts.
  • Excessive group membership.
  • Old accounts not disabled.
  • Service accounts without owners.
  • Secrets stored in files or repositories.
  • Privilege creep.
  • Poor identity logging.
  • Overly broad default roles.

10.3 Defensive strategies

  • Use unique named accounts for humans.
  • Use least privilege and need-to-know access.
  • Use separate standard and administrative accounts.
  • Use MFA for sensitive and remote access.
  • Review group membership regularly.
  • Disable accounts promptly during leaver events.
  • Rotate and protect service-account secrets.
  • Prefer managed identities or short-lived credentials where possible.
  • Log authentication, authorization, group changes, and privileged actions.
  • Require approval for high-risk privilege changes.

10.4 Detection methods

  • New account created outside normal process.
  • Account added to privileged group.
  • Disabled account used or re-enabled unexpectedly.
  • Service account logging in interactively.
  • Many failed logins across many accounts.
  • Successful login from unusual geography, device, or time.
  • Old account suddenly becomes active.
  • Privilege use without a change ticket or approval.

10.5 Mitigation and best practices

Identity defense works best as a lifecycle, not a one-time setup. Create identities carefully, grant minimum required access, monitor use, review regularly, and remove access quickly when it is no longer needed.

12. Research Perspective

12.1 Current research directions

  • Usable authentication: making secure login methods easier and less error-prone.
  • Passwordless identity: passkeys, hardware-backed credentials, and phishing-resistant authentication.
  • Federated identity risk: how trust relationships across organizations can fail.
  • Cloud identity analysis: detecting overprivileged roles and dangerous permission paths.
  • Identity graph security: modeling paths from one identity to another privilege level.
  • Continuous authentication: using context and behavior after login without harming privacy.
  • Decentralized identity: exploring user-controlled credentials and verifiable claims.
  • Machine identity: securing certificates, keys, service accounts, workloads, and automation agents.

12.2 Open problems

  • How can organizations minimize access without blocking legitimate work?
  • How can service-account ownership be kept accurate over many years?
  • How can identity systems be both privacy-preserving and auditable?
  • How can cloud IAM complexity be explained to humans accurately?
  • How can universities secure highly open environments with many temporary users?

12.3 Notable research topics to explore

Useful research areas include access-control graph analysis, role mining, least-privilege recommendation, OAuth and OpenID Connect security, SAML federation failures, passkey usability, phishing-resistant authentication, and detection of identity-based lateral movement.

13. Case Studies

13.1 Colonial Pipeline ransomware

What happened: public reporting and government-backed IAM guidance describe compromised credentials as a major factor in the 2021 Colonial Pipeline incident.

Root cause theme: a valid identity was abused. This shows that attackers do not always need to break cryptography or exploit a memory bug; sometimes an account is enough.

Impact: operational disruption and broad public consequences.

Lessons learned: protect remote access, use MFA, monitor identity activity, and remove or secure unused accounts.

13.2 SolarWinds-related credential and identity concerns

What happened: the SolarWinds campaign is best known as a supply-chain compromise, but incident guidance also emphasized credential review, identity infrastructure, and trust relationships after compromise.

Root cause theme: once an attacker is inside, identity systems and credentials become major paths for persistence and lateral movement.

Lessons learned: assume important credentials may be exposed during deep compromise, review privileged credentials, and understand federation trust.

13.3 Fictional university stale-account incident

What happened: a former teaching assistant’s account remained active after graduation. The account still belonged to a lab-admin group. Months later, the account was used to download private lab submissions.

Root cause: failed leaver process, privilege creep, weak access review, and insufficient alerts for old account activity.

Impact: student privacy breach, grade-integrity questions, incident-response workload, and loss of trust.

Lessons learned: disable accounts during leaver events, transfer ownership, review groups, and alert on stale privileged-account activity.

Identity means password.A password is only one type of authenticator. Identity includes accounts, attributes, identifiers, groups, and lifecycle.
If login succeeds, the user should be allowed.Authentication only proves control of an account. Authorization still decides what the account may do.
Groups are only for convenience.Groups are a core security control for scalable, reviewable access.
Service accounts are less risky because no human uses them.Service accounts often hold powerful long-lived credentials and run continuously.
Admin access is harmless if the user is trusted.Trusted users can be phished, infected, coerced, mistaken, or compromised.
Disabling a person’s email account removes all access.Access may remain in local systems, cloud roles, API keys, service accounts, or external tools.
Names are good enough for access control.Names can change or collide. Stable identifiers are safer.
Identity security is only an IT help-desk issue.Identity is central to incident response, cloud security, compliance, software design, and research.

15. Summary

Identity is how systems represent people, services, machines, and workloads. Users, groups, roles, identifiers, attributes, and credentials allow systems to decide who is making a request and what that subject may do. Groups make access manageable, but they also create risk when membership is excessive or undocumented. Service accounts and privileged accounts require special care because they can cause high-impact compromise. Strong identity security depends on lifecycle management, least privilege, account review, logging, and clear ownership.

16. Glossary

Account
A system record representing a user, service, device, or workload.
Attribute
A property of an identity, such as department, role, status, or expiry date.
Credential
Evidence used to authenticate an identity, such as a password, passkey, certificate, or token.
Directory Service
A centralized system that stores and manages identity information.
Entitlement
A permission or right granted to an identity.
GID
Group Identifier; a numeric Linux identifier for a group.
Group
A collection of users, computers, services, or other groups used to manage access.
Identifier
A stable value used to distinguish one identity from another.
Identity
A digital representation of a person, service, machine, or workload.
Joiner-Mover-Leaver
The identity lifecycle covering onboarding, role changes, and departure.
Local Account
An account stored and managed on one device.
Machine Account
An identity representing a computer or device.
Orphaned Account
An account without a valid current owner.
Privileged Account
An account with elevated administrative or sensitive access.
Security Principal
An entity, such as a user or group, that can be authenticated and assigned permissions.
Service Account
An account used by software rather than by a human.
SID
Security Identifier; a Windows identifier for a security principal.
Stale Account
An account that has not been used recently or has outdated purpose.
UID
User Identifier; a numeric Linux identifier for a user.
Workload Identity
An identity assigned to software running in cloud, container, serverless, or automation environments.

Part D: Analyze fictional identity risk with Python

import csv

# Simple classroom scoring model.
# Higher score means the account should be reviewed sooner.
def score(row):
    privilege = int(row["privilege"])
    exposure = int(row["exposure"])
    staleness = 5 if int(row["last_login_days"]) > 180 else 3 if int(row["last_login_days"]) > 90 else 1
    return privilege * exposure * staleness

with open("fictional-accounts.csv", newline="") as f:
    accounts = list(csv.DictReader(f))

for account in sorted(accounts, key=score, reverse=True):
    print(f"{account['account']}: score={score(account)} owner={account['owner']} status={account['status']}")
