Week 2 · Operating Systems, Identity, and Access Control

Lesson 2.1: Operating Systems from First Principles

An operating system is the control layer between people, programs, and hardware. Cybersecurity depends on this layer because it decides who can run programs, read files, use memory, access devices, start services, write logs, and change the machine.

Week 2 Context

Overview: Week 2 explains how computers enforce identity and access control. Lesson 2.1 builds the base: what an operating system is, what a kernel does, how processes run, how files and devices are represented, how users and groups are modeled, how permissions protect resources, and how logs support accountability.

Learning objectives: by the end, you should be able to explain the OS security boundary, distinguish kernel mode from user mode, inspect users, processes, files, services, and logs in a safe lab, and connect OS concepts to cybersecurity controls.

Required background: no operating-systems background is assumed. Helpful Week 0 concepts: Linux terminal, networking basics, Python files, and Git documentation.

Estimated reading time: 180–260 minutes. Difficulty: begins at zero and ends with graduate-level design and research questions.

1. Introduction

Plain-English idea: an operating system, often shortened to OS, is the main software that manages a computer. It helps programs use hardware without letting every program directly control the machine.

Analogy: imagine a large university building. Students, teachers, administrators, and guests need rooms, electricity, printers, doors, and records. Without building management, everyone would fight over space and resources. The operating system is like the building management office, security desk, key system, maintenance team, and activity log combined.

Why the topic exists: hardware is complex and dangerous to share directly. If every program could write anywhere in memory, read any file, or talk directly to every device, one bug or malicious program could destroy the system. The OS provides controlled access.

Why it matters for cybersecurity: many attacks try to make the OS enforce the wrong decision: run unwanted code, give a user more privileges, hide a process, steal a file, disable logging, or abuse a service. Many defenses work by strengthening the OS: patching, permissions, least privilege, service isolation, logging, backups, and hardening.

Where it is used: laptops, phones, servers, routers, cloud virtual machines, containers, hospital systems, bank systems, industrial control systems, research clusters, and security appliances.

Who uses it: ordinary users, system administrators, software developers, cloud engineers, security analysts, malware analysts, forensic investigators, penetration testers working under written authorization, and researchers.

Safety boundary

This lesson uses only your own lab machine or fictional examples. Do not inspect, modify, scan, or attempt to access accounts, files, logs, services, or systems that you do not own or have written authorization to test.

2. Learning Outcomes

  • Define operating system, kernel, user space, kernel space, process, thread, memory, filesystem, device, driver, service, user, group, permission, privilege, and log.
  • Explain why an OS exists and why security depends on it.
  • Describe the difference between hardware, firmware, kernel, system services, applications, and users.
  • Explain how a program becomes a process and why process isolation matters.
  • Inspect safe lab information about processes, files, users, groups, services, and logs.
  • Explain the difference between authentication, authorization, access control, privilege, and accountability.
  • Compare beginner-level Linux and Windows security models.
  • Identify common OS weaknesses: unpatched systems, excessive privileges, weak permissions, unnecessary services, poor logging, exposed secrets, and unsafe configuration.
  • Apply basic defensive practices: least privilege, patching, secure defaults, logging, service minimization, backups, and configuration review.
  • Discuss research directions in operating-system security, isolation, memory safety, capabilities, sandboxing, and formal verification.

3. Prerequisites

3.1 Hardware

Hardware means the physical parts of a computer: central processing unit, memory, disk, keyboard, screen, network card, and other devices. Hardware performs work, but it needs instructions.

3.2 Software

Software means instructions that hardware executes. A calculator app, browser, text editor, database, and operating system are all software.

3.3 Program versus process

A program is code stored somewhere, usually on disk. A process is a running instance of a program. The same program can run many times as many separate processes.

3.4 File

A file is a named unit of data. A file may contain text, an image, a program, a log, a configuration, or many other forms of data.

3.5 Identity

Identity is how a system represents a person, service, or machine. On a laptop, a username such as samad may represent a person. On a server, a service account such as www-data may represent a web service.

3.6 Safe lab assumption

All commands in this lesson are for your own local lab computer or virtual machine. Use test files only. Do not run commands against real organizational systems unless a qualified authority has explicitly authorized that exact work.

4. Core Concepts

4.1 What an operating system does

An operating system manages four major things:

  1. Compute: which programs get CPU time.
  2. Memory: which data each program can read and write.
  3. Storage: how files are named, saved, read, and protected.
  4. Devices and communication: how programs use keyboards, screens, disks, cameras, printers, and networks.

The OS also provides security mediation. Mediation means the OS stands between a requester and a resource, then decides whether the request is allowed.

4.2 Kernel

The kernel is the most privileged core of the OS. It controls memory, processes, devices, filesystems, and system calls. If the kernel is compromised, the whole machine may be compromised.

Intuition: applications ask; the kernel decides. A text editor cannot directly write random bytes to the disk controller. It asks the OS to save a file. The OS checks permissions and coordinates the disk operation.

4.3 User mode and kernel mode

Modern processors support privilege levels. In simple terms:

  • User mode: ordinary applications run here. They have limited power.
  • Kernel mode: the kernel runs here. It has broad power over the machine.

This separation limits damage. If a browser tab crashes in user mode, the whole OS should not crash. If code runs in kernel mode, mistakes are much more dangerous.

4.4 System calls

A system call is a controlled request from a program to the kernel. Examples include opening a file, reading data, creating a process, writing to the network, or changing permissions.

Analogy: a student cannot directly open the university vault. The student submits a request at the authorized office. The office checks identity and permission before acting.

4.5 Processes and threads

A process is a running program with its own memory, open files, identity, environment, and execution state. A thread is a path of execution inside a process. One process may have one thread or many threads.

Security idea: process isolation helps stop one application from reading another application’s private memory. Isolation is not perfect, but it is fundamental.

4.6 Memory

Memory, often RAM, stores data while programs run. The OS gives each process a view of memory. Programs normally cannot read another process’s memory without special privilege.

Memory bugs matter because an unsafe program may accidentally or maliciously read, overwrite, or execute data in ways the developer did not intend. Later weeks study memory safety in more depth.

4.7 Filesystems

A filesystem organizes stored data into files and directories. A directory is like a folder. A path names where a file is located, such as /home/student/report.txt on Linux or C:\Users\Student\report.txt on Windows.

Filesystems also store metadata, such as owner, group, permissions, size, timestamps, and sometimes extended attributes.

4.8 Users, groups, and service accounts

A user is an identity that can own files and run processes. A group collects users so permissions can be assigned to many users at once. A service account is an identity used by software rather than a person.

Security intuition: do not run every program as the most powerful account. If a web server is compromised, it should not automatically gain full administrator power.

4.9 Permissions

Permissions define what an identity can do to a resource. Basic Linux file permissions are read, write, and execute. Windows commonly uses Access Control Lists, or ACLs, that store more detailed allow, deny, and audit rules.

Permissions are security decisions made durable. A permission setting can silently protect data for years or silently expose data for years.

4.10 Privilege

Privilege means power granted to an identity or process. Administrative privileges can install software, change system settings, manage accounts, read protected data, and stop security tools.

Least privilege means giving only the access needed for the task, no more.

4.11 Services and daemons

A service is a background program that provides functionality. Linux often calls background services daemons. Examples include web servers, database servers, SSH servers, print services, and update services.

Services matter because they often listen for requests, run continuously, and hold privileges. Unnecessary services increase attack surface.

4.12 Logs

A log is a record of events. Logs may record logins, service starts, configuration changes, errors, network connections, and security alerts.

Logs support accountability and incident response. Without logs, defenders may not know what happened.

4.13 Linux and Windows at a beginner level

Linux and Windows are both operating systems, but they expose different administrative models and command styles. Cybersecurity students should understand both at a conceptual level.

4.14 Boot process, simplified

Booting is the process of starting a computer. A simplified path is:

  1. Firmware starts hardware checks.
  2. Bootloader loads the operating-system kernel.
  3. Kernel initializes memory, devices, and core services.
  4. System services start.
  5. Users log in and applications run.

Boot security matters because attackers who control early boot stages may control everything after them.

4.15 Isolation and security boundaries

A security boundary is a separation that should prevent one actor from accessing another actor’s resources without permission. Examples include user accounts, process boundaries, virtual machines, containers, browser sandboxes, and network segments.

Good security asks: What is the boundary? Who controls it? What happens if it fails?

7. Mathematical Foundations

Operating systems are not only code. They also involve measurable security and reliability decisions.

7.1 Permission combinations

Linux basic file permissions use three actions: read, write, execute. Each action can be either off or on. That gives:

Number of combinations = 2^3 = 8
BinaryDecimalMeaning
0000no permissions
0011execute only
0102write only
0113write + execute
1004read only
1015read + execute
1106read + write
1117read + write + execute

Linux stores separate permission sets for owner, group, and other. A mode such as 750 means:

7 = owner: read + write + execute
5 = group: read + execute
0 = other: no permissions

7.2 Attack surface counting

A simple exposure count can help beginners reason about service minimization:

Exposed services = count of services reachable by untrusted users

If a server exposes SSH, HTTP, HTTPS, database, file sharing, and admin panel to an untrusted network, then:

Exposed services = 6

If the database and admin panel are moved behind a private network and file sharing is disabled:

New exposed services = 3
Exposure reduction = (6 - 3) / 6 = 0.5 = 50%

This is not a full risk formula, but it teaches why unnecessary services matter.

7.3 Availability and downtime

Availability can be estimated as:

Availability = uptime / total time

If a lab server is expected to be available for 100 hours and is down for 2 hours:

Uptime = 98 hours
Availability = 98 / 100 = 0.98 = 98%

7.4 Log volume estimate

Defenders must plan storage for logs:

Daily log volume = events per day × average event size

If a machine produces 40,000 log events per day and each event averages 600 bytes:

40,000 × 600 = 24,000,000 bytes ≈ 24 MB per day

For 90 days:

24 MB × 90 = 2160 MB ≈ 2.16 GB

Linux examples

Run these only on your own lab Linux VM.

# Show operating system information
uname -a

Python example: safe local OS inventory

This script collects basic information from the local machine only. It does not scan networks or access other systems.

# os_inventory.py
# Purpose: collect safe local operating-system information for a lab report.

import os                 # Provides access to OS functions such as getcwd().
import platform           # Provides information about the operating system.
import getpass            # Provides the current username in a portable way.
from pathlib import Path  # Provides safer object-oriented file paths.

# Store output inside the student's home directory.
lab_dir = Path.home() / "os-security-lab"
lab_dir.mkdir(exist_ok=True)

# Build a small report as plain text.
report = []
report.append(f"User: {getpass.getuser()}")
report.append(f"Operating system: {platform.system()}")
report.append(f"OS release: {platform.release()}")
report.append(f"Machine architecture: {platform.machine()}")
report.append(f"Current working directory: {os.getcwd()}")

# Save the report locally.
output_file = lab_dir / "inventory.txt"
output_file.write_text("\n".join(report), encoding="utf-8")

print(f"Wrote local inventory to: {output_file}")

Line-by-line security notes: the script writes only to the user's home directory, avoids collecting secrets, avoids network activity, avoids privileged operations, and uses Path.home() instead of hardcoded system paths.

Network example

An OS owns the network stack. Even a browser depends on OS-managed sockets, DNS resolution, routing, and firewall decisions.

# Linux: show listening network sockets on your own lab machine
ss -tuln

# Windows PowerShell: show listening TCP sockets on your own lab machine
Get-NetTCPConnection -State Listen | Select-Object -First 20
Web example

A web server is a process, often running as a service account, listening on a network port, reading files, and writing logs. OS permissions decide what files that process can access.

Browser request → OS network stack → web server process → filesystem read → response → log entry
Cloud example

A cloud virtual machine still has an OS. Cloud providers manage the underlying physical infrastructure, but customers usually remain responsible for OS patching, user accounts, firewall rules, service configuration, logs, and application security depending on the service model.

9. Code Examples

9.1 Secure file permission check in Python

This example checks whether a sensitive local lab file is readable by group or other users on Unix-like systems. It is defensive and local-only.

# check_file_permissions.py
# Purpose: warn when a local lab file is too permissive.

import os
import stat
from pathlib import Path

# Use a file inside the user's home directory, not a system file.
target = Path.home() / "os-security-lab" / "report.txt"

# Stop with a clear message if the file does not exist.
if not target.exists():
    raise SystemExit(f"File not found: {target}")

# Read the file's metadata. Metadata includes permission mode bits.
mode = target.stat().st_mode

# Check whether group or other users can read the file.
group_can_read = bool(mode & stat.S_IRGRP)
other_can_read = bool(mode & stat.S_IROTH)

if group_can_read or other_can_read:
    print("Warning: file may be readable by users beyond the owner.")
    print("Suggested Linux command: chmod 600", target)
else:
    print("Good: file is not readable by group or other users.")

9.2 Explanation of every important line

  • import os: imports operating-system functions. It is included here for common OS work, though this exact script mainly uses pathlib and stat.
  • import stat: imports constants for reading Unix permission bits.
  • from pathlib import Path: imports a safer way to build paths across systems.
  • target = Path.home() / ...: builds a path inside the user's home directory, reducing risk of modifying sensitive system files.
  • target.exists(): checks whether the file exists before reading metadata.
  • target.stat().st_mode: reads metadata, including permissions.
  • mode & stat.S_IRGRP: uses a bitwise operation to test the group-read permission.
  • mode & stat.S_IROTH: tests the other-read permission.
  • print(...): reports a defensive recommendation without changing the system automatically.

9.3 Common coding mistakes

  • Hardcoding sensitive paths: beginner scripts sometimes target /etc or system directories unnecessarily.
  • Changing permissions automatically: defensive tools should often report first and change only with explicit approval.
  • Collecting secrets: inventory scripts should avoid passwords, tokens, private keys, and private user data.
  • Assuming Linux and Windows are identical: permission models differ.
  • Running as administrator by default: elevated scripts can cause more damage if wrong.

9.4 Why secure coding matters

Security scripts often run with more access than ordinary applications. A poorly written script can delete evidence, expose private data, change permissions incorrectly, or create new vulnerabilities. Defensive automation must be careful, minimal, logged, reversible, and understandable.

10. Security Perspective

10.1 Common attacks

  • Privilege escalation: gaining more power than intended.
  • Credential theft: stealing passwords, tokens, hashes, or private keys from the OS or files.
  • Persistence: configuring a program to restart after reboot or login without authorization.
  • Defense evasion: disabling logs, hiding processes, changing timestamps, or tampering with security tools.
  • Data exfiltration: copying protected files to an unauthorized location.
  • Denial of service: exhausting CPU, memory, disk, or process limits.

10.2 Common vulnerabilities

  • Unpatched kernel or service vulnerabilities.
  • Weak file permissions.
  • Users running daily work as administrator.
  • Shared accounts with no accountability.
  • Unnecessary exposed services.
  • Secrets stored in plain text.
  • Missing or short-retention logs.
  • Insecure default configurations.

10.3 Defensive strategies

  • Patch the OS and applications.
  • Use least privilege.
  • Separate administrator accounts from daily-use accounts.
  • Disable unnecessary services.
  • Use strong authentication and multi-factor authentication where possible.
  • Protect logs and send important logs to a separate system.
  • Use disk encryption for portable devices.
  • Back up critical data and test restore procedures.
  • Use configuration baselines and hardening guides.

10.4 Detection methods

  • Monitor failed and successful logins.
  • Monitor new users, new groups, and privilege changes.
  • Monitor service creation, service changes, and startup items.
  • Monitor suspicious process behavior.
  • Monitor unexpected network listeners.
  • Monitor changes to sensitive files and directories.

10.5 Mitigation and best practices

A strong OS baseline combines preventive, detective, and recovery controls. Prevention reduces the chance of compromise. Detection reveals suspicious activity. Recovery limits damage when prevention fails.

Good OS security baseline = patching + least privilege + hardened services + protected logs + tested backups + documented configuration

12. Research Perspective

Operating-system security is a major research field because the OS is both powerful and difficult to secure completely.

12.1 Current research directions

  • Microkernels: reducing the amount of code running with maximum privilege.
  • Formal verification: mathematically proving properties of critical OS components.
  • Memory safety: reducing bugs caused by unsafe memory access.
  • Sandboxing: limiting what applications can do even if compromised.
  • Capability systems: granting explicit, narrow authority instead of broad ambient authority.
  • Container isolation: improving boundaries between workloads sharing a kernel.
  • Trusted execution environments: protecting selected computations even from some privileged software.
  • Endpoint telemetry: detecting suspicious behavior without excessive false positives or privacy invasion.

12.2 Open problems

  • How can ordinary users make correct security decisions without becoming system administrators?
  • How can operating systems minimize damage from compromised applications?
  • How can logging be useful for defenders while preserving privacy?
  • How can cloud and container systems provide strong isolation while remaining fast and usable?
  • How can high-assurance OS designs be adopted in ordinary commercial systems?

12.3 Notable paper and topic families

Relevant research areas include classic protection principles, access-control models, capability-based systems, microkernel design, formally verified kernels, sandboxing, information-flow control, memory-safe systems programming, and endpoint detection. Students should focus on the ideas, assumptions, threat models, and limitations rather than copying implementation details.

13. Case Studies

13.1 WannaCry and unpatched systems

What happened: WannaCry spread globally in 2017 and disrupted many organizations, including healthcare services. It exploited weaknesses in unpatched Windows systems and caused ransomware-style disruption.

Root cause: exposed vulnerable systems, delayed patching, and insufficient containment contributed to impact.

Impact: service disruption, operational delays, recovery cost, and public safety concerns.

Lessons learned: OS patching, network segmentation, backups, incident response readiness, and asset visibility matter.

13.2 Misconfigured cloud server permissions

What happened: many real breaches have involved exposed storage, overly broad service permissions, or compromised cloud instances.

Root cause: weak access control, excessive privileges, poor monitoring, and misunderstanding of shared responsibility.

Impact: data exposure, compliance problems, customer notification, and reputational damage.

Lessons learned: cloud systems still depend on identity, permissions, OS patching, logging, and least privilege.

13.3 Sudo vulnerability as privilege-risk example

What happened: past vulnerabilities in privilege-management tools have shown that small mistakes in highly privileged software can be serious.

Root cause: privileged code is high-value and difficult to get perfectly right.

Impact: local privilege escalation may allow a lower-privileged user or process to gain administrative power on affected systems.

Lessons learned: minimize privileged code, patch quickly, monitor privilege changes, and avoid unnecessary local accounts.

14. Common Misconceptions

  • Misconception: “The OS is just the desktop.” Correction: the desktop is only one interface. The kernel, services, filesystems, users, permissions, and logs are deeper OS components.
  • Misconception: “Administrator access is convenient, so I should always use it.” Correction: administrator access increases damage when mistakes or malware occur.
  • Misconception: “If a file is hidden, it is secure.” Correction: hidden is not the same as permission-protected or encrypted.
  • Misconception: “Linux has no malware.” Correction: Linux can be attacked; many servers run Linux, making it highly relevant to attackers and defenders.
  • Misconception: “Windows security is only antivirus.” Correction: Windows has identity, ACLs, UAC, logging, Defender, application control, event auditing, and enterprise management features.
  • Misconception: “Logs always tell the truth.” Correction: logs can be missing, incomplete, misconfigured, or tampered with.
  • Misconception: “Containers are the same as virtual machines.” Correction: containers often share the host kernel; virtual machines usually have stronger hardware-assisted isolation.
  • Misconception: “A patched OS is fully secure.” Correction: patching is necessary but not sufficient; configuration, identity, monitoring, and user behavior also matter.

15. Summary

An operating system manages hardware, programs, memory, files, devices, users, services, and logs. For cybersecurity, the OS is a central enforcement point. It authenticates identities, authorizes access, isolates processes, controls privileges, records activity, and exposes configuration choices that can either reduce or increase risk.

The key security lesson is simple: every process runs with some identity, every identity has some permissions, every permission creates possible access, and every access decision should be intentional, minimal, and auditable.

18. Practice Lab

Lab title

Local OS Inventory, Permissions, and Logging Lab

Prerequisites

  • Your own Linux VM from Week 0, or your own Windows lab machine.
  • No administrator actions are required for the basic lab.
  • Do not use real organizational machines or shared systems.

Objectives

  • Identify local OS version, user identity, processes, services, files, permissions, and logs.
  • Create safe test files and adjust permissions.
  • Write a short report connecting observations to security goals.

Part A: Linux lab

mkdir -p ~/os-security-lab
cd ~/os-security-lab

# 1. Record identity and OS information
whoami > identity.txt
id >> identity.txt
uname -a > os.txt

# 2. Record a small process sample
ps aux | head -15 > processes.txt

# 3. Create a test file and inspect permissions
echo "fictional lab secret" > secret.txt
ls -l secret.txt > permissions-before.txt

# 4. Restrict permissions to owner read/write only
chmod 600 secret.txt
ls -l secret.txt > permissions-after.txt

# 5. Record recent logs if journalctl is available
journalctl -n 20 --no-pager > recent-logs.txt 2>/dev/null || echo "journalctl unavailable" > recent-logs.txt

Part B: Windows lab

New-Item -ItemType Directory -Path "$HOME\os-security-lab" -Force
Set-Location "$HOME\os-security-lab"

# 1. Record identity and OS information
whoami | Out-File identity.txt
Get-ComputerInfo | Select-Object WindowsProductName, WindowsVersion, OsArchitecture | Out-File os.txt

# 2. Record a small process sample
Get-Process | Select-Object -First 15 | Out-File processes.txt

# 3. Create a test file and inspect permissions
"fictional lab secret" | Out-File secret.txt
Get-Acl .\secret.txt | Format-List | Out-File permissions.txt

# 4. Record recent system events if permitted
Get-EventLog -LogName System -Newest 10 | Out-File recent-events.txt

Part C: Analysis report

Create os-lab-report.md with these headings:

# Local OS Security Lab Report

## Scope

## Identity

## Processes

## Files and Permissions
What changed before and after permission hardening?

## Logs

## Security Lessons
List five defensive lessons from the lab.

## Uncertainty
List anything you could not confirm.

Expected outcomes

  • You can explain what your OS is doing at a beginner level.
  • You can inspect local identity, processes, files, permissions, and logs.
  • You can connect OS observations to confidentiality, integrity, availability, accountability, and least privilege.
