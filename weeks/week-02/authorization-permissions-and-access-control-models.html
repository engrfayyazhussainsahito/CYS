Lesson 2.4: Authorization, Permissions, and Access Control Models

Authentication proves who or what is making a request. Authorization decides whether that authenticated subject may perform a specific action on a specific resource under current conditions. This lesson starts from everyday permission ideas and builds toward operating-system permissions, ACLs, DAC, MAC, RBAC, ABAC, least privilege, policy evaluation, and broken access-control defense.

Difficulty levelBeginner → MS foundation
Required backgroundLessons 2.1–2.3 recommended

Week 2 Context

Overview: Week 2 explains how computers represent identities and enforce access. Lesson 2.3 focused on authentication. Lesson 2.4 focuses on authorization: the decision that turns identity into allowed or denied action.

Learning objectives: after this lesson you should be able to explain permissions, policy evaluation, least privilege, ACLs, RBAC, ABAC, DAC, MAC, access reviews, broken access control, and safe authorization testing in fictional systems.

Required background: no previous access-control background is required. You should understand users, groups, service accounts, authentication, files, and processes from Lessons 2.1–2.3.

Estimated reading time: 220–330 minutes. Difficulty: begins with everyday examples and ends with graduate-level access-control design and research questions.

1. Introduction

Plain-English idea: authorization asks, “Now that the system knows who or what is asking, what is this subject allowed to do?” A student may log in successfully but still be blocked from editing another student’s grade. A nurse may open patient charts for assigned patients but not payroll records. A backup service may read files but not delete user accounts.

Why the topic exists: shared systems need boundaries. Without authorization, every authenticated user could read, modify, delete, or administer everything. Authorization is the mechanism that turns broad identity into controlled access.

Why it matters: many severe breaches happen after a valid login because access is too broad, stale, hidden, inherited incorrectly, or checked in the wrong place. Strong authorization limits blast radius when accounts, applications, or tokens are compromised.

Where it is used: operating systems, file servers, databases, web applications, APIs, hospitals, banking platforms, cloud IAM, Kubernetes, universities, mobile apps, source-code repositories, HR systems, and security tools.

Who uses it: users experience it, developers implement it, system administrators configure it, architects design it, auditors review it, incident responders investigate it, and researchers analyze how it fails.

Analogy: a university ID card proves identity. It does not automatically open every classroom, server room, finance office, and exam database. Authorization is the rule system that decides which doors open for that card, at what time, and for what purpose.

Safety boundary

This lesson is defensive and educational. Do not try to bypass permissions, access another user’s data, test real APIs, alter production IAM, or probe real systems. Labs use only local files, fictional policy tables, and documentation IP/user examples.

2. Learning Outcomes

  • Define subject, object, action, permission, privilege, policy, authorization, access control, ACL, ACE, capability, role, attribute, decision, enforcement, owner, inheritance, and least privilege.
  • Distinguish authentication, authorization, accountability, and audit.
  • Explain how a basic authorization decision is made using subject, object, action, and context.
  • Read and interpret Linux permission strings and octal permission values.
  • Explain Windows ACLs at a conceptual level: security descriptors, DACLs, SACLs, ACEs, inheritance, and effective access.
  • Compare DAC, MAC, RBAC, ABAC, rule-based, relationship-based, and capability-based access-control ideas.
  • Explain why least privilege, separation of duties, need to know, and deny by default reduce risk.
  • Identify common authorization failures such as broken object-level authorization, horizontal privilege escalation, vertical privilege escalation, excessive permissions, and stale access.
  • Design a simple access matrix and risk-based access review for a fictional university service.
  • Write safe local Python code that evaluates fictional authorization rules without touching real systems.

3. Prerequisites

3.1 Identity

An identity is how a system represents a person, service, device, or workload. Authorization decisions usually start with identity attributes such as user ID, group, role, department, account status, or service name.

3.2 Authentication

Authentication is proof that the requester controls a valid authenticator. Authorization normally happens after authentication, but some systems also authorize anonymous users to public resources.

3.3 Resource

A resource, also called an object, is something protected: a file, folder, database row, API endpoint, server, queue, key, document, container, patient chart, payroll record, or cloud bucket.

3.4 Action

An action is what the subject wants to do: read, write, execute, delete, approve, share, administer, list, create, update, reset, or assume a role.

3.5 Policy

A policy is a rule or set of rules describing what is allowed or denied. Policy may be written as file permissions, access-control entries, IAM statements, application code, database grants, or organizational procedures.

4. Core Concepts

4.1 The shortest authorization model

Every authorization decision can be simplified into four questions:

  1. Who or what is asking? The subject.
  2. What are they trying to access? The object or resource.
  3. What do they want to do? The action or operation.
  4. Under what conditions? The context, such as time, location, device health, network, emergency status, or request purpose.

4.2 Subject

A subject is an active entity requesting access. Examples: human user, process, service account, workload identity, API client, container, device, or scheduled job.

Intuition: a subject is the “actor” in a security decision.

4.3 Object or resource

An object is the protected thing. In an operating system, an object might be a file or process. In a database, it might be a table or row. In a web API, it might be an invoice, profile, message, or payment method.

Intuition: an object is the “thing being acted on.”

4.4 Action or operation

An action is the operation requested. Common actions include read, write, execute, delete, list, approve, update, share, export, and administer. Good access control is action-specific. A user may be allowed to read a report but not edit or delete it.

4.5 Permission and privilege

A permission is a specific allowed action on a resource. A privilege is often a broader ability, such as “can install software,” “can manage users,” or “can bypass normal file checks.” In everyday usage the words overlap, but precise security design should define them clearly.

4.6 Authorization versus access control

Authorization is the decision: allow or deny. Access control is the broader mechanism that stores policy, evaluates requests, enforces decisions, and records evidence. In a small system they may look identical. In large systems they are often separated into policy administration, decision, enforcement, and audit components.

4.7 Policy decision point and policy enforcement point

A policy decision point, or PDP, evaluates policy and produces a decision. A policy enforcement point, or PEP, blocks or allows the actual request. If the PEP is missing, a correct policy decision may not matter because nothing enforces it.

Example: an API gateway may enforce a decision made by an authorization service. A file system may enforce decisions using kernel-level permission checks.

4.8 Deny by default

Deny by default means access is denied unless an explicit rule allows it. This is safer than allow-by-default because mistakes tend to fail closed instead of exposing resources.

4.9 Least privilege

Least privilege means a subject receives only the permissions needed to perform an approved task, for the required time, under required conditions. Least privilege limits damage if a subject is compromised or misused.

Bad example: every intern can edit all payroll files because it is easier to manage one group. Better example: interns can access only training data; HR analysts can read assigned HR records; payroll administrators can modify payroll data only through approved workflows.

4.10 Need to know

Need to know means access should depend on whether the user needs the information for an authorized purpose, not merely whether the user has a high rank. This is especially important in healthcare, defense, legal, research, and investigations.

4.11 Separation of duties

Separation of duties divides sensitive work so one person cannot complete a high-risk action alone. For example, one employee creates a payment request and another approves it. This reduces fraud and accidental damage.

4.12 Access-control matrix

An access-control matrix is a table where rows are subjects, columns are objects, and cells list allowed actions.

SubjectGrades filePayroll fileCourse notes
StudentRead own grade onlyDenyRead
InstructorRead/write assigned course gradesDenyRead/write
Payroll officerDenyRead/write payrollDeny

The matrix is conceptually simple, but large organizations have too many subjects and objects to manage a raw matrix manually. Access-control models help compress the matrix into manageable rules.

4.13 Access-control list

An access-control list, or ACL, stores permissions with the object. A file might say: Alice can read, instructors can write, everyone else denied. Each row is often called an access-control entry, or ACE.

Object-centered view: “For this file, who can do what?”

4.14 Capability list

A capability stores authority with the subject or token. A token might say: this process can write to backup storage and read log files. Capability systems are useful because possession of the capability authorizes action, but capabilities must be protected carefully.

Subject-centered view: “For this subject, what can it do?”

4.15 Linux permissions

Basic Linux file permissions use three subject classes: user, group, and other. Each class may have read, write, and execute bits.

-rw-r----- 1 alice analysts 2048 Jun 28 report.txt
│││││││││
│││││││└─ other execute: no
││││││└── other write: no
│││││└─── other read: no
││││└──── group execute: no
│││└───── group write: no
││└────── group read: yes
│└─────── owner write: yes
└──────── owner read: yes

For files, read means view contents, write means modify contents, and execute means run the file as a program or script. For directories, the meaning changes: read lists names, write changes entries, and execute allows traversal into the directory.

4.16 Octal permissions

Linux permissions are often written as octal numbers. Read is 4, write is 2, execute is 1. Add the values for each class.

ModeOwnerGroupOtherMeaning
600rw-------Owner can read/write; nobody else
640rw-r-----Owner read/write; group read
755rwxr-xr-xOwner full; others read/execute
777rwxrwxrwxEveryone full access; usually dangerous

4.17 Special Linux permission bits

Linux also has special bits: setuid, setgid, and sticky bit. They are powerful and should be understood before use.

  • setuid: executable may run with the file owner’s effective privileges.
  • setgid: executable may run with group privileges, or directories may cause new files to inherit group ownership.
  • sticky bit: often used on shared directories so users cannot delete other users’ files.

4.18 Windows access control

Windows uses securable objects, security descriptors, owners, ACLs, ACEs, inheritance, and auditing. A DACL controls allowed and denied access. A SACL controls audit logging for access attempts. Effective access can be complex because permissions may come from users, groups, inherited entries, explicit entries, deny entries, and privileges.

4.19 Discretionary access control

Discretionary access control, or DAC, lets object owners or authorized users decide who else can access an object. Traditional Unix file permissions and many Windows object permissions are discretionary in this sense.

Strength: flexible. Weakness: users may accidentally grant too much access.

4.20 Mandatory access control

Mandatory access control, or MAC, enforces centrally defined labels and rules that normal users cannot override. Military classification systems and SELinux-style enforcement are common examples.

Strength: strong central policy. Weakness: can be hard to configure and may interfere with usability if labels or rules are wrong.

4.21 Role-based access control

Role-based access control, or RBAC, assigns permissions to roles, then assigns users to roles. Instead of giving Alice 50 direct permissions, Alice becomes a “course instructor” and receives the role’s permissions.

Strength: easier administration and review. Weakness: role explosion can occur when too many narrow roles are created.

4.22 Attribute-based access control

Attribute-based access control, or ABAC, evaluates attributes of the subject, object, action, and environment. Example: allow a doctor to read a patient record if the doctor is assigned to that patient, the record is in the hospital system, the action is read, the account is active, and the request comes from an approved device.

Strength: expressive and context-aware. Weakness: depends on accurate attributes and well-governed policies. Incorrect attributes can create incorrect access decisions.

4.23 Rule-based and context-aware access

Rule-based access uses explicit rules such as “only allow administrative changes from the corporate network during business hours.” Context-aware access includes signals such as device compliance, location, risk score, emergency mode, or data sensitivity.

4.24 Relationship-based access control

Relationship-based access control, sometimes called ReBAC, makes decisions based on relationships. A social network might allow a user to view a post if the user is a friend of the author. A healthcare system might allow access if the clinician is on the patient’s care team.

4.25 Cloud IAM policies

Cloud identity and access management policies usually combine principals, actions, resources, and conditions. A small permission mistake can expose storage, databases, keys, snapshots, or management functions. Cloud authorization must be reviewed continuously because infrastructure changes quickly.

4.26 Kubernetes RBAC

Kubernetes commonly uses RBAC to authorize API actions against cluster resources. The model includes roles, cluster roles, role bindings, cluster role bindings, subjects, resources, and verbs such as get, list, create, update, patch, and delete.

4.27 Application authorization

Application authorization is often the hardest part because business rules are specific. A banking app must know whether a user owns an account, whether a transfer exceeds a limit, whether approval is required, and whether the request is suspicious. A generic login check is not enough.

4.28 Broken access control

Broken access control occurs when a system allows a subject to do something outside intended policy. Common forms include:

  • Horizontal privilege escalation: one normal user accesses another normal user’s data.
  • Vertical privilege escalation: a normal user performs administrator actions.
  • Broken object-level authorization: an API checks login but not ownership or permission for the specific object.
  • Missing function-level authorization: a hidden or undocumented route performs sensitive actions without proper checks.
  • Insecure direct object reference: a user-controlled ID directly identifies an object without a valid object-level access check.

4.29 Access review

An access review checks whether current access is still appropriate. Reviews should ask: Does the person still work here? Did their role change? Is this service account still needed? Are permissions excessive? Who approved this access? When should it expire?

4.30 Just-in-time and temporary access

Just-in-time access grants powerful access only when needed, often with approval, time limits, logging, and automatic removal. It reduces standing privilege, which is privilege that remains active all the time.

7. Mathematical Foundations

7.1 Access-control matrix size

If there are S subjects and O objects, a simple matrix has S × O cells. If each cell can contain multiple actions, management grows quickly.

Worked example: 2,000 users and 8,000 files create 16,000,000 subject-object cells. If each cell can contain read, write, execute, delete, and share, direct management becomes unrealistic.

7.2 Permission-bit arithmetic

Linux octal permissions use read = 4, write = 2, execute = 1. Add values for each subject class.

  • rw- = 4 + 2 + 0 = 6.
  • r-x = 4 + 0 + 1 = 5.
  • --- = 0 + 0 + 0 = 0.

Worked example: 750 means owner 7 = rwx, group 5 = r-x, other 0 = ---.

7.3 Risk scoring for excessive access

A simple teaching formula is:

Access Risk = Sensitivity × Privilege Level × Exposure × Duration

This is not a universal standard. It is a classroom model for prioritization.

Worked example: a service account can delete production backups. Sensitivity = 5, privilege = 5, exposure = 4, duration = 5. Risk score = 5 × 5 × 4 × 5 = 500. A temporary read-only account for test data might score 1 × 1 × 1 × 1 = 1.

7.4 Role explosion

If every department, project, region, and seniority level gets a separate role, the number of roles can multiply:

roles = departments × projects × regions × seniority_levels

Worked example: 12 departments × 30 projects × 5 regions × 4 levels = 7,200 roles. This is difficult to review and often indicates a need for attributes or better role design.

8. Practical Examples

8.1 Linux examples

# Create a local practice folder. Use only your own lab machine.
mkdir -p access-lab
cd access-lab

# Create a practice file.
echo "fictional course notes" > notes.txt

# Show permissions, owner, and group.
ls -l notes.txt

# Allow owner read/write, group read, others no access.
chmod 640 notes.txt
ls -l notes.txt

# Create a private folder for the owner only.
mkdir private
chmod 700 private
ls -ld private

Line-by-line explanation: mkdir -p creates a folder safely. cd enters it. echo writes fictional text. ls -l displays permissions. chmod 640 sets owner read/write, group read, and no access for others. chmod 700 makes a directory private to its owner.

8.2 Windows examples

# Run in a personal lab folder, not on system directories.
New-Item -ItemType Directory -Path .\AccessLab
Set-Location .\AccessLab
"fictional course notes" | Out-File .\notes.txt

# View the access-control list.
Get-Acl .\notes.txt | Format-List

# View current identity and group memberships.
whoami
whoami /groups

Line-by-line explanation: New-Item creates a folder. Set-Location moves into it. Out-File writes harmless text. Get-Acl displays Windows access-control information. whoami shows the current account. whoami /groups shows group memberships that may affect access.

8.3 Python example: safe local authorization evaluator

# This program evaluates fictional authorization rules.
# It does not connect to any real system.

policies = [
    {
        "role": "student",
        "action": "read",
        "resource_type": "course_notes",
        "decision": "allow"
    },
    {
        "role": "instructor",
        "action": "update",
        "resource_type": "gradebook",
        "decision": "allow"
    },
    {
        "role": "student",
        "action": "update",
        "resource_type": "gradebook",
        "decision": "deny"
    }
]

request = {
    "user": "amina",
    "role": "student",
    "action": "update",
    "resource_type": "gradebook"
}

def authorize(request, policies):
    """Return allow or deny for a fictional request."""
    for policy in policies:
        same_role = policy["role"] == request["role"]
        same_action = policy["action"] == request["action"]
        same_resource = policy["resource_type"] == request["resource_type"]

        if same_role and same_action and same_resource:
            return policy["decision"]

    # Deny by default: no matching allow means no access.
    return "deny"

print(authorize(request, policies))

Explanation: the policies list stores fictional rules. The request dictionary describes one access attempt. The function compares role, action, and resource type. If a rule matches, the function returns its decision. If no rule matches, it returns deny. This demonstrates deny by default.

8.4 Network example

A network firewall is also an access-control system. It may allow outbound HTTPS from student laptops but deny inbound connections from the internet to internal lab machines.

8.5 Web example

A safe web authorization check should verify both login and object ownership or role permission.

def can_view_invoice(user, invoice):
    if user["role"] == "admin":
        return True
    return invoice["owner_user_id"] == user["id"]

user = {"id": 10, "role": "customer"}
invoice = {"id": 501, "owner_user_id": 10}
print(can_view_invoice(user, invoice))  # True

Common mistake: checking only that the user is logged in, then returning any invoice whose numeric ID appears in the request.

9. Code Examples

9.1 Insecure application pattern

# Insecure teaching example: do not copy into real applications.
def get_grade(request, database):
    # Problem: this checks only that someone is logged in.
    if not request["user"]:
        return "login required"

    # Problem: grade_id comes from the user request.
    # Without an object-level authorization check, one user may request another grade.
    grade_id = request["grade_id"]
    return database[grade_id]

What is wrong: the function does not check whether the requested grade belongs to the user or whether the user is an instructor for that course.

9.2 Safer application pattern

def can_read_grade(user, grade):
    """Return True only when the user may read this fictional grade."""
    if user["role"] == "registrar":
        return True

    if user["role"] == "instructor":
        return grade["course_id"] in user["assigned_course_ids"]

    if user["role"] == "student":
        return grade["student_id"] == user["id"]

    return False


def get_grade(request, database):
    user = request.get("user")
    if user is None:
        return "login required"

    grade_id = request["grade_id"]
    grade = database.get(grade_id)
    if grade is None:
        return "not found"

    if not can_read_grade(user, grade):
        return "forbidden"

    return grade

Line-by-line explanation: can_read_grade centralizes the rule. Registrars can read all grades because their role requires it. Instructors can read grades only for assigned courses. Students can read only their own grades. Unknown roles are denied. get_grade checks login, retrieves the object, checks object-level authorization, and returns the grade only after the check succeeds.

9.3 Common coding mistakes

  • Checking authentication but not authorization.
  • Hiding buttons in the user interface but leaving server actions unprotected.
  • Trusting user-supplied object IDs without ownership checks.
  • Using broad administrator roles because fine-grained policy is inconvenient.
  • Failing open when policy services are unavailable.
  • Scattering authorization logic across many files without consistent review.

9.4 Secure coding practices

  • Use deny by default.
  • Centralize authorization checks where possible.
  • Check object-level authorization for every protected object.
  • Perform checks on the server side, not only in the user interface.
  • Log authorization denials and sensitive grants.
  • Write tests for allowed and denied paths.
  • Review changes to roles, policies, and default permissions.

10. Security Perspective

Common attacks

  • Using another user’s object ID in a request.
  • Calling an administrative endpoint as a normal user.
  • Reusing overly privileged service tokens.
  • Exploiting wildcard cloud permissions.
  • Abusing stale access after a job change.
  • Modifying client-side fields to request higher privileges.

Common vulnerabilities

  • Missing object-level checks.
  • Excessive default permissions.
  • Weak role governance.
  • Inaccurate attributes in ABAC systems.
  • Hidden inheritance paths.
  • Privilege creep.
  • No access expiry.

Defensive strategies

  • Least privilege and deny by default.
  • Centralized policy evaluation for sensitive actions.
  • Object-level checks in APIs and applications.
  • Access reviews and recertification.
  • Temporary elevation and just-in-time access.
  • Separation of duties for critical workflows.

Detection methods

  • Repeated denied access attempts.
  • Unexpected access to sensitive objects.
  • Privilege changes outside change windows.
  • Service account use from unusual systems.
  • Bulk reads, exports, or deletes.
  • Access after termination or role change.

Mitigation and best practices

  • Design access before coding features.
  • Document who can do what and why.
  • Use groups or roles instead of direct user grants where practical.
  • Avoid broad wildcards such as “all resources” and “all actions.”
  • Use break-glass accounts only with strong logging and review.
  • Test negative cases: normal users must fail when requesting unauthorized objects.

12. Research Perspective

Access control remains an active research area because real systems are large, distributed, dynamic, and full of exceptions. The core problem is not only “write a rule”; it is designing rules that are correct, understandable, enforceable, auditable, usable, and scalable.

Current research themes

  • Policy mining: learning candidate access rules from existing access logs and configurations.
  • ABAC migration: moving from legacy groups and ACLs to attribute-based policy without breaking business workflows.
  • Broken access-control testing: automatically finding object-level and function-level authorization failures in APIs.
  • Zero Trust authorization: using continuous context rather than one-time perimeter assumptions.
  • Usable access governance: making access reviews understandable enough for managers to make accurate decisions.
  • Privacy-preserving authorization: proving eligibility without revealing unnecessary personal attributes.

Open problems

  • How can systems explain authorization decisions clearly without revealing sensitive policy internals?
  • How can policy engines remain available without failing open during outages?
  • How can organizations measure whether access reviews actually reduce risk?
  • How can AI-assisted development avoid generating missing authorization checks?
  • How can dynamic emergency access be safe, fast, and accountable?

Notable directions: attribute-based access control, relationship-based access control, policy-as-code, authorization logic verification, API authorization testing, and least-privilege recommendation systems.

13. Case Studies

13.1 Broken object-level authorization in APIs

What happened: many API incidents involve a valid user changing an object identifier and receiving another user’s data or performing another user’s action.

Root cause: the application checks login but fails to check authorization for the specific object.

Impact: exposure or modification of private records, invoices, messages, profiles, bookings, or health data.

Lessons learned: every object access needs server-side authorization. Do not rely on hidden UI elements or unguessable IDs alone.

13.2 Excessive cloud permissions

What happened: cloud incidents often become worse when a compromised workload or token has broad permissions to read sensitive storage or call management APIs.

Root cause: service identities receive more access than required, sometimes through wildcards or inherited roles.

Impact: large-scale data exposure, unauthorized snapshots, key misuse, or infrastructure changes.

Lessons learned: restrict service roles, monitor unusual use, remove wildcards where possible, and review permissions after architecture changes.

13.3 Stale employee access

What happened: a user changes job roles or leaves an organization, but old access remains active.

Root cause: weak joiner-mover-leaver process, no access expiry, direct grants instead of managed groups, and poor review discipline.

Impact: insider risk, accidental access to restricted data, and compliance failures.

Lessons learned: automate deprovisioning, review access periodically, and require owners for sensitive roles.

13.4 Overly broad shared folders

What happened: sensitive spreadsheets, research data, or HR files are placed in shared folders readable by too many users.

Root cause: convenience-driven sharing, inherited permissions, poor data classification, and lack of folder ownership.

Impact: privacy violations, data leakage, and loss of trust.

Lessons learned: classify data, assign owners, use least privilege, and audit high-risk shares.

14. Common Misconceptions

  • “If the user is logged in, access is safe.” Login only proves identity. Authorization must still check the requested action and object.
  • “Admin users do not need authorization checks.” Admin actions are high-risk and need strong authorization, logging, and often step-up verification.
  • “Hiding the button is enough.” Client-side controls improve usability but do not enforce server-side security.
  • “RBAC solves everything.” RBAC simplifies administration but may struggle with dynamic context and fine-grained object rules.
  • “ABAC is always better than RBAC.” ABAC is more expressive, but poor attributes or unclear policies can make it dangerous.
  • “A random UUID means object authorization is unnecessary.” Hard-to-guess IDs reduce enumeration risk but do not prove permission.
  • “777 permissions fix Linux problems.” They may make errors disappear by exposing files to everyone, which is usually unsafe.
  • “One annual access review is enough.” Sensitive systems often need event-driven reviews after role changes, incidents, and major system changes.
  • “Deny logs always mean an attack.” Denials may also be misconfiguration, user error, expired access, or broken workflows.
  • “Least privilege means users cannot do their jobs.” Good least privilege gives necessary access, not zero access.

15. Summary

Authorization decides whether an authenticated subject may perform a specific action on a specific resource under current conditions. Access control implements that decision through policies, permissions, ACLs, roles, attributes, enforcement points, and audit logs. Strong authorization uses least privilege, deny by default, need to know, separation of duties, object-level checks, and periodic review. Weak authorization creates horizontal privilege escalation, vertical privilege escalation, excessive access, stale access, and data exposure. The practical goal is not to memorize model names; it is to design access so people and services can do approved work without receiving unnecessary power.

16. Glossary

Access Control
The mechanisms that decide, enforce, and record who or what may access resources.
Access-Control Entry
A single rule inside an ACL that grants, denies, or audits access for a trustee or subject.
Access-Control List
A list of entries attached to an object that describes who can do what.
ABAC
Attribute-based access control; authorization based on subject, object, action, and environmental attributes.
Authorization
The process of granting or denying a specific access request.
Capability
A token or unforgeable reference that carries authority to perform an action.
DAC
Discretionary access control; owners or authorized subjects can grant access.
DACL
Discretionary access-control list in Windows that controls allowed or denied access.
Deny by Default
A design where access is denied unless an explicit rule allows it.
Least Privilege
Granting only the access needed for an approved task.
MAC
Mandatory access control; centrally enforced labels and policy that users cannot override.
Object
A protected resource, such as a file, record, API object, or cloud asset.
Permission
A specific allowed action on a resource.
PEP
Policy enforcement point; the component that permits or blocks the request.
PDP
Policy decision point; the component that evaluates policy and returns allow or deny.
Privilege Creep
Gradual accumulation of access that is no longer needed.
RBAC
Role-based access control; permissions are assigned to roles and subjects are assigned to roles.
ReBAC
Relationship-based access control; decisions depend on relationships between subjects and objects.
SACL
System access-control list in Windows that controls auditing of access attempts.
Subject
An active entity requesting access, such as a user, process, service, or workload.

Part C: Evaluate fictional requests

import csv

requests = [
    {"subject": "amina", "role": "student", "resource": "course_notes", "action": "read"},
    {"subject": "amina", "role": "student", "resource": "gradebook", "action": "update"},
    {"subject": "backup-svc", "role": "service", "resource": "users", "action": "delete"},
    {"subject": "unknown", "role": "guest", "resource": "gradebook", "action": "read"}
]

with open("access-matrix.csv", newline="") as f:
    policies = list(csv.DictReader(f))

def decide(request):
    for policy in policies:
        if (
            policy["role"] == request["role"] and
            policy["resource"] == request["resource"] and
            policy["action"] == request["action"]
        ):
            return policy["decision"]
    return "deny"

for request in requests:
    print(request, "=>", decide(request))
