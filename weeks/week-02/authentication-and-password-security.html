Lesson 2.3: Authentication and Password Security

Authentication answers the question: can this subject prove control of the authenticator bound to the account? This lesson starts from zero and builds toward modern password security, safe password storage, multi-factor authentication, passkeys, account recovery, brute-force defenses, and authentication monitoring.

Estimated reading time210–320 minutes
Difficulty levelBeginner → MS foundation
Required backgroundLessons 2.1 and 2.2 recommended
Practice time5–8 hours

Week 2 Context

Overview: Week 2 explains how systems identify subjects and enforce access. Lesson 2.2 showed how systems represent users, groups, service accounts, and identifiers. Lesson 2.3 focuses on authentication: proving that the requester controls the right authenticator.

Learning objectives: after this lesson you should be able to explain password authentication, account recovery, multi-factor authentication, passkeys, safe password storage, authentication attacks, and defensive monitoring.

Required background: no previous authentication background is required. You should understand user accounts, groups, service accounts, and identifiers from Lesson 2.2.

Estimated reading time: 210–320 minutes. Difficulty: begins with everyday login ideas and ends with graduate-level authentication design and research questions.

1. Introduction

Plain-English idea: authentication is the process of checking whether a claimant can prove control of something linked to an account. A claimant says, “I am this account.” The system asks for evidence. If the evidence is valid, the system authenticates the claimant.

Why the topic exists: systems cannot safely grant access just because someone types a username. A username is only a claim. Authentication exists because the system needs evidence before it treats the requester as that identity.

Why it matters: stolen, guessed, reused, phished, leaked, or poorly stored passwords remain one of the most common ways attackers gain initial access. Strong authentication reduces the chance that a stolen secret alone becomes a full compromise.

Where it is used: laptops, servers, email, banking apps, government portals, cloud dashboards, source-code repositories, university learning systems, hospital records, VPNs, databases, APIs, containers, and administrative consoles.

Who uses it: every user who logs in, developers who build login systems, system administrators who configure policy, help desks that reset access, auditors who review logs, incident responders who investigate suspicious sign-ins, and researchers who design stronger authentication.

Analogy: entering a building often requires showing a badge, using a key, or passing a guard check. A username is like saying your name at the door. A password, hardware key, or biometric unlock provides evidence that you are allowed to enter.

Safety boundary

This lesson is defensive and educational. Do not test passwords, attempt logins, enumerate accounts, bypass MFA, intercept credentials, or run authentication tools against systems you do not own or administer with explicit written authorization. Labs use only fictional data and local demonstration code.

  • Define claimant, subscriber, verifier, authenticator, credential, password, passphrase, MFA, OTP, passkey, password hash, salt, pepper, rate limiting, account lockout, password spraying, brute force, and credential stuffing.

3. Prerequisites

3.1 Identity

An identity is the system’s representation of a person, service, device, or workload. Authentication proves control of an authenticator bound to that identity.

3.2 Account

An account is a system record. It may store an identifier, username, status, attributes, and references to authenticators.

3.3 Request

A request is an attempted action: logging in, opening a file, reading email, calling an API, or changing settings.

3.4 Secret

A secret is information that must be kept confidential. A password is a secret. A private key is also a secret, but it is used differently.

3.5 Hash function

A hash function maps input data to a fixed-size output. For files, fast hashes help detect changes. For passwords, fast hashes alone are dangerous because attackers can test guesses quickly if a password database is stolen.

3.6 Cryptographic key

A cryptographic key is a value used by an algorithm. Passkeys and hardware security keys use public-key cryptography: the server stores a public key, while the private key remains with the user’s authenticator.

4. Core Concepts

4.1 Identification, authentication, authorization

Identification is claiming an identity, such as typing a username. Authentication is proving that claim. Authorization is deciding what the authenticated subject may do.

StepQuestionExample
IdentificationWho are you claiming to be?Username: amina
AuthenticationCan you prove it?Password + authenticator app
AuthorizationWhat are you allowed to access?Read course notes, not payroll

Common mistake: “The user logged in, so they can do anything.” Authentication does not grant unlimited access. Authorization must still enforce limits.

4.2 Authentication actors

  • Claimant: the subject trying to authenticate.
  • Subscriber: the person or entity enrolled with an account and authenticators.
  • Verifier: the component that checks authentication evidence.
  • Relying party: the application or service that relies on authentication results.
  • Identity provider: a system that authenticates users and sends assertions to applications.

4.3 Authenticator

An authenticator is something used to authenticate. Examples: password, authenticator app, hardware security key, smart card, device-bound passkey, recovery code, or certificate-backed key.

4.4 Authentication factors

Authentication factors are broad categories of evidence:

  • Something you know: password, passphrase, PIN.
  • Something you have: phone, hardware key, smart card, authenticator app, cryptographic device.
  • Something you are: biometric characteristic used locally to unlock an authenticator, such as fingerprint or face recognition.

Important precision: two passwords are not two factors. They are two examples of the same factor: something you know.

4.5 Passwords and passphrases

A password is a secret string. A passphrase is usually longer and composed of several words or a sentence-like structure. Length matters because every extra character increases the number of possible guesses.

Beginner intuition: a short clever password is often weaker than a long memorable passphrase generated or stored safely by a password manager.

4.6 Modern password policy

Modern password policy should avoid rules that make users choose predictable patterns. Rules such as “must include uppercase, lowercase, number, symbol” often produce passwords like Summer2026!. A better baseline emphasizes length, blocklists of known weak or compromised passwords, password managers, rate limiting, MFA, and reset only when compromise is suspected.

  • Require enough length for the use case.
  • Block common, expected, context-specific, and compromised passwords.
  • Allow password managers, paste, and long passwords.
  • Do not rely on arbitrary periodic password changes as a primary control.
  • Force change when there is evidence of compromise.
  • Protect login with TLS and secure session management.

4.7 Password managers

A password manager stores unique passwords and helps users avoid reuse. It can generate long random passwords that humans cannot remember. The user protects the vault with a strong master secret and, preferably, MFA.

Risk trade-off: a password manager centralizes many secrets, so the vault itself must be strongly protected. The benefit is that password reuse drops dramatically.

4.8 Password reuse

Password reuse means using the same or similar password on multiple services. If one service is breached, attackers may try the same username-password pair elsewhere. That attack is called credential stuffing.

4.9 Password storage: plaintext, encryption, hashing

Password storage is one of the most important implementation details in authentication.

Storage methodMeaningSecurity problem
PlaintextStore the password exactly as typedDatabase theft immediately exposes passwords
EncryptionStore a reversible encrypted passwordIf the decryption key is stolen, all passwords are recoverable
Fast hash onlyStore a fast hash such as raw SHA-256Attackers can test huge numbers of guesses quickly
Slow salted password hashStore output from a password-hashing scheme with salt and costBest normal pattern; each guess is intentionally expensive

4.10 Salt

A salt is a unique random value stored with each password hash. It prevents two equal passwords from producing the same stored hash and makes precomputed lookup tables much less useful.

Analogy: if two students choose the same secret phrase, the salt is like giving each student a different random mixing ingredient before creating the stored fingerprint.

4.11 Pepper

A pepper is an additional secret value kept separately from the password database, often in a secrets manager or hardware security module. A pepper is defense in depth. It does not replace salting or slow hashing.

4.12 Work factor

A work factor, or cost factor, controls how expensive each password verification is. Higher cost makes offline guessing harder after database theft, but it also consumes more server resources during legitimate login.

Design tension: the verifier must be slow enough to hurt attackers but fast enough for real users and resilient enough against login-flood denial-of-service.

4.13 Recommended password-hashing schemes

Modern systems should use password-hashing schemes designed for password storage, such as Argon2id, scrypt, bcrypt, or PBKDF2 where compliance requires it. The exact configuration depends on platform, risk, performance, and compliance needs.

Do not design your own password-hashing algorithm. Use a reputable library and keep parameters reviewable.

4.14 Online versus offline guessing

Online guessing means the attacker sends login attempts to the real service. Defenses include rate limiting, lockout, risk-based challenges, MFA, and monitoring. Offline guessing means the attacker has stolen password hashes and can test guesses without contacting the service. Defenses include slow salted hashing and strong passwords.

4.15 Brute force, dictionary attacks, spraying, stuffing

  • Brute force: trying many possible passwords.
  • Dictionary attack: trying likely words or common patterns.
  • Password spraying: trying one or a few common passwords across many accounts to avoid simple lockouts.
  • Credential stuffing: trying username-password pairs leaked from another service.

4.16 Rate limiting and account lockout

Rate limiting slows repeated attempts. Account lockout temporarily blocks login after too many failures. These controls reduce online guessing but must be designed carefully: attackers may abuse lockout to deny service to real users.

Safer designs often combine account-based counters, IP or device signals, progressive delays, MFA, known-device checks, alerting, and help-desk procedures.

4.17 Multi-factor authentication

Multi-factor authentication, or MFA, requires proof using more than one factor. A password plus an authenticator app is stronger than a password alone because stolen password knowledge is not enough.

However, not all MFA is equal. SMS codes can be vulnerable to SIM-swap and interception risks. Push approval can be vulnerable to fatigue attacks if users approve prompts without checking. Phishing-resistant MFA uses cryptography that binds authentication to the real service.

4.18 One-time passwords

A one-time password, or OTP, is a short code used once. A time-based OTP changes every time window, often every 30 seconds. OTPs reduce risk from static password theft, but they can still be phished if a user enters the code into a fake login page.

4.19 Push approval and MFA fatigue

Push approval asks the user to approve a sign-in on a trusted device. A common attack pattern is MFA fatigue, where attackers repeatedly trigger prompts until a tired or confused user approves. Number matching and context display reduce this risk but do not make push approval fully phishing-resistant.

4.20 Passkeys, FIDO2, and WebAuthn

A passkey is a user-friendly name for a FIDO2/WebAuthn-style credential. The basic idea is public-key authentication:

  1. The user registers an authenticator with a service.
  2. The authenticator creates a key pair.
  3. The server stores the public key, not a shared password.
  4. During login, the server sends a challenge.
  5. The authenticator signs the challenge with the private key.
  6. The server verifies the signature using the public key.

Because the private key is not typed into a website, passkeys can resist common phishing flows better than passwords and OTPs.

4.21 Biometrics

Biometrics such as fingerprint or face recognition are usually used to unlock a local authenticator. In well-designed systems, the remote website does not receive your fingerprint or face. The authenticator proves cryptographic control after local user verification succeeds.

4.22 Account recovery

Account recovery is the process of regaining access after losing an authenticator, forgetting a password, or replacing a device. Recovery is security-critical because attackers often target the reset path instead of the normal login path.

Good recovery uses multiple signals, notifications, recovery codes, waiting periods for high-risk accounts, help-desk verification, and careful logging. Security questions are weak because answers can be guessed, researched, or socially engineered.

4.23 Session after authentication

Authentication usually creates a session, such as a cookie or token, so the user does not re-enter credentials on every page. Session security will be studied later, but the key point is this: authenticating securely is not enough if the session token can be stolen or misused.

4.24 Authentication logs

Authentication logs record events such as successful login, failed login, MFA challenge, password reset, account lockout, new device, new location, impossible travel, and recovery-code use. These logs are essential for incident detection and investigation.

7. Mathematical Foundations

7.1 Search space

If a password has length L and each position can be one of N possible symbols, the maximum search space is:

Search space = N^L

Example: a 4-digit PIN has N = 10 and L = 4.

10^4 = 10,000 possible PINs

7.2 Entropy approximation

Password entropy is often approximated in bits:

Entropy bits = log2(N^L) = L × log2(N)

Example: 12 random lowercase letters have N = 26 and L = 12.

Entropy = 12 × log2(26)
        ≈ 12 × 4.70
        ≈ 56.4 bits

This estimate assumes truly random selection. Human-chosen passwords are usually much less random than the formula suggests.

7.3 Time to test guesses

If an attacker can test G guesses per second and the search space is S, the worst-case time is:

Worst-case time = S / G seconds
Average time ≈ S / (2G) seconds

For online guessing, defenders try to make G very small using rate limiting. For offline guessing, defenders use slow password hashing to make each guess expensive.

7.4 Safe worked example

Suppose a fictional service allows 5 failed attempts per hour for an account. A common password list contains 1,000 likely guesses.

Hours to try 1,000 guesses online = 1000 / 5 = 200 hours

This simple calculation shows why rate limiting helps against online guessing. It does not protect against credential stuffing if the attacker already has the correct password from another breach, which is why MFA and unique passwords matter.

8.3 Python examples

The following example demonstrates password hashing concepts locally. It is not a complete production login system.

import hashlib
import hmac
import os

# Choose a slow password-hashing configuration for demonstration.
# Production systems should use a reviewed library such as Argon2id, bcrypt, scrypt,
# or PBKDF2 with parameters selected for the environment and compliance needs.
ITERATIONS = 600_000
HASH_NAME = "sha256"
SALT_BYTES = 16


def hash_password(password: str) -> tuple[bytes, bytes]:
    """Return a random salt and PBKDF2-HMAC password hash."""
    salt = os.urandom(SALT_BYTES)  # Unique random salt for this password.
    password_bytes = password.encode("utf-8")  # Convert text to bytes.
    password_hash = hashlib.pbkdf2_hmac(
        HASH_NAME,         # Internal hash function.
        password_bytes,    # Password bytes supplied by the user.
        salt,              # Unique salt stored with the hash.
        ITERATIONS         # Work factor: more iterations means slower guessing.
    )
    return salt, password_hash


def verify_password(password: str, salt: bytes, expected_hash: bytes) -> bool:
    """Verify a password without comparing hashes using unsafe equality."""
    candidate_hash = hashlib.pbkdf2_hmac(
        HASH_NAME,
        password.encode("utf-8"),
        salt,
        ITERATIONS
    )
    return hmac.compare_digest(candidate_hash, expected_hash)


salt, stored_hash = hash_password("correct horse battery staple")
print("Stored salt:", salt.hex())
print("Stored hash:", stored_hash.hex())
print("Correct password accepted:", verify_password("correct horse battery staple", salt, stored_hash))
print("Wrong password accepted:", verify_password("password123", salt, stored_hash))

8.4 Network example

A login form should send credentials only over an authenticated encrypted channel such as HTTPS. In a real deployment, the browser should verify the server certificate, send credentials over TLS, and receive a secure session cookie after successful authentication.

Browser --HTTPS/TLS--> Login endpoint --verifier--> Account store

8.5 Web example

Safe pseudo-code for login flow:

def login(username, password, mfa_code):
    # 1. Use a generic failure message to avoid account enumeration.
    account = lookup_account_by_username(username)
    if account is None:
        log_event("login_failed", reason="generic")
        return "Invalid username or password"

    # 2. Apply rate limiting before expensive checks.
    if too_many_attempts(account.id):
        log_event("login_throttled", account_id=account.id)
        return "Try again later"

    # 3. Verify the password with the stored salt and password-hashing parameters.
    if not verify_password(password, account.salt, account.password_hash):
        record_failed_attempt(account.id)
        return "Invalid username or password"

    # 4. Require MFA when policy says it is needed.
    if account.mfa_enabled and not verify_mfa(account.id, mfa_code):
        log_event("mfa_failed", account_id=account.id)
        return "Additional verification failed"

    # 5. Create a secure session after all checks pass.
    session = create_secure_session(account.id)
    log_event("login_success", account_id=account.id)
    return session

8.6 Cloud example

Cloud providers usually centralize authentication through an identity provider. Strong designs use MFA for users, phishing-resistant MFA for administrators, conditional access for risky sign-ins, workload identities instead of long-lived static keys, and logs for every sign-in and reset event.

9. Code Examples

9.1 Line-by-line explanation of the Python hashing demo

  • import hashlib: loads Python’s hashing and key-derivation functions.
  • import hmac: provides compare_digest, which avoids unsafe direct hash comparison patterns.
  • import os: provides os.urandom for cryptographically suitable random bytes.
  • ITERATIONS = 600_000: sets the PBKDF2 work factor for the demonstration.
  • os.urandom(SALT_BYTES): creates a unique random salt for this stored password.
  • password.encode("utf-8"): converts a text password into bytes.
  • hashlib.pbkdf2_hmac(...): derives a password hash using PBKDF2-HMAC.
  • hmac.compare_digest(...): compares two hash outputs safely.

9.2 Common coding mistakes

MistakeWhy it is dangerousBetter approach
Store plaintext passwordsDatabase theft exposes every passwordStore slow salted password hashes
Use raw SHA-256 for passwordsToo fast for offline guessing resistanceUse Argon2id, scrypt, bcrypt, or PBKDF2
Use the same salt for all usersSame passwords reveal same hashesUse unique random salt per password

9.3 Secure coding practices

  • Never print, log, email, or store raw passwords.
  • Use vetted authentication libraries instead of custom login logic where possible.
  • Use TLS for all credential submission.
  • Use generic login errors.
  • Store password-hashing parameters with the hash so they can be upgraded later.
  • Protect password reset flows as carefully as login flows.

10. Security Perspective

10.1 Common attacks

  • Phishing: tricking a user into entering credentials on a fake page.
  • Credential stuffing: using leaked username-password pairs from another breach.
  • Password spraying: trying a small set of common passwords across many accounts.
  • Brute force: repeated guessing against one or more accounts.
  • Offline hash guessing: testing guesses after stealing password hashes.
  • MFA fatigue: repeatedly sending push prompts until the user approves.
  • SIM swap: taking over a phone number to receive SMS codes.
  • Account recovery abuse: bypassing the normal login by manipulating reset workflows.
  • Session theft: stealing a session token after authentication succeeds.

10.2 Common vulnerabilities

  • Weak or reused passwords.
  • No MFA on privileged accounts.
  • Plaintext password storage.
  • Fast password hashes without salt or work factor.
  • Exposed reset links or recovery codes.
  • Help-desk reset process based on weak verbal checks.
  • No login throttling.
  • Different error messages for missing users and wrong passwords.
  • Missing logs for authentication and recovery events.

10.3 Defensive strategies

  • Use phishing-resistant MFA for administrators and high-risk users.
  • Require MFA broadly where feasible.
  • Allow and encourage password managers.
  • Block known weak, common, breached, and context-specific passwords.
  • Use slow salted password hashing with approved libraries.
  • Rate-limit failed attempts and detect spraying patterns.
  • Use generic failure messages.
  • Protect account recovery with stronger checks, notifications, and logging.
  • Review privileged authentication events regularly.

10.4 Detection methods

  • Many failed logins across many accounts from one source.
  • One account failing from many locations.
  • Successful login after many failures.
  • Impossible travel or abnormal country/device changes.
  • Repeated MFA denials followed by approval.
  • Password reset followed by new device enrollment.
  • Login to privileged account outside normal hours.
  • High-volume attempts against disabled or stale accounts.

10.5 Mitigation and best practices

The strongest authentication strategy is layered: strong unique passwords or passwordless authentication, MFA, phishing-resistant options for high-risk accounts, secure recovery, logging, monitoring, least privilege, and user education that focuses on practical behavior rather than blame.

12. Research Perspective

Authentication research sits at the intersection of cryptography, usable security, human-computer interaction, systems security, privacy, and economics. A technically strong method can still fail if users cannot understand it or recover from device loss safely.

12.1 Current research directions

  • Passwordless authentication: passkeys, hardware-backed authenticators, and enterprise deployment models.
  • Usable security: making secure choices easier than insecure choices.
  • Phishing resistance: domain-bound and channel-bound authentication.
  • Risk-based authentication: adjusting requirements based on device, location, behavior, and transaction risk.
  • Privacy-preserving identity: reducing unnecessary tracking across services.
  • Account recovery: balancing recovery availability with resistance to social engineering.
  • Authentication telemetry: detecting attacks without excessive surveillance or false positives.

12.2 Open problems

  • How to make phishing-resistant authentication usable for all populations.
  • How to recover high-value accounts when all authenticators are lost.
  • How to secure synchronized passkeys without over-trusting platform ecosystems.
  • How to measure authentication risk without collecting unnecessary personal data.
  • How to protect users who cannot reliably use phones, biometrics, or hardware keys.

12.3 Notable research areas

Study classic and modern work on password cracking economics, password-composition policies, password managers, FIDO/WebAuthn, authentication usability, adaptive authentication, and secure recovery protocols. Do not reproduce copyrighted papers; read abstracts, methodology, results, and limitations, then summarize in your own words.

13. Case Studies

13.1 Credential stuffing against consumer services

What happened: attackers used credentials leaked from one service to attempt logins against other services. Root cause: password reuse and weak detection. Impact: account takeover, fraud, privacy loss, and support costs. Lessons learned: unique passwords, password managers, MFA, breach monitoring, and anomaly detection reduce risk.

13.2 Password spraying against enterprise email

What happened: attackers tried a few common passwords across many enterprise accounts to avoid per-account lockouts. Root cause: weak passwords, limited MFA, and insufficient sign-in monitoring. Impact: mailbox access, data theft, internal phishing, and lateral movement. Lessons learned: MFA, blocklists, conditional access, and cross-account detection matter.

13.3 Plaintext or weakly hashed password database exposure

What happened: a password database was exposed. Root cause: poor password storage such as plaintext, reversible encryption, or fast unsalted hashes. Impact: users had to reset passwords; reused passwords created downstream risk. Lessons learned: use slow salted password hashing, protect databases, monitor access, and plan credential-reset response.

13.4 MFA fatigue and social engineering

What happened: users received repeated approval prompts or help-desk requests after attackers obtained passwords. Root cause: push-based approval without enough context, weak help-desk verification, and user confusion. Impact: unauthorized access despite MFA. Lessons learned: phishing-resistant MFA, number matching, user education, prompt limits, and recovery-process controls are needed.

14. Common Misconceptions

  1. “A complex password is always strong.” Predictable complexity patterns can be weak. Length and randomness matter.
  2. “Changing passwords every month always improves security.” Arbitrary forced changes often lead to predictable modifications. Change should be forced when compromise is suspected.
  3. “Hashing and encryption are the same.” Encryption is reversible with a key. Hashing is one-way. Password verification normally needs hashing, not reversible encryption.
  4. “SHA-256 is secure, so it is enough for passwords.” SHA-256 is fast. Password storage needs slow, salted password-hashing schemes.
  5. “MFA means impossible to hack.” Some MFA can be phished, socially engineered, or bypassed through recovery weaknesses.
  6. “Security questions are a second factor.” They are another knowledge factor and often have guessable or researchable answers.
  7. “Biometrics are passwords.” In many systems, biometrics unlock a local authenticator rather than being sent to the remote service.
  8. “Lockout is always good.” Poorly designed lockout can be abused for denial of service.
  9. “Account recovery is separate from authentication security.” Recovery can become the easiest bypass path.
  10. “Logs are optional.” Without authentication logs, defenders cannot reliably detect or investigate suspicious sign-ins.

15. Summary

Authentication proves control of an authenticator bound to an identity. Passwords are still common, but they must be handled carefully: encourage long unique secrets, block known weak passwords, support password managers, avoid arbitrary forced rotation, and store only slow salted password hashes. MFA reduces risk, but phishing-resistant methods such as passkeys and hardware-backed authenticators are stronger than SMS or basic push approval. Account recovery, rate limiting, logging, and detection are part of authentication security, not afterthoughts.

Trying leaked username-password pairs from one service against another service.
Hash
Fixed-size output produced from input data by a hash function.
MFA
Multi-factor authentication; authentication using more than one factor.
OTP
One-time password; a code intended for one authentication event or short time window.
Passkey
A user-friendly form of FIDO2/WebAuthn public-key authentication.
Password hash
A stored output from a password-hashing scheme used to verify future password entries.
Password spraying
Trying one or a few common passwords across many accounts.
Rate limiting
Restricting the number or speed of attempts.
Salt
A unique random value stored with a password hash to prevent identical passwords from having identical stored hashes.
Verifier
The system component that checks authentication evidence.
Work factor
A cost parameter that makes password hashing slower and more expensive to guess offline.

Part B: Analyze the fictional log

import csv
from collections import Counter, defaultdict

with open("fictional-auth-log.csv", newline="") as f:
    rows = list(csv.DictReader(f))

failed_by_source = Counter()
mfa_denials_by_account = Counter()
events_by_account = defaultdict(list)

for row in rows:
    events_by_account[row["account"]].append(row)
    if row["result"] == "failed":
        failed_by_source[row["source_ip"]] += 1
    if row["mfa_result"] == "denied":
        mfa_denials_by_account[row["account"]] += 1

print("Failed logins by source:")
for source, count in failed_by_source.items():
    print(source, count)

print("\nRepeated MFA denials:")
for account, count in mfa_denials_by_account.items():
    if count >= 2:
        print(account, count)

print("\nAccounts with password reset events:")
for account, events in events_by_account.items():
    if any(e["event"] == "password_reset" for e in events):
        print(account)
