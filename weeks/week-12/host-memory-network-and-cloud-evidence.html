Week 12 · Lesson 12.3

Host, Memory, Network, and Cloud Evidence

This lesson teaches how to choose, preserve, and interpret evidence sources during defensive incident response. The goal is not to collect everything. The goal is to answer incident questions with the right evidence, in the right order, with known limitations.

Incident Response & ForensicsVolatility-aware evidence planningSafe local lab

1. Learning Objectives

By the end of this lesson, you should be able to:

  • Explain the evidence value and limitations of host, memory, network, cloud, SaaS, identity, and container sources.
  • Use volatility, retention, authorization, privacy sensitivity, and business impact to prioritize collection.
  • Distinguish control-plane evidence, data-plane evidence, endpoint artifacts, telemetry summaries, and full-content evidence.
  • Build an investigation plan that maps each incident question to evidence sources and preservation actions.
  • Document uncertainty when a source is incomplete, lossy, delayed, sampled, disabled, or outside authorized scope.
  • Practice evidence-source prioritization with a safe fictional local Python lab.

2. Why This Matters

Incident responders often fail in two opposite ways. One team collects too little and loses volatile evidence. Another team collects too much and creates privacy, legal, storage, and analysis problems without improving the answer. Professional evidence handling sits between these extremes: define the incident question, select the minimum evidence needed to answer it, preserve what may disappear first, and document every limitation.

Important boundary: This lesson is defensive and educational. Do not acquire memory, inspect devices, export user content, copy cloud logs, review SaaS records, or query another person’s system unless you have explicit authorization, approved scope, and an evidence-handling process.

3. Research Basis

This lesson is based on current and authoritative defensive references, with practical interpretation for a student course:

  • NIST SP 800-86 frames computer and network forensics for incident response and uses collection, examination, analysis, and reporting as the core process.
  • NIST IR 8354 emphasizes that digital investigation techniques have limitations and should be used with awareness of tool behavior, changing artifacts, and incomplete discovery.
  • NIST IR 8006 summarizes cloud forensic challenges, including architecture, data collection, analysis, role management, legal, and standards concerns in cloud ecosystems.
  • NIST SP 800-92 supports disciplined log management, including planning, collection, storage, analysis, and retention.
  • NIST SP 800-61 Rev. 3 provides the current incident-response context aligned to CSF 2.0 risk-management outcomes.
  • AWS Security Incident Response Guide highlights preparation steps such as enabling logs, defining forensic artifacts, preparing access, building playbooks, and running simulations.
  • AWS log-source guidance identifies CloudTrail, VPC Flow Logs, and Route 53 Resolver query logs as a basic logging trio for AWS security investigations.
  • Microsoft Azure Monitor Activity Log documentation explains activity-log retrieval and export to destinations such as Log Analytics, Event Hubs, and Storage.
  • Microsoft Sentinel Entra ID connector documentation describes collecting sign-in, audit, provisioning, risky-user, service-principal, and Microsoft 365 audit data.
  • Google Cloud Audit Logs documentation describes Admin Activity, Data Access, System Event, and Policy Denied audit logs.
  • Google Cloud VPC Flow Logs documentation describes flow logs as useful for network monitoring, forensics, security analysis, and cost optimization.

4. First Principle: Evidence Must Answer a Question

Evidence planning starts with the question, not the tool. A memory capture may be valuable for active malware, but excessive for a simple password-reset audit. A cloud control-plane log may be essential for infrastructure change, but insufficient for data-content exposure. A DNS log may show a lookup, but not prove payload transfer.

Incident questionHigh-value evidence familiesCommon mistake
Which account was used?Identity sign-in logs, SaaS audit logs, cloud audit logs, endpoint user session records.Assuming source IP alone identifies a person.
What changed?Cloud control-plane logs, configuration history, admin audit records, CI/CD logs, endpoint process telemetry.Looking only at alerts and ignoring control-plane state changes.
What executed?EDR telemetry, process lineage, file metadata, prefetch or execution artifacts, memory where justified.Treating file presence as proof of execution.
Where did traffic go?Network flows, DNS logs, proxy logs, cloud flow logs, endpoint network telemetry.Assuming flow metadata proves content exfiltration.
Was data accessed?SaaS audit logs, storage access logs, database audit logs, cloud data-plane logs, application logs.Using control-plane logs when data-plane logging was required but disabled.
What is still active?Live response telemetry, memory, active sessions, running processes, network connections, cloud session/token state.Rebooting or isolating before preserving volatile evidence when preservation was justified.

6. Order of Volatility

Order of volatility means some sources disappear faster than others. It is not an excuse to collect everything immediately. It is a decision aid that must be balanced against authorization, active harm, safety, business continuity, and privacy.

Very highMemory, current processes, open network connections, active sessions.Preserve quickly only when justified and approved.

MediumEndpoint logs, firewall logs, DNS logs, proxy logs, Kubernetes audit events.Record source, time window, retention, filters, and export method.
LowArchived logs, case records, stored disk images, cloud configuration snapshots.Still preserve integrity and custody; do not assume completeness.

7. Host Evidence

Host evidence describes what happened on a device, server, or virtual machine. It can connect identity, process execution, files, services, persistence, and local network activity. The evidence value depends on logging configuration, EDR health, time synchronization, and whether the host was altered after the incident.

Host sourceUseful questionPreservation note
Security and system logsWho logged in? Was a service created? Did a privileged action occur?Export with source host, UTC conversion, event IDs/categories, and time window.
Process telemetryWhat executed? What parent process launched it?Preserve raw records before enrichment changes or alert grouping hide detail.
File metadataWhat file appeared, changed, or was deleted?Record path, timestamps, hashes, owner, and acquisition method.
Persistence locationsWill something run again?Do not clean before documenting the mechanism and related context.
Endpoint agent stateWas telemetry available and trustworthy?Record agent version, policy, last check-in, and known outages.
Interpretation rule: File creation, file presence, and detection of a tool are not the same as confirmed execution. Link execution claims to process events, command history, service creation, scheduled execution, prefetch/execution artifacts, or other corroborating records.

8. Memory Evidence

Memory is valuable because it can show active state: running code, injected modules, network sockets, decrypted material, and process relationships that may not exist on disk. It is risky because it is volatile, sensitive, and collection changes the system. Treat memory capture as a scoped forensic action, not a routine curiosity.

Use memory when...Avoid or defer memory when...
There is a credible hypothesis of in-memory malware, active credential theft, or volatile session abuse.The question can be answered from less intrusive logs.
The system can remain stable long enough for approved collection.Collection could cause unacceptable operational harm.
Legal/privacy approval and evidence custody are defined.The team lacks authorization or a documented procedure.
The capture will materially improve containment or scope decisions.The team is collecting it only because “forensics usually does memory.”
Privacy warning: Memory can contain credentials, messages, documents, browser data, personal information, and privileged material. Minimize scope and restrict access.

9. Network Evidence

Network evidence helps responders understand direction, timing, destinations, rough volume, and control paths. It often cannot reveal content because encryption and metadata-only logging are common. That limitation is not a defect; it is a property to document.

SourceEvidence valueLimitation
Flow logsSource, destination, protocol, port, bytes, action, and time.May be sampled or aggregated; usually lacks payload.
DNS logsClient, query, response, resolver, and timing.Encrypted DNS, caching, and shared clients can obscure behavior.
Proxy logsURL or domain access, user, user agent, category, action.May miss direct connections or non-web protocols.
IDS/IPS alertsKnown signatures, suspicious patterns, packet metadata.Alert does not automatically prove compromise.
Packet captureHigh-detail network evidence when scoped and authorized.High privacy sensitivity and storage volume; encryption limits content analysis.

10. Cloud Evidence

Cloud investigations differ from traditional host investigations because many meaningful actions happen through APIs. Control-plane logs answer who changed resources, roles, policies, keys, networks, instances, storage, and logging configuration. Data-plane logs answer whether data objects, records, or service operations were read or written. Both may require explicit enablement, retention planning, and provider-specific interpretation.

Cloud evidence typeTypical questionExamples of fields to preserve
Control-plane audit/activity logsWho created, deleted, modified, or authorized a resource?Principal, API method, resource ID, source IP, user agent, region, result, time.
Data-plane access logsWas data read, copied, written, or deleted?Object/table/resource, actor, operation, bytes, request ID, result, time.
Cloud identity logsHow did the identity authenticate and what roles changed?User/service principal, MFA, conditional access, token/session indicators, role change.
Network flow logsWhich resources communicated and how much?VPC/subnet/interface, source/destination, ports, protocol, bytes, allow/deny, time.
Snapshots and imagesWhat state existed at a point in time?Snapshot ID, source resource, region, collection time, encryption, custody, hash/metadata.
Logging configuration historyWas monitoring disabled or altered?Who changed sinks, trails, diagnostic settings, retention, filters, or storage policy.

12. Containers, Kubernetes, and Ephemeral Workloads

Containers and orchestration platforms create evidence problems because workloads can be short-lived. A pod may be replaced before an analyst opens a ticket. Node autoscaling can remove local traces. Image tags can move. Service accounts can be more important than human usernames.

SourceUseWeak point
Kubernetes audit logsWho requested what resource action in the cluster?Audit policy may omit useful fields.
Container runtime logsWhat did the container output or report?Logs may rotate quickly or be centralized with delay.
Image metadataWhich image digest ran?Tags are mutable; prefer digests when available.
Namespace and RBAC recordsWhich identities could act?Current policy may differ from incident-time policy.
Cloud workload identity eventsWhich workload assumed cloud permissions?Correlation requires cluster, cloud, and identity timestamps.

13. Evidence Selection Matrix

A selection matrix prevents noisy, unbounded collection. Each evidence request should have an incident question, expected answer, source owner, authorization scope, retention risk, and limitation.

FactorWhy it mattersDecision question
RelevanceEvidence should answer a defined question.Which hypothesis will this source confirm or weaken?
VolatilitySome sources disappear quickly.Will the evidence be lost before normal collection?
RetentionLogs may expire or be overwritten.What is the retention window and export deadline?
AuthorizationEvidence collection can invade privacy or exceed role authority.Who approved collection and what is the scope?
IntegrityExported records must be traceable and tamper-evident.How will the team preserve, hash, store, and verify?
InterpretabilityFields can be ambiguous or provider-specific.What does this field actually mean in this service/version?
Business impactCollection or containment can disrupt operations.Can evidence be preserved without increasing harm?

14. Correlation and Timeline Discipline

No single source is perfect. Reliable conclusions usually come from correlation. A cloud API call, sign-in event, endpoint process, and network flow may tell a consistent story, or they may expose gaps. Correlation requires disciplined timestamps and clear source semantics.

  • Record original timezone and normalized UTC time.
  • Separate event time, collection time, ingestion time, and analyst note time.
  • Do not mix alert creation time with the time of the underlying event.
  • Record query filters, API endpoints, page tokens, export format, and tenant/account/region/project scope for cloud evidence.
  • Preserve raw exports before parsing and enrichment.
  • Label each statement as fact, inference, assumption, or open question.

FailureWhy it hurtsBetter practice

16. Evidence Planning Playbook Pattern

A repeatable playbook for evidence planning should include:

  1. Declare the incident question: Example: “Was the admin account used to change cloud storage policy?”
  2. List candidate sources: Identity sign-in logs, cloud audit logs, storage access logs, admin workstation telemetry.
  3. Rank by volatility and retention: Prioritize sources that will disappear soon.
  4. Check authorization: Confirm who can collect, who approves, and what privacy boundaries apply.
  5. Preserve raw evidence: Export raw records, record filters, hash files, and store master copies.
  6. Analyze working copies: Parse, normalize, enrich, and correlate without altering the preserved master.
  7. Record limitations: Missing logs, disabled data-plane logging, sampled flows, unclear user agent, clock skew, or provider-specific ambiguity.
  8. Update the timeline: Add facts, inferences, assumptions, and open questions.

18. Good Reporting Language

Forensic reporting should be precise. Avoid overstating what a source proves.

Weak wordingBetter wording
“The user stole data.”“The SaaS audit log records 247 file download events attributed to the user account between 09:22 and 09:41 UTC. The available logs do not prove who physically controlled the account.”
“There was no exfiltration.”“No exfiltration was observed in the reviewed proxy and flow logs. Data-plane storage logs were not enabled, so this conclusion is limited.”
“Malware ran on the server.”“Endpoint telemetry records process execution for file X with hash Y at 14:03 UTC under account Z. Memory evidence was not collected.”
“The cloud was hacked.”“Cloud audit logs show unauthorized role and storage-policy changes after a suspicious sign-in sequence. Scope is limited to the reviewed account and region.”

Step 2: Create evidence_source_planner.py

import json
from collections import Counter, defaultdict

VOLATILITY_WEIGHT = {
    "very_high": 40,
    "high": 30,
    "medium": 18,
    "low": 8,
}

COMPLEXITY_PENALTY = {
    "low": 0,
    "medium": 6,
    "high": 14,
}

SENSITIVITY_PENALTY = {
    "low": 0,
    "medium": 5,
    "high": 12,
}

RISK_WARNING = {
    "low": "standard custody controls still required",
    "moderate": "record query scope, collector, storage, and hash for exported files",
    "high": "requires explicit approval, minimal scope, documented method, and evidence custodian review",
}


def load_sources(path="evidence_sources.json"):
    with open(path, "r", encoding="utf-8") as handle:
        return json.load(handle)


def score_source(source, incident_question):
    direct_fit = incident_question in source.get("question_fit", [])
    fit_score = 55 if direct_fit else 12
    volatility_score = VOLATILITY_WEIGHT.get(source.get("volatility"), 0)
    retention_days = int(source.get("retention_days") or 0)
    retention_score = 20 if retention_days <= 7 else 12 if retention_days <= 30 else 6
    complexity_penalty = COMPLEXITY_PENALTY.get(source.get("collection_complexity"), 0)
    sensitivity_penalty = SENSITIVITY_PENALTY.get(source.get("privacy_sensitivity"), 0)
    score = fit_score + volatility_score + retention_score - complexity_penalty - sensitivity_penalty
    return max(score, 0)


def recommended_collection_posture(source):
    volatility = source.get("volatility")
    sensitivity = source.get("privacy_sensitivity")
    if volatility in {"very_high", "high"} and sensitivity == "high":
        return "seek approval immediately; preserve only if the incident question justifies high volatility and sensitivity"
    if volatility in {"very_high", "high"}:
        return "prioritize preservation before reboot, shutdown, reimage, or containment that destroys state"
    if source.get("retention_days", 999) <= 7:
        return "export promptly because retention is short"
    return "collect with documented scope, UTC time window, source system, and custody record"


def plan_for_question(sources, incident_question, top_n=5):
    ranked = []
    for source in sources:
        score = score_source(source, incident_question)
        ranked.append({
            "evidence_id": source["evidence_id"],
            "source_name": source["source_name"],
            "source_family": source["source_family"],
            "score": score,
            "volatility": source["volatility"],
            "retention_days": source["retention_days"],
            "authorization_scope": source["authorization_scope"],
            "collection_posture": recommended_collection_posture(source),
            "preservation_warning": RISK_WARNING.get(source["preservation_risk"], "review with evidence custodian"),
            "key_fields": source["key_fields"],
            "limitations": source["limitations"],
        })
    ranked.sort(key=lambda item: (-item["score"], item["retention_days"], item["source_name"]))
    return ranked[:top_n]


def summarize_coverage(sources):
    by_family = Counter(source["source_family"] for source in sources)
    by_question = defaultdict(list)
    for source in sources:
        for question in source.get("question_fit", []):
            by_question[question].append(source["evidence_id"])
    return {
        "source_families": dict(sorted(by_family.items())),
        "questions_with_coverage": {k: sorted(v) for k, v in sorted(by_question.items())},
    }


def main():
    sources = load_sources()
    questions = [
        "account_takeover",
        "malware_execution",
        "lateral_movement",
        "possible_data_exfiltration",
        "cloud_resource_change",
    ]
    report = {
        "coverage_summary": summarize_coverage(sources),
        "plans": {question: plan_for_question(sources, question) for question in questions},
    }
    print(json.dumps(report, indent=2))


if __name__ == "__main__":
    main()

20. How to Interpret the Lab Output

The planner should usually rank cloud audit and identity logs highly for account takeover. It should rank memory evidence highly for malware execution, but also warn that memory requires explicit approval and careful handling. It should use network, cloud, and SaaS sources for possible data exfiltration instead of pretending one source can answer everything.

This lab deliberately uses simple scoring. Real teams should not blindly follow a score. They should use the score to expose tradeoffs, then document the actual decision with scope, authorization, and evidence limitations.
