Lesson 4.4: Secrets, Dependencies, and SBOMs

Modern software is assembled from your code, third-party packages, build tools, containers, cloud services, and secrets. This lesson teaches how to protect secrets, understand dependency risk, and use SBOMs without pretending that an inventory alone solves security.

Estimated reading time280–380 minutes
Difficulty levelBeginner → MS foundation
Required backgroundLessons 0.1–4.3 recommended; no package-management background required
Practice time5–8 hours

Supply-chain security begins with knowing what you use and protecting what gives access.

Secrets unlock systems. Dependencies bring outside code into your system. SBOMs record components so teams can ask better questions: What do we run? Who supplied it? Is it vulnerable? Can we patch it?

Week 4 Context

Lesson 4.1 introduced secure software engineering as a lifecycle. Lesson 4.2 connected requirements to secure design. Lesson 4.3 covered safe input, output, and error handling. Lesson 4.4 focuses on what enters your software project from outside and what must never leak from inside: dependencies and secrets.

Safety boundary: this lesson does not teach how to steal secrets, poison public package registries, or attack real supply chains. Labs use local toy files only.

1. Introduction

Plain-English explanation

Secrets are values that grant access, such as passwords, API keys, private keys, database credentials, session-signing keys, and cloud tokens. Dependencies are external pieces of software your project uses. A Software Bill of Materials, or SBOM, is a structured inventory of software components and their relationships.

Analogy: a restaurant needs to know its ingredients and suppliers, and it must protect keys to the kitchen and cash register. A software team needs to know its code ingredients and protect the keys that unlock systems.

Why the topic exists

Very little modern software is written entirely from scratch. Applications import libraries, frameworks, container images, build plugins, cloud services, and developer tools. These external components increase speed, but they also create dependency, licensing, availability, and security risk.

Why the topic matters

A leaked secret can let an attacker access data or infrastructure. A vulnerable dependency can expose an application even when the team’s own code is well written. An inaccurate dependency inventory can delay emergency response because the organization cannot answer, “Are we affected?”

Where it is used

  • Application development, DevSecOps, cloud operations, procurement, vulnerability management, compliance, and incident response.
  • Package ecosystems such as PyPI, npm, Maven, NuGet, Go modules, RubyGems, crates.io, and container registries.
  • Vendor risk reviews, software acquisition, government systems, healthcare platforms, banking systems, and critical infrastructure.

Who uses it

  • Developers choose libraries, configure dependencies, and remove hardcoded secrets.
  • Security engineers run software composition analysis and define secret-handling rules.
  • Platform teams manage CI/CD credentials, registries, artifact signing, and deployment secrets.
  • Risk and procurement teams ask vendors for SBOMs and supply-chain evidence.
  • Incident responders use inventories to determine exposure during urgent vulnerability events.

2. Learning Outcomes

After finishing this lesson, you should be able to:

  • Describe basic secret lifecycle steps: creation, storage, access, rotation, revocation, and audit.
  • Explain how dependencies enter a project and why transitive dependencies matter.
  • Compare manual dependency inventories, lockfiles, SCA tools, SBOMs, and vulnerability feeds.
  • Explain the difference between vulnerability severity and local organizational priority.

  • Design a safe local lab process for reviewing fake secrets and toy dependencies.

3. Prerequisites

Source code

Source code is human-readable instructions written by programmers. It is usually stored in a version-control repository such as Git.

Package

A package is a reusable unit of software distributed through a package manager or registry. It may contain code, metadata, documentation, tests, and build instructions.

Build

A build turns source code and dependencies into a runnable artifact, such as a binary, container image, web bundle, or deployable application.

Hash

A hash is a fixed-size fingerprint of data. Hashes can help identify files and detect accidental or malicious changes, but only when the expected hash is trusted.

Trust boundary

A trust boundary is a point where assumptions change. A package downloaded from the internet crosses a trust boundary before becoming part of your project.

4. Core Concepts

4.1 Secret

A secret is any value that must remain confidential because it grants access or protects integrity. Examples include passwords, API keys, private keys, OAuth client secrets, database URLs with credentials, cloud access tokens, signing keys, session secrets, and encryption keys.

Intuition: if another person could use the value to impersonate a user, access a system, decrypt data, modify infrastructure, or sign software, treat it as a secret.

4.2 Secret leakage

Secret leakage happens when a secret appears somewhere it should not: source code, Git history, terminal screenshots, CI logs, error reports, Slack messages, support tickets, browser bundles, mobile apps, or public repositories. Removing the visible line is not always enough because Git history, logs, forks, caches, and backups may preserve the value.

4.3 Secret lifecycle

A mature secret lifecycle includes:

  1. Generate: create a high-entropy value using approved tooling.
  2. Store: keep it in a secret manager, key vault, hardware-backed store, or protected environment.
  3. Distribute: provide it only to authorized workloads or people.
  4. Use: read it at runtime without printing or committing it.
  5. Rotate: replace it periodically or after suspected exposure.
  6. Revoke: disable it when no longer needed or compromised.
  7. Audit: record access and changes.

4.4 Dependency

A dependency is software your project relies on. A direct dependency is one you explicitly choose, such as a package in requirements.txt or package.json. A transitive dependency is brought in by another dependency. Transitive dependencies can be invisible to beginners but still run inside the application.

4.5 Package manager and registry

A package manager installs and updates dependencies. A package registry stores packages. Examples include pip/PyPI for Python, npm for JavaScript, Maven Central for Java, NuGet for .NET, and container registries for images.

Common risk: the team may trust a package because it is easy to install, not because it has been reviewed, maintained, and verified.

4.6 Lockfile

A lockfile records exact dependency versions selected by the package manager. It improves reproducibility because future installs can use the same dependency versions. A lockfile is not a security guarantee by itself, but it helps teams know what they intended to install.

4.7 Software Composition Analysis

Software Composition Analysis, or SCA, identifies third-party components and checks them against vulnerability, license, and policy data. SCA is useful, but it can produce false positives, miss hidden components, or report vulnerabilities that are not reachable in a specific application.

4.8 SBOM

An SBOM is a formal inventory of software components and relationships. A useful SBOM may include component names, versions, suppliers, dependency relationships, hashes, licenses, package URLs, and metadata about how the SBOM was created.

Important limitation: an SBOM tells you what is inside or claimed to be inside. It does not automatically prove that components are safe, up to date, reachable, correctly configured, or free of malicious behavior.

4.9 SPDX and CycloneDX

SPDX and CycloneDX are common SBOM standards. SPDX has strong roots in licensing and component metadata. CycloneDX is widely used for security-oriented bill-of-materials use cases. The choice depends on ecosystem, tooling, regulatory need, and information-sharing workflow.

4.10 VEX

VEX, or Vulnerability Exploitability eXchange, is a structured way for a producer to state whether a known vulnerability affects a product. This matters because a vulnerable component may be present but not reachable or not used in the vulnerable configuration. VEX should be evidence-backed; it should not be used as a shortcut to ignore vulnerability management.

4.11 Provenance and integrity

Provenance is information about where software came from and how it was built. Integrity means the artifact has not been altered unexpectedly. Hashes, signatures, attestations, reproducible builds, and protected CI/CD pipelines support provenance and integrity.

4.12 Dependency risk is more than CVSS

Severity scores are useful, but local priority also depends on exposure, exploit availability, whether the vulnerable code path is used, privileges, compensating controls, patch availability, business impact, and whether the dependency is direct or buried deep in the tree.

4.13 Secrets and dependencies interact

A build plugin may read CI secrets. A dependency installation script may execute code. A leaked package-token may let an attacker publish malicious updates. A compromised maintainer account may affect thousands of downstream projects. For this reason, software supply-chain security must connect identity, secrets, build systems, dependencies, and monitoring.

7. Mathematical Foundations

7.1 Entropy for secrets

Entropy measures uncertainty. If a random token can be one of N equally likely values, its entropy is:

H = log₂(N) bits.

If a secret is 16 random bytes, then there are 2^(8×16) = 2^128 possible values. That is 128 bits of entropy, assuming the random generator is cryptographically secure.

Worked example

A six-digit one-time code has one million possibilities:

log₂(1,000,000) ≈ 19.93 bits

This is why short codes need rate limiting and expiration. Their security does not come from high entropy alone.

7.2 Dependency graph size

Dependencies form a graph. If an application has 8 direct dependencies and each direct dependency brings in an average of 6 unique transitive dependencies, a rough upper estimate is:

Total ≈ 1 application + 8 direct + 8×6 transitive = 57 components

Real graphs can share dependencies, so this is an estimate, not a precise count.

7.3 Prioritization model

A simple teaching model can combine severity and context:

Priority = min(10, Severity + Exposure + Privilege + FixBonus)

This is not CVSS. It is a local triage model to illustrate that risk decisions need context beyond a public severity score.

Python examples

This safe script searches only local files for fake secret patterns and redacts suspected values before printing.

# Safe local example: simple defensive secret-pattern review.
# This script scans only files in the folder where you run it.
# It is not a replacement for professional secret-scanning tools.

from pathlib import Path
import re

# 1. Define patterns that look like common accidental secrets.
#    These are intentionally broad and may produce false positives.
PATTERNS = {
    "possible_api_key_assignment": re.compile(r"(?i)(api[_-]?key|token|secret)\s*=\s*['\"][^'\"]{8,}['\"]"),
    "possible_private_key_header": re.compile(r"-----BEGIN (RSA |EC |OPENSSH )?PRIVATE KEY-----"),
    "possible_password_assignment": re.compile(r"(?i)password\s*=\s*['\"][^'\"]{6,}['\"]"),
}

# 2. Define folders that should not be scanned in this beginner lab.
SKIP_DIRS = {".git", "__pycache__", ".venv", "node_modules"}


def should_skip(path: Path) -> bool:
    """Return True when the path is inside a directory we do not want to scan."""
    return any(part in SKIP_DIRS for part in path.parts)


def scan_file(path: Path) -> list[tuple[int, str, str]]:
    """Scan one text file and return findings as line number, pattern name, and line."""
    findings = []
    try:
        text = path.read_text(encoding="utf-8")
    except UnicodeDecodeError:
        return findings

    for line_number, line in enumerate(text.splitlines(), start=1):
        for name, pattern in PATTERNS.items():
            if pattern.search(line):
                # 3. Redact the line before printing. Do not expose the suspected secret.
                findings.append((line_number, name, "[redacted suspected secret line]"))
    return findings


root = Path(".")
for path in root.rglob("*"):
    if path.is_file() and not should_skip(path):
        for line_number, name, preview in scan_file(path):
            print(f"{path}:{line_number}: {name}: {preview}")

Network example

A web service may call an external API using a bearer token. The token belongs in a secret manager or protected environment variable, not in JavaScript sent to the browser, not in a URL query string, and not in access logs.

Web example

A server-rendered web app may depend on a template engine. If the template engine has a vulnerability, the team needs to know whether that version is present, whether the vulnerable feature is used, whether untrusted users can reach it, and whether a fixed version is available.

Cloud example

A cloud function should read secrets from a managed secret service at runtime using a workload identity. The CI/CD pipeline should not print secrets, and the deployment artifact should not contain plaintext credentials.

9.2 Educational SBOM-like inventory

# Safe local example: create a tiny educational SBOM-like inventory.
# This does not contact the internet and does not claim to be a complete SBOM.

import json
from pathlib import Path

# 1. A toy requirements file for a fictional local project.
requirements = Path("requirements.txt")
requirements.write_text("""
flask==3.0.0
requests==2.32.0
pytest==8.2.0
""".strip() + "\n", encoding="utf-8")

# 2. Read the declared direct dependencies.
components = []
for line in requirements.read_text(encoding="utf-8").splitlines():
    if "==" not in line:
        continue
    name, version = line.split("==", 1)
    components.append({
        "type": "library",
        "name": name.strip(),
        "version": version.strip(),
        "scope": "direct",
        "supplier": "unknown in this lab",
    })

# 3. Build a minimal educational inventory.
inventory = {
    "bomFormat": "Educational-SBOM",
    "specVersion": "0.1",
    "metadata": {
        "component": {
            "type": "application",
            "name": "fictional-student-portal",
            "version": "0.0.1"
        }
    },
    "components": components,
    "limitations": [
        "Does not include transitive dependencies.",
        "Does not include hashes, licenses, package URLs, or provenance.",
        "Does not check vulnerabilities."
    ]
}

Path("educational-sbom.json").write_text(json.dumps(inventory, indent=2), encoding="utf-8")
print(json.dumps(inventory, indent=2))

Line-by-line reasoning: the script creates a toy dependency file, reads direct dependencies, splits each pinned requirement into name and version, builds a JSON inventory, writes it to disk, and records limitations. It deliberately states what it does not do because honest security evidence must describe scope and gaps.

9.3 Contextual dependency prioritization

# Safe local example: prioritize dependency risk using fictional data.
# Do not treat this as a real vulnerability scanner.

components = [
    {"name": "alpha-lib", "version": "1.0", "severity": 9.8, "internet_exposed": True, "privileged": False, "fix_available": True},
    {"name": "beta-tool", "version": "2.3", "severity": 7.5, "internet_exposed": False, "privileged": False, "fix_available": True},
    {"name": "gamma-core", "version": "4.1", "severity": 6.5, "internet_exposed": True, "privileged": True, "fix_available": False},
]


def priority_score(item: dict) -> float:
    """Combine severity with local context. This is a teaching model, not CVSS."""
    score = item["severity"]
    if item["internet_exposed"]:
        score += 1.0
    if item["privileged"]:
        score += 1.0
    if item["fix_available"]:
        score += 0.5
    return min(score, 10.0)

for component in sorted(components, key=priority_score, reverse=True):
    print(component["name"], component["version"], "priority", priority_score(component))

Line-by-line reasoning: each component has fictional context fields. The scoring function adds context to public severity, caps the score at 10, then sorts components. The lesson is not the exact formula; the lesson is that vulnerability priority changes when exposure, privilege, exploitability, and fix availability change.

Common coding mistakes

  • Putting API_KEY = "real-value" in source code.
  • Logging full authorization headers.
  • Using broad version ranges without testing updates.
  • Assuming a lockfile proves safety.
  • Treating all SCA findings as equal without reachability or exposure analysis.
  • Generating an SBOM once and never updating it.

10. Security Perspective

Common attacks

  • Secret leakage: credentials exposed through source code, logs, screenshots, or misconfigured storage.
  • Dependency confusion: a build tool fetches a package from the wrong registry or namespace.
  • Typosquatting: a malicious package uses a name similar to a popular package.
  • Maintainer compromise: an attacker gains control of a legitimate maintainer account or release process.
  • Malicious update: a package update introduces harmful behavior.
  • Build-pipeline compromise: CI/CD credentials or build steps are abused to alter releases.

Common vulnerabilities

  • Hardcoded secrets.
  • Overprivileged CI tokens.
  • Unpinned or poorly reviewed dependencies.
  • Missing lockfiles or unreviewed lockfile changes.
  • No inventory of deployed components.
  • No process for emergency patching.

Defensive strategies

  • Use secret managers and workload identity where possible.
  • Apply least privilege to developer, CI/CD, and runtime credentials.
  • Run secret scanning before commit and in CI.
  • Pin versions and review dependency updates.
  • Use SCA and generate SBOMs per release.
  • Verify artifact integrity with signatures or trusted hashes.
  • Monitor vulnerability feeds and maintain patch playbooks.

Detection methods

  • Repository secret-scanning alerts.
  • Unexpected package-version changes.
  • CI logs showing secret-like strings.
  • Cloud audit logs showing unusual token use.
  • SCA findings for new vulnerabilities in deployed components.
  • Runtime alerts for unexpected outbound traffic from applications.

Mitigation and best practices

  • When a secret leaks: revoke it, rotate dependent credentials, search logs and history, assess access, and preserve evidence.
  • When a dependency is vulnerable: identify affected versions, check deployed inventory, test fixed versions, mitigate exposure, and record residual risk.
  • When an SBOM is incomplete: document limitations, improve generation at build time, and compare multiple sources when necessary.

11. Industry Perspective

  • Banks: manage strict third-party risk, protect payment credentials, demand vendor evidence, and prioritize dependencies affecting internet-facing systems.
  • Governments: require supply-chain transparency, vulnerability handling, procurement controls, and evidence for software assurance.
  • Cloud providers: operate secret managers, identity platforms, container registries, artifact signing services, and vulnerability scanning at large scale.
  • Healthcare: must protect patient data while managing vendor software, medical-device software, and emergency patching constraints.
  • Defense: emphasizes provenance, tamper resistance, classified boundaries, controlled build environments, and supplier assurance.
  • Universities: often run open research systems, student projects, public repositories, and mixed-maturity software, making secret leakage and vulnerable dependencies common risks.
  • Large enterprises: centralize SCA, secret scanning, dependency approval, SBOM intake, vendor risk, and patch governance across thousands of applications.

12. Research Perspective

Current research studies whether SBOMs accurately represent real software, how to identify hidden or vendored dependencies, how to combine SBOMs with VEX and reachability analysis, how to protect sensitive SBOM contents, and how to verify provenance across registries, mirrors, proxies, and air-gapped environments.

Open problems include package identity, dependency graph completeness, malicious-package detection, maintainer-account security, secure build reproducibility, privacy-preserving SBOM sharing, vulnerability reachability, AI-generated dependency risk, and usability of signing and verification tools.

Research directions

  • Can SBOM generation be made complete and reproducible across ecosystems?
  • How can tools distinguish “component present” from “vulnerable code reachable”?
  • How can organizations share SBOM evidence without exposing sensitive architecture?
  • How can cryptographic provenance survive mirrors, proxies, and offline distribution?
  • How should dependency risk change for AI/ML packages and generated code?

13. Case Studies

Log4Shell and dependency visibility

What happened: a critical vulnerability in Apache Log4j forced organizations to determine quickly whether they used affected versions. Root cause: a vulnerable logging component was widely embedded across applications and products. Impact: many teams struggled to find where Log4j existed in direct, transitive, or vendor-supplied software. Lesson: inventories, SBOMs, and deployment visibility matter most during urgent response.

SolarWinds supply-chain compromise

What happened: malicious code was inserted into legitimate software updates. Root cause: a compromised build/release chain produced trusted-looking artifacts. Impact: customers installed updates that became part of a broader espionage campaign. Lesson: dependency lists are not enough; build integrity, provenance, signing, monitoring, and vendor assurance matter.

GitHub secret exposure patterns

What happened: developers frequently commit tokens, keys, or credentials by accident. Root cause: local convenience, poor secret storage, weak review, and missing scanning. Impact: exposed tokens can be used quickly for cloud abuse, data access, or repository modification. Lesson: prevention, detection, and fast rotation are all necessary.

Event-stream package compromise

What happened: a popular JavaScript package chain included malicious code through a dependency-maintenance compromise. Root cause: trust in package maintainers and transitive dependencies. Impact: downstream applications inherited risk from code they did not directly write. Lesson: maintainer trust, package review, version control, and monitoring matter.

14. Common Misconceptions

  • “A secret in a private repository is safe.” Private repositories still have users, integrations, backups, forks, logs, and compromise risk.
  • “Deleting a secret from the latest commit fixes the leak.” The secret may still exist in history, logs, caches, forks, and third-party systems. Rotate or revoke it.
  • “Open source is automatically safer because many people can inspect it.” Visibility helps, but inspection is uneven and maintainers can be compromised.
  • “Pinned versions are always secure.” Pinning improves reproducibility but can freeze vulnerable versions.
  • “SBOM means secure.” SBOM means inventory. Security still requires analysis, patching, verification, and operations.
  • “A vulnerability in a dependency always means the product is exploitable.” It may be present but unreachable; evidence is needed.
  • “SCA tools give final answers.” They provide signals. Teams must validate applicability and priority.
  • “Transitive dependencies are someone else’s problem.” If they run in your product, they are part of your risk.

15. Summary

Secrets are access-bearing values and must be generated, stored, used, rotated, revoked, and audited carefully. Dependencies are external code that become part of your software risk. SBOMs provide structured component visibility, but they do not automatically prove safety. Effective supply-chain security combines secret management, SCA, SBOMs, lockfiles, provenance, signing, vulnerability prioritization, and disciplined incident response.

16. Glossary

Artifact
A built software output such as a binary, package, image, or release file.
API Key
A token used by software to authenticate to an API.
Dependency
External software that a project relies on.
Direct Dependency
A dependency explicitly declared by the project.
Transitive Dependency
A dependency brought in by another dependency.
Lockfile
A file recording exact dependency versions selected for reproducible installation.
Package Registry
A service that stores and distributes software packages.
SCA
Software Composition Analysis; tooling and process for identifying software components and associated risks.
SBOM
Software Bill of Materials; a structured inventory of software components and relationships.
SPDX
A standard for representing software component, license, and security metadata.
CycloneDX
A bill-of-materials standard focused on supply-chain and security use cases.
VEX
Vulnerability Exploitability eXchange; structured statements about whether vulnerabilities affect products.
Provenance
Evidence about where software came from and how it was produced.
Secret Manager
A system for storing, controlling, rotating, and auditing access to secrets.
Rotation
Replacing an old secret with a new one.
Revocation
Disabling a secret or credential so it can no longer be used.
