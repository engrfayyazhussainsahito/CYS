Lesson 4.1: Secure Software Engineering from First Principles

Software security begins before anyone writes code. It starts with understanding what software is supposed to do, what could go wrong, who can change it, how it is tested, how it is deployed, and how it is maintained after release.

Estimated reading time220–320 minutes
Difficulty levelBeginner → MS foundation
Required backgroundWeeks 0–3 recommended; no software-engineering background required
Practice time5–8 hours

Security is built into the lifecycle.

A secure product is not created by one final scan. It comes from repeated decisions: what is required, how the system is designed, how code is reviewed, how dependencies are selected, how tests run, how releases are protected, and how incidents feed back into engineering.

BuildRequirements, design, code, dependencies.
VerifyReview, test, scan, threat model.
OperateDeploy, monitor, patch, learn.

Week 4 Context

Week 4 moves from cryptographic building blocks into software engineering. Most modern cybersecurity incidents do not happen because nobody knew the word “security.” They happen because requirements were vague, code handled input unsafely, secrets leaked, dependencies were trusted blindly, build systems were weak, or nobody owned security after release.

This lesson introduces secure software engineering from zero. Later Week 4 lessons will go deeper into secure requirements, input validation, dependency management, software bills of materials, testing, fuzzing, CI/CD, and release integrity.

Boundary: all examples in this lesson are defensive and local. Do not scan public repositories, test third-party applications, or search for real secrets in systems you do not own or have written authorization to assess.

1. Introduction

Plain-English explanation

Software engineering is the organized process of building and maintaining software. Secure software engineering means building software so that it still behaves safely when users make mistakes, attackers send hostile input, components fail, dependencies change, or administrators misconfigure something.

Analogy: building software is like building a hospital. It is not enough for the doors to open and lights to work. The building also needs fire exits, restricted medicine storage, emergency power, hygiene rules, staff badges, maintenance records, and incident drills. Secure software needs the digital equivalent: access control, input validation, logging, safe error handling, dependency control, tests, monitoring, and patching.

Why the topic exists

Software controls banking, healthcare, cloud platforms, vehicles, universities, identity systems, industrial systems, and national infrastructure. A small code defect or design mistake can leak data, corrupt records, interrupt service, or become a stepping stone into a larger environment.

Why the topic matters

Security teams cannot inspect every line of software after it is already deployed. The lower-cost and higher-impact approach is to reduce defects early, make risky decisions visible, automate checks where possible, and design systems that are easier to review and recover.

Where it is used

  • Web and mobile applications.
  • Cloud services and APIs.
  • Operating-system and embedded software.
  • Banking, healthcare, education, government, and defense systems.
  • Open-source projects and commercial products.
  • CI/CD pipelines and software supply chains.

Who uses it

  • Developers use it to avoid introducing vulnerabilities.
  • Security engineers use it to define secure patterns, review code, and build guardrails.
  • DevOps and platform engineers use it to secure builds, deployments, and runtime environments.
  • Product managers use it to prioritize security requirements and risk reduction.
  • Auditors and governance teams use it to evaluate process evidence.
  • Researchers use it to study vulnerability classes, developer behavior, automated testing, and supply-chain assurance.

2. Learning Outcomes

After finishing this lesson, you should be able to:

  • Explain software, programs, source code, binaries, builds, releases, and deployments from first principles.
  • Describe the software development life cycle and where security controls fit.
  • Distinguish functional requirements, non-functional requirements, security requirements, abuse cases, and acceptance criteria.
  • Explain why secure software is a lifecycle problem rather than a final checklist.
  • Identify common software vulnerability classes at a beginner level.
  • Explain secure design principles such as least privilege, deny by default, fail safely, complete mediation, secure defaults, and separation of duties.
  • Perform a safe local review of a small program for input validation, error handling, logging, and secret-handling issues.
  • Write a simple secure coding checklist and map it to development stages.
  • Describe how secure software engineering is used in banks, governments, cloud providers, healthcare, defense, universities, and enterprises.
  • Summarize research directions in secure software engineering and software supply-chain security.

3. Prerequisites

This lesson assumes no previous software-engineering course. It re-explains the required concepts here.

Program

A program is a set of instructions a computer can run. A calculator app, web server, mobile app, and login script are all programs.

Source code

Source code is human-readable program text written in a programming language such as Python, JavaScript, Java, Go, Rust, or C. Humans write source code. Tools translate or interpret it so computers can execute it.

Input and output

Input is data the program receives: keyboard text, files, web requests, API parameters, environment variables, database records, or network packets. Output is what the program produces: screen text, files, HTTP responses, database updates, logs, or messages.

Bug

A bug is a mistake in software behavior. Not every bug is a security vulnerability. A spelling mistake in a label may be a bug, but usually not a vulnerability. A bug that lets one student read another student’s grades is a security vulnerability.

Vulnerability

A vulnerability is a weakness that can be used to violate a security goal such as confidentiality, integrity, availability, privacy, or safety.

Repository

A repository stores project files and their history, usually managed by Git. It helps teams track changes, review code, and recover older versions.

Build

A build converts source code and dependencies into something that can run: a binary, container image, package, or deployable web artifact.

Deployment

A deployment places software into an environment where users or other systems can use it, such as a web server, cloud service, mobile app store, or internal network.

Mini-explanation: why “it works” is not enough

A program can work correctly for friendly input and fail dangerously for hostile input. Secure engineering asks: What if the input is too long, missing, malformed, repeated, encoded strangely, sent too fast, or submitted by the wrong identity?

4. Core Concepts

4.1 Software as a system of decisions

Beginners often think software is only code. In professional security, software is a system of decisions:

  • What should the software do?
  • Who is allowed to use each function?
  • What data is accepted?
  • Where are secrets stored?
  • Which third-party packages are trusted?
  • How is code reviewed?
  • How are builds protected?
  • How are logs created and monitored?
  • How are defects patched after release?

Each decision can reduce risk or create risk.

4.2 The software development life cycle

The software development life cycle, often shortened to SDLC, is the process used to create and maintain software. Different organizations use different models, but most include these stages:

StagePlain meaningSecurity questionExample control

4.3 Secure SDLC

A secure SDLC adds security activities to the ordinary SDLC. The goal is not to slow development for no reason. The goal is to catch expensive security mistakes when they are still cheaper to fix.

Example: finding a missing authorization check during design may require one diagram change. Finding it after breach investigation may require emergency patching, incident response, customer notification, legal review, and reputational repair.

4.4 Requirements

A requirement states what the system must do. A functional requirement describes a feature. A security requirement describes how the system must behave to preserve security goals.

Weak requirementBetter requirementWhy better?
The system should be secure.Only authenticated students may view their own grade records.Specific asset, subject, object, and action.
Use encryption.All grade records must be encrypted in transit with TLS and at rest using managed keys.Defines where and how protection applies.
Log errors.Authentication failures, authorization denials, admin changes, and grade updates must be logged with timestamp, actor, object, action, result, and request ID.Supports detection and investigation.

4.5 Abuse cases

A use case describes how a legitimate user uses a feature. An abuse case describes how someone could misuse the feature to cause harm.

Use case: “A student downloads their own transcript.” Abuse case: “A student changes the student ID in the URL and downloads someone else’s transcript.”

4.6 Secure design principles

Secure design principles are reusable ideas that help systems remain safe under stress.

PrincipleMeaningExample
Least privilegeGive only the access needed to perform a task.A reporting job can read reports but cannot modify grades.
Deny by defaultIf access is not explicitly allowed, reject it.Unknown API role receives no access.
Secure defaultsThe safest configuration should be the default.New storage bucket is private unless explicitly published.
Fail safelyWhen an error occurs, do not expose sensitive data or grant extra access.If authorization service is unavailable, deny grade changes.
Complete mediationCheck authorization every time a protected object is accessed.Every transcript download verifies ownership.

Defense in depthUse multiple independent controls.Authentication, authorization, logging, rate limits, and backups.
Minimize attack surfaceRemove unnecessary interfaces and features.Disable unused admin endpoints.

4.7 Common vulnerability classes

A vulnerability class is a family of similar security mistakes. Knowing classes helps you recognize patterns.

  • Injection: untrusted input is interpreted as code or command.
  • Broken access control: a user can perform an action or access an object they should not.
  • Authentication weakness: login or identity proof can be bypassed or abused.
  • Cryptographic failure: sensitive data is exposed because crypto is missing or misused.
  • Insecure design: the feature itself has unsafe assumptions.
  • Security misconfiguration: software is deployed with unsafe settings.
  • Vulnerable dependency: a third-party component contains a known weakness.
  • Insufficient logging and monitoring: attacks occur but are not noticed quickly.

4.8 Secure coding

Secure coding means writing code that resists common mistakes. It includes input validation, output encoding, authorization checks, safe error handling, secure defaults, secret management, and careful dependency use.

Secure coding does not mean the developer must be afraid of every line. It means the developer uses safe patterns and avoids known dangerous shortcuts.

4.9 Code review

Code review is a human inspection of code before it is merged. Security-aware review asks questions such as:

  • Can input reach a sensitive operation?
  • Is authorization checked on every protected object?
  • Are secrets hardcoded?
  • Are errors exposing sensitive information?
  • Are logs useful without leaking private data?
  • Does this dependency add unnecessary risk?

4.10 Automated analysis

Automated tools help, but they are not complete security proof.

Tool typeWhat it checksLimitation

4.11 DevSecOps

DevSecOps means integrating security into development and operations workflows. It is not a tool name. It is a way of making secure behavior easier for teams by embedding checks, templates, guardrails, and feedback into normal engineering work.

4.12 Software supply chain

The software supply chain includes everything used to create, build, test, package, distribute, and run software: developers, source repositories, dependencies, package managers, CI runners, build scripts, artifacts, containers, registries, deployment tools, and cloud infrastructure.

Supply-chain security asks: Can someone tamper with source code, dependencies, build systems, or release artifacts before users receive the software?

4.13 SBOM

A software bill of materials, or SBOM, is an inventory of software components. It is like an ingredient list for software. It helps teams answer: “Are we using a vulnerable component?”

An SBOM does not automatically make software secure. It improves visibility. Security still requires prioritization, patching, verification, and risk management.

4.14 Secure by design

Secure by design means security is treated as a product responsibility from the beginning, not pushed entirely onto customers or administrators after release. It emphasizes safe defaults, reducing whole classes of vulnerabilities, and making the secure path the easiest path.

4.15 Security debt

Security debt is accumulated risk from postponed security work: old dependencies, missing tests, temporary exceptions, weak architecture, unclear ownership, or undocumented assumptions. Like financial debt, it can grow until it becomes expensive to fix.

7. Mathematical Foundations

Secure software engineering uses simple quantitative reasoning. Numbers do not replace judgment, but they help compare risks and track improvement.

7.1 Expected loss

A basic risk estimate can be written as:

Expected Loss = Probability of Event × Impact of Event

Example: a team estimates a 5% annual chance that an authorization flaw exposes grade records. Estimated impact is $200,000 in investigation, notification, and remediation costs.

Expected Loss = 0.05 × 200,000 = 10,000 dollars per year

This does not prove the exact future. It helps compare whether a control costing $2,000 and reducing the probability significantly may be justified.

7.2 Defect density

Defect density is often measured as defects per thousand lines of code, abbreviated KLOC.

Defect Density = Number of Confirmed Defects / Thousand Lines of Code

If a project has 40 confirmed defects in 20,000 lines of code:

20,000 lines = 20 KLOC
Defect Density = 40 / 20 = 2 defects per KLOC

Security caution: fewer reported defects does not always mean safer software. It may also mean weak testing or low reporting.

7.3 Defect removal efficiency

Defect removal efficiency, or DRE, estimates the percentage of defects found before release.

DRE = Defects found before release / (Defects found before release + Defects found after release)

If 90 defects are found before release and 10 after release:

DRE = 90 / (90 + 10) = 0.90 = 90%

Security interpretation: teams want more serious defects found before release, but they must avoid hiding or reclassifying post-release vulnerabilities to make metrics look better.

7.4 Test coverage

Test coverage measures how much code or behavior is exercised by tests.

Line Coverage = Executed Lines / Total Lines

If tests execute 800 lines out of 1,000:

Line Coverage = 800 / 1000 = 80%

Security caution: 100% line coverage can still miss security flaws if the tests do not include hostile input, authorization checks, race conditions, and abuse cases.

7.5 Mean time to remediate

Mean time to remediate, or MTTR, measures average time to fix confirmed issues.

MTTR = Sum of remediation times / Number of issues

If three vulnerabilities take 2, 5, and 8 days to fix:

MTTR = (2 + 5 + 8) / 3 = 5 days

7.6 Prioritization formula

A simple internal prioritization score can combine severity, exposure, asset importance, and exploit evidence:

Priority = Severity × Exposure × Asset Criticality × Exploit Evidence

Example scale: each factor ranges from 1 to 5.

Severity = 4
Exposure = 5
Asset Criticality = 5
Exploit Evidence = 3
Priority = 4 × 5 × 5 × 3 = 300

This is not a universal standard. It is a teaching formula to show that risk is contextual. A severe bug in an unreachable lab system may be lower priority than a medium-severity bug in an internet-facing payment system.

8.3 Python example: safe input validation

This toy function validates a student ID format. It does not contact any real system.

import re

# Compile a regular expression once.
# This pattern allows exactly: S, followed by 6 digits.
STUDENT_ID_PATTERN = re.compile(r"^S\d{6}$")

def is_valid_student_id(value: str) -> bool:
    """Return True only when value matches the approved student ID format."""
    if not isinstance(value, str):
        # Reject non-string input instead of converting unexpected objects silently.
        return False

    if STUDENT_ID_PATTERN.fullmatch(value) is None:
        # fullmatch means the entire string must match, not just part of it.
        return False

    return True

# Safe local tests
examples = ["S123456", "123456", "S123", "S123456; DROP TABLE grades", "S000001"]
for item in examples:
    print(item, is_valid_student_id(item))

Common mistake: using a weak check such as if "S" in value. That checks whether the letter appears somewhere, not whether the input has the expected structure.

8.4 Network example

In a safe lab, network-facing software should be treated as exposed to untrusted input. Even a simple HTTP endpoint receives strings controlled by clients.

Client request
  method: GET
  path: /grades?student=S123456
  header: User-Agent: anything the client wants
  source: untrusted network

Secure design question:
  Is the authenticated user allowed to access this exact student record?

8.5 Web example

Unsafe output can create cross-site scripting risk. When displaying untrusted text in HTML, encode it rather than inserting it as raw HTML.

import html

comment = "<script>alert('not safe')</script>"

# Escape special characters before placing untrusted text into HTML.
safe_comment = html.escape(comment)

print(safe_comment)
# Output: &lt;script&gt;alert(&#x27;not safe&#x27;)&lt;/script&gt;

8.6 Cloud example

Cloud software often depends on managed identities, secrets managers, storage permissions, network rules, and CI/CD pipelines. A secure cloud release asks:

  • Which identity deploys the software?
  • Which identity runs the software?
  • Where are secrets stored?
  • Can the build artifact be verified?
  • Are logs available for security investigation?
  • Can vulnerable images or packages be blocked before deployment?

9. Code Examples

9.1 Insecure example: hardcoded secret

# Do not use this pattern in real applications.
# The secret is stored directly in source code.
API_KEY = "demo-realistic-looking-key-123"

def call_service():
    # Anyone who can read the repository can read the key.
    print("Calling service with", API_KEY)

Why this is dangerous: source repositories are copied, forked, backed up, cached, indexed, reviewed, and sometimes leaked. Once a real secret is committed, deleting it from the latest version is not enough. The secret must be revoked and rotated.

9.2 Better local teaching pattern: read from environment variable

import os

# Read a secret from the environment instead of hardcoding it.
# This example is still simplified; production systems often use a secrets manager.
API_KEY = os.environ.get("DEMO_API_KEY")

if API_KEY is None:
    # Fail closed: do not continue without required configuration.
    raise RuntimeError("DEMO_API_KEY is not set")

# Do not print secrets.
print("API key loaded from environment")

Line-by-line explanation:

  • import os loads Python’s operating-system interface module.
  • os.environ.get reads an environment variable.
  • The code checks whether the variable exists before continuing.
  • The code prints only a safe status message, not the secret value.

9.3 Defensive code review helper

This script searches only the current local lab folder for teaching patterns. It is not a professional scanner and must not be used against systems you do not own.

from pathlib import Path
import re

# Search patterns for local teaching only.
PATTERNS = {
    "possible_password": re.compile(r"password\s*=\s*['\"].+['\"]", re.IGNORECASE),
    "possible_api_key": re.compile(r"api[_-]?key\s*=\s*['\"].+['\"]", re.IGNORECASE),
    "debug_true": re.compile(r"debug\s*=\s*True", re.IGNORECASE),
}

# Only scan files in this folder.
root = Path.cwd()

for path in root.glob("*.py"):
    text = path.read_text(encoding="utf-8", errors="replace")
    for line_number, line in enumerate(text.splitlines(), start=1):
        for name, pattern in PATTERNS.items():
            if pattern.search(line):
                print(f"{path}:{line_number}: {name}: {line.strip()}")

Secure coding lesson: automated pattern matching can help find obvious mistakes, but it cannot understand all context. Human review and secret rotation are still required when a real secret is exposed.

9.4 Safe error handling

import logging

logging.basicConfig(level=logging.INFO)

class AuthorizationError(Exception):
    """Raised when a user is not allowed to perform an action."""

def read_grade(requesting_user_id: str, target_student_id: str) -> str:
    # Complete mediation: check authorization for this exact object access.
    if requesting_user_id != target_student_id:
        logging.warning(
            "authorization_denied action=read_grade actor=%s target=%s",
            requesting_user_id,
            target_student_id,
        )
        # Return a generic error to the caller.
        # Do not reveal whether the target record exists.
        raise AuthorizationError("Access denied")

    return "Demo grade record"

try:
    print(read_grade("S123456", "S654321"))
except AuthorizationError as error:
    print(str(error))

Why this matters: logs should help defenders investigate, but user-facing errors should avoid leaking sensitive details.

10. Security Perspective

Common attacks

  • Input-based attacks: hostile input attempts to trigger injection, parsing errors, memory bugs, or business-logic flaws.
  • Authorization attacks: attacker accesses another user’s object or performs a restricted action.
  • Dependency attacks: attacker abuses a vulnerable or malicious third-party package.
  • Secret exposure: API keys, tokens, or passwords leak through source code, logs, screenshots, or build output.
  • Build tampering: attacker changes source, dependencies, or artifacts before release.
  • Misconfiguration: debug mode, permissive CORS, public storage, weak headers, or overly broad cloud roles reach production.

Common vulnerabilities

  • Missing input validation.
  • Missing object-level authorization.
  • Hardcoded credentials.
  • Unsafe error messages.
  • Verbose logs containing secrets or personal data.
  • Outdated dependencies.
  • Weak session handling.
  • Insecure defaults.
  • Unprotected CI/CD secrets.

Defensive strategies

  • Write specific security requirements.
  • Threat model high-risk features before implementation.
  • Use safe framework defaults.
  • Perform peer review for security-sensitive code.
  • Use automated checks in CI/CD.
  • Maintain dependency inventories.
  • Protect secrets with secret managers and least privilege.
  • Sign and verify release artifacts where appropriate.
  • Log security-relevant events consistently.

Detection methods

  • SAST findings for unsafe patterns.
  • SCA alerts for vulnerable dependencies.
  • Secret-scanning alerts.
  • DAST findings against test environments.
  • Runtime logs showing authorization denials, unusual error rates, or unexpected input patterns.
  • CI/CD audit logs showing unusual build or release activity.

Mitigation

  • Patch vulnerable code and dependencies.
  • Rotate exposed secrets.
  • Add missing tests and monitoring.
  • Reduce privileges and attack surface.
  • Improve secure defaults and documentation.
  • Update threat models after incidents or architecture changes.

Best practices

  • Make security requirements testable.
  • Prefer well-maintained frameworks over custom security code.
  • Never trust client-side checks as the only enforcement.
  • Use central authorization and input-validation patterns.
  • Keep secrets out of source code and logs.
  • Make build and release evidence reproducible.
  • Treat security exceptions as temporary, owned, and tracked.

11. Industry Perspective

Banks

Banks use secure software engineering for transaction systems, mobile banking, fraud detection, regulatory reporting, and customer identity systems. Access control, audit trails, change approval, secure coding, and segregation of duties are critical.

Governments

Government systems handle citizen records, public services, tax data, defense contracts, and critical infrastructure. Secure engineering supports resilience, procurement assurance, secure configuration, and vulnerability disclosure.

Cloud providers

Cloud providers build multi-tenant systems where one customer’s activity must not affect another customer’s confidentiality or integrity. Secure defaults, isolation, identity, logging, automated deployment controls, and secure supply chains are central.

Healthcare

Healthcare software handles clinical records, billing, medical devices, scheduling, and lab systems. Security failures can affect privacy, patient care, and operational continuity.

Defense

Defense systems require assurance, controlled changes, supply-chain scrutiny, secure communications, and resistance to sophisticated adversaries. Software provenance and tamper resistance matter heavily.

Universities

Universities operate learning systems, research platforms, identity systems, payment systems, and student records. Secure development reduces risks such as grade tampering, account compromise, research-data exposure, and service outages.

Large enterprises

Enterprises manage thousands of applications and dependencies. They need repeatable secure engineering processes, central policies, developer education, application inventories, vulnerability management, and secure CI/CD platforms.

12. Research Perspective

Secure software engineering is an active research area because software ecosystems are too large for manual inspection alone, and because human engineering behavior strongly affects security outcomes.

Current research directions

  • Usable security for developers: designing APIs, tools, and warnings that developers can understand and act on.
  • Program analysis: improving static and dynamic analysis to reduce false positives and false negatives.
  • Fuzzing: automatically generating inputs that find crashes, parser errors, and unexpected behavior.
  • Formal methods: mathematically verifying properties of critical software.
  • Software supply-chain security: improving dependency trust, build provenance, reproducible builds, and artifact integrity.
  • AI-assisted coding security: studying whether code-generation tools introduce, detect, or fix vulnerabilities.
  • Vulnerability prediction: using project history, code structure, and dependency data to prioritize review.
  • Security metrics: finding useful measurements that improve decisions without creating perverse incentives.

Open problems

  • How can tools explain security findings clearly enough for beginners without oversimplifying?
  • How can organizations verify that a binary truly came from reviewed source code?
  • How can security be integrated into fast development without becoming ignored noise?

13. Case Studies

Heartbleed

What happened: Heartbleed was a serious vulnerability in OpenSSL’s TLS heartbeat implementation that could allow memory disclosure from affected systems.

Root cause: a missing bounds check allowed a request to claim a larger payload size than was actually supplied.

Impact: affected services risked leaking sensitive memory such as session data or key material.

Lessons learned: memory safety, code review, testing, dependency inventory, and rapid patching all matter. A vulnerability in a widely used library can affect many systems.

Equifax breach

What happened: attackers exploited an unpatched Apache Struts vulnerability in a public-facing application.

Root cause: vulnerable dependency management, patching failure, asset visibility gaps, and detection gaps contributed to the incident.

Impact: sensitive personal information of many individuals was exposed.

Lessons learned: dependency inventories, patch governance, monitoring, and accountability are secure software engineering concerns, not only operations concerns.

SolarWinds supply-chain compromise

What happened: attackers compromised the software build/distribution process of a widely used IT management product, resulting in trojanized updates.

Root cause: a trusted supply-chain path was abused.

Impact: downstream customers installed compromised software through a trusted update channel.

Lessons learned: build system security, artifact integrity, provenance, monitoring, and supplier risk management are critical.

Log4Shell

What happened: a vulnerability in the Log4j Java logging library created widespread emergency patching across many organizations.

Root cause: unsafe feature behavior in a widely used dependency, combined with broad exposure and difficulty identifying all affected systems.

Impact: organizations had to rapidly identify, patch, mitigate, and monitor affected applications.

Lessons learned: SBOMs, dependency inventories, secure defaults, and rapid response capacity are essential.

14. Common Misconceptions

MisconceptionWhy it is incorrect
“Security is handled after coding.”Late fixes are more expensive and often miss design flaws.
“If tests pass, the software is secure.”Tests only check what they are designed to check.
“A scanner will find everything.”Tools miss context, business logic, and many design flaws.
“Open source means secure because everyone can review it.”Visibility does not guarantee review quality, funding, or maintenance.
“Closed source means secure because attackers cannot see the code.”Security by obscurity is weak; attackers can still test behavior or reverse engineer.
“Developers are the only people responsible.”Security also depends on management, design, operations, procurement, and users.
“Security slows everything down.”Poorly designed security can slow teams; good guardrails reduce rework and emergencies.
“Dependencies are someone else’s problem.”If your product ships or runs the dependency, your users inherit the risk.
“No known vulnerabilities means no risk.”Unknown vulnerabilities, insecure design, misconfiguration, and weak monitoring can still exist.
“Security requirements can be vague.”Vague requirements cannot be tested or verified.

15. Summary

Secure software engineering is the practice of building, testing, releasing, and maintaining software so that it resists misuse and failure. It begins with clear requirements and threat models, continues through secure design and coding, and extends into testing, build security, deployment, monitoring, patching, and learning from incidents.

The most important idea is that secure software is not produced by one final scan. It is produced by repeated, evidence-backed decisions across the entire lifecycle.
