Lesson 4.2: Requirements, Abuse Cases, and Secure Design

Security failures often begin before code exists. A vague requirement, missing abuse case, or unsafe design assumption can become a breach months later. This lesson teaches how to define security behavior before implementation.

Estimated reading time240–340 minutes
Difficulty levelBeginner → MS foundation
Required backgroundLessons 0.1–4.1 recommended; no prior design course required
Practice time5–8 hours

Design is where security becomes specific.

A secure design translates vague goals such as “protect grades” into precise behavior: who may access which object, under what conditions, how denials are logged, how misuse is detected, and how the system fails safely.

RequirementWhat must be true?
Abuse caseHow could it be misused?
Design controlWhat prevents, detects, or limits harm?

Week 4 Context

Lesson 4.1 introduced secure software engineering as a lifecycle discipline. Lesson 4.2 moves earlier in that lifecycle: requirements and design. This matters because many severe vulnerabilities are not caused by a missing library call. They are caused by unclear authorization rules, unsafe trust assumptions, missing logging, weak defaults, or designs that put too much responsibility on users.

The main output of this lesson is a secure design packet: a small set of artifacts that can be reviewed before code is written. It includes assets, security requirements, abuse cases, a data-flow diagram, trust boundaries, controls, acceptance tests, and residual risks.

Boundary: all examples use fictional systems or your own local files. Do not review, test, or probe real systems unless you have written authorization and a defined scope.

1. Introduction

Plain-English explanation

A requirement says what a system must do. A security requirement says what a system must do to protect something. An abuse case says how a feature could be misused. A secure design is the plan that makes the required protection realistic.

Analogy: suppose a university builds a room for storing exam papers. A weak requirement says, “Make the room safe.” A better requirement says, “Only authorized exam officers may enter the room; every entry must be logged; exam papers must stay locked when unattended.” An abuse case says, “A student tries to enter by following an officer through the door.” A secure design adds badge access, anti-tailgating procedure, cameras, locked cabinets, and audit review.

Why the topic exists

Software teams need a way to decide security behavior before code is written. Without explicit security requirements, developers may implement different assumptions. Without abuse cases, teams may test only friendly behavior. Without secure design, controls are often bolted on late and inconsistently.

Why the topic matters

Design flaws are expensive because they shape the whole system. A missing input check may require a small patch. A missing authorization model may require redesigning APIs, databases, caches, logs, user interfaces, and administrative workflows.

Where it is used

  • Product planning and feature design.
  • Architecture review boards.
  • Secure software development life cycle gates.
  • Threat modeling workshops.
  • Pull-request and design-document reviews.
  • Compliance evidence and audit preparation.

Who uses it

  • Developers use requirements and design notes to implement the correct behavior.
  • Security engineers use abuse cases to find missing controls early.
  • Architects use trust boundaries and data flows to choose safe structures.
  • Product managers use security acceptance criteria to prioritize work.
  • Quality assurance teams use abuse cases as negative tests.
  • Auditors use the artifacts as evidence that security was considered before release.

2. Learning Outcomes

After finishing this lesson, you should be able to:

  • Define functional, non-functional, security, privacy, safety, and operational requirements.
  • Convert vague security goals into specific, testable requirements.
  • Write abuse cases that describe realistic misuse without giving instructions for unauthorized activity.
  • Draw a simple data-flow diagram and identify trust boundaries.
  • Explain how secure design principles guide architecture choices.
  • Map assets, subjects, objects, actions, threats, controls, tests, and residual risks.
  • Distinguish prevention, detection, response, and recovery controls in design.
  • Write security acceptance criteria that can be tested before release.
  • Evaluate a small feature design for missing authorization, logging, data validation, and failure-mode decisions.
  • Create a complete safe design-review packet for a fictional feature.

3. Prerequisites

This lesson assumes no prior design or software-engineering course. It uses these concepts:

Feature

A feature is a capability a system provides. Example: “download transcript,” “reset password,” or “invite a new user.”

User story

A user story is a short description of what a legitimate user wants. Example: “As a student, I want to download my transcript so that I can apply for a scholarship.”

Asset

An asset is anything worth protecting: data, identity, money, service availability, safety, reputation, or system control.

Subject, object, and action

In access-control thinking, a subject is the actor, an object is the protected thing, and an action is what the subject wants to do. Example: student Alice reads transcript T100.

Trust boundary

A trust boundary is a place where data or control crosses from one trust level to another. Example: a browser request enters a server. The server should not blindly trust client-provided values.

Control

A control is a safeguard that prevents, detects, limits, or helps recover from harm. Examples: authentication, authorization, encryption, logging, rate limiting, backups, and approval workflows.

Mini-explanation: friendly path vs hostile path

The friendly path asks, “What should happen when the correct user follows the instructions?” The hostile path asks, “What could go wrong if input is wrong, identity is false, timing is unusual, or a user tries to access something they should not?” Secure design needs both paths.

4. Core Concepts

4.1 Requirements are security contracts

A good requirement is a contract between the system and its stakeholders. It should be clear enough that two people can test whether the system satisfies it.

Requirement typePlain meaningExample
Functional requirementWhat the system should doA student can download their transcript.
Security requirementWhat must be protected and howA student can download only their own transcript.
Privacy requirementHow personal data should be limited and protectedTranscript access logs must not expose grades to help-desk staff.
Safety requirementHow the system avoids physical or serious human harmA medical device must fail into a clinically safe state.
Operational requirementHow the system is run and maintainedSecurity patches must be deployable without manual database edits.
Assurance requirementHow confidence is demonstratedAuthorization tests must cover owner, non-owner, admin, and unauthenticated cases.

4.2 Testable requirements

A requirement is testable when a reviewer can check it using inspection, test, demonstration, analysis, or audit evidence.

VagueTestableTest method
Protect student data.Only the transcript owner, registrar role, or approved advisor may read a transcript.Authorization unit and integration tests.
Log important things.Every denied transcript access must log actor, object, action, decision, timestamp, and request ID.Integration test and log review.
Use secure defaults.New transcript-sharing links must be disabled unless an authorized user creates them with an expiry time.Default-configuration test.
Prevent abuse.The API must reject transcript requests where authenticated_user_id does not match transcript_owner_id unless the user has registrar approval.Negative authorization tests.

4.3 Acceptance criteria

Acceptance criteria are conditions that must be true before a feature is accepted as done. Security acceptance criteria should include normal behavior and denial behavior.

  • Owner access succeeds.
  • Non-owner access fails.
  • Unauthenticated access fails.
  • Registrar access succeeds only with the correct role.
  • Denied requests create audit events.
  • Error messages do not reveal sensitive object details.

4.4 Abuse cases

An abuse case is a story about misuse. It should be specific enough to guide controls and tests, but it should not become an instruction for attacking real systems.

FeatureUse caseAbuse caseDesign response
Download transcriptStudent downloads own transcriptStudent changes transcript ID to another student’s IDServer-side object-level authorization
Password resetUser resets forgotten passwordAttacker triggers reset spam or takes over weak recoveryRate limiting, strong recovery proof, event logging
Admin exportRegistrar exports recordsInsider exports excessive dataLeast privilege, approval, export limits, monitoring
File uploadUser uploads profile documentUser uploads unexpected file type or oversized contentType validation, size limits, storage isolation, malware scanning

4.6 Trust boundaries

A trust boundary appears when control changes. The browser is less trusted than the server. A third-party email service is outside the university’s direct control. An admin interface has more privilege than a student interface. Every boundary needs validation, authentication, authorization, logging, and failure handling appropriate to the risk.

4.7 Secure design principles

Secure design principles are reusable rules of judgment. They do not replace analysis, but they prevent common bad decisions.

PrincipleMeaningDesign question
Least privilegeGive only the access neededDoes this component need write access or only read access?
Deny by defaultReject unless explicitly allowedWhat happens when a role is missing or unknown?
Secure defaultsDefault state should be safeAre new sharing links off unless deliberately enabled?
Complete mediationCheck every protected accessIs authorization checked on each transcript request?
Fail safelyFailure should not create excess accessIf the policy service is down, does the app allow or deny?
Separation of dutiesDivide sensitive powersCan one person both approve and execute a high-risk export?
Defense in depthUse multiple layersIf one control fails, what still limits harm?
Minimize attack surfaceExpose less functionalityIs an admin API reachable from the public internet?
Psychological acceptabilitySecurity should be usable enough to followWill users bypass the control because it is too painful?

4.8 Prevention, detection, response, and recovery by design

Secure design should not contain only prevention. Some failures will occur. The design should also detect misuse, support response, and enable recovery.

  • Prevention: authorization checks block non-owner transcript access.
  • Detection: audit logs reveal repeated denied access attempts.
  • Response: security staff can disable a suspicious account.
  • Recovery: backups and change history help restore corrupted records.

4.9 Traceability

Traceability means linking requirements to design decisions, controls, tests, and evidence. Without traceability, teams cannot easily answer, “Which test proves this requirement?” or “Which control addresses this abuse case?”

flowchart LR
  Asset[Asset: transcript] --> Req[Requirement: owner-only read]
  Req --> Abuse[Abuse case: ID tampering]
  Abuse --> Control[Control: object authorization]
  Control --> Test[Test: non-owner denied]
  Test --> Evidence[Evidence: CI test result and review note]

4.10 Residual risk

Residual risk is the risk that remains after controls are applied. Secure design is not a promise of zero risk. It is a disciplined way to reduce risk, make assumptions visible, and decide what remains acceptable.

7. Mathematical Foundations

Secure design is partly qualitative, but basic quantitative thinking helps prioritize work. The numbers below are teaching models, not precise predictions.

Risk as expected loss

A simple model is:

Expected loss = probability of event × impact if event occurs

If a feature has a 5% annual chance of exposing transcript data and the estimated impact is $200,000, then:

Expected annual loss = 0.05 × 200,000 = $10,000

This does not mean the organization will lose exactly $10,000. It means the average expected loss under the model is $10,000 per year.

Control benefit

If object-level authorization and audit review reduce estimated probability from 5% to 1%, then:

New expected annual loss = 0.01 × 200,000 = $2,000

Expected reduction = 10,000 − 2,000 = $8,000

Prioritization score

Teams often need rough prioritization. One simple score is:

Priority = asset value × abuse likelihood × design exposure × control gap

Use a 1–5 scale for each factor.

FeatureAsset valueAbuse likelihoodExposureControl gapPriority
Transcript download5443240
Theme color setting12214

The transcript feature should receive more design attention because the protected asset and possible harm are greater.

Availability math

If a design has one required component with 99% availability, the feature’s availability cannot exceed that component’s availability. If two required independent components are both 99% available, the simple series estimate is:

0.99 × 0.99 = 0.9801 = 98.01%

This is why design must consider dependencies. Adding components can add capability, but it can also add failure paths.

Python example

This local toy example shows a common design issue: checking that an object exists is not the same as checking that the current subject may access it.

# Defensive example: checking authorization before returning a record.
# This is a toy local example. It does not connect to a real service.

records = {
    "student-100": {"owner": "alice", "grade": "A"},
    "student-200": {"owner": "bob", "grade": "B"},
}

def get_grade_insecure(current_user, record_id):
    # Mistake: the function checks that the record exists,
    # but it does NOT check whether the current user owns it.
    return records.get(record_id)

def get_grade_secure(current_user, record_id):
    # Step 1: find the requested object.
    record = records.get(record_id)

    # Step 2: fail closed if the object does not exist.
    # Returning None avoids exposing details in this toy example.
    if record is None:
        return None

    # Step 3: authorize this specific subject-object-action.
    # Subject: current_user
    # Object: record
    # Action: read grade
    if record["owner"] != current_user:
        return None

    # Step 4: return only the allowed data.
    return {"grade": record["grade"]}

print("Insecure:", get_grade_insecure("alice", "student-200"))
print("Secure:", get_grade_secure("alice", "student-200"))

Network example

Requirement: “The admin API must not be reachable from the public internet.” Design response: place the admin API behind a private network boundary, require administrator identity, log access, and block public routes. Test: from a non-admin network segment, the admin endpoint must be unreachable.

Web example

Requirement: “A user may edit only their own profile.” Abuse case: “A user changes user_id in a request and attempts to edit another profile.” Design response: the server derives the acting user from the authenticated session, not from a client-supplied user ID, and checks object ownership on every edit.

Cloud example

Requirement: “Build jobs may deploy only to the test environment unless approved.” Abuse case: “A compromised build token deploys directly to production.” Design response: separate cloud roles for test and production, require protected branch rules and human approval for production, use short-lived workload credentials, and log deployment actions.

9. Code Examples

Line-by-line: authorization check

The secure function in the earlier Python example follows a design requirement: “A student may read only their own grade record.”

  1. records is a toy in-memory database. Real systems would use a database and stronger controls.
  2. current_user represents the authenticated subject.
  3. record_id identifies the requested object.
  4. The insecure function returns the object if it exists, regardless of ownership.
  5. The secure function first fails closed if the object does not exist.
  6. It then checks that record["owner"] matches current_user.
  7. If the check fails, it returns nothing rather than exposing another user’s data.
  8. If the check succeeds, it returns only the allowed field.

Simple requirements-as-data example

# Tiny local requirements checker.
# It does NOT prove a system is secure. It only shows how requirements
# can be written in a testable form.

security_requirements = [
    {
        "id": "SEC-AUTHZ-001",
        "statement": "A student may read only their own grade record.",
        "asset": "grade record",
        "subject": "student",
        "action": "read",
        "verification": "unit test + code review"
    },
    {
        "id": "SEC-LOG-001",
        "statement": "Every denied grade-record access must create an audit event.",
        "asset": "audit log",
        "subject": "application",
        "action": "write audit event",
        "verification": "integration test + log inspection"
    }
]

for req in security_requirements:
    print(f"{req['id']}: {req['statement']}")
    print(f"  Verification: {req['verification']}")

This code does not enforce security by itself. Its purpose is to show how requirements can be written clearly enough to be reviewed, tested, and traced.

Common mistakes

  • Trusting client-side checks: browser JavaScript can improve user experience, but server-side authorization must enforce security.
  • Checking role but not object: “is user a student?” is not enough; the system must check whether this student may access this object.
  • Returning detailed denials: error messages may reveal whether a private record exists.
  • Logging secrets: audit logs should record security-relevant events without exposing passwords, tokens, or sensitive content.
  • Missing negative tests: tests must prove that forbidden actions fail.

Secure coding practices tied to design

Secure coding is stronger when it implements explicit design decisions. A checklist helps reviewers connect code to the design:

Security review checklist for a pull request that adds "download transcript":

[ ] Requirement states who can download which transcript.
[ ] Abuse case covers changing transcript_id to another user's ID.
[ ] Server-side authorization checks every download request.
[ ] Client-side checks are not treated as security controls.
[ ] Error messages do not reveal whether another transcript exists.
[ ] Successful and denied downloads are audit logged.
[ ] Tests include owner access, non-owner denial, unauthenticated denial, and admin workflow.
[ ] Design document shows data flow and trust boundary.
[ ] Sensitive files are not cached publicly.
[ ] Rollback plan exists if the feature leaks data.

10. Security Perspective

Common attacks

  • Broken access control: a user performs an action or accesses an object they should not access.
  • Insecure direct object reference: a client-supplied object ID is trusted without object-level authorization.
  • Privilege escalation: a lower-privilege user gains higher-privilege capability.
  • Data overexposure: an API returns more data than the user needs.
  • Workflow bypass: a user skips approval steps by calling a backend endpoint directly.
  • Rate abuse: repeated use of a feature creates spam, lockouts, resource exhaustion, or enumeration.

Common vulnerabilities

  • Vague requirements such as “make it secure.”
  • No abuse cases for high-risk features.
  • No server-side authorization decision.
  • Missing trust boundary documentation.
  • Unsafe defaults that expose data unless users configure protection.
  • Audit logs that are missing, excessive, or not reviewable.

Defensive strategies

  • Use security requirements and abuse cases during planning.
  • Require design review for high-risk features.
  • Document subjects, objects, actions, and allowed conditions.
  • Mark trust boundaries and validate data at boundaries.
  • Write negative tests for denied behavior.
  • Use secure defaults and deny by default.

Detection methods

  • Monitor denied authorization attempts.
  • Alert on unusual admin exports or repeated object-ID access patterns.
  • Review audit logs for high-risk workflow bypass indicators.
  • Track changes to authorization code, policy files, and admin workflows.

Mitigation and best practices

  • Fail closed when policy decisions cannot be made safely.
  • Keep authorization on the server side.
  • Separate user identity, role, object ownership, and context.
  • Use central policy helpers where appropriate to avoid inconsistent checks.
  • Document residual risks rather than hiding them.

12. Research Perspective

Secure requirements and design remain active research areas because human judgment, complex systems, and changing attacker behavior make full automation difficult.

Current research directions

  • Automated threat modeling: deriving likely threats from architecture diagrams, code, cloud configurations, or infrastructure-as-code.
  • Security requirements engineering: improving how teams discover, specify, verify, and maintain security requirements.
  • Usable security for developers: studying how developers understand warnings, APIs, libraries, and review feedback.
  • Policy-as-code: expressing authorization and infrastructure rules in testable machine-readable form.
  • Secure-by-design metrics: measuring whether products reduce customer security burden.
  • AI-assisted design review: using language models to find missing requirements, unsafe assumptions, and untested abuse cases, while managing hallucination risk.

Open problems

  • How to make threat modeling lightweight enough that teams actually do it.
  • How to keep design documents accurate after systems change.
  • How to measure whether a requirement is complete.
  • How to verify authorization behavior across microservices and cloud policy layers.
  • How to balance privacy, auditability, safety, and operational usability.

Notable research themes

Students should look for papers on requirements engineering for security, threat modeling methods, security design patterns, formal methods for access control, usable secure APIs, and empirical studies of developer security behavior. Do not copy copyrighted papers into reports; summarize the ideas and cite them properly.

13. Case Studies

Case Study 1: Broken object-level authorization in web APIs

What happened: many real web and API incidents occur when a user changes an object identifier and the server returns data belonging to someone else.

Root cause: the design checks that the user is logged in, but not whether that user is authorized for the specific object.

Impact: confidential data can be exposed or modified.

Lessons learned: requirements must specify object-level authorization, tests must include non-owner access, and server-side policy must not rely on client-side controls.

Case Study 2: Unsafe defaults and exposed storage

What happened: organizations have repeatedly exposed sensitive data through misconfigured cloud storage or overly broad sharing settings.

Root cause: design and operational assumptions placed too much responsibility on users or administrators to configure protection correctly.

Impact: public exposure of private records, intellectual property, backups, or credentials.

Lessons learned: secure defaults, least privilege, policy guardrails, monitoring, and clear ownership must be part of design.

Case Study 3: Payment and approval workflow bypass

What happened: in many business systems, attackers or insiders try to bypass approval workflows by calling backend endpoints directly or abusing missing state checks.

Root cause: the design treats the user interface as the workflow authority instead of enforcing workflow state on the server.

Impact: unauthorized transfers, fraudulent approvals, or unauthorized administrative changes.

Lessons learned: workflows need server-side state machines, separation of duties, audit logs, and tests for skipped or repeated steps.

14. Common Misconceptions

MisconceptionWhy it is wrong
“We can add security after the feature works.”Some controls depend on architecture and data model choices. Late security often becomes expensive and incomplete.
“Authentication means the user can access everything.”Authentication proves identity; authorization decides allowed actions.
“Client-side checks are enough.”Users can modify client-side requests. Servers must enforce security decisions.
“If we use encryption, the design is secure.”Encryption does not solve authorization, logging, input validation, workflow abuse, or unsafe defaults.
“Threat modeling is only for experts.”Beginners can use simple questions: what are we protecting, who might misuse it, where does trust change, and what should fail closed?
“A requirement can say ‘must be secure.’”That is not testable. Requirements must specify asset, actor, action, condition, and verification.
“Logging everything is always good.”Logs can expose sensitive data. Good logging records security events while minimizing secrets and unnecessary personal data.
“Secure design guarantees no breaches.”It reduces risk and improves evidence; it cannot eliminate all uncertainty.

15. Summary

Secure software begins before coding. Requirements define what security behavior must exist. Abuse cases reveal how features can be misused. Secure design chooses controls, architecture, trust boundaries, failure behavior, logging, and verification methods. The strongest designs connect assets to requirements, requirements to abuse cases, abuse cases to controls, controls to tests, and tests to evidence.

The practical habit to build: never accept “make it secure” as a requirement. Ask what asset is being protected, who may do what, what abuse is likely, where trust changes, what should fail closed, and how the team will prove the control works.

User story
A short description of what a legitimate user wants to accomplish.

20 MCQs

1. What is a security requirement?

Answer: B. It should be specific and verifiable.

2. Which requirement is most testable?

Answer: C. It defines subject, action, and object.

3. What is an abuse case?

Answer: D. Abuse cases help teams design controls and negative tests.

4. What does a trust boundary show?

Answer: A. Boundaries are where validation and policy enforcement matter.

5. Why are negative tests important?

Answer: B. Security requires testing denials, not only success.

6. What is the main problem with “make it secure”?

Answer: C. Security must be made specific.

7. What does deny by default mean?

Answer: A. This reduces accidental exposure.

8. Which is an example of complete mediation?

Answer: D. Every protected access needs a decision.

9. What is residual risk?

Answer: B. No design removes all risk.

10. What does traceability connect?

Answer: C. Traceability helps prove intent was implemented and tested.

11. Which control is mainly preventive?

Answer: A. It blocks unauthorized access before it happens.

12. Which control is mainly detective?

Answer: D. It helps find suspicious behavior.

13. What is a data-flow diagram useful for?

14. Why is server-side enforcement required?

Answer: C. Client-side checks are not authoritative controls.

Answer: A. Failure must not become a privilege escalation path.

18. Which is a secure default?

Answer: C. The safe state is the starting state.

19. What is separation of duties?

18. Practice Lab

Lab title

Design a secure transcript-download feature for a fictional university portal.

Safety statement

This lab uses a fictional system and local documents only. Do not test a real university portal, cloud account, website, or API.

Prerequisites

  • Basic file editing.
  • Basic Git knowledge from Lesson 0.5.
  • Basic authorization concepts from Week 2.
  • Basic threat modeling concepts from Lesson 1.4.

Objectives

  • Identify assets and stakeholders.
  • Write testable security requirements.
  • Write abuse cases.
  • Draw a data-flow diagram.
  • Identify trust boundaries.
  • Choose controls and acceptance tests.
  • Document residual risks.

Scenario

CampusOne is a fictional university portal. Students can download transcripts. Registrars can approve official transcript exports. Advisors may view transcripts only for assigned advisees. The system sends email notifications after successful downloads.

Tasks

  1. Create a folder named secure-design-lab.
  2. Create assets.md listing at least five assets.
  3. Create requirements.md with at least eight testable security requirements.
  4. Create abuse-cases.md with at least eight abuse cases.
  5. Create design.md containing a data-flow diagram in Mermaid.
  6. Create controls.md mapping each abuse case to a prevention or detection control.
  7. Create tests.md with positive and negative acceptance tests.
  8. Create risk.md describing residual risks and assumptions.
  9. Commit the files with Git.

Expected outcomes

  • A reviewer can tell exactly what security behavior the feature requires.
  • At least one abuse case maps to each high-value asset.
  • Every important trust boundary has a control.
  • Tests include allowed and denied behavior.
  • Residual risk is visible rather than hidden.

Optional extension

Add a traceability matrix with columns: asset, requirement ID, abuse case ID, control ID, test ID, evidence.
