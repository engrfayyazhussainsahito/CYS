Lesson 4.5: Testing, Fuzzing, CI/CD, and Release Integrity

Secure releases need more than working code. Testing checks expected behavior. Fuzzing explores unexpected inputs. CI/CD makes checks repeatable. Release integrity gives evidence that the artifact users receive is the artifact the team intended to ship.

Week 4 Context

Lessons 4.1–4.4 explained secure software lifecycle, secure design, input/output safety, secrets, dependencies, and SBOMs. Lesson 4.5 closes Week 4 by turning those ideas into a repeatable release process.

Safety boundary: this lesson only uses local toy applications and example CI files for repositories you own. Do not run tests, fuzzers, scanners, or build-integrity checks against systems, repositories, packages, or services you are not authorized to assess.

1. Introduction

Plain-English explanation

Testing checks whether software behaves as expected. Fuzzing sends many unusual inputs to software to discover crashes or strange behavior. CI/CD automates build, test, package, and deployment steps. Release integrity gives evidence that the released artifact came from the intended source and was not quietly changed.

Why this topic exists

Humans forget steps. Manual releases are inconsistent. Attackers target the gap between source code and production: build scripts, CI runners, secrets, package registries, artifact stores, approval workflows, and deployment tooling. Secure teams therefore treat the pipeline as a security-sensitive system.

Why this topic matters

A software product can have clean source code but still ship a compromised artifact if the build system is tampered with. A team can write secure validation logic but miss an edge case if no test exercises it. A signed artifact can still be risky if the signing key is stolen or the wrong identity is trusted.

Where it is used

  • Web applications, mobile apps, APIs, cloud services, embedded systems, operating-system packages, container images, and internal enterprise tools.
  • Software vendors, banks, hospitals, universities, defense contractors, government agencies, and cloud providers.
  • DevSecOps programs, vulnerability management, release engineering, compliance evidence, and incident response.

Who uses it

  • Developers write tests and fix defects.
  • Quality engineers design test strategies and automation.
  • Security engineers add abuse-case tests, fuzzing, SAST, DAST, SCA, and release controls.
  • Platform engineers operate CI/CD runners, artifact registries, signing systems, and deployment gates.
  • Auditors and risk teams ask for evidence that releases were controlled.

3. Prerequisites

Program

A program is a set of instructions that a computer executes. A program has inputs, logic, outputs, errors, and states.

Source code and commit

Source code is the human-readable form of a program. A commit is a recorded change in version control.

Build artifact

An artifact is a built output, such as a package, binary, installer, container image, or generated website.

Hash and digital signature

A hash is a fingerprint of data. A digital signature uses cryptography to bind a key or identity to data so others can verify integrity and origin.

Secret

A secret is a credential or sensitive value that grants access. CI/CD systems often need secrets, so they are high-value targets.

4. Core Concepts

4.1 Testing

Testing runs software under selected conditions and checks results. A test is a question: “Given this input and setup, does the software produce the expected result?”

Analogy: testing a bridge with known weights does not prove the bridge can survive every possible situation, but it gives evidence about expected conditions and engineering assumptions.

4.2 Unit, integration, and end-to-end tests

  • Unit test: checks a small function or module in isolation.
  • Integration test: checks whether multiple components work together.
  • End-to-end test: checks a whole user workflow, such as login → add item → checkout.
  • Regression test: checks that a fixed bug does not return.
  • Security test: checks security behavior, such as authorization, validation, rate limiting, and safe error handling.

4.3 Test oracle

A test oracle is the rule that decides whether a test passed. Weak oracles create weak tests.

4.4 Coverage

Coverage measures how much code or behavior was exercised by tests. High line coverage does not prove the system is secure because a test can execute code without checking the right security property.

4.5 Fuzzing

Fuzzing automatically generates many inputs to find crashes, hangs, assertion failures, parser bugs, memory errors, and unexpected behavior. It is especially useful for parsers, file formats, network protocols, APIs, compilers, image processors, and low-level code.

4.6 Fuzzing styles

  • Mutation-based fuzzing: starts with valid examples and mutates them.
  • Generation-based fuzzing: creates inputs from a grammar or model.
  • Coverage-guided fuzzing: uses feedback from code coverage to search deeper paths.

4.7 Crash triage

Crash triage means deciding whether a crash is reproducible, security-relevant, already known, and worth fixing urgently.

4.8 Continuous Integration

Continuous Integration, or CI, automatically builds and tests code when changes are proposed or merged.

4.9 Continuous Delivery and Continuous Deployment

Continuous Delivery keeps software ready for release, often with a human approval step. Continuous Deployment automatically deploys successful changes to production. The approval difference matters.

4.10 Pipeline and runner

A pipeline is an automated workflow. A runner is the machine or container that executes CI/CD jobs. If a runner has secrets or deployment permissions, untrusted code running on that runner becomes dangerous.

4.11 Pipeline as a privileged system

CI/CD can build, sign, publish, deploy, and access secrets. Secure pipelines use least privilege, isolated runners, protected branches, reviewed workflow changes, pinned actions, short-lived credentials, and separation between untrusted pull requests and trusted release jobs.

4.12 Release gate

A release gate is a required control before release, such as passing tests, vulnerability thresholds, code-owner approval, signed commits, artifact signing, SBOM generation, or manual approval.

4.13 Artifact integrity

Artifact integrity means the built release has not been unexpectedly modified. Hashes detect changes. Digital signatures make changes detectable and link the artifact to a signing key or identity.

4.14 Provenance and attestation

Provenance records where software came from and how it was built. An attestation is a signed statement about an artifact, such as which source commit, builder, dependency set, and build command produced it.

4.15 Release evidence

Release evidence includes test results, build logs, SBOM, vulnerability scan output, approvals, artifact hashes, signatures, provenance, deployment logs, and rollback plans.

7. Mathematical Foundations

7.1 Test pass rate

Pass rate = passing tests / total tests. If 148 of 150 tests pass, then 148 / 150 = 98.67%. This is useful evidence, but one failing test may still block release if it covers a critical security property.

7.2 Coverage ratio

Coverage = executed lines / total instrumented lines. If 720 of 900 lines execute, coverage is 80%. Security interpretation: coverage shows what ran, not whether authorization, validation, or error handling was checked correctly.

7.3 Fuzzing discovery probability

If each generated input has probability q of triggering a specific bug, the chance of finding the bug at least once after n independent inputs is 1 − (1 − q)n. Example: q = 0.001 and n = 3000 gives approximately 95.03%.

7.4 Hash collision intuition

A 256-bit hash has 2256 possible outputs. Random collision finding is infeasible under ordinary assumptions, but hashes only help if the expected hash comes from a trusted source.

7.5 Teaching risk score

Release risk = impact × likelihood × exposure − control strength. Example: 5 × 4 × 3 − 7 = 53. This is not a standard; it teaches that release decisions combine defects, exposure, and controls.

Python examples

Use unit tests to check validation behavior and a toy fuzzer to explore unexpected input.

# Safe local example: unit tests for a tiny validation function.
# This code does not connect to the internet or attack anything.

import re


def is_valid_student_id(value: str) -> bool:
    """Return True only for IDs like STU-123456."""
    if not isinstance(value, str):
        return False
    return re.fullmatch(r"STU-[0-9]{6}", value) is not None


def test_accepts_valid_id():
    assert is_valid_student_id("STU-123456") is True


def test_rejects_missing_prefix():
    assert is_valid_student_id("123456") is False


def test_rejects_letters_in_number_part():
    assert is_valid_student_id("STU-12AB56") is False


def test_rejects_extra_characters():
    assert is_valid_student_id("STU-123456<script>") is False
# Safe local example: beginner-friendly fuzzing of a parser function.
# The target is our own toy code, not a real website or third-party service.

import random
import string


def parse_score(text: str) -> int:
    """Parse a score from 0 to 100. This function should not crash unexpectedly."""
    value = int(text)
    if value < 0 or value > 100:
        raise ValueError("score must be between 0 and 100")
    return value


def random_input(max_length: int = 8) -> str:
    """Generate random text using digits, letters, and symbols."""
    alphabet = string.digits + string.ascii_letters + "-+!@#$%^&*()"
    size = random.randint(0, max_length)
    return "".join(random.choice(alphabet) for _ in range(size))


for i in range(1000):
    sample = random_input()
    try:
        parse_score(sample)
    except ValueError:
        pass  # Expected for invalid scores.
    except Exception as error:
        print("Unexpected crash on input:", repr(sample), "error:", error)
        break
else:
    print("No unexpected crash found in 1000 generated inputs.")

9. Code Examples

9.1 Unit test example

# Safe local example: unit tests for a tiny validation function.
# This code does not connect to the internet or attack anything.

import re


def is_valid_student_id(value: str) -> bool:
    """Return True only for IDs like STU-123456."""
    if not isinstance(value, str):
        return False
    return re.fullmatch(r"STU-[0-9]{6}", value) is not None


def test_accepts_valid_id():
    assert is_valid_student_id("STU-123456") is True


def test_rejects_missing_prefix():
    assert is_valid_student_id("123456") is False


def test_rejects_letters_in_number_part():
    assert is_valid_student_id("STU-12AB56") is False


def test_rejects_extra_characters():
    assert is_valid_student_id("STU-123456<script>") is False

Line-by-line explanation

  • import re loads Python's regular-expression library.
  • is_valid_student_id defines a function that returns a boolean answer.
  • isinstance prevents non-string values from being processed as text.
  • re.fullmatch requires the whole input to match, not just part of it.
  • The test functions check valid input, missing prefix, invalid characters, and extra content.

Common mistakes

  • Using re.match when fullmatch is required.
  • Testing only the happy path.
  • Writing tests that check implementation details instead of security behavior.

9.3 CI workflow example

# .github/workflows/ci.yml
# Safe educational CI workflow for a repository you own.

name: Safe CI Demo

on:
  push:
    branches: ["main"]
  pull_request:
    branches: ["main"]

permissions:
  contents: read

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - name: Check out repository
        uses: actions/checkout@v4

      - name: Set up Python
        uses: actions/setup-python@v5
        with:
          python-version: "3.12"

      - name: Install test tool
        run: python -m pip install pytest

      - name: Run tests
        run: pytest -q

Line-by-line explanation

  • name labels the workflow.
  • on defines when it runs.
  • permissions: contents: read starts with minimal repository permission.
  • runs-on chooses a runner environment.
  • actions/checkout checks out your code.
  • actions/setup-python installs Python.
  • The final step runs tests.

Common mistakes

  • Giving workflows write permissions by default.
  • Running untrusted pull-request code with secrets available.
  • Using unreviewed third-party actions without pinning or trust review.
  • Letting deployment happen automatically from every branch.

10. Security Perspective

Common attacks

  • Malicious workflow modification to exfiltrate secrets or publish a malicious artifact.
  • Secret exposure in CI logs.
  • Dependency tampering or malicious build plugins.
  • Runner compromise.
  • Unsigned release replacement.
  • Untrusted pull-request execution with privileged tokens.
  • Cache poisoning.

Common vulnerabilities

  • Over-permissive CI tokens.
  • Long-lived cloud credentials in CI variables.
  • Unprotected release branches and tags.
  • Manual release steps with no evidence trail.
  • No artifact signing or verification.
  • No SBOM or dependency inventory at release time.
  • No separation between build, test, release, and deploy permissions.

Defensive strategies

  • Use least-privilege workflow permissions.
  • Separate untrusted PR workflows from trusted release workflows.
  • Require code review for workflow and build-script changes.
  • Use protected branches, protected tags, and environment approvals.
  • Prefer short-lived OIDC-based credentials over static keys.
  • Generate SBOMs and provenance during the build.
  • Sign artifacts and verify signatures before deployment.

Detection methods

  • Alert on workflow file changes and new CI secrets.
  • Monitor artifact hash or signature mismatches.
  • Review unusual release times, new maintainers, and unexpected deployments.
  • Inspect CI logs for unusual outbound connections.

Mitigation

  • Revoke exposed tokens and rotate secrets.
  • Disable compromised runners and rebuild from trusted infrastructure.
  • Rebuild artifacts from known-good source commits.
  • Invalidate affected releases and audit downstream deployments.

Best practices

  • Make releases reproducible and reviewable.
  • Store release evidence with the artifact.
  • Use policy-as-code for release gates.
  • Keep tests fast enough that developers actually run them.
  • Treat CI/CD configuration as production code.

11. Industry Perspective

  • Banks: strict release approvals, segregation of duties, audit evidence, fraud-risk tests, and rollback plans.
  • Governments: traceability, secure build practices, procurement evidence, vulnerability handling, and controlled deployment.
  • Cloud providers: automated pipelines with strong identity, artifact signing, canary releases, monitoring, and rollback.
  • Healthcare: patient-safety tests, regulated data protection, and deployment evidence.
  • Defense: provenance, controlled build environments, software assurance, tamper resistance, and strict change control.
  • Universities: protection for identity systems, learning platforms, payroll, grant data, and student records.
  • Large enterprises: centralized CI/CD platforms, code-owner review, artifact repositories, vulnerability gates, and developer guardrails.

12. Research Perspective

Current research and engineering work focuses on making secure releases stronger and easier to adopt.

  • Fuzzing at scale: coverage guidance, grammar inference, hybrid fuzzing, and automatic crash triage.
  • Property-based testing: generating structured inputs from specifications and checking invariants.
  • Formal verification: mathematically proving selected security properties.
  • Reproducible builds: allowing independent verification that source code produces the same artifact.
  • Provenance usability: making signing, verification, and attestation practical for ordinary developers.
  • CI/CD threat modeling: modeling runner compromise, cache poisoning, untrusted PRs, and secret exposure.
  • AI-assisted development risk: studying how automated agents affect tests, workflows, review quality, and release safety.

Open problems include measuring release trust, preventing build-system compromise, making provenance meaningful to consumers, reducing false positives in gates, and balancing fast delivery with strong verification.

13. Case Studies

Codecov Bash Uploader incident

What happened: a third party modified a Bash Uploader used in CI environments. The modified uploader could export environment variables from users' CI environments.

Root cause: Codecov reported that an attacker extracted a key from an intermediate Docker image layer and used it to modify the uploader in storage.

Impact: CI secrets, tokens, keys, and repository metadata could have been exposed for affected users.

Lessons learned: verify downloaded tooling, minimize CI secrets, protect build artifacts, monitor script integrity, and rotate exposed credentials quickly.

SolarWinds Orion software supply-chain compromise

What happened: attackers compromised the software supply chain for SolarWinds Orion and distributed malicious updates to customers.

Root cause: the attack involved the trusted software build/update process, not only individual customer systems.

Impact: many public and private organizations had to investigate exposure from trusted vendor updates.

Lessons learned: signed software is not automatically safe if the build process is compromised. Provenance, build isolation, monitoring, vendor risk management, and detection remain necessary.

xz Utils backdoor

What happened: malicious code was introduced into xz Utils release artifacts in a sophisticated open-source supply-chain attack.

Root cause: the incident involved social, maintainer, release, and build-process weaknesses rather than a simple coding mistake.

Impact: Linux distributions and security teams had to urgently identify affected versions and prevent distribution.

Lessons learned: review release artifacts, protect maintainers, improve reproducible builds, monitor unusual build behavior, and avoid relying on trust reputation alone.

14. Common Misconceptions

  • “All tests passed, so the software is secure.” Passing tests only shows that selected checks passed.
  • “High coverage means good security.” Coverage measures execution, not security-property quality.
  • “Fuzzing is only for hackers.” Fuzzing is a defensive quality and security technique.
  • “CI/CD automatically makes releases safe.” Automation can scale mistakes and attacks if permissions and trust boundaries are weak.
  • “A signed artifact is always trustworthy.” A signature proves integrity relative to a key or identity; it does not prove the source, build, or signer was uncompromised.
  • “SBOMs replace testing.” SBOMs provide inventory, not behavioral assurance.
  • “Only production pipelines matter.” Development and staging pipelines may contain secrets or deployment paths.
  • “Manual approval fixes everything.” Approval without evidence is weak control.

15. Summary

Testing checks expected behavior. Fuzzing explores unexpected inputs. CI/CD makes checks and releases repeatable. Release integrity creates evidence that software came from the intended source, passed required checks, and was not altered unexpectedly. Secure release engineering connects coding, secrets, dependencies, build infrastructure, artifact signing, provenance, monitoring, and risk decisions.

16. Glossary

Artifact
A built software output such as a package, binary, installer, container image, or generated website.
Attestation
A signed statement about an artifact or process, such as how software was built.
Build
The process of turning source code and dependencies into a runnable or distributable artifact.
CI
Continuous Integration; automated building and testing of code changes.
CD
Continuous Delivery or Continuous Deployment, depending on whether human release approval is required.
Coverage
A measurement of what code or behavior was exercised by tests.
Fuzzing
Automated testing that feeds many unexpected inputs to a target to find crashes or abnormal behavior.
Pipeline
An automated sequence of software delivery steps.
Provenance
Evidence of where software came from and how it was produced.
Release gate
A required control that must pass before software is released or deployed.
Runner
The machine or container that executes CI/CD jobs.
Test oracle
The rule that decides whether a test passed or failed.

18. Practice Lab

Title

Safe Local Release Pipeline Evidence Lab

Prerequisites

  • Python 3 installed locally.
  • Basic terminal or PowerShell usage.
  • No internet target, cloud account, or external system required.

Objectives

  • Create a tiny Python function and tests.
  • Run tests locally.
  • Run a toy fuzzer against local code.
  • Create a release artifact.
  • Record its SHA-256 hash.
  • Create a simple release manifest with evidence.

Linux/macOS setup

# Safe local lab setup for Linux/macOS.
mkdir -p testing-fuzzing-cicd-lab
cd testing-fuzzing-cicd-lab
python3 -m venv .venv
. .venv/bin/activate
python -m pip install --upgrade pip pytest
cat > validator.py <<'PYCODE'
import re

def is_valid_student_id(value: str) -> bool:
    if not isinstance(value, str):
        return False
    return re.fullmatch(r"STU-[0-9]{6}", value) is not None
PYCODE
cat > test_validator.py <<'PYCODE'
from validator import is_valid_student_id

def test_valid():
    assert is_valid_student_id("STU-123456")

def test_invalid():
    assert not is_valid_student_id("123456")
PYCODE
pytest -q
