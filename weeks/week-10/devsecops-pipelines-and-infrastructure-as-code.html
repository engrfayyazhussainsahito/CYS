Lesson 10.4: DevSecOps Pipelines and Infrastructure as Code

DevSecOps is not a slogan and Infrastructure as Code is not automatically secure. This lesson starts from zero, explains how software moves from an idea to production, then shows how security controls can be built into the pipeline so cloud infrastructure changes become reviewable, repeatable, auditable, and reversible.

1. Introduction

Modern cloud systems change quickly. Developers commit code. Automated systems test it. Build systems create artifacts. Deployment systems push those artifacts into environments. Infrastructure teams define networks, storage, identities, databases, and permissions as configuration files. This speed is useful, but it creates a security problem: a single bad change can be copied into production faster than a human can manually inspect every detail.

DevSecOps means integrating security into development and operations instead of treating it as a final gate after everything is built. The term combines development, security, and operations. The important idea is not the name. The important idea is feedback: security checks should happen early, repeatedly, automatically where possible, and with human review where judgment is needed.

Infrastructure as Code, often shortened to IaC, means infrastructure is described in machine-readable files. Instead of manually clicking through a cloud console to create a network, storage bucket, firewall rule, or identity role, the team writes configuration and lets a tool create or update the infrastructure. IaC matters because it makes infrastructure reviewable, version-controlled, repeatable, testable, and auditable.

This topic exists because cloud infrastructure is now software-like. Banks use pipelines to deploy transaction systems. Governments use controlled release workflows for citizen services. Cloud providers use automation at massive scale. Healthcare organizations use change records for compliance and safety. Defense organizations need controlled promotion and evidence. Universities use pipelines to maintain learning platforms, identity systems, and research computing environments.

Core warning: automation does not automatically create security. It can also automate mistakes. DevSecOps is useful only when the pipeline itself is trustworthy.

3. Prerequisites

You need six ideas. First, source code is human-written instructions for software. Second, Git records changes to files. Third, cloud infrastructure includes compute, storage, networks, identities, keys, logs, and managed services. Fourth, identity and authorization decide who or what can perform actions. Fifth, testing checks whether software behaves as expected. Sixth, logging provides evidence after actions happen.

If those ideas are new, use this analogy. A pipeline is like a factory line. Source code is the design. Tests are quality checks. Scanners are safety inspections. A release gate is the supervisor's approval point. IaC is the blueprint for the building itself. Monitoring is the security camera. Rollback is the emergency plan for returning to a safe state.

4. Core Concepts

4.1 From DevOps to DevSecOps

DevOps tries to reduce the gap between building software and operating it. DevSecOps adds security into that same flow. Security should not appear only as a late audit. Late security feedback is expensive, slow, and often ignored under release pressure.

The goal is not to replace security professionals. The goal is to make good security practices part of normal engineering work: secure defaults, automated checks, clear ownership, fast feedback, evidence, and continuous improvement.

4.2 CI/CD from first principles

Continuous Integration, or CI, means code changes are integrated frequently and checked automatically. A CI system may run unit tests, linting, dependency scanning, secret scanning, build steps, and artifact creation.

Continuous Delivery means validated changes are kept ready for release, usually requiring a human approval before production. Continuous Deployment means validated changes are automatically deployed to production. Continuous deployment is powerful but risky for high-impact systems unless controls are very mature.

4.3 Pipeline, job, step, and runner

A pipeline is a sequence of automated work. A job is one group of pipeline tasks. A step is one command or action inside a job. A runner is the machine or container that executes the job. Runners are security-sensitive because they may receive source code, credentials, build artifacts, and deployment permissions.

4.4 Trust boundaries in a pipeline

A trust boundary is a place where control changes. Examples: developer laptop to source repository, source repository to CI runner, CI runner to artifact registry, registry to production deployment, deployment system to cloud control plane. Each boundary needs authentication, authorization, logging, and integrity checks.

4.5 Artifacts, provenance, and attestations

An artifact is an output of a build, such as a container image, binary, package, or deployment bundle. Provenance is evidence about how that artifact was produced. An attestation is a signed statement about some property of the artifact, such as which source commit and build system created it. These help answer: "Is this the artifact we intended to ship?"

They do not prove the application is vulnerability-free. They strengthen integrity and traceability.

4.6 Release gates

A release gate is a control point before a change moves to another environment. A gate can be automatic, manual, or mixed. Examples include passing unit tests, no high-severity unreviewed dependency findings, clean secret scan, reviewed Terraform plan, signed artifact, approved change ticket, and rollback plan.

4.7 Infrastructure as Code

IaC describes infrastructure using files. Common IaC tools include Terraform-style declarative configuration, cloud-native templates, Kubernetes manifests, and configuration-management tools. A declarative file says what the desired state should be. The tool compares desired state with actual state and creates, changes, or deletes resources.

IaC improves security when it is reviewed, scanned, tested, and governed. It increases risk when teams copy untrusted modules, commit secrets, use wildcard permissions, skip reviews, or auto-apply changes without understanding impact.

4.8 Plan and apply

A plan previews intended infrastructure changes. An apply executes them. For production, a safer workflow usually requires a reviewed plan before apply. Auto-approval may be acceptable in tightly controlled low-risk environments, but it is dangerous as a blanket default.

4.9 Policy as code

Policy as code expresses rules in files that machines can evaluate. A policy might reject public storage, privileged containers, wildcard administrator policies, missing encryption, or missing tags. Policy as code is useful because the rules can be version-controlled, tested, reviewed, and applied consistently.

4.10 Secrets in pipelines

A secret is sensitive material such as an API key, token, password, private key, or cloud credential. Pipelines need secrets to deploy, but secrets must be scoped, rotated, logged carefully, and protected from untrusted code. Pull requests from untrusted sources should not receive powerful production secrets.

4.11 Deployment identities

A deployment identity is the user, service account, role, or workload identity used by the pipeline to change infrastructure. It should have least privilege. One global administrator key for all deployments is a major failure point.

4.12 Drift

Drift is the difference between intended configuration and actual deployed state. Drift can happen when someone manually changes cloud settings, emergency fixes bypass IaC, resources are edited by another tool, or deployment fails halfway. Drift matters because documentation and reality disagree.

4.13 Environments and promotion

Common environments include development, testing, staging, and production. Promotion moves a change through these environments. The point is to catch problems before production. Production should have stricter permissions, approvals, logging, and rollback requirements than development.

4.14 Rollback and recovery

Rollback means returning to a known-good version or configuration. Not every infrastructure change is safely reversible. Deleting a database, changing identity policies, or rotating keys can have irreversible effects unless backups and recovery plans exist.

7. Mathematical Foundations

DevSecOps decisions often use simple measurement. The mathematics is not advanced, but it forces clarity.

7.1 Pipeline pass rate

If 80 out of 100 pipeline runs pass, the pass rate is 80 / 100 = 0.80 = 80%. A low pass rate may mean poor test quality, unstable dependencies, flaky environments, or unclear requirements.

7.2 Change failure rate

If 6 out of 50 production deployments require rollback or urgent fixes, the change failure rate is 6 / 50 = 12%. This matters because frequent unsafe deployments reduce trust in automation.

7.3 Mean time to recovery

If three incidents take 30, 60, and 90 minutes to recover, the mean time to recovery is (30 + 60 + 90) / 3 = 60 minutes. Better rollback design should reduce this value.

7.4 Simple release risk score

A beginner score can use:

release risk = impact × likelihood × exposure × reversibility factor

Example: impact = 5, likelihood = 3, exposure = 4, reversibility factor = 2 gives 5 × 3 × 4 × 2 = 120. The score is not universal truth. It is a decision aid for comparing changes.

8.3 Python defensive automation

The following script reviews local training files for beginner pipeline and IaC risks. It does not connect to any service.

# iac_review.py
# Purpose: perform a beginner static review of local CI/CD and IaC files.
# Safety: this script reads local files only. It does not contact GitHub, a cloud
# provider, Terraform Cloud, a registry, or any network target.

from pathlib import Path
import re

CHECKS = [
    ("workflow uses broad write-all permissions", r"permissions:\s*write-all"),
    ("workflow pulls a shell script directly into bash", r"curl\s+[^\n|]+\|\s*bash"),
    ("workflow runs auto-approved infrastructure change", r"terraform\s+apply\s+.*-auto-approve"),
    ("action reference is not pinned to a version-like tag", r"uses:\s*[^\s]+@(main|master)"),
    ("possible secret written to file", r"(SECRET|TOKEN|KEY|PASSWORD)\s*="),
    ("public storage protection disabled", r"block_public_(acls|policy)\s*=\s*false"),
    ("wildcard IAM action and resource", r"Action\s*=\s*\"\*\".*Resource\s*=\s*\"\*\""),
]

for path in sorted(Path('.').glob('**/*')):
    if path.is_dir() or path.suffix not in {'.yml', '.yaml', '.tf'}:
        continue

    text = path.read_text(encoding='utf-8', errors='replace')
    findings = []

    for label, pattern in CHECKS:
        if re.search(pattern, text, flags=re.IGNORECASE | re.DOTALL):
            findings.append(label)

    if findings:
        print(f"\n{path}")
        for finding in findings:
            print(f" - {finding}")

8.4 Network example

A deployment pipeline should not have unrestricted network access. A safer design limits the runner to required repositories, registries, package mirrors, and cloud APIs. Logs should record outbound destinations used during builds.

8.5 Web example

A web application pipeline might run unit tests, dependency scanning, SAST, API contract checks, DAST against a staging URL, security-header checks, and then require approval before production deployment.

8.6 Cloud example

A cloud IaC workflow should produce a plan, check public exposure, check encryption, check IAM privilege expansion, require approval for production, apply with a scoped deployment role, then verify logs and alerts.

9. Code Examples

9.1 Risky workflow for review

# unsafe-demo.yml
# This workflow is intentionally risky and is for review only.
# Do not use it in a real repository.
name: unsafe-demo
on: [push]
jobs:
  deploy:
    runs-on: ubuntu-latest
    permissions: write-all
    steps:
      - uses: actions/checkout@main
      - run: echo "DEPLOY_KEY=example-secret" >> .env
      - run: curl https://example.invalid/install.sh | bash
      - run: terraform apply -auto-approve

Line-by-line concerns: on: [push] runs for every push. permissions: write-all is too broad. checkout@main follows a moving branch. Writing a key-like value into .env risks leakage. Piping downloaded content into bash trusts remote code without review. terraform apply -auto-approve can change infrastructure without human review.

9.2 Safer workflow pattern

# safer-demo.yml
# Purpose: demonstrate safer CI/CD structure for a training repository.
# Safety: this example uses placeholder domains and a plan-only Terraform command.
name: safer-demo

on:
  pull_request:
    branches: ["main"]

permissions:
  contents: read
  security-events: write

jobs:
  validate:
    runs-on: ubuntu-24.04
    steps:
      - name: Check out repository at the reviewed commit
        uses: actions/checkout@v4

      - name: Set up Python for local validation scripts
        uses: actions/setup-python@v5
        with:
          python-version: "3.12"

      - name: Run unit tests
        run: python -m unittest discover -s tests

      - name: Validate Terraform formatting
        run: terraform fmt -check -recursive

      - name: Create a Terraform plan only
        run: terraform plan -input=false -out=tfplan

      - name: Upload review artifact
        uses: actions/upload-artifact@v4
        with:
          name: terraform-plan
          path: tfplan

This example uses pull requests, narrower permissions, specific action versions, tests, formatting checks, a plan-only Terraform step, and an uploaded review artifact. It still needs real-world additions such as pinned action digests where appropriate, secret handling, environment protection, artifact signing, and deployment approvals.

9.3 Risky Terraform-style example

# risky-storage.tf
# Intentionally risky example for review only.
resource "aws_s3_bucket" "training" {
  bucket = "northbridge-public-training-data"
}

resource "aws_s3_bucket_public_access_block" "training" {
  bucket                  = aws_s3_bucket.training.id
  block_public_acls       = false
  block_public_policy     = false
  ignore_public_acls      = false
  restrict_public_buckets = false
}

resource "aws_iam_policy" "too_broad" {
  name = "training-too-broad"
  policy = jsonencode({
    Version = "2012-10-17"
    Statement = [{
      Effect   = "Allow"
      Action   = "*"
      Resource = "*"
    }]
  })
}

The storage public-access controls are disabled, and the IAM policy grants all actions on all resources. This is a training example of excessive exposure and excessive privilege.

9.4 Safer Terraform-style pattern

# safer-storage.tf
# Safer pattern for a fictional training bucket.
# This file is not meant to be applied to a real cloud account in this course.
resource "aws_s3_bucket" "training" {
  bucket = "northbridge-private-training-data-example"
}

resource "aws_s3_bucket_public_access_block" "training" {
  bucket                  = aws_s3_bucket.training.id
  block_public_acls       = true
  block_public_policy     = true
  ignore_public_acls      = true
  restrict_public_buckets = true
}

resource "aws_s3_bucket_server_side_encryption_configuration" "training" {
  bucket = aws_s3_bucket.training.id

  rule {
    apply_server_side_encryption_by_default {
      sse_algorithm = "AES256"
    }
  }
}

This pattern blocks public access and enables basic server-side encryption. It is not complete production security, but it shows the direction: secure defaults, explicit settings, and reviewable configuration.

9.5 Policy-as-code example

# training_policy.rego
# Simplified policy-as-code example for teaching only.
# It demonstrates the idea of denying risky infrastructure changes.
package training.iac

deny[msg] {
  input.resource_type == "storage_bucket"
  input.public_access == true
  msg := "Public storage buckets require a documented exception."
}

deny[msg] {
  input.resource_type == "iam_policy"
  input.action == "*"
  input.resource == "*"
  msg := "Wildcard administrator policies are not allowed by default."
}

This simplified policy demonstrates the idea of denying risky conditions. Production policies need testing, versioning, exceptions, ownership, and clear error messages.

10. Security Perspective

Common attacks

  • Stealing CI/CD secrets or deployment tokens.
  • Changing pipeline definitions to run malicious steps.
  • Compromising a runner and tampering with artifacts.
  • Abusing broad cloud deployment permissions.
  • Injecting malicious dependencies or build scripts.
  • Using IaC misconfigurations to expose storage, databases, or management interfaces.

Common vulnerabilities

  • Long-lived administrator credentials in pipelines.
  • Unprotected branches and no peer review.
  • Untrusted pull requests receiving secrets.
  • Runners reused across trust levels.
  • No artifact signing or provenance.
  • IaC with public access, wildcard IAM, disabled logs, missing encryption, and no rollback plan.

Defensive strategies

  • Use protected branches and required reviews.
  • Use least-privilege, short-lived deployment credentials.
  • Separate build, test, staging, and production identities.
  • Scan secrets, dependencies, containers, and IaC.
  • Require plan review and policy-as-code checks.
  • Sign artifacts and collect provenance.
  • Monitor pipeline logs, cloud audit logs, and deployment events.

Detection methods

  • Alert on pipeline definition changes.
  • Alert on new secrets, new deployment identities, or new wildcard permissions.
  • Compare IaC state with cloud reality to find drift.
  • Detect unusual runner network destinations.
  • Detect production deploys outside approved windows or approval paths.

Mitigation and best practices

Use defense in depth: strong repository governance, trusted build systems, minimal runner privileges, isolated environments, secret management, artifact integrity, IaC review, policy checks, audit logging, and rehearsed rollback.

11. Industry Perspective

  • Banks: use controlled promotion, separation of duties, audit evidence, approval workflows, and rollback plans for transaction systems.
  • Governments: need traceable changes, compliance evidence, software supply-chain controls, and strict identity governance.
  • Cloud providers: rely on automation, policy enforcement, telemetry, and blast-radius reduction at large scale.
  • Healthcare: must protect patient systems and records while maintaining availability and change traceability.
  • Defense: emphasizes provenance, controlled build environments, least privilege, and strong authorization boundaries.
  • Universities: use pipelines for websites, identity services, research platforms, and teaching systems; the challenge is often decentralized ownership.
  • Large enterprises: need standard templates, reusable modules, governance, exception handling, and pipeline observability across many teams.

12. Research Perspective

Current research explores how to verify policies before deployment, how to generate least-privilege IaC automatically, how to measure SBOM and provenance quality, how to detect build-system compromise, and how to reduce false positives in IaC scanners. Another active area is AI-assisted infrastructure generation. Large language models can generate syntactically valid configuration, but semantic correctness, least privilege, and hidden security assumptions remain difficult.

Open problems include: proving that a pipeline executed exactly as claimed, creating portable policy languages across clouds, detecting drift without overwhelming teams, securing ephemeral runners, protecting secrets during untrusted contribution workflows, and making supply-chain frameworks easier to adopt in small organizations.

13. Case Studies

13.1 SolarWinds

The SolarWinds compromise showed how damaging build and software supply-chain compromise can be. The broad lesson for this course is that trusted build and release systems are high-value security assets, not background automation.

13.2 Codecov

The Codecov incident involved unauthorized modification of a script used in CI environments. The lesson is that CI/CD scripts can become a path to secrets and downstream systems when teams run remote tooling with broad access.

13.3 GitHub Actions and leaked tokens

Public repositories often expose risky workflow patterns, broad token permissions, and accidental secrets. The lesson is to treat workflow configuration as production code and review it with the same seriousness as application code.

13.4 Cloud storage exposure

Many organizations have accidentally exposed storage because of public access rules, weak review, or manual changes. IaC can prevent this only when secure templates, policy checks, and drift detection are actually used.

14. Common Misconceptions

  • Misconception: DevSecOps means developers do all security work. Correction: security expertise still matters; DevSecOps integrates that expertise into engineering flow.
  • Misconception: IaC is automatically secure. Correction: IaC makes infrastructure reviewable; it can also repeatedly deploy insecure settings.
  • Misconception: Passing a scanner means the release is safe. Correction: scanners are useful but incomplete.
  • Misconception: Production auto-deploy is always mature. Correction: automatic production deployment requires strong testing, rollback, observability, and risk tolerance.
  • Misconception: A signed artifact has no vulnerabilities. Correction: a signature helps verify origin and integrity, not absence of defects.
  • Misconception: Manual cloud console changes are safer than code. Correction: manual changes often lack review, reproducibility, and evidence.

15. Summary

DevSecOps integrates security into software delivery and operations. CI/CD pipelines run tests, scans, builds, release gates, deployments, and monitoring. Infrastructure as Code makes cloud resources reviewable and repeatable, but it must be paired with policy checks, least privilege, evidence, and drift detection. The secure pattern is not blind automation. It is controlled automation with clear ownership, small privileges, documented exceptions, observable outcomes, and practiced rollback.

iac_review.py
# iac_review.py
# Purpose: perform a beginner static review of local CI/CD and IaC files.
# Safety: this script reads local files only. It does not contact GitHub, a cloud
# provider, Terraform Cloud, a registry, or any network target.

from pathlib import Path
import re

CHECKS = [
    ("workflow uses broad write-all permissions", r"permissions:\s*write-all"),
    ("workflow pulls a shell script directly into bash", r"curl\s+[^\n|]+\|\s*bash"),
    ("workflow runs auto-approved infrastructure change", r"terraform\s+apply\s+.*-auto-approve"),
    ("action reference is not pinned to a version-like tag", r"uses:\s*[^\s]+@(main|master)"),
    ("possible secret written to file", r"(SECRET|TOKEN|KEY|PASSWORD)\s*="),
    ("public storage protection disabled", r"block_public_(acls|policy)\s*=\s*false"),
    ("wildcard IAM action and resource", r"Action\s*=\s*\"\*\".*Resource\s*=\s*\"\*\""),
]

for path in sorted(Path('.').glob('**/*')):
    if path.is_dir() or path.suffix not in {'.yml', '.yaml', '.tf'}:
        continue

    text = path.read_text(encoding='utf-8', errors='replace')
    findings = []

    for label, pattern in CHECKS:
        if re.search(pattern, text, flags=re.IGNORECASE | re.DOTALL):
            findings.append(label)

    if findings:
        print(f"\n{path}")
        for finding in findings:
            print(f" - {finding}")
