Lesson 10.2: Cloud IAM and Identity Governance

Cloud security usually fails through identity before it fails through packets. This lesson teaches cloud identity and access management from zero: who can request access, how the cloud decides, why permissions become excessive, and how governance keeps access justified, temporary, observable, and reviewable.

1. Introduction

Cloud IAM means cloud identity and access management. It answers a security question: who or what is allowed to perform which action on which cloud resource under which conditions?

Plain-English analogy: a university building may have students, lecturers, cleaners, researchers, administrators, and emergency staff. They should not all have the same keys. Some people may enter classrooms, some may enter labs, some may approve grade changes, and some emergency keys should work only under strict logging and approval. Cloud IAM is the digital version of this key system, but it is faster, larger, and API-driven.

Cloud IAM exists because cloud environments contain many resources: projects, subscriptions, accounts, networks, databases, storage buckets, virtual machines, serverless functions, logs, keys, secrets, and deployment pipelines. Each resource can be changed through a control plane. If the wrong identity receives too much permission, one compromised account can create servers, copy data, disable logs, delete backups, or change security settings.

Cloud IAM matters because identity is often the real perimeter in cloud systems. Traditional network location still matters, but a cloud API may accept requests from anywhere if the credential is valid and authorized. That makes identity governance central to banks, governments, healthcare, defense, universities, cloud providers, and large enterprises.

Who uses IAM? Developers use it to deploy applications. Security engineers use it to reduce attack paths. Cloud administrators use it to assign roles. Auditors use it to prove access is justified. Incident responders use it to reconstruct what happened. Researchers study it because large IAM systems are hard for humans to reason about correctly.

2. Learning Outcomes

After finishing this lesson, you should be able to:

  1. Define identity, principal, subject, group, role, permission, policy, resource, condition, and credential.
  2. Distinguish authentication from authorization in cloud systems.
  3. Explain how human identities differ from service accounts and workload identities.
  4. Describe how temporary credentials reduce the risk of long-lived key compromise.
  5. Explain least privilege, privilege creep, access reviews, and just-in-time privileged access.
  6. Read a simple IAM binding and identify risky permissions.
  7. Explain how federation, single sign-on, SAML, OpenID Connect, and SCIM fit into identity governance.
  8. Identify common cloud IAM attack paths and defensive controls.
  9. Design a basic identity-governance process for a fictional cloud environment.
  10. Produce an evidence-based IAM review report without touching real cloud accounts.

3. Prerequisites

Identity is a digital representation of a person, system, device, or workload. Authentication proves that an identity is being used by the right claimant. Authorization decides what the authenticated identity may do. Logging records evidence of security-relevant events. Cloud control plane means the APIs and consoles that create, modify, or delete cloud resources.

Before cloud IAM, imagine a local Linux system. A user belongs to groups. Groups have permissions on files. A process runs as a user. Logs record logins and privileged actions. Cloud IAM generalizes these ideas across thousands of remote resources and APIs.

4. Core Concepts

4.1 The IAM decision question

Every cloud access decision can be reduced to five parts:

  1. Principal: who or what is making the request.
  2. Action: what operation is requested, such as read object, create VM, delete key, or update policy.
  3. Resource: the target of the action, such as a bucket, database, virtual machine, project, or secret.
  4. Context: extra facts, such as time, source network, device posture, MFA status, request tags, or workload identity.
  5. Policy: the rule set used to allow or deny the request.

Intuition: IAM is a permission engine. It does not ask whether the request is morally good. It checks whether the rules allow it. Bad rules can allow dangerous actions.

4.2 Principals

A principal is an entity that can request access. Examples include a human user, group, application, virtual machine, serverless function, container workload, external partner, CI/CD pipeline, or emergency administrator account.

Beginner mistake: thinking only people have identity. In cloud systems, software often has identity too. A web application may use a workload role to read a database. A backup job may use a service account to write backup objects. A CI/CD pipeline may use a deployment identity to release software.

4.3 Credentials and authenticators

A credential is evidence used to authenticate. Examples include passwords, passkeys, hardware security keys, certificates, OAuth tokens, access keys, SSH keys, and temporary security tokens. A stolen credential can let an attacker act as the identity until it expires, is revoked, or is blocked.

Long-lived static access keys are risky because they may be copied into source code, build logs, laptops, chat messages, tickets, or screenshots. Temporary credentials are safer because they expire automatically and are usually issued after a policy decision.

4.4 Roles, permissions, and policies

A permission allows an action. A role is a named collection of permissions. A policy attaches permissions, roles, conditions, and sometimes deny rules to principals or resources. A binding connects a principal to a role on a resource.

Example: group:portal-developers has the role app.deployer on project/student-portal-dev. That does not automatically mean the group should deploy to production. Scope matters.

4.5 Resource hierarchy and scope

Cloud resources usually sit inside a hierarchy: organization, account, folder, subscription, project, resource group, service, and individual resource. Permissions granted high in the hierarchy may flow down to many child resources. This inheritance is useful, but dangerous if broad roles are assigned too high.

4.6 Federation and single sign-on

Federation allows one identity system to trust assertions from another identity system. A university may use an identity provider for single sign-on to cloud platforms. Protocols such as Security Assertion Markup Language (SAML) and OpenID Connect (OIDC) are common federation technologies.

Federation reduces password sprawl, but it increases dependence on identity-provider security. A compromised identity provider or misconfigured federation trust can affect many applications at once.

4.7 Workload identity

A workload identity is an identity assigned to software. It lets applications access cloud resources without embedding static secrets in code. For example, a container running the student portal can receive short-lived credentials to read only the database rows it needs.

Workload identity is safer than hardcoded keys, but only when permissions are narrow, metadata access is protected, tokens are short-lived, and logs are monitored.

4.8 Privileged access

Privileged access means access that can cause major harm, such as administrator, owner, billing admin, key admin, security admin, or policy admin. Privileged access should be rare, time-limited, strongly authenticated, approved when appropriate, and heavily logged.

Just-in-time access grants privilege only when needed. Just-enough-administration grants only the specific permissions needed for the task. These reduce standing privilege.

4.9 Identity lifecycle

Identity governance follows the joiner-mover-leaver lifecycle:

  • Joiner: a person or workload is created and receives initial access.
  • Mover: a role changes, so access must change.
  • Leaver: a person leaves or a workload retires, so access must be removed.

Failures in this lifecycle create stale accounts, orphaned service accounts, privilege creep, and unmanaged keys.

4.10 Access reviews and governance

An access review asks whether current access is still necessary, appropriate, and approved. Good reviews include asset owner sign-off, evidence of last use, business justification, risk rating, and remediation tracking. A weak review is just an email asking “Is this okay?” with no evidence.

4.11 Deny by default and conditions

Deny by default means access is blocked unless a specific rule allows it. Conditions make decisions more precise: allow deployment only from an approved pipeline, only to development, only with MFA, only during an approved window, or only from a managed device.

4.12 IAM attack paths

An attack path is a chain of permissions and resources that lets an attacker move from one compromised identity to a more valuable asset. Example: a developer can update a build pipeline; the pipeline has production deployment rights; production can read secrets; the secrets can decrypt database backups. Each step may look reasonable alone, but together they create excessive power.

7. Mathematical Foundations

7.1 Set model of permissions

Think of permissions as sets. If a user belongs to two groups, effective permissions are often the union of permissions from both groups, minus explicit denies or constraints.

Simple notation:

EffectivePermissions = (RoleA ∪ RoleB ∪ DirectGrants) - ExplicitDenies

If RoleA = {read, write} and RoleB = {deploy}, then the union is {read, write, deploy}. If an explicit deny removes deploy, the effective result is {read, write}.

7.2 Risk scoring for IAM findings

A beginner-safe risk score can use a simple product:

Risk = Impact × Exposure × Likelihood

Worked example: an unused contractor account has owner access to all resources. Impact is 5 because it can change critical systems. Exposure is 5 because the scope is global. Likelihood is 3 because the account is stale and not monitored. The score is 5 × 5 × 3 = 75. That should be prioritized before a low-impact read-only role on a test project.

7.3 Privilege reduction metric

If a role originally has 120 permissions and the redesigned role has 18, the reduction is:

Reduction percentage = ((120 - 18) / 120) × 100 = 85%

This metric is useful, but incomplete. A role with only three dangerous permissions may still be high risk. Count permissions, but also examine what each permission can do.

8.3 Python example: local IAM policy review

# cloud_iam_review.py
# Purpose: review a fictional cloud IAM policy file for governance risks.
# Safety: this script reads a local JSON file only. It does not connect to any cloud account.

import json
from pathlib import Path

policy_file = Path("fictional-iam-bindings.json")

with policy_file.open("r", encoding="utf-8") as f:
    data = json.load(f)

high_risk_terms = ["admin", "owner", "*", "administrator"]

for binding in data.get("bindings", []):
    principal = binding.get("principal", "unknown")
    role = binding.get("role", "unknown")
    resource = binding.get("resource", "unknown")
    conditions = binding.get("conditions", [])
    last_used_days = binding.get("last_used_days", None)
    mfa = binding.get("mfa_required", False)

    findings = []
    role_lower = role.lower()

    if any(term in role_lower for term in high_risk_terms):
        findings.append("privileged or broad role")
    if resource == "*" or resource.endswith("/*"):
        findings.append("broad resource scope")
    if not conditions:
        findings.append("no contextual conditions")
    if principal.startswith("user:") and not mfa:
        findings.append("human principal without MFA requirement")
    if last_used_days is not None and last_used_days > 90:
        findings.append("stale access not used in over 90 days")

    if findings:
        print(f"REVIEW: {principal} -> {role} on {resource}")
        for item in findings:
            print(f" - {item}")
    else:
        print(f"OK: {principal} -> {role} on {resource}")

8.4 Network example

Network controls and IAM controls answer different questions. A firewall asks whether traffic is allowed to reach an endpoint. IAM asks whether the requester may perform an action. In cloud systems, a request from an allowed network can still be denied by IAM, and a valid identity may still be blocked by network segmentation.

8.5 Web example

A web application may redirect a user to an identity provider. After login, the application receives claims such as subject, email, groups, and authentication method. The application must still authorize each action. Being logged in does not mean the user may download every record.

8.6 Cloud example

A storage administrator role on a development bucket may be acceptable for a small team. The same role at organization scope may allow access to sensitive production data. Scope changes risk.

9.2 Teaching-only authorization function

# This is educational pseudocode, not a full OIDC implementation.
# In real systems, use a maintained identity library and validate issuer,
# audience, signature, expiration, nonce, and claims.

def decide_access(claims, requested_resource, requested_action):
    """Return a teaching-only decision for a fictional cloud request."""
    subject = claims.get("sub")
    groups = claims.get("groups", [])
    tenant = claims.get("tenant")

    if tenant != "northbridge-training":
        return False, "wrong tenant"

    if requested_action == "deploy" and "student-portal-developers" in groups:
        if requested_resource == "student-portal-dev":
            return True, f"{subject} may deploy to development"
        return False, "developers may not deploy to this resource"

    return False, "no matching authorization rule"

Every line of the function is deliberately simple. claims represents identity facts from a trusted login process. requested_resource and requested_action represent the operation. The function checks tenant, group membership, action, and resource. Real systems need maintained libraries, signed-token validation, expiry checks, nonce checks, issuer checks, audience checks, and centralized policy management.

9.3 Common coding mistakes

  • Trusting claims without validating tokens: attackers may forge unsigned or incorrectly validated tokens.
  • Authorizing only at the page level: APIs and object access still require checks.
  • Hardcoding secrets: source repositories and logs can leak them.
  • Using broad service roles: one compromised workload can affect unrelated resources.
  • Skipping logs: no evidence exists when access is abused.

10. Security Perspective

Common attacks and vulnerabilities

  • Leaked long-lived access keys.
  • Over-permissive administrator or owner roles.
  • Stale contractor accounts and orphaned service accounts.
  • Privilege creep after role changes.
  • Weak federation trust or incorrect token validation.
  • MFA fatigue and weak privileged-account protection.
  • Service account impersonation abuse.
  • Metadata-service credential theft from vulnerable applications.
  • CI/CD identities with production administrator rights.
  • Disabled or incomplete audit logging.

Defensive strategies

  • Use centralized identity federation for humans.
  • Require phishing-resistant MFA for privileged access where feasible.
  • Prefer roles and short-lived credentials over static access keys.
  • Use least privilege, resource scoping, and conditions.
  • Separate development, test, and production identities.
  • Use JIT and approval workflows for privileged access.
  • Automate joiner-mover-leaver lifecycle processes.
  • Review stale access and unused permissions.
  • Protect break-glass accounts with strict monitoring and tested procedures.
  • Continuously log and alert on privileged changes.

Detection methods

  • Alert on new administrator or owner grants.
  • Alert on access-key creation or use after long inactivity.
  • Alert on policy changes that introduce wildcards or organization-wide scope.
  • Alert on disabled identity logs or deleted audit sinks.
  • Monitor unusual sign-in location, impossible travel, device changes, and unfamiliar user agents.
  • Detect service accounts performing unusual actions or accessing new resources.

Mitigation

Mitigation usually means contain the identity, revoke sessions, rotate credentials, remove risky grants, preserve logs, verify affected resources, restore least privilege, and fix the governance process that allowed the access to exist.

11. Industry Perspective

  • Banks: use IAM governance to separate trading, payment, fraud, and administrative duties, with strict access reviews and audit evidence.
  • Governments: rely on federation, identity assurance, privileged-access controls, and policy enforcement for citizen services and classified or regulated workloads.
  • Cloud providers: build large IAM systems that enforce policies across millions of tenants and resources.
  • Healthcare: must control access to patient data, clinical systems, billing platforms, and research datasets.
  • Defense: emphasizes strong authentication, separation of duties, mission-specific authorization, and monitored privileged operations.
  • Universities: face complex identity governance because students, faculty, researchers, alumni, guests, and contractors frequently change roles.
  • Large enterprises: use identity governance, privileged access management, cloud IAM analysis, and automated certification campaigns to manage scale.

12. Research Perspective

Cloud IAM research is active because real IAM systems are expressive, distributed, and difficult to analyze manually. Important research directions include:

  • Policy verification: proving whether policies accidentally permit unsafe access.
  • Graph-based attack-path analysis: modeling identities, resources, roles, and trust relationships as graphs.
  • Usable least privilege: recommending smaller roles without breaking operations.
  • Continuous authorization: adapting access based on context, behavior, risk, and device state.
  • Federation security: analyzing SAML, OIDC, token handling, trust configuration, and session lifetime.
  • Workload identity: replacing static secrets with verifiable workload identity in containers, serverless, and CI/CD systems.
  • Identity forensics: reconstructing events across identity providers, cloud APIs, SaaS platforms, and endpoint telemetry.

Open problems include reducing role explosion, measuring effective permissions accurately, explaining policy decisions to humans, preventing toxic permission combinations, and creating governance workflows that are secure without blocking productivity.

13. Case Studies

Case Study 1: Capital One cloud breach

What happened: Public reporting described unauthorized access to cloud-hosted data through a chain involving application-layer weakness, cloud metadata access, and permissions.

Root cause: The incident is commonly used to teach how application flaws and cloud IAM scope can combine.

Impact: Large data exposure, regulatory scrutiny, legal consequences, and major remediation effort.

Lessons learned: Protect metadata access, scope workload permissions narrowly, monitor unusual data access, and test cloud-specific attack paths.

Case Study 2: Identity-provider compromise and downstream risk

What happened: Organizations increasingly depend on identity providers for many applications. If federation, support workflows, or privileged identity operations are compromised, many downstream applications may be affected.

Root cause: Centralized identity creates efficiency but also concentration risk.

Impact: Single sign-on environments can become broad access paths if not segmented and monitored.

Lessons learned: Protect identity providers as critical infrastructure, monitor support access, restrict privileged sessions, and maintain emergency revocation procedures.

Case Study 3: CI/CD secret exposure

What happened: Build pipelines sometimes contain tokens, signing keys, cloud credentials, or deployment roles.

Root cause: Automation identities are often overprivileged because teams prioritize deployment speed.

Impact: A pipeline compromise can become a production compromise.

Lessons learned: Use workload federation, short-lived credentials, least privilege, signed builds, protected branches, and auditable release approvals.

14. Common Misconceptions

  • Misconception: “IAM is only usernames and passwords.” Correction: IAM also includes workload identity, policies, roles, federation, lifecycle, privileged access, and logging.
  • Misconception: “If MFA is enabled, permissions do not matter.” Correction: MFA reduces credential-abuse risk but does not make excessive authorization safe.
  • Misconception: “Service accounts are less risky than users.” Correction: service accounts can be very risky because they may have production permissions and no human watching them.
  • Misconception: “Admin access is fine for automation because it is convenient.” Correction: compromised automation can operate at machine speed.
  • Misconception: “Access reviews are compliance paperwork.” Correction: good access reviews remove real attack paths.
  • Misconception: “Network controls replace IAM.” Correction: cloud API authorization still matters even when network paths are restricted.

15. Summary

Cloud IAM controls who or what can perform actions on cloud resources. The basic ingredients are principals, credentials, roles, permissions, policies, resources, and conditions. The major security goals are least privilege, strong authentication, temporary credentials, scoped workload identities, access reviews, privileged access governance, and reliable audit logs. The central lesson is simple: in the cloud, identity is a primary security boundary.

16. Glossary

Access Review
A periodic check that existing access is still necessary, approved, and appropriate.
Binding
A connection between a principal, role, and resource.
Break-Glass Account
An emergency account used when normal access paths fail, protected by strict controls.
Credential
Evidence used to authenticate an identity, such as a token, key, password, or certificate.
Federation
A trust relationship that lets one system rely on identity assertions from another system.
IAM
Identity and access management; the discipline of managing identities and permissions.
JIT Access
Just-in-time access; privilege granted temporarily when needed.
Least Privilege
Granting only the minimum access needed for a specific task.
OIDC
OpenID Connect; an identity layer on top of OAuth 2.0.
Principal
An entity that can request access, such as a user, group, service account, or workload.
Privilege Creep
Access accumulation over time as roles change without old permissions being removed.
Role
A named collection of permissions.
SAML
Security Assertion Markup Language; a federation standard used for exchanging authentication and authorization assertions.
SCIM
System for Cross-domain Identity Management; a standard often used to automate user provisioning and deprovisioning.
Service Account
A non-human identity used by software or automation.
Temporary Credential
A credential with a limited lifetime, usually issued after authentication and authorization checks.
Workload Identity
An identity assigned to running software such as a VM, container, function, or pipeline.

18. Practice Lab

Lab title

Fictional Cloud IAM Governance Review

Safety rules

  • Use only local fictional files.
  • Do not log into real cloud accounts.
  • Do not test real tenants, subscriptions, projects, or organizations.
  • Do not use real credentials, emails, tokens, or customer data.

Prerequisites

  • A text editor.
  • Python 3.
  • Basic JSON understanding.
  • Lesson 10.1 recommended.

Objectives

  1. Create a fictional IAM binding file.
  2. Run a local review script.
  3. Identify excessive privilege, broad scope, missing conditions, missing MFA, and stale access.
  4. Write a remediation plan.

Step 1: Create fictional-iam-bindings.json

{
  "environment": "Northbridge University training cloud",
  "bindings": [
    {"principal": "group:student-portal-developers", "role": "app.deployer", "resource": "project/student-portal-dev", "conditions": ["business_hours"], "last_used_days": 3, "mfa_required": true},
    {"principal": "user:alex.contractor@example.test", "role": "owner", "resource": "*", "conditions": [], "last_used_days": 147, "mfa_required": false},
    {"principal": "serviceAccount:ci-build@example.test", "role": "storage.admin", "resource": "project/student-portal-prod/*", "conditions": [], "last_used_days": 2, "mfa_required": false},
    {"principal": "workload:portal-api", "role": "database.reader", "resource": "database/student-records/read-only", "conditions": ["source_workload_is_portal_api"], "last_used_days": 0, "mfa_required": false}
  ]
}

Step 2: Create cloud_iam_review.py

Use the Python script in Section 8.3.

Step 3: Run the script

python3 cloud_iam_review.py
sha256sum fictional-iam-bindings.json cloud_iam_review.py

Expected findings

  • The contractor user has owner access at global scope, no conditions, no MFA requirement, and stale use.
  • The CI build service account has a broad storage administrator role over production resources.
  • The portal workload has narrower read-only access and a contextual condition.

Deliverable

Submit a two-page IAM governance review with an asset summary, risky bindings, risk scores, recommended replacement roles, lifecycle actions, detection alerts, and evidence hashes.
