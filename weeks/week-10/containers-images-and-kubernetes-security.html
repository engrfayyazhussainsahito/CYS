Lesson 10.3: Containers, Images, and Kubernetes Security

Containers make applications portable. Kubernetes makes many containers manageable. Security fails when teams treat that portability as automatic safety. This lesson starts from zero and builds toward graduate-level reasoning about container isolation, image supply chains, workload identity, cluster policy, runtime monitoring, and Kubernetes hardening.

Difficulty levelBeginner → MS foundation
Required backgroundWeeks 0–9 and Lessons 10.1–10.2 recommended; key ideas are reintroduced

Overview

Containers package an application and its runtime dependencies so it can run consistently across environments. Kubernetes coordinates containers across many machines. Together they support modern cloud-native systems, but they also create new places for security mistakes: vulnerable base images, leaked secrets, excessive container privileges, broad service-account permissions, weak registry controls, exposed control planes, missing network policies, and poor runtime visibility.

The core idea is simple: containers are not magic boxes. They are ordinary processes with packaging, namespaces, cgroups, filesystem layers, and runtime configuration. Kubernetes is not magic automation. It is a distributed control system that tries to make the actual state match the desired state. Security comes from controlling what gets built, what gets admitted, what runs, what it can access, how it communicates, and what evidence is recorded.

1. Introduction

Plain English: a container is a way to run an application in a controlled environment that carries its files and dependencies with it. Imagine a lunchbox: the meal, fork, napkin, and instructions are packed together. The lunchbox is not the building. It is a portable package. A container image is the packed lunchbox template. A running container is one opened lunchbox being used.

Containers exist because software often fails when moved between machines. A developer says, “It works on my laptop.” The production server has a different library version, environment variable, file path, or operating system configuration. Containers reduce this problem by packaging application dependencies and declaring runtime behavior.

They matter because banks, governments, healthcare systems, universities, cloud providers, defense teams, and large enterprises use containers for web APIs, data pipelines, machine-learning services, microservices, batch jobs, security tools, and internal platforms.

Kubernetes exists because running one container is easy, but running thousands reliably is hard. Teams need scheduling, health checks, scaling, service discovery, rollout, rollback, identity, secrets, storage, and policy. Kubernetes provides a common orchestration model.

Who uses these technologies? Developers build images. Platform engineers operate clusters. Security engineers write guardrails and detections. DevSecOps teams scan and sign artifacts. Incident responders investigate suspicious workloads. Researchers study misconfiguration detection, runtime isolation, policy verification, and supply-chain integrity.

3. Prerequisites

You need five ideas. First, a process is a running program. Second, an operating system kernel controls access to CPU, memory, files, and network. Third, identity controls who or what can access resources. Fourth, networking lets services communicate. Fifth, software supply chain means the chain from source code and dependencies to build artifacts and deployed software.

We will reintroduce each concept in container form. If you know Linux permissions, cloud IAM, and secure software, containers are a composition of those earlier topics.

4. Core Concepts

4.1 Containers as processes, not tiny virtual machines

A container usually runs as one or more processes on a host kernel. It may have isolated views of files, process IDs, users, network interfaces, and resource usage. That isolation is provided by operating-system features, not by a full guest operating system in the usual container model.

A virtual machine runs a guest operating system on virtualized hardware. A container shares the host kernel. This makes containers fast and efficient, but security depends heavily on kernel isolation, runtime configuration, privilege settings, and host hardening.

4.2 Images, layers, and registries

A container image is a read-only template. It contains a filesystem, metadata, default command, environment settings, and dependency layers. A layer is a filesystem change stacked on previous layers. A registry stores and distributes images.

Intuition: an image is a recipe plus packed ingredients. A registry is the warehouse. A running container is the meal prepared from that recipe.

4.3 Dockerfile security

A Dockerfile describes how to build an image. Dangerous patterns include using floating base tags, copying secrets into the image, installing unnecessary packages, running as root, skipping vulnerability scans, and failing to pin dependencies. Safer patterns include minimal base images, explicit versions, non-root users, no secrets in builds, reproducible dependencies, and clear runtime commands.

4.4 Image supply chain

Images inherit risk from base operating-system packages, language packages, build scripts, source repositories, CI/CD runners, registries, and signing systems. A vulnerability in a base image can affect every application that uses it. An attacker who can push an image to a trusted registry may compromise many deployments.

Defenders use scanning, SBOMs, provenance, signatures, registry access control, immutable tags or digests, and deployment policy. These controls reduce risk, but none proves an image is safe forever. New vulnerabilities can be discovered after an image is built.

4.5 Container runtime and host security

The container runtime starts and manages containers. The host still matters. If the host kernel, runtime, kubelet, or node credentials are compromised, container boundaries may not protect workloads. Host hardening, patching, logging, resource limits, and least privilege remain essential.

4.6 Kubernetes architecture

A Kubernetes cluster contains a control plane and worker nodes. The control plane exposes the API, stores cluster state, schedules workloads, and runs controllers. Worker nodes run pods through the kubelet, container runtime, and network components. etcd stores cluster state and must be protected carefully.

4.7 Desired state and reconciliation

Kubernetes works by desired state. You submit objects such as Deployments, Services, ConfigMaps, Secrets, and NetworkPolicies. Controllers continually compare desired state with actual state and make changes. Security policies should therefore control both configuration before admission and behavior after runtime.

4.8 Pods, deployments, services, and namespaces

A Pod is the smallest commonly scheduled Kubernetes unit. It can contain one or more containers that share network and storage context. A Deployment manages replica Pods and rollout. A Service provides stable network access. A Namespace groups resources and can help separate teams, environments, or applications.

4.9 Kubernetes identity and RBAC

Kubernetes has human users and service accounts. A service account gives a workload an identity. RBAC uses Roles, ClusterRoles, RoleBindings, and ClusterRoleBindings to control who can perform verbs such as get, list, create, update, delete, or watch on resources. Broad permissions can become privilege-escalation paths.

4.10 Security contexts

A security context defines privilege and access-control settings for Pods or containers. Examples include run as non-root, disallow privilege escalation, drop Linux capabilities, use a read-only root filesystem, set seccomp profile, and avoid privileged mode.

4.11 Secrets and ConfigMaps

A ConfigMap stores non-sensitive configuration. A Secret stores sensitive values such as tokens or passwords, but Kubernetes Secrets require careful handling. They are not automatically safe just because the object is named Secret. Encryption at rest, RBAC, workload scoping, external secret managers, and audit logs matter.

4.12 NetworkPolicy

Kubernetes networking is often permissive by default. NetworkPolicy can restrict pod-to-pod and pod-to-external communication when the cluster network plugin supports it. A common design is deny by default, then allow only required flows.

4.13 Admission control and policy as code

Admission control checks API requests before objects are stored. Admission policies can reject privileged Pods, require signed images, enforce non-root execution, block hostPath volumes, require resource limits, or require approved registries. Policy as code makes these decisions reviewable and testable.

4.14 Runtime detection

Build-time controls are not enough. Runtime monitoring watches process behavior, file writes, network connections, DNS queries, API calls, and unusual resource usage. Detection helps identify cryptomining, suspicious shell execution, unexpected package installation, lateral movement, or abnormal service-account use.

7. Mathematical Foundations

Container security uses practical measurement. The numbers are imperfect, but they help prioritize work.

7.1 Image risk score

A simple teaching model:

Image Risk = Vulnerability Severity × Exposure × Privilege × Deployment Count

Worked example: an image has a high-severity vulnerability scored 8 on a 0–10 teaching scale. It is internet-facing, so exposure is 3. It runs as non-root, so privilege is 1. It runs in 20 replicas.

Risk = 8 × 3 × 1 × 20 = 480

If the same image runs privileged with host access, privilege might be 5:

Risk = 8 × 3 × 5 × 20 = 2400

The vulnerability did not change. The deployment context changed the risk.

7.2 Vulnerability density

Vulnerability Density = Number of Findings / Image Size in MB

If image A has 30 findings and is 300 MB, density is 0.10 findings per MB. If image B has 15 findings and is 50 MB, density is 0.30 findings per MB. This does not prove B is worse; it only gives another review signal. Severity, exploitability, reachability, and business exposure matter more than raw count.

7.3 Availability and resource limits

Resource limits reduce noisy-neighbor and denial-of-service risk. If each Pod requests 250 millicores of CPU and a node has 4000 millicores allocatable, a rough upper bound is:

Maximum requested Pods = 4000 / 250 = 16 Pods

This is a scheduling estimate, not a security proof. Memory, storage, networking, and daemon overhead also matter.

Python example

The following script reviews local manifest files for beginner-pattern risks. It is intentionally simple and is not a replacement for professional policy tools.

# manifest_review.py
# Purpose: review local Kubernetes YAML-like text for beginner security risks.
# Safety: this script reads local files only. It does not connect to a cluster,
# cloud account, registry, or network service.

from pathlib import Path
import re

FILES = ["safe-pod.yaml", "risky-pod.yaml", "network-policy.yaml"]

CHECKS = [
    ("privileged container", r"privileged:\s*true"),
    ("runs as root or UID 0", r"runAsUser:\s*0\b"),
    ("privilege escalation allowed", r"allowPrivilegeEscalation:\s*true"),
    ("host namespace requested", r"host(Network|PID|IPC):\s*true"),
    ("host path volume", r"hostPath:"),
    ("latest tag used", r"image:\s*[^\s]+:latest"),
    ("missing resource limits", r"kind:\s*Pod(?:(?!limits:).)*$"),
]

for name in FILES:
    path = Path(name)
    if not path.exists():
        print(f"SKIP {name}: file not found")
        continue

    text = path.read_text(encoding="utf-8")
    print(f"\nReviewing {name}")
    findings = []

    for label, pattern in CHECKS:
        if re.search(pattern, text, flags=re.IGNORECASE | re.DOTALL):
            findings.append(label)

    if findings:
        print("Findings:")
        for finding in findings:
            print(f" - {finding}")
    else:
        print("No beginner-pattern findings detected. Manual review is still required.")

Network example

A web Pod should not automatically talk to every Pod. A safer design permits only required flows.

apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-web-to-api-only
spec:
  podSelector:
    matchLabels:
      app: api
  policyTypes:
    - Ingress
  ingress:
    - from:
        - podSelector:
            matchLabels:
              app: web
      ports:
        - protocol: TCP
          port: 8080

Web example

A containerized web API should receive configuration from the environment or a secret manager, not from hardcoded secrets in the image. Logs should avoid secrets, session tokens, and full request bodies containing private data.

Cloud example

A managed Kubernetes service still follows shared responsibility. The provider may operate parts of the control plane, but the customer usually remains responsible for workload configuration, image security, IAM, network policy, secrets handling, and application security.

9. Code Examples

9.1 Safer Dockerfile

# Use a small, specific base image tag instead of a floating "latest" tag.
FROM python:3.12-slim

# Create a non-root user for the application process.
RUN useradd --create-home --shell /usr/sbin/nologin appuser

# Set a working directory inside the image.
WORKDIR /app

# Copy only dependency metadata first to support repeatable builds.
COPY requirements.txt ./

# Install dependencies without retaining package-manager cache.
RUN pip install --no-cache-dir -r requirements.txt

# Copy the application code after dependencies.
COPY app.py ./

# Drop root privileges before the application starts.
USER appuser

# Document the port used by the application.
EXPOSE 8080

# Start the application with an explicit command.
CMD ["python", "app.py"]

Line-by-line reasoning: the base image is specific; a non-root user is created; dependencies are installed before code to support repeatable builds; package cache is not retained; root privilege is dropped before runtime; the startup command is explicit.

9.2 Risky Dockerfile

FROM python:latest
WORKDIR /app
COPY . .
RUN pip install -r requirements.txt
USER root
CMD python app.py

Common mistakes: latest is floating; COPY . . may include secrets or unnecessary files; package versions are not clearly controlled; the application runs as root; the command uses shell form, which can complicate signal handling and parsing.

9.3 Safer Pod manifest

apiVersion: v1
kind: Pod
metadata:
  name: safe-training-pod
  labels:
    app: training
spec:
  automountServiceAccountToken: false
  securityContext:
    runAsNonRoot: true
    seccompProfile:
      type: RuntimeDefault
  containers:
    - name: app
      image: example.invalid/training-app:1.0.0
      ports:
        - containerPort: 8080
      securityContext:
        allowPrivilegeEscalation: false
        readOnlyRootFilesystem: true
        capabilities:
          drop: ["ALL"]
      resources:
        requests:
          cpu: "100m"
          memory: "128Mi"
        limits:
          cpu: "500m"
          memory: "256Mi"

Security reasoning: the service-account token is not mounted automatically; the pod runs as non-root; seccomp uses the runtime default; privilege escalation is disabled; Linux capabilities are dropped; the root filesystem is read-only; resource requests and limits are declared.

9.4 Risky Pod manifest for review only

apiVersion: v1
kind: Pod
metadata:
  name: risky-training-pod
spec:
  hostNetwork: true
  containers:
    - name: app
      image: example.invalid/training-app:latest
      securityContext:
        privileged: true
        allowPrivilegeEscalation: true
        runAsUser: 0
      volumeMounts:
        - name: host-root
          mountPath: /host
  volumes:
    - name: host-root
      hostPath:
        path: /

This file is intentionally unsafe for local review. Do not apply it to a cluster. The risks include host networking, privileged mode, root execution, privilege escalation, a floating image tag, and host filesystem mounting.

10. Security Perspective

Common attacks and weaknesses

  • Image poisoning: a trusted-looking image contains malicious code or vulnerable dependencies.
  • Secret leakage: credentials are baked into images, environment variables, build logs, or manifests.
  • Overprivileged containers: workloads run as root, privileged, or with unnecessary Linux capabilities.
  • Host escape risk: a vulnerability or misconfiguration allows access beyond the container boundary.
  • Weak RBAC: service accounts can read secrets, create pods, or modify roles unnecessarily.
  • Exposed control plane: API endpoints or dashboards are reachable from untrusted networks.
  • Unrestricted pod traffic: missing network policy permits lateral movement.
  • Supply-chain drift: old images continue running after patches are available.
  • Runtime abuse: attackers use containers for cryptomining, scanning, data staging, or command execution.

Defensive strategies

  • Use trusted registries, image signatures, SBOMs, provenance, and vulnerability scanning.
  • Prefer minimal base images and rebuild regularly.
  • Run containers as non-root and drop unnecessary capabilities.
  • Set resource requests and limits.
  • Use admission control to enforce baseline policies.
  • Use RBAC least privilege and review service accounts.
  • Use NetworkPolicy to reduce lateral movement.
  • Protect Kubernetes API server, etcd, kubelet, and node credentials.
  • Collect audit logs, container runtime logs, DNS logs, and cloud control-plane logs.

Detection methods

  • Alert on privileged Pod creation, hostPath mounts, or host networking.
  • Alert on service accounts reading many secrets.
  • Alert on new ClusterRoleBinding to broad roles.
  • Monitor unusual CPU use, network connections, DNS queries, and shell execution.
  • Compare running images against approved digests.
  • Detect workloads launched from unapproved registries.

11. Industry Perspective

Banks use containers to deploy APIs and fraud-detection services, but they require strong identity, audit logs, image provenance, and segregation of duties. Governments use Kubernetes for digital services and often emphasize hardening baselines, supply-chain evidence, and policy enforcement. Cloud providers offer managed Kubernetes, registries, image scanning, workload identity, policy engines, and logging integrations.

Healthcare uses containers for patient portals, analytics, and interoperability APIs; privacy and recovery requirements are central. Defense environments care about strict isolation, signed artifacts, least privilege, disconnected or constrained networks, and tamper evidence. Universities use containers for research platforms and teaching labs; multi-tenant separation and cost controls matter. Large enterprises use platform teams to provide paved roads: approved images, templates, CI/CD controls, and cluster guardrails.

12. Research Perspective

Current research includes container escape analysis, Kubernetes misconfiguration detection, policy verification, SBOM accuracy, artifact signing and provenance, runtime anomaly detection, side-channel risks in multi-tenant environments, confidential containers, eBPF-based observability, and least-privilege synthesis for RBAC and NetworkPolicy.

Open problems remain. Scanners can disagree. SBOMs can be incomplete. Runtime behavior baselines can produce false positives. Kubernetes policy languages can be difficult to use correctly. Workload identity can be hard to reason about across cloud IAM, Kubernetes RBAC, and application-level authorization. Confidential-computing approaches can reduce some infrastructure trust assumptions, but they add complexity and do not automatically fix application vulnerabilities.

13. Case Studies

13.1 Tesla cloud cryptojacking report

Researchers reported that an exposed Kubernetes administrative console was abused in a cloud cryptomining incident. The lesson is not “Kubernetes is bad.” The lesson is that exposed admin surfaces, weak authentication, secrets in workloads, and poor runtime visibility can combine into a cloud compromise path.

13.2 Vulnerable public images

Many organizations inherit risk from base images and language packages. A container can be newly built but still contain old vulnerable packages. Security teams therefore need image inventories, rebuild processes, vulnerability triage, and ownership.

13.3 Misconfigured service accounts

In many cluster incidents, the first compromised workload is less important than what its service account can do. If a web Pod can read secrets or create privileged Pods, a small application flaw can become a cluster-level incident.

13.4 Missing network policy

Flat cluster networks allow compromised workloads to reach internal services. NetworkPolicy does not fix vulnerable applications, but it reduces blast radius by limiting unnecessary communication.

14. Common Misconceptions

  • “Containers are the same as virtual machines.” Incorrect. They usually share the host kernel and require different security assumptions.
  • “If the image scans clean, it is safe.” Incorrect. Scanners miss issues and new vulnerabilities appear later.
  • “Kubernetes secrets are automatically secure.” Incorrect. Access control, encryption, external secret management, and logs still matter.
  • “NetworkPolicy always works by default.” Incorrect. It depends on network plugin support and correct policy design.
  • “Non-root containers remove all risk.” Incorrect. They reduce privilege but do not fix vulnerable applications or broad identity permissions.
  • “Managed Kubernetes means the provider handles everything.” Incorrect. Shared responsibility still leaves workload, IAM, network, and data controls to the customer.
  • “Admission control replaces monitoring.” Incorrect. Admission prevents some bad states; runtime monitoring detects behavior after deployment.

15. Summary

Containers package software and dependencies. Kubernetes orchestrates containers across machines. Security depends on the full lifecycle: source, build, image, registry, admission, runtime, identity, networking, logging, and recovery. The safest mental model is layered defense: build trustworthy images, admit only policy-compliant workloads, run with least privilege, restrict communication, monitor behavior, patch continuously, and review access.

16. Glossary

Admission Controller
A Kubernetes component that can validate or mutate API requests before objects are stored.
Base Image
The starting image layer used to build another image.
Cluster
A group of machines and control-plane components managed as one Kubernetes environment.
Container
A packaged runtime environment for an application process using operating-system isolation features.
Container Image
A read-only template used to create containers.
Container Runtime
Software that starts and manages containers on a host.
etcd
Kubernetes key-value store that holds cluster state.
Image Layer
A filesystem change stacked as part of a container image.
Kubelet
Node agent that communicates with the Kubernetes API server and manages Pods on a node.
Kubernetes
An open-source system for automating deployment, scaling, and management of containerized applications.
Namespace
A Kubernetes grouping mechanism for resources.
NetworkPolicy
A Kubernetes resource used to control network traffic to or from selected Pods when supported by the network plugin.
Pod
The smallest commonly scheduled Kubernetes workload unit, usually containing one or more containers.
Registry
A service that stores and distributes container images.
Security Context
Pod or container privilege and access-control settings.
Service Account
A Kubernetes identity commonly used by workloads.
SBOM
Software Bill of Materials; an inventory of software components.
Workload Identity
A digital identity assigned to software or a runtime workload.

Lab: Local Container Manifest Review

Safety boundary: this lab uses only local text files. Do not connect to any real cluster. Do not scan internet systems. Do not pull or run untrusted images.

Prerequisites: a text editor and Python 3. Docker and kubectl are optional for local version checks and client-side manifest validation.

Objectives:

  1. Create three local files: safe-pod.yaml, risky-pod.yaml, and network-policy.yaml.
  2. Create manifest_review.py.
  3. Run the script locally.
  4. Write a report explaining each finding, risk, and safer alternative.

Expected outcomes: students should be able to identify privileged containers, root execution, hostPath mounts, host networking, floating image tags, missing resource limits, and network-policy intent.

manifest_review.py

# manifest_review.py
# Purpose: review local Kubernetes YAML-like text for beginner security risks.
# Safety: this script reads local files only. It does not connect to a cluster,
# cloud account, registry, or network service.

from pathlib import Path
import re

FILES = ["safe-pod.yaml", "risky-pod.yaml", "network-policy.yaml"]

CHECKS = [
    ("privileged container", r"privileged:\s*true"),
    ("runs as root or UID 0", r"runAsUser:\s*0\b"),
    ("privilege escalation allowed", r"allowPrivilegeEscalation:\s*true"),
    ("host namespace requested", r"host(Network|PID|IPC):\s*true"),
    ("host path volume", r"hostPath:"),
    ("latest tag used", r"image:\s*[^\s]+:latest"),
    ("missing resource limits", r"kind:\s*Pod(?:(?!limits:).)*$"),
]

for name in FILES:
    path = Path(name)
    if not path.exists():
        print(f"SKIP {name}: file not found")
        continue

    text = path.read_text(encoding="utf-8")
    print(f"\nReviewing {name}")
    findings = []

    for label, pattern in CHECKS:
        if re.search(pattern, text, flags=re.IGNORECASE | re.DOTALL):
            findings.append(label)

    if findings:
        print("Findings:")
        for finding in findings:
            print(f" - {finding}")
    else:
        print("No beginner-pattern findings detected. Manual review is still required.")
