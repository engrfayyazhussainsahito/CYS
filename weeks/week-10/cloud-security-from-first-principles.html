Lesson 10.1: Cloud Security from First Principles

This lesson teaches cloud security from zero. You will learn what cloud computing is, why responsibility is shared, how cloud systems are built, what commonly goes wrong, and how defenders design cloud environments that are observable, governed, resilient, and safe to operate.

1. Introduction

Cloud computing means using computing resources over a network instead of owning every server, disk, router, and data center yourself. A cloud provider offers resources such as virtual machines, storage buckets, managed databases, identity services, networking, serverless functions, logging, and deployment tools. Customers use provider APIs or consoles to create and manage those resources.

Plain-English analogy: renting an apartment is not the same as owning a building. The building owner may maintain the elevator, power, and walls. You still lock your door, protect your belongings, decide who gets a key, and report suspicious activity. Cloud security works similarly: the provider protects some layers; the customer protects others.

Cloud security exists because cloud systems are programmable, distributed, multi-tenant, internet-connected, and fast-moving. A developer can create infrastructure in minutes. That speed is useful, but it also means mistakes scale quickly. A public storage bucket, leaked access key, over-permissive role, disabled log, or exposed database can become an incident.

Cloud security matters because banks run payment systems in the cloud, governments host citizen services, hospitals process patient data, universities manage research platforms, cloud providers operate global infrastructure, defense organizations use cloud workloads with strict controls, and large enterprises depend on SaaS, IaaS, PaaS, containers, and CI/CD pipelines.

2. Learning Outcomes

After finishing this lesson, you should be able to:

  1. Define cloud computing, cloud provider, tenant, region, availability zone, service model, deployment model, and shared responsibility.
  2. Explain the difference between infrastructure as a service, platform as a service, software as a service, serverless, and managed services.
  3. Draw a basic cloud architecture with users, identity provider, load balancer, compute, database, storage, network boundaries, logging, and key management.
  4. Explain why cloud identity and APIs are central security boundaries.
  5. Identify common cloud misconfigurations without testing real systems.
  6. Map cloud risks to preventive, detective, corrective, and recovery controls.
  7. Explain why cloud logging, resource inventory, and configuration history are necessary for incident response.
  8. Review a fictional cloud design and produce a practical risk register.

3. Prerequisites

3.1 Server

A server is a computer that provides a service to other computers. In a cloud environment, the server may be a virtual machine, container, managed database, serverless function, or SaaS application rather than a physical box you can touch.

3.2 Network

A network moves data between systems. In cloud environments, networks are often virtual. You still think about addresses, routes, ports, firewalls, DNS, and segmentation, but you configure many of them through cloud APIs.

3.3 Identity

An identity represents a person, service, workload, or automation. In cloud security, identity is often more important than network location because almost every cloud action is an API call authorized by an identity.

3.4 Log

A log is an event record. Cloud logs can show who changed a firewall rule, who read an object, which function ran, which database connection failed, or which API call created a new key.

4. Core Concepts

4.1 Cloud provider

A cloud provider operates data centers, hardware, networks, virtualization systems, storage platforms, identity services, APIs, and managed products. Examples include major public cloud providers, private cloud platforms, and specialized SaaS providers.

4.2 Tenant

A tenant is a customer or organizational boundary inside a cloud platform. Multi-tenant platforms host many customers on shared provider infrastructure while keeping their data and control logically separated.

4.3 Virtualization

Virtualization lets one physical machine host multiple isolated virtual machines. The intuition is apartment units inside one building: each tenant has a separate unit even though the building infrastructure is shared.

4.4 Region and availability zone

A region is a geographic cloud area. An availability zone is a separate facility or failure domain within a region. Using multiple zones can improve availability, but it does not automatically make an application resilient. The application, database, networking, backups, and recovery procedures must be designed for failure.

4.5 Service models

  • IaaS: infrastructure as a service. The customer manages virtual machines, operating systems, applications, data, and many network settings.
  • PaaS: platform as a service. The provider manages more of the operating platform. The customer still manages application code, identity, data, configuration, and access policies.
  • SaaS: software as a service. The provider operates the application. The customer still manages users, data, configuration, integrations, retention, and governance.
  • Serverless: the provider runs code without the customer managing servers directly. The customer still secures code, identities, secrets, events, permissions, dependencies, and logs.

4.6 Shared responsibility

Shared responsibility means security duties are split between the provider and the customer. The exact split changes by service model. In IaaS, the customer usually has more responsibility for the operating system and application stack. In SaaS, the provider manages more of the stack, but the customer remains responsible for data use, identity, access policies, integrations, and user behavior.

Important distinction: cloud does not remove customer responsibility. It changes which responsibilities remain with the customer and which are delegated to the provider.

4.7 Control plane and data plane

The control plane is used to create, modify, delete, and configure resources. The data plane is where application data is stored, processed, or transmitted. Example: creating a storage bucket is a control-plane action; reading a file from that bucket is a data-plane action.

4.8 Cloud API

A cloud API is a programmable interface for cloud actions. Almost everything in cloud security flows through APIs: creating servers, changing firewall rules, reading secrets, starting functions, attaching roles, viewing logs, and deleting resources. API permissions therefore become a major security boundary.

4.9 Identity and access management

Identity and access management, or IAM, decides who or what can perform which action on which resource under which conditions. Cloud IAM commonly includes users, groups, roles, service accounts, workload identities, policies, permissions, and trust relationships.

4.10 Compute

Compute means processing power. Cloud compute can appear as virtual machines, containers, managed container services, serverless functions, batch jobs, or platform runtimes. Security questions include: What code runs? Which identity does it use? Which network can reach it? Which secrets can it read? Which logs does it produce?

4.11 Storage

Cloud storage includes object storage, block storage, file storage, databases, snapshots, backups, and data warehouses. Storage security requires access control, encryption, retention, classification, backup, deletion policy, and monitoring for unusual access.

4.12 Virtual private cloud

A virtual private cloud, often called VPC or virtual network, is a logically isolated cloud network. It may contain subnets, route tables, gateways, network ACLs, security groups, private endpoints, and peering connections.

4.13 Public exposure

A resource is publicly exposed when it can be reached from the internet or by a broad unauthenticated audience. Public exposure is not always wrong: public websites need public entry points. The problem is accidental exposure of management interfaces, storage, databases, debug services, or internal APIs.

4.14 Encryption and key management

Cloud services often support encryption at rest and in transit. Encryption is not complete without key management. Defenders must know who controls keys, who can use keys, who can rotate keys, who can disable keys, and which logs prove key usage.

4.15 Cloud logging

Cloud logging includes management-plane audit logs, network flow logs, DNS logs, load balancer logs, storage access logs, database audit logs, identity logs, application logs, and security findings. Without logs, cloud incident response becomes guesswork.

4.16 Landing zone

A landing zone is a governed cloud foundation. It usually includes account/project structure, identity federation, network layout, logging, monitoring, encryption defaults, policy guardrails, tagging standards, budgets, and incident-response access.

4.17 Policy as code

Policy as code means writing security and compliance rules in machine-readable form so they can be checked automatically. Example rule: storage buckets must not be public unless approved and tagged.

4.18 Cloud-native risk pattern

Many cloud incidents follow a chain: exposed resource → weak identity or secret → excessive permissions → insufficient logging → slow detection → high impact. Strong cloud security breaks that chain at multiple points.

7. Mathematical Foundations

7.1 Availability

Availability can be expressed as a proportion between 0 and 1. If a service is available 99.9% of the time, then its availability is 0.999.

Approximate yearly downtime is:

downtime_minutes = (1 - availability) × 365 × 24 × 60

Worked example: 99.9% availability means (1 - 0.999) × 525600 = 525.6 minutes of downtime per year, or about 8.76 hours.

7.2 Dependency availability

If a cloud application requires several independent components to work at the same time, a simplified approximation is:

A_total = A1 × A2 × A3 × ... × An

If the application, database, and identity provider are each 99.9% available, the rough combined availability is 0.999 × 0.999 × 0.999 = 0.997003, or about 99.7003%. This example is simplified, but it teaches a key design lesson: dependencies reduce overall availability unless you design redundancy, failover, caching, graceful degradation, and recovery.

7.3 Expected loss

A simple risk-estimation formula is:

expected_loss = probability_of_event × impact_cost

If a public-storage exposure has a 5% annual probability and an estimated impact of $200,000, the expected annualized loss is 0.05 × 200000 = $10,000. This number is not exact; it helps compare controls, such as logging, access review, policy enforcement, and encryption.

7.4 Cost anomaly intuition

Cloud resources are billed by usage. Security teams sometimes monitor cost because unexpected usage can indicate misconfiguration, abuse, or account compromise. A simple anomaly rule is:

alert if today_cost > average_daily_cost + 3 × standard_deviation

This rule is imperfect, but it introduces the idea of baseline behavior and deviation.

8. Practical Examples

8.1 Linux example: inspect a fictional cloud inventory file

cat cloud-inventory.json
python3 -m json.tool cloud-inventory.json
sha256sum cloud-inventory.json

The first command displays the file. The second checks that the JSON is readable. The third creates a hash so you can prove the file did not change during your lab report.

8.2 Windows example: read the same inventory with PowerShell

Get-Content .\cloud-inventory.json | ConvertFrom-Json
Get-FileHash .\cloud-inventory.json -Algorithm SHA256

ConvertFrom-Json turns JSON into PowerShell objects. Get-FileHash calculates an evidence hash.

8.3 Python example: find public resources in fictional data

import json

with open("cloud-inventory.json", "r", encoding="utf-8") as file:
    inventory = json.load(file)

for resource in inventory["resources"]:
    name = resource["name"]
    kind = resource["type"]
    public = resource.get("public", False)
    if public:
        print(f"Review required: {kind} {name} is marked public")

This script is safe because it reads a local fictional file. It does not connect to a real cloud account.

8.4 Network example

A secure cloud design usually places public entry points in public subnets or managed edge services, while application servers, databases, and internal services remain private. Network exposure should match business need.

8.5 Web example

A public web application may use DNS, TLS, a load balancer, a web service, a database, and object storage. The secure design question is not only “Is HTTPS enabled?” It is also: Who can deploy code? Who can read storage? Are sessions secure? Are logs retained? Are secrets outside source code?

8.6 Cloud example

A storage bucket that hosts public images may intentionally be public for read access. A bucket containing student records should not be public. Same technology, different data classification, different access policy.

9. Code Examples

9.1 Safe cloud policy review script

The following script checks a fictional IAM-style policy for risky wildcard permissions. It is intentionally local and educational.

# cloud_policy_review.py
# Purpose: review a local fictional policy document for risky wildcard access.
# Safety: this script does not connect to any cloud provider or real account.

import json                         # Loads Python's built-in JSON parser.
from pathlib import Path             # Helps work with file paths safely.

policy_path = Path("sample-policy.json")   # Name of the local policy file to review.

with policy_path.open("r", encoding="utf-8") as file:  # Open the file as text.
    policy = json.load(file)                         # Convert JSON text into Python data.

statements = policy.get("Statement", [])             # Read the Statement list or use an empty list.

for index, statement in enumerate(statements, start=1):  # Loop through each statement with a number.
    effect = statement.get("Effect", "Unknown")          # Read Allow or Deny.
    actions = statement.get("Action", [])                 # Read actions.
    resources = statement.get("Resource", [])             # Read resources.

    if isinstance(actions, str):                          # Policies may use a string or list.
        actions = [actions]                               # Convert string to list for consistent logic.
    if isinstance(resources, str):                        # Same normalization for resources.
        resources = [resources]

    risky_action = "*" in actions or any(a.endswith(":*") for a in actions)
    risky_resource = "*" in resources

    if effect == "Allow" and risky_action and risky_resource:
        print(f"Statement {index}: HIGH REVIEW - allows broad actions on all resources")
    elif effect == "Allow" and risky_action:
        print(f"Statement {index}: MEDIUM REVIEW - allows broad actions")
    elif effect == "Allow" and risky_resource:
        print(f"Statement {index}: MEDIUM REVIEW - applies to all resources")
    else:
        print(f"Statement {index}: no wildcard risk found by this simple check")

9.3 Common mistakes

  • Mistake: hardcoding cloud keys in code. Why wrong: source code, logs, build artifacts, and screenshots can leak secrets.
  • Mistake: relying only on network location. Why wrong: cloud control planes are API-driven; identity and policy matter heavily.
  • Mistake: disabling logs to reduce cost. Why wrong: incident response needs evidence.
  • Mistake: using administrator permissions for automation. Why wrong: one compromised pipeline can control the environment.

10. Security Perspective

Common attacks and failures

  • Leaked access keys or tokens.
  • Over-permissive IAM roles.
  • Public storage containing sensitive data.
  • Exposed databases or management ports.
  • Insecure serverless function permissions.
  • Weak SaaS administrator controls.
  • Unmonitored cloud control-plane changes.
  • Insecure CI/CD credentials.
  • Supply-chain compromise of images, dependencies, or deployment pipelines.
  • Incomplete backups or untested recovery plans.

Defensive strategies

  • Use identity federation and multi-factor authentication for human users.
  • Use short-lived workload credentials where possible.
  • Apply least privilege and review access regularly.
  • Separate production, development, and testing environments.
  • Enable management-plane, data-plane, network, DNS, and application logs where appropriate.
  • Use encryption, key management, and secret managers.
  • Automate configuration checks with policy as code.
  • Use infrastructure as code and peer review for changes.
  • Test backups and recovery procedures.
  • Monitor cost, resource creation, public exposure, and privileged actions.

Detection methods

  • Alert on new administrator roles or policy changes.
  • Alert on disabled logging or deleted audit trails.
  • Alert on public storage changes.
  • Alert on unusual API geography, time, source address, or user agent.
  • Alert on new access keys, long-lived keys, or unused keys becoming active.
  • Alert on abnormal data downloads or unusual database access.

Mitigation and best practices

Cloud mitigation is layered: prevent risky changes, detect suspicious activity, contain compromised identities, rotate secrets, preserve logs, restore clean systems, and improve guardrails. The strongest designs assume mistakes will happen and make dangerous mistakes harder, more visible, and faster to reverse.

11. Industry Perspective

  • Banks: use cloud for analytics, fraud detection, customer applications, and backup, while enforcing strict identity, encryption, audit, and regulatory controls.
  • Governments: use cloud for citizen services, data platforms, and resilient infrastructure, with procurement, sovereignty, logging, and compliance requirements.
  • Cloud providers: secure global facilities, hardware, virtualization layers, managed services, DDoS protection, and foundational identity systems.
  • Healthcare: uses cloud for electronic health systems, imaging, research, and telehealth while protecting patient privacy and availability.
  • Defense: uses cloud with strict segmentation, identity controls, logging, supply-chain assurance, and mission resilience.
  • Universities: use cloud for learning platforms, research computing, student systems, storage, and collaboration tools. The challenge is balancing openness with privacy and security.
  • Large enterprises: use hybrid and multi-cloud environments, SaaS platforms, containers, and DevSecOps pipelines. Governance and inventory are major operational problems.

12. Research Perspective

Cloud security research is active because cloud environments are large, dynamic, multi-tenant, and API-driven. Important directions include:

  • Cloud forensics: collecting trustworthy evidence when systems are distributed, ephemeral, and provider-managed.
  • Policy verification: proving that identity, network, and storage policies do not permit unsafe paths.
  • Confidential computing: protecting data while it is being processed, not only when stored or transmitted.
  • Serverless security: reasoning about event-driven code, short-lived execution, and least privilege.
  • Container and Kubernetes security: securing images, runtimes, orchestration, admission control, and workload identity.
  • Multi-cloud governance: maintaining consistent controls across different providers and SaaS systems.
  • Cloud-native detection: using API events, identity graphs, resource relationships, and behavioral baselines to identify attacks.

Open problems include usable least privilege, secure defaults at scale, high-confidence cloud attack-path analysis, verifiable incident evidence, and reducing alert fatigue in large cloud estates.

13. Case Studies

Case Study 1: Capital One cloud breach

What happened: Public reporting and legal records describe a breach involving cloud infrastructure, a web application firewall configuration issue, server-side request forgery-style access to cloud metadata, and unauthorized access to data.

Root cause: The incident is commonly used to teach how application-layer flaws, cloud metadata services, IAM permissions, and data access controls can combine.

Impact: Customer data exposure, regulatory scrutiny, legal action, and major incident-response cost.

Lessons learned: Harden metadata access, minimize role permissions, monitor unusual data access, segment sensitive data, and test cloud-specific abuse paths.

Case Study 2: Public cloud storage exposure

What happened: Many organizations have accidentally exposed sensitive data through public object storage or overly broad sharing settings.

Root cause: Misunderstood access policies, weak review, missing classification, missing monitoring, or broad default access.

Impact: Confidentiality loss, privacy exposure, regulatory notification, and reputational damage.

Lessons learned: Use deny-by-default storage policies, public-access blocks, classification, automated checks, access logs, and exception approvals.

Case Study 3: Cloud cost abuse

What happened: Compromised cloud credentials are sometimes used to create expensive compute resources or run unauthorized workloads.

Root cause: Exposed keys, no MFA, excessive permissions, no budget alarms, and weak monitoring.

Impact: Financial loss, service disruption, and possible data compromise.

Lessons learned: Use short-lived credentials, least privilege, budget alerts, anomaly detection, and automated containment for suspicious resource creation.

14. Common Misconceptions

  • Misconception: “The provider handles all security.” Correction: responsibility is shared and depends on service model and configuration.
  • Misconception: “Private subnet means secure.” Correction: identity, routes, peering, endpoints, credentials, and application flaws can still expose data.
  • Misconception: “Encryption solves cloud risk.” Correction: encrypted data can still be accessed by identities with key and data permissions.
  • Misconception: “SaaS needs no security work.” Correction: customers still manage users, data, sharing, retention, integrations, and monitoring.
  • Misconception: “More clouds means more resilience automatically.” Correction: multi-cloud can add complexity and operational risk if not governed well.
  • Misconception: “Logs are optional.” Correction: without logs, you may not know who changed what, what data was accessed, or how to respond.

15. Summary

Cloud computing gives organizations fast access to scalable computing, storage, networking, and managed services. Cloud security requires understanding service models, shared responsibility, identity, control planes, data planes, networks, storage, encryption, logging, automation, and recovery. The core idea is simple: cloud resources are controlled by APIs, so secure identity, configuration, and evidence are central. Good cloud security makes risky actions difficult, visible, reversible, and accountable.

16. Glossary

Availability Zone
A separate cloud failure domain inside a region.
Cloud API
A programmable interface used to manage cloud resources.
Cloud Provider
An organization that delivers cloud infrastructure, platforms, or software services.
Control Plane
The management layer used to create, configure, update, and delete resources.
Data Plane
The layer where application data is stored, processed, or transmitted.
IaaS
Infrastructure as a Service; virtualized compute, storage, and networking where the customer manages more of the stack.
PaaS
Platform as a Service; managed platform runtime where customers deploy applications and configuration.
SaaS
Software as a Service; provider-operated application accessed by customers.
Serverless
A model where the provider manages server infrastructure and customers deploy code or event-driven logic.
Shared Responsibility
The split of security duties between provider and customer.
Tenant
A customer or organizational boundary in a cloud platform.
VPC/VNet
A logically isolated virtual network in a cloud environment.

18. Practice Lab

Lab title

Fictional Cloud Architecture Review

Safety rules

  • Use only the local files provided below or created by you.
  • Do not log into real cloud accounts.
  • Do not scan, probe, or test public systems.
  • Do not use real credentials or real student data.

Prerequisites

  • A text editor.
  • Python 3.
  • Basic JSON understanding.
  • Lessons 1.1–9.5 recommended.

Objectives

  1. Create a fictional cloud inventory.
  2. Identify public exposure, missing logs, weak encryption, and broad permissions.
  3. Map each issue to CIA/security goals.
  4. Write a risk register and mitigation plan.

Step 1: Create a local file named cloud-inventory.json

{
  "environment": "Northbridge University training cloud",
  "resources": [
    {"name": "student-portal-lb", "type": "load-balancer", "public": true, "logs_enabled": true},
    {"name": "student-portal-app", "type": "app-service", "public": false, "identity": "portal-app-role", "logs_enabled": true},
    {"name": "transcript-db", "type": "database", "public": false, "encrypted": true, "logs_enabled": false},
    {"name": "transcript-exports", "type": "object-storage", "public": true, "encrypted": false, "logs_enabled": false},
    {"name": "dev-admin-role", "type": "iam-role", "permissions": "administrator", "mfa_required": false}
  ]
}

Step 2: Run a safe review script

import json

with open("cloud-inventory.json", "r", encoding="utf-8") as f:
    data = json.load(f)

for r in data["resources"]:
    issues = []
    if r.get("public") and r["type"] != "load-balancer":
        issues.append("Unexpected public exposure")
    if r.get("encrypted") is False:
        issues.append("Encryption disabled")
    if r.get("logs_enabled") is False:
        issues.append("Logging disabled")
    if r.get("permissions") == "administrator":
        issues.append("Administrator-level permission")
    if r.get("mfa_required") is False:
        issues.append("MFA not required")

    if issues:
        print(r["name"])
        for issue in issues:
            print(" -", issue)

Expected findings

  • transcript-db needs logging.
  • transcript-exports should not be public, should be encrypted, and needs access logs.
  • dev-admin-role has excessive permissions and no MFA requirement.

Deliverable

Submit a one-page report with: architecture summary, asset inventory, shared-responsibility map, risk register, recommended controls, and evidence hash of your JSON file.
