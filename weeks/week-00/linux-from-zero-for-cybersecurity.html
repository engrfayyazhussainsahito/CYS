Lesson 0.2: Linux from Zero for Cybersecurity

Linux is the operating system behind many servers, cloud workloads, containers, routers, developer machines, and security labs. This lesson starts at zero and builds the Linux knowledge needed for safe cybersecurity work.

Estimated reading time90–130 minutes
Difficulty levelBeginner → Intermediate bridge
Required backgroundLesson 0.1 recommended
Practice time3–5 hours

Week 0 Context

Overview: Week 0 builds the missing technical foundation required before security topics. This lesson focuses on Linux because later lessons use Linux for logs, permissions, processes, networking, scripts, web servers, containers, and security monitoring.

Learning objectives: after this lesson you should be able to navigate Linux, inspect files safely, understand permissions, view processes, read logs, run simple shell scripts, and explain why Linux knowledge matters in cybersecurity.

Required background: no prior Linux experience is required. You should understand from Lesson 0.1 that all practice must stay inside your own computer, course VM, or explicitly authorized lab.

Estimated reading time: 90–130 minutes. Difficulty: starts beginner, ends with intermediate operational concepts.

1. Introduction

Plain-English idea: Linux is an operating system. An operating system is the main software layer that lets people and programs use the computer. It manages files, users, memory, programs, devices, and network connections.

Linux exists because computers need a reliable, flexible, and controllable system for running software. It is especially common on servers because it is stable, scriptable, automatable, and widely supported by cloud and infrastructure tools.

Linux matters in cybersecurity because many real security tasks happen on or against Linux-based systems: server hardening, cloud administration, log analysis, malware triage, network monitoring, container security, incident response, and forensic investigation. Even when a company uses Windows desktops, its web servers, cloud workloads, firewalls, or security tools may run Linux.

Linux is used by system administrators, cloud engineers, DevOps engineers, security operations center analysts, digital forensic investigators, penetration testers, malware analysts, researchers, software engineers, and students. In this course, Linux is your learning workbench.

Safety boundary

All commands in this lesson are local inspection, local file creation, or local configuration exercises. Do not run commands against systems, IP addresses, websites, accounts, or networks that are not yours or not explicitly assigned by the course.

2. Learning Outcomes

After finishing this lesson, you should be able to:

  • Explain what Linux is and why cybersecurity learners use it.
  • Distinguish between kernel, shell, terminal, command, program, and process.
  • Navigate the Linux filesystem using safe basic commands.
  • Create, read, copy, move, and remove files inside a practice folder.
  • Interpret Linux file permissions and explain least privilege.
  • Explain users, groups, root, sudo, and ownership.
  • Inspect running processes without harming the system.
  • View local network information safely.
  • Read basic log files and identify meaningful events.
  • Write and run a small safe Bash script.
  • Describe common Linux security mistakes and defensive practices.
  • Use Linux terminology accurately in later cybersecurity lessons.

3. Prerequisites

This lesson assumes no Linux background. It introduces prerequisite concepts before using them.

Computer hardware

Hardware means the physical parts of a computer: processor, memory, disk, keyboard, screen, and network card.

Software

Software means instructions that run on hardware. Examples include operating systems, browsers, text editors, servers, and scripts.

Operating system

The operating system is the coordinator. Without it, every program would need to know how to directly control disks, memory, network cards, screens, and users. The operating system gives programs a controlled way to request resources.

Text interface

A terminal is a text window where you type commands. A shell is the program that reads your commands and starts other programs. In many Linux systems, the default shell is Bash.

Path

A path is the address of a file or directory. Example: /home/student/notes.txt. A directory is a folder.

Privilege

Privilege means permission to perform an action. In Linux, normal users have limited privileges. The root account has full administrative power. Cybersecurity uses the principle of least privilege: use only the access needed for the task.

4. Core Concepts

4.1 Linux distribution

Linux by itself usually refers to the Linux kernel. A complete user-ready system is called a distribution, or distro. A distribution packages the kernel with command-line tools, system services, package managers, documentation, and default settings.

Examples include Ubuntu, Debian, Fedora, Arch Linux, Kali Linux, and Rocky Linux. For beginners, Ubuntu is usually easier than specialized security distributions. Kali is useful later, but it is not a substitute for understanding Linux fundamentals.

4.2 Kernel, shell, terminal, and commands

The kernel is the core of the operating system. It manages hardware, memory, processes, filesystems, and network interfaces. The shell is a command interpreter. The terminal is the window or interface where the shell runs. A command is text you type to ask the shell to run something.

flowchart TD

  Terminal[Terminal window]
  Shell[Shell, usually Bash]

  Kernel[Linux kernel]

Example:

pwd

The shell receives pwd, finds the pwd program or shell built-in, runs it, and prints your current directory.

4.3 Filesystem hierarchy

Linux organizes files into a tree starting at /, called the root directory. This is not the same as the root user. The root directory is the top of the file tree; the root user is the administrator account.

4.4 Absolute and relative paths

An absolute path starts from /. Example: /home/student/labs. A relative path starts from your current location. Example: if you are already in /home/student, then labs points to /home/student/labs.

pwd                 # show current location
cd /home/student    # move using an absolute path
cd labs             # move using a relative path from current location
cd ..               # move up one directory

Linux commands are small tools that can be combined. This is one reason Linux is powerful for cybersecurity. You can inspect files, search logs, filter output, and automate repetitive work.

Do not copy random rm -rf commands from the internet. rm removes files. With recursive and force options, it can destroy large parts of a system if misused.

4.6 Users, groups, and root

Linux is multi-user. A user is an identity. A group is a collection of users. File permissions can allow or deny actions for the owner, the group, and everyone else.

The root user is the administrator. Root can modify nearly everything. sudo lets a permitted normal user run a specific command with administrative privileges.

Security intuition: using root all the time is like leaving every door unlocked because it is convenient. It makes mistakes more damaging.

4.7 Permissions

Linux permissions describe who can read, write, or execute a file.

  • Read means view file contents or list a directory.
  • Write means modify a file or create/remove files in a directory.
  • Execute means run a file as a program or enter a directory.
ls -l notes.txt

Example output:

-rw-r--r-- 1 student student 45 Jun 28 10:30 notes.txt

Breakdown:

PartMeaning
-Regular file. A directory would show d.
rw-Owner can read and write.
r--Group can read.
r--Others can read.
student studentOwner and group.

Numeric permissions use r=4, w=2, x=1. Add them:

  • 7 = 4+2+1 = rwx
  • 6 = 4+2 = rw-
  • 5 = 4+1 = r-x
  • 4 = r--
  • 0 = ---

Example: chmod 600 private.txt means owner can read/write, group and others have no permission.

4.8 Processes

A process is a running program. If a text editor is open, it is a process. If a web server is running, it is a process. Linux gives each process a process ID, or PID.

ps aux | head

This shows a short list of running processes. The pipe symbol | sends output from one command into another command. Here, head shows only the first lines.

4.9 Services and systemd

A service is a program meant to run in the background, often starting automatically. Web servers, database servers, logging tools, and SSH are examples. Many Linux distributions manage services with systemd and its command systemctl.

systemctl status ssh

This asks Linux whether the SSH service exists and whether it is running. On some systems it may be named sshd or may not be installed.

4.10 Logs

Logs are records of events. Linux logs may show logins, authentication failures, service starts, kernel events, package changes, and application errors. Logs help defenders reconstruct what happened.

journalctl -n 20

This asks the system journal for the most recent 20 log entries. Some systems require sudo for certain logs.

4.11 Package management

A package manager installs, updates, and removes software from trusted repositories. Ubuntu and Debian use apt. Fedora uses dnf. Arch uses pacman.

sudo apt update      # refresh package information
sudo apt upgrade     # install available updates

Security meaning: unpatched software often contains known vulnerabilities. Package management is part of system hardening.

4.12 Shell scripting

A shell script is a text file containing commands. Scripts help automate repeated administrative or security tasks, such as checking file permissions or extracting log lines.

Beginner rule: scripts should be small, readable, and tested in a safe folder before being used anywhere important.

7. Mathematical Foundations

Linux permissions have a simple mathematical representation. Each permission has a value:

read = 4
write = 2
execute = 1

To describe a permission set, add the values.

Worked example 1: file readable and writable by owner only

Owner needs read and write: 4 + 2 = 6. Group gets nothing: 0. Others get nothing: 0.

Permission = 600
Owner: rw-
Group: ---
Others: ---

This is common for private files.

Worked example 2: script executable by owner, readable by others

Owner needs read, write, execute: 4 + 2 + 1 = 7. Group needs read and execute: 4 + 1 = 5. Others need read and execute: 4 + 1 = 5.

Permission = 755
Owner: rwx
Group: r-x
Others: r-x

This is common for executable programs and directories.

Boolean logic in permissions

Access decisions are often yes/no questions. For example:

Can read file = (user is owner AND owner_read) OR
                (user is in group AND group_read) OR
                (user is other AND other_read)

This prepares you for later access-control models, firewalls, and policy rules.

Python example: check file permissions conceptually

# permission_math.py
# Goal: Explain numeric Linux permissions using simple addition.

permissions = {
    "read": 4,
    "write": 2,
    "execute": 1
}

owner_value = permissions["read"] + permissions["write"]
group_value = permissions["read"]
other_value = permissions["read"]

print("Owner:", owner_value)   # 6 means read + write
print("Group:", group_value)   # 4 means read only
print("Other:", other_value)   # 4 means read only
print("Full numeric mode:", f"{owner_value}{group_value}{other_value}")

This script does not change permissions. It teaches the arithmetic behind permission notation.

Network example: inspect only your own network interfaces

ip address show

These commands show local network interfaces and routes. They do not scan or contact outside systems.

9. Code Examples

The following Bash script performs a safe local system summary. It does not modify files, install software, scan networks, or contact external targets.

#!/usr/bin/env bash
# linux_summary.sh
# Goal: Print a beginner-friendly summary of the local Linux environment.
# Safety: This script only reads local information and prints it.

set -u
# set -u makes Bash stop if we use a variable that was never defined.
# This helps catch mistakes in scripts.

echo "=== Current user ==="
# echo prints text to the terminal.

whoami
# whoami prints the username running this script.

echo "=== Current directory ==="
pwd
# pwd prints the current working directory.

echo "=== Kernel and operating system information ==="
uname -a
# uname prints system information.
# -a means show all available summary information.

echo "=== First five visible files in this directory ==="
ls -la | head -n 5
# ls -la lists files with details, including hidden files.
# The pipe sends that output to head.
# head -n 5 shows only the first five lines.

echo "=== First five running processes ==="
ps aux | head -n 5
# ps aux lists running processes.
# head limits output so the screen is not flooded.

echo "=== Local IP address information summary ==="
ip -brief address
# ip -brief address shows local network interfaces compactly.

echo "Summary complete."

How to run it safely

cd ~/ms-cybersecurity-labs/week-00/linux
nano linux_summary.sh       # paste the script, save, and exit
chmod 700 linux_summary.sh  # allow only you to read/write/execute it
./linux_summary.sh          # run from current directory

Line-by-line security lessons

  • #!/usr/bin/env bash tells Linux to run the file with Bash.
  • set -u makes errors more visible by catching undefined variables.
  • whoami confirms privilege level. If this says root unexpectedly, stop and check why.
  • uname -a helps identify the system during documentation.
  • ps aux helps understand running programs.
  • ip -brief address shows local network configuration without scanning anything.

Common mistakes

  • Running scripts with sudo when sudo is not needed.
  • Copying scripts from unknown sources without reading them.
  • Using broad deletion commands during experiments.
  • Not recording what commands were run.
  • Confusing local inspection with permission to inspect other systems.

Secure coding practices

Security scripts should be readable, minimal, tested on harmless data, and documented. Avoid hidden network connections, hardcoded secrets, silent deletion, unnecessary privilege, and unclear error handling.

10. Security Perspective

Common attacks involving Linux systems

  • Password guessing: attackers try weak or reused credentials against remote login services.
  • Privilege escalation: attackers gain limited access and then try to obtain higher privileges.
  • Persistence: attackers configure a program, service, user, or scheduled task to survive reboot.
  • Log tampering: attackers attempt to hide evidence by changing or deleting logs.
  • Misconfiguration abuse: attackers exploit world-writable files, exposed services, weak permissions, or outdated packages.

Common vulnerabilities and mistakes

  • Using root for daily tasks.
  • Giving files permissions such as 777 without understanding the risk.
  • Leaving unnecessary services running.
  • Failing to update packages.
  • Storing secrets in shell history, scripts, or public repositories.
  • Ignoring logs until after an incident.

Defensive strategies

  • Use least privilege and sudo only when necessary.
  • Patch regularly through the package manager.
  • Disable services that are not needed.
  • Use strong authentication and multi-factor authentication where available.
  • Protect private keys and configuration files with restrictive permissions.
  • Monitor logs for unusual login, process, and service activity.

Detection methods

  • Review authentication logs for repeated failures or unexpected successful logins.
  • Compare running services with expected services.
  • Check unusual processes, parent-child process relationships, and command lines.
  • Inspect recent file changes in sensitive directories during authorized investigations.
  • Use centralized logging in enterprise environments so local tampering does not erase all evidence.

Mitigation and best practices

Baseline Linux defense begins with patching, least privilege, service reduction, safe configuration, log collection, file-permission hygiene, secure authentication, backups, and tested recovery. Later lessons will convert these ideas into formal hardening checklists and detection rules.

12. Research Perspective

Linux security is also a research area. Researchers study how to make operating systems safer, more observable, and more resilient against adversaries.

Current research themes

  • Kernel attack surface reduction: reducing bugs reachable from untrusted users and programs.
  • Memory safety: preventing memory-corruption classes of bugs through safer languages, sanitizers, and isolation.
  • Linux sandboxing: improving mechanisms such as namespaces, cgroups, capabilities, seccomp, and containers.
  • Provenance and audit: tracking how processes, files, and network events are causally related.
  • Runtime detection: detecting suspicious process behavior, privilege changes, and persistence mechanisms.
  • Supply-chain security: improving package integrity, build reproducibility, and dependency trust.

Open problems

  • How can systems provide strong isolation without becoming too complex for administrators?
  • How can defenders collect enough evidence without violating privacy?
  • How can package ecosystems prevent malicious or compromised dependencies?
  • How can Linux be hardened for small organizations without dedicated security teams?

Notable research directions to explore later

Later in the course, students can explore operating-system protection, capability systems, intrusion detection, audit log causality, container isolation, fuzzing, memory safety, and reproducible builds. Do not worry if these terms feel advanced now; this lesson only prepares the vocabulary.

13. Case Studies

Case Study 1: Misconfigured permissions expose sensitive files

What happened: an organization stores configuration files on a Linux server. A file containing credentials is readable by more users than necessary.

Root cause: permissions were too broad. The file should have been readable only by the service account or administrator.

Impact: a user or compromised low-privilege process could read secrets and use them to access other systems.

Lessons learned: use least privilege, review sensitive file permissions, avoid storing secrets in plain text, and rotate exposed credentials.

Case Study 2: Unpatched Linux service is compromised

What happened: a server runs an outdated network-facing service. Attackers use a known vulnerability against that service.

Root cause: patch management failed, and the service was exposed without sufficient compensating controls.

Impact: attackers gain access, create persistence, and attempt lateral movement.

Lessons learned: maintain an asset inventory, patch regularly, remove unnecessary services, monitor logs, and segment systems.

Case Study 3: Logs reveal repeated failed login attempts

What happened: a defender notices many failed login attempts followed by one successful login from an unusual location.

Root cause: weak authentication controls or credential exposure may have allowed account compromise.

Impact: unauthorized access to a Linux account may allow data theft or further attacks.

Lessons learned: monitor authentication logs, use strong passwords or keys, apply multi-factor authentication where possible, and disable unnecessary remote login access.

14. Common Misconceptions

MisconceptionWhy it is wrong
“Linux is only for hackers.”Linux is a general-purpose operating system used in servers, phones, embedded devices, clouds, research, and education.
“Kali Linux teaches cybersecurity by itself.”Kali packages tools. It does not teach operating systems, networks, ethics, or secure reasoning.
“Using sudo is always better.”Sudo increases privilege and therefore increases potential damage from mistakes.
“chmod 777 fixes permission problems.”It often creates security problems by giving everyone read, write, and execute permissions.
“Logs are only useful after attacks.”Logs support monitoring, debugging, compliance, and routine operations before, during, and after incidents.
“The terminal is dangerous by default.”The terminal is precise. Risk depends on what command is run, with what privilege, and on what system.
“Linux commands must be memorized before learning security.”Understanding concepts matters more. Commands become familiar through safe repeated practice.

15. Summary

Linux is a core cybersecurity workbench because it exposes the building blocks of systems: users, files, permissions, processes, services, packages, networking, and logs. You learned the difference between the kernel, shell, and terminal; how the filesystem is organized; how permissions work; why root and sudo require caution; and how local commands can support defensive investigation. Later lessons will reuse these Linux ideas when teaching networking, secure software, cryptography, web security, cloud security, incident response, and detection engineering.

Bash
A common Linux shell that reads and executes typed commands and scripts.
Command
Text typed into a shell to run a program or request an action.
Directory
A folder that can contain files and other directories.
Distribution
A complete Linux-based operating system package containing the Linux kernel and supporting software.
Execute permission
Permission to run a file as a program or enter a directory.
Filesystem
The structure used to organize files and directories.
Group
A collection of users used for permission management.
Kernel
The core part of an operating system that manages hardware and system resources.
Least privilege
The principle that users and programs should have only the permissions required for their task.
Log
A record of events produced by systems, services, or applications.
Package manager
A tool that installs, updates, and removes software packages.
Path
The address of a file or directory.
Permission
A rule that determines whether a user or process can read, write, or execute something.
Process
A running program with a process ID.
Root
The administrative user account with broad system privileges; also separately the name of the top filesystem directory /.
Service
A background program usually managed by the operating system.
Shell
A program that interprets commands.
Sudo
A mechanism that allows authorized users to run specific commands with elevated privileges.
Terminal
A text interface used to interact with a shell.

Steps

  1. Create the lab folder: mkdir -p ~/ms-cybersecurity-labs/week-00/linux.
  2. Move into it: cd ~/ms-cybersecurity-labs/week-00/linux.
  3. Create three folders: mkdir notes scripts evidence.
  4. Create a note: echo "Linux lab started" > notes/lab-00-02.md.
  5. List permissions: ls -la notes.
  6. Create a private file: echo "private practice data" > evidence/private.txt.
  7. Restrict it: chmod 600 evidence/private.txt.
  8. Verify: ls -l evidence/private.txt.
  9. Create scripts/linux_summary.sh using the script from this lesson.
  10. Set permission: chmod 700 scripts/linux_summary.sh.
  11. Run it: ./scripts/linux_summary.sh.
  12. Append the output summary to your lab notes in your own words.
  13. Write a paragraph explaining why 600 is safer than 644 for private files.

Expected outcomes

  • You can navigate directories and inspect files.
  • You can explain basic permissions.
  • You can create and protect a private local file.
  • You can run a small Bash script without sudo.
  • You have a written lab record.

Safety restrictions

  • Do not use sudo unless your instructor explicitly says it is required.
  • Do not change permissions outside your own lab folder.
  • Do not scan, connect to, or test external systems.
  • Do not run unknown scripts from the internet.
