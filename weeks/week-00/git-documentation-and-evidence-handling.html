Lesson 0.5: Git, Documentation, and Evidence Handling

Cybersecurity work is only useful when another competent person can understand what you did, reproduce your result, and trust that evidence was not casually changed. This lesson teaches Git, documentation, hashing, evidence notes, and chain-of-custody thinking from zero knowledge.

Estimated reading time120–180 minutes
Difficulty levelBeginner → Intermediate bridge
Required backgroundLessons 0.1–0.4 recommended
Practice time5–8 hours

Week 0 Context

Overview: Week 0 prepares students for safe, reproducible cybersecurity learning. This lesson adds version control and evidence-aware documentation so future labs are organized, auditable, and explainable.

Learning objectives: after this lesson you should be able to create a Git repository, track file changes, write useful commit messages, document lab evidence, compute hashes, understand chain of custody, and explain why Git helps reproducibility but does not replace forensic evidence storage.

Required background: no Git or forensics background is required. Linux, networking, and Python lessons make the commands and scripts easier.

Estimated reading time: 120–180 minutes. Difficulty: starts beginner, ends with intermediate evidence and reproducibility concepts.

1. Introduction

Plain-English idea: Git records the history of changes in project files. Documentation explains what happened. Evidence handling preserves and describes information so conclusions can be checked.

These topics exist because security work is easy to misunderstand. A student may run a command and forget what it meant. An analyst may collect logs but fail to record where they came from. A developer may fix a bug but leave no explanation. A researcher may publish a result that nobody can reproduce. Git and disciplined documentation reduce these problems.

This matters because cybersecurity decisions can affect money, privacy, safety, employment, legal processes, and public trust. A weak screenshot, unclear note, or modified evidence file may not support a serious conclusion. In technical work, the quality of the record often determines whether the finding is usable.

Git is used by software engineers, cloud engineers, security researchers, DevSecOps teams, malware analysts, detection engineers, digital forensic analysts, students, universities, banks, defense teams, healthcare organizations, and governments. Evidence handling is used by incident responders, forensic investigators, auditors, compliance teams, security operations centers, researchers, and legal teams.

Important boundary

Git is useful for lab notes, source code, detection rules, reports, and reproducible research. Git alone is not a complete forensic evidence vault. Real investigations may require write blockers, forensic imaging, chain-of-custody forms, access-controlled storage, legal review, and organization-specific procedures.

  • Define version control, repository, working tree, staging area, commit, branch, merge, diff, tag, remote, clone, README, issue, pull request, hash, checksum, provenance, metadata, chain of custody, and audit trail.

  • Write useful commit messages and security lab notes.
  • Use .gitignore to avoid committing generated files or sensitive local data.
  • Compute SHA-256 hashes on Linux, Windows, and Python.

  • Describe common documentation failures in incident response and research.
  • Use Git and documentation safely in a lab without exposing secrets or real personal data.

3. Prerequisites

Files and folders

A file stores data. A folder, also called a directory, organizes files. Git watches selected files inside a project folder and records changes over time.

Terminal basics

You need to run commands such as git status and git commit. If terminal commands feel unfamiliar, review Lesson 0.2 first.

Text editing

Git works best with text files because text changes are easy to compare. Source code, Markdown notes, configuration files, and JSON logs are text-friendly. Binary files such as screenshots and disk images can be stored, but Git cannot show their internal changes as clearly.

Time and identity

Evidence records depend on who did something and when. Computer timestamps can be wrong, time zones can be confusing, and account names may not prove the real human actor. Always record limitations.

Authorization

Only document and handle data you are allowed to access. Do not put real passwords, private keys, student records, patient data, or employer data into a public GitHub repository.

4. Core Concepts

4.1 What is version control?

Definition: version control is a system for recording changes to files over time so earlier versions can be reviewed or restored.

Intuition: imagine a notebook where every important edit is saved with the author, time, and reason. Git is a powerful version of that notebook for files.

Security example: when you write a detection rule, Git can show when the rule was created, who changed it, what changed, and why the change was made.

4.2 What is Git?

Git is a distributed version control system. Distributed means each copy of a repository can contain the project history, not only the latest files. This makes Git useful for offline work, collaboration, and review.

Git is not the same as GitHub. Git is the tool. GitHub is a hosting and collaboration platform that can store Git repositories online. You can use Git locally without creating a GitHub account.

4.3 Repository

A repository, often called a repo, is a project folder tracked by Git. The hidden .git directory stores Git metadata and history.

git init
ls -la

After git init, Git creates the internal tracking structure. Do not manually edit the .git folder as a beginner.

4.4 Working tree, staging area, and commit history

The working tree is the files you see and edit. The staging area is the set of changes selected for the next saved snapshot. The commit history is the record of saved snapshots.

Analogy: working tree = your messy desk, staging area = documents you placed in an outgoing folder, commit = the official archived packet.

git status
git add README.md
git commit -m "Add lab README"

4.5 Commit

A commit is a saved record of selected changes. It includes metadata such as author, timestamp, parent commit, and a message.

A good commit message explains the reason for the change, not only the file name. Weak: update. Better: Add SHA-256 evidence verification notes.

4.6 Diff

A diff shows what changed. Added lines are often marked with +; removed lines are often marked with -.

git diff
git diff --staged

Security teams use diffs during code review, detection-rule review, infrastructure-as-code review, and incident-report review.

4.7 Branch

A branch is an independent line of work. You can experiment without changing the main line immediately.

git branch experiment-notes
git switch experiment-notes
# edit files safely
git switch main

Branches are useful for capstone projects, research experiments, draft reports, and proposed fixes.

4.8 Merge

A merge combines changes from one branch into another. If two branches changed the same lines, Git may ask you to resolve a conflict.

Security intuition: merge conflicts force humans to inspect competing changes. That is useful when the file is a policy, firewall rule, detection query, or incident timeline.

4.9 Tag

A tag marks a specific commit with a name. In this course, a tag can mark a submitted lab version.

git tag week0-lab-submission
git log --oneline --decorate

4.10 Remote and clone

A remote is another copy of the repository, usually hosted on a service such as GitHub, GitLab, or a private enterprise server. Clone means copy a repository from a remote to your machine.

Do not push sensitive lab evidence or secrets to public repositories. Use private repositories or local-only repositories when data is sensitive.

4.11 README and documentation

A README is the front page of a project. It should answer: What is this? Who is it for? How do I run it? What is in scope? What are the limitations?

Cybersecurity documentation should separate observation from interpretation. Observation: “Log line shows five failed logins.” Interpretation: “This may indicate password guessing.” Conclusion: “More evidence is required before claiming compromise.”

4.12 Markdown

Markdown is a lightweight text format for headings, lists, links, tables, and code blocks. It is widely used for Git project documentation.

# Lab Notes

## Objective
Analyze sample logs.

## Evidence
- File: sample-auth.log
- SHA-256: replace-with-hash

## Limitation
Sample data only; not proof of real compromise.

4.13 Evidence

Evidence is information used to support a conclusion. In cybersecurity, evidence may include logs, packet captures, screenshots, disk images, memory captures, command output, configuration files, source code, cloud audit records, and witness notes.

Evidence is not automatically reliable. You must ask: Where did it come from? Who handled it? Could it have changed? Is the time correct? Is it complete? Does it support the conclusion?

4.14 Chain of custody

Chain of custody is a record of evidence handling. It documents who collected evidence, where it came from, when it was handled, what was done, where it was stored, and who received it next.

Analogy: a package delivery record shows each handoff. Chain of custody does the same for evidence.

4.15 Provenance

Provenance means origin and history. For a log file, provenance includes the system that created it, collection method, export time, transfer path, and processing steps.

Weak provenance: “I found this file.” Stronger provenance: “Exported auth.log from training VM lab-ubuntu-01 at 2026-06-28 10:00 UTC using read-only copy; SHA-256 recorded before analysis.”

4.16 Hashes and integrity

A cryptographic hash is a fixed-length fingerprint of data. If a file changes, its hash should change unpredictably. SHA-256 is a common hash algorithm used for integrity checks.

A hash helps answer: “Is this file byte-for-byte the same as when I computed the hash?” It does not prove the file is true, legal, malware-free, or collected correctly.

4.17 Metadata

Metadata is data about data: filename, size, timestamps, owner, permissions, camera model, source path, or log source. Metadata can be useful but fragile. Copying, exporting, or editing files can change metadata.

4.18 Audit trail

An audit trail is a chronological record of activities. Git history is one form of audit trail for project files, but security-grade audit trails may require stronger controls such as append-only logging, access control, signed events, and monitored storage.

4.19 Reproducibility

Reproducibility means another person can follow your steps and get the same result from the same input. Reproducibility is essential in labs, research, incident reports, and professional security assessments.

A reproducible note includes environment, commands, input files, tool versions, timestamps, assumptions, outputs, and limitations.

4.20 Secrets and public repositories

A secret is sensitive authentication material such as a password, API token, SSH private key, cloud credential, or database connection string. Never commit secrets to Git. If a secret is committed and pushed, deleting the visible line is not enough because the secret may remain in history.

7. Mathematical Foundations

7.1 Hash function intuition

A hash function maps input data of any length to a fixed-length output.

H(data) = digest

For SHA-256, the digest is 256 bits. It is commonly displayed as 64 hexadecimal characters because each hexadecimal character represents 4 bits, and 64 × 4 = 256.

7.2 Integrity check formula

To check whether a file changed:

  1. Compute original hash: h1 = H(file at collection)
  2. Compute later hash: h2 = H(file later)
  3. If h1 = h2, the file is probably unchanged byte-for-byte.
  4. If h1 ≠ h2, the file changed or a different file was hashed.

The word “probably” matters because cryptography discusses collision probability, but for normal SHA-256 lab integrity checking, accidental matching after a real change is not a practical concern.

7.3 Birthday-bound intuition

For an ideal n-bit hash, collision resistance is roughly around 2^(n/2) work because of the birthday effect. For SHA-256, this rough bound is 2^128, which is far beyond normal lab capability.

Beginner takeaway: use modern standard hashes such as SHA-256, not weak legacy hashes for integrity-sensitive work.

7.4 Set difference and diffs

A Git diff compares two versions. At a simple conceptual level: added lines = lines in new version but not old version; removed lines = lines in old version but not new version. Real diff algorithms preserve ordering and context, but the intuition is change comparison.

7.5 Timeline ordering

Incident notes rely on time ordering. If events have timestamps t1, t2, t3, a timeline sorts them from earliest to latest. Problems occur when systems use different time zones, clocks are wrong, or logs are delayed.

Worked example: 10:00 UTC, 15:00 Asia/Karachi, and 05:00 New York time may describe different local times. Convert to one reference, usually UTC, before comparing incident events.

7.6 Risk scoring for documentation gaps

A simple teaching formula is: documentation risk = likelihood of misunderstanding × impact of wrong conclusion. Example: if likelihood is 4 out of 5 and impact is 5 out of 5, the score is 4 × 5 = 20. This is not a universal formula; it is a learning tool for prioritizing documentation quality.

8. Practical Examples

Linux example: create a Git lab

mkdir -p ~/ms-cybersecurity-labs/week-00/git-evidence
cd ~/ms-cybersecurity-labs/week-00/git-evidence
git init
git config user.name "Student Analyst"
git config user.email "student@example.invalid"

This creates a local lab directory, initializes Git, and sets local identity for commits inside this repository.

Windows example: create a Git lab

New-Item -ItemType Directory -Force -Path "$HOME\ms-cybersecurity-labs\week-00\git-evidence"
Set-Location "$HOME\ms-cybersecurity-labs\week-00\git-evidence"
git init
git config user.name "Student Analyst"
git config user.email "student@example.invalid"

This performs the same task in PowerShell. Use Git Bash if your Windows terminal does not recognize Git commands.

Python example: hash a lab file

# hash_file.py
# Purpose: compute a SHA-256 hash for a local lab file.
# Safety: use this only on files you own or are authorized to handle.

from hashlib import sha256
from pathlib import Path

# Choose a lab file. Path() represents a filesystem path.
file_path = Path("sample-evidence.txt")

# Create a hash object. It starts empty.
digest = sha256()

# Open the file in binary mode so bytes are read exactly.
with file_path.open("rb") as file:
    # Read in chunks so the script can handle larger files.
    while True:
        chunk = file.read(8192)
        if not chunk:
            break
        digest.update(chunk)

# Print the final hexadecimal hash string.
print(digest.hexdigest())

This script reads a local file in binary chunks and prints its SHA-256 digest.

Network example: document a localhost-only result

# localhost_note_server.py
# Purpose: serve only the current lab folder on your own machine.
# It binds to 127.0.0.1 so it is not meant for public network exposure.

from http.server import ThreadingHTTPServer, SimpleHTTPRequestHandler

server_address = ("127.0.0.1", 8000)
httpd = ThreadingHTTPServer(server_address, SimpleHTTPRequestHandler)
print("Serving local lab files at http://127.0.0.1:8000")
httpd.serve_forever()

Run this only in your lab folder. It serves local files on 127.0.0.1 for a safe documentation exercise. It does not scan or attack any network.

Web example: document a local HTTP observation

curl -I http://127.0.0.1:8000
# Record status code, date header, server header, and command used.
# Do not use this against public websites for course labs unless assigned.

The point is not to attack the server. The point is to record the exact command and result so another student can reproduce the local observation.

Cloud example: structured cloud observation note

# cloud-evidence-note.yaml
# This is not a real cloud configuration.
# It is a structured note showing how to document a cloud observation safely.

This is a safe simulated note. Real cloud evidence may contain account identifiers, user names, IP addresses, and sensitive business context, so it must be protected according to policy.

9. Code Examples

9.1 Create and commit a README

cat > README.md <<'EOF'
# Week 0 Git Evidence Lab

This repository stores lab notes, scripts, and sample evidence metadata.

## Scope
Local training files only. No real personal, employer, university, or third-party data.

## Safety
Do not commit secrets, real evidence, private keys, passwords, or tokens.
EOF

git status
git add README.md
git commit -m "Add lab README and safety scope"

Line explanation: cat > README.md writes text into a file; the lines between EOF markers become the file content; git status checks the repository state; git add stages the file; git commit records the staged change with a message.

9.2 Create an evidence log

cat > evidence-log.md <<'EOF'
# Evidence Log

| ID | Time UTC | Source | Collector | Action | SHA-256 | Notes |
|---|---|---|---|---|---|---|
| E001 | 2026-06-28T10:00:00Z | sample-evidence.txt | Student Analyst | Created sample lab file | pending | Training data only |
EOF

git add evidence-log.md
git commit -m "Add evidence log template"

This creates a table for evidence notes. In real work, use your organization’s required format.

9.3 Compute file hash with Linux

echo "Sample training evidence" > sample-evidence.txt
sha256sum sample-evidence.txt

echo creates a sample file. sha256sum prints the SHA-256 hash and filename. This is safe because it uses local training data.

9.4 Compute file hash with Windows PowerShell

Set-Content -Encoding UTF8 sample-evidence.txt "Sample training evidence"
Get-FileHash .\sample-evidence.txt -Algorithm SHA256

Set-Content writes the sample file. Get-FileHash computes the SHA-256 digest.

9.5 Use .gitignore

cat > .gitignore <<'EOF'
# Generated files
*.tmp
*.log

# Local secrets - never commit
.env
*.key
*.pem

# Large local evidence copies
evidence-originals/
EOF

git add .gitignore
git commit -m "Add gitignore for generated files and secrets"

.gitignore helps prevent accidental tracking. It is not a security boundary. A file already committed remains in history unless special cleanup is performed.

9.6 Common mistakes

  • Mistake: committing passwords or API tokens. Fix: use environment variables, local ignored files, or secret managers.
  • Mistake: editing original evidence. Fix: preserve originals and analyze verified copies.
  • Mistake: writing vague commit messages. Fix: explain why the change was made.
  • Mistake: relying only on screenshots. Fix: record commands, outputs, hashes, and limitations.
  • Mistake: rewriting Git history after submission. Fix: treat submitted tags and commits as stable records.

10. Security Perspective

Common attacks and abuses

  • Secret leakage: passwords, tokens, SSH keys, cloud keys, or database URLs accidentally committed.
  • History manipulation: commits rewritten to hide changes or remove context.
  • Evidence tampering: files modified after collection without recording the change.
  • Misleading documentation: conclusions written stronger than the evidence supports.
  • Supply-chain compromise: malicious changes inserted into code, dependencies, build scripts, or configuration.

Common vulnerabilities

  • Public repositories containing secrets or private data.
  • No review process for security-sensitive changes.
  • No signed releases or tag verification.
  • No retention policy for investigation notes.
  • No access control over evidence storage.

Defensive strategies

  • Use private repositories for sensitive course work or professional work.
  • Use .gitignore, secret scanning, and code review.
  • Keep real evidence in approved evidence storage; commit metadata and analysis notes where appropriate.
  • Hash evidence before and after transfer.
  • Use clear commit messages, tags, and release notes.
  • Separate facts, assumptions, interpretation, and recommendations.

Detection methods

  • Scan repositories for committed secrets.
  • Review unusual force-pushes, deleted branches, or rewritten history.
  • Monitor changes to security-sensitive files such as CI/CD workflows, access policies, infrastructure templates, and detection rules.
  • Compare evidence hashes against recorded values.
  • Review audit logs for unauthorized repository access.

Mitigation and best practices

  • Rotate exposed secrets immediately; do not only delete the visible file.
  • Use least privilege for repository access.
  • Protect important branches and require review for security-critical changes.
  • Use signed commits or signed tags where the organization requires stronger provenance.
  • Back up repositories and evidence records according to retention policy.

12. Research Perspective

Research depends on reproducibility and provenance. A security paper may propose a detection method, measurement study, exploit mitigation, privacy technique, or forensic process. Readers need to know what data was used, how it was processed, which code version ran, and what limitations apply.

Current research directions

  • Reproducible security research: packaging code, data, scripts, and environments so results can be checked.
  • Artifact evaluation: assessing whether research artifacts are available, functional, reusable, and documented.
  • Software supply-chain security: improving provenance, build integrity, dependency tracking, and tamper detection.
  • Tamper-evident logging: designing logs where unauthorized changes are detectable.
  • Secure collaborative development: protecting code review, CI/CD workflows, and repository access.
  • Digital evidence integrity: improving methods for collection, storage, verification, and court/organizational admissibility.

Open problems

  • How can small teams achieve strong provenance without expensive infrastructure?
  • How can public research share enough artifacts for reproducibility without exposing sensitive data?
  • How can AI-generated code and reports be documented transparently and reviewed safely?
  • How can incident responders preserve volatile cloud evidence before it disappears?
  • How can organizations make audit trails trustworthy without making systems unusable?

Notable research areas to explore later include transparency logs, secure build systems, reproducible builds, software bills of materials, signed artifacts, supply-chain attacks, provenance graphs, forensic timelines, and privacy-preserving evidence sharing.

13. Case Studies

Case Study 1: Secret exposure in source repositories

What happened: many organizations have suffered incidents because developers accidentally committed credentials or tokens into repositories. Once pushed to a shared or public repository, a secret may be copied by bots, humans, logs, mirrors, or backups.

Root cause: weak secret-handling process, missing secret scanning, poor training, and misunderstanding that deleting a line removes it from all history.

Impact: exposed credentials can allow unauthorized access to cloud accounts, databases, source code, or internal services.

Lessons learned: never commit secrets; scan repositories; rotate exposed credentials; restrict permissions; protect history; use secret managers.

Case Study 2: Supply-chain compromise and build trust

What happened: major software supply-chain incidents have shown that attackers may target build systems, update mechanisms, repositories, or dependencies instead of attacking every victim directly.

Root cause: insufficient control over the path from source code to build artifact, weak monitoring of trusted systems, or inadequate provenance checks.

Impact: many downstream organizations can be affected because they trust the compromised software or update channel.

Lessons learned: protect repositories and CI/CD systems, review changes, monitor build pipelines, sign releases, verify dependencies, and document provenance.

Case Study 3: Incident notes that cannot support conclusions

What happened: in many investigations, responders find useful clues but fail to record commands, timestamps, tool versions, source systems, or hashes. Later, reviewers cannot distinguish facts from assumptions.

Root cause: pressure during incidents, lack of templates, no evidence-handling process, and overreliance on memory or screenshots.

Impact: conclusions may be challenged, response may be delayed, and lessons learned may be weak.

Lessons learned: use evidence logs, standardized timelines, controlled storage, hash verification, peer review, and clear uncertainty statements.

14. Common Misconceptions

  • “GitHub and Git are the same.” Incorrect. Git is the version control tool; GitHub is one hosting platform.
  • “Git is a backup system.” Weak assumption. Git records project history, but it does not replace tested backups or evidence storage.
  • “If I delete a secret from the latest file, it is gone.” Incorrect. It may remain in Git history, forks, logs, caches, or clones.
  • “A screenshot is enough evidence.” Often weak. Screenshots lack full provenance and are harder to verify than structured logs and hashes.
  • “A matching hash proves the evidence is true.” Incorrect. It proves integrity relative to a previous hash, not truth or correct collection.
  • “Commit messages do not matter.” Incorrect. They are part of the audit trail and help reviewers understand intent.
  • “Rewriting history is always bad.” Too absolute. It can be useful before publishing, but it is dangerous after collaboration, submission, or evidence recording.
  • “More documentation is always better.” Incorrect. Documentation should be complete, accurate, searchable, and relevant, not bloated.
  • “Metadata is always reliable.” Incorrect. Metadata can be changed by copying, exporting, editing, or clock errors.
  • “Public repositories are safe if nobody knows the URL.” Incorrect. Public data should be treated as discoverable.

15. Summary

Git helps track changes, compare versions, collaborate, and preserve a project-level history. Documentation makes technical work understandable and reviewable. Evidence handling protects the reliability of observations used in cybersecurity conclusions.

The most important distinction is this: Git is excellent for lab notes, scripts, reports, and reproducible work, but real evidence may need specialized storage and process controls. A good cybersecurity learner records source, time, commands, tool versions, hashes, assumptions, limitations, and conclusions separately.

18. Practice Lab

Lab title

Evidence-Aware Git Repository for Cybersecurity Labs

Safety scope

  • Use only files you create in this lab.
  • Do not use real passwords, private keys, tokens, student data, employer data, patient data, or third-party evidence.
  • Do not push to a public repository unless every file is safe and intentionally public.
  • Do not modify real system logs for this lab.

Prerequisites

  • Git installed.
  • Terminal or PowerShell access.
  • Python 3 available for optional hash script.
  • Lessons 0.1–0.4 completed or reviewed.

Objectives

  • Create a Git repository.
  • Create a README with scope and safety notes.
  • Create a sample evidence file.
  • Compute and record its SHA-256 hash.
  • Create an evidence log.
  • Commit changes with meaningful messages.
  • Use a branch for a draft report.
  • Tag the final lab version.

Step 1: Create the repository

mkdir -p ~/ms-cybersecurity-labs/week-00/git-evidence
cd ~/ms-cybersecurity-labs/week-00/git-evidence
git init
git config user.name "Student Analyst"
git config user.email "student@example.invalid"

Windows users may use the PowerShell setup shown earlier.

Step 2: Create safe lab files

cat > sample-evidence.txt <<'EOF'
This is a fake evidence file for a cybersecurity documentation lab.
It contains no real personal data, no secrets, and no external system data.
EOF

cat > README.md <<'EOF'
# Evidence-Aware Git Lab

## Purpose
Practice Git, documentation, hashes, and chain-of-custody thinking.

## Scope
Training files created locally by the student.

## Safety
No real credentials, private keys, personal data, employer data, or third-party evidence.
EOF

Step 3: Compute the hash

sha256sum sample-evidence.txt

Windows alternative:

Get-FileHash .\sample-evidence.txt -Algorithm SHA256

Step 4: Record evidence metadata

cat > evidence-log.md <<'EOF'
# Evidence Log

| ID | Time UTC | Source | Collector | Action | SHA-256 | Limitations |
|---|---|---|---|---|---|---|
| E001 | replace-time | sample-evidence.txt | replace-name | Created fake training evidence file | replace-hash | Training data only |
EOF

Replace replace-time, replace-name, and replace-hash with your actual lab values.

Step 5: Commit the first version

git status
git add README.md evidence-log.md sample-evidence.txt
git commit -m "Add safe training evidence and documentation"
git log --oneline

Step 6: Create a draft report branch

git switch -c draft-report
cat > report.md <<'EOF'
# Draft Report

## Finding
The sample evidence file was created and hashed successfully.

## Evidence
See evidence-log.md.

## Limitations
This is training data only. It does not prove any real incident.
EOF

git add report.md
git commit -m "Draft lab report with limitations"

Step 7: Merge and tag

git switch main
git merge draft-report
git tag week0-lesson05-complete
git log --oneline --decorate --graph --all

Expected outcomes

  • You can initialize and inspect a Git repository.
  • You can create clear documentation for a security lab.
  • You can compute a hash and record it as evidence metadata.
  • You can use commits, branches, merges, and tags.
  • You can explain what Git proves and what it does not prove.

Lab report checklist

  • Operating system and Git version.
  • Repository path.
  • Hash command used.
  • SHA-256 value.
  • Screenshot or copied text of git log --oneline --decorate --graph --all.
  • Explanation of chain of custody in your own words.
  • Statement confirming no real secrets or third-party data were used.
  • One paragraph explaining why Git is helpful but not a complete forensic evidence system.
