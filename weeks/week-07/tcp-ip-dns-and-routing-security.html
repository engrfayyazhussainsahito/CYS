Lesson 7.2: TCP/IP, DNS, and Routing Security

This lesson separates three foundational network systems: TCP/IP packet delivery, DNS name resolution, and routing path selection. You will learn how they work, why they fail, and how defenders secure and monitor them.

Estimated reading time390–570 minutes
Difficulty levelBeginner → MS foundation
Required backgroundLesson 7.1 recommended; core ideas are reintroduced
Practice time6–10 hours

1. Introduction

Most network communication depends on three related systems. TCP/IP moves data between applications across networks. DNS, the Domain Name System, maps human-readable names to network information. Routing chooses where packets should go next.

Plain English: DNS answers “what address should I use?”, TCP/IP carries the conversation, and routing decides the path that packets take. Security problems in any one of these can send traffic to the wrong place, expose services, hide malicious behavior, or break availability.

This lesson exists because defenders cannot secure networks by memorizing port numbers. They need to reason about packet headers, connection state, name resolution, resolver behavior, routing tables, gateway choices, route changes, and logs. These foundations are used by network engineers, security engineers, cloud engineers, SOC analysts, incident responders, malware analysts, and internet infrastructure researchers.

Where it is used: every bank, hospital, government agency, cloud provider, university, defense network, and enterprise depends on TCP/IP, DNS, and routing. A web request, database connection, software update, VPN session, authentication redirect, and cloud API call all depend on these systems.

Safety boundary: this lesson is defensive. Do not scan networks, spoof traffic, poison DNS, test routers, or probe public systems without written authorization. Labs use your own machine, localhost, known documentation domains, or fictional logs.

2. Learning Outcomes

  • Explain TCP/IP, DNS, and routing using first-principles language.
  • Distinguish IP addresses, TCP/UDP ports, DNS names, routes, gateways, and services.
  • Describe what happens when a browser connects to a website from DNS lookup through TCP connection and routing.
  • Interpret basic IP headers, TCP/UDP concepts, DNS record types, TTL, caching, and routing tables.
  • Explain DNS security risks including spoofing, cache poisoning, tunneling, malicious domains, and resolver misconfiguration.
  • Explain routing risks including wrong default routes, route leaks, route hijacks, asymmetric paths, and BGP trust limitations.
  • Identify useful defensive telemetry: DNS logs, flow logs, route-change logs, firewall logs, proxy logs, and endpoint network logs.
  • Run safe local commands to inspect addresses, routes, DNS resolution, and local sockets.
  • Write a short defensive report that connects observed network behavior to security risk.

3. Prerequisites

No prior networking knowledge is assumed. Key concepts are reintroduced here:

  • Bit: the smallest unit of digital information, either 0 or 1.
  • Byte: usually 8 bits.
  • Packet: a small unit of network data with control information.
  • Header: metadata at the beginning of a protocol unit. It tells systems how to process the data.
  • Payload: the data carried after the header.
  • Host: a device that sends or receives network traffic.
  • Client: the side that starts a request.
  • Server: the side that waits for requests.
  • Protocol: a shared communication rule, like a grammar for machines.

Analogy: sending a package requires a recipient name, street address, route through postal facilities, and tracking information. Networks do something similar with names, IP addresses, routes, protocol headers, and logs.

4. Core Concepts

4.1 TCP/IP is a protocol family, not one protocol

TCP/IP is a family of protocols that make internet-style networking possible. IP handles addressing and packet forwarding. TCP and UDP sit above IP and help applications exchange data. DNS, HTTP, TLS, SSH, SMTP, and many other application protocols sit above those layers.

4.2 Encapsulation

Encapsulation means one protocol wraps data from a higher layer inside its own header. An HTTP message can be carried inside TCP, which is carried inside IP, which is carried inside an Ethernet or Wi-Fi frame. Each layer adds information needed by that layer.

4.3 IP addressing

An IP address identifies an interface or endpoint for packet forwarding. IPv4 addresses look like 192.0.2.10. IPv6 addresses look like 2001:db8::10. Addresses are not proof of identity; they are routing information and can change, be shared, or be translated.

4.4 Packets and headers

An IP packet contains an IP header and payload. The header includes information such as source address, destination address, protocol number, and hop limit or time-to-live. Defenders use header metadata to understand who talked to whom, when, and through which protocol, even when payload content is encrypted.

4.5 TCP

Transmission Control Protocol provides reliable, ordered byte-stream delivery between applications. It uses sequence numbers, acknowledgments, retransmission, flow control, and congestion control. A beginner intuition: TCP is like a careful conversation where each side tracks what was sent, what arrived, and what must be resent.

4.6 TCP three-way handshake

Before data exchange, TCP commonly begins with SYN, SYN-ACK, and ACK messages. This sets up connection state. Firewalls often use this state to allow responses to legitimate outbound connections while denying unrelated inbound traffic.

4.7 UDP

User Datagram Protocol sends datagrams without TCP-style connection setup or delivery guarantees. This does not make UDP bad. It is useful for DNS, real-time media, and protocols that implement reliability differently. Security implication: defenders must understand application behavior instead of assuming every flow has a TCP-style session.

4.8 Ports

A port is a transport-layer number used to direct traffic to an application endpoint. TCP port 443 commonly indicates HTTPS, but port numbers alone do not prove what is running. A malicious program can use common ports, and a legitimate service can be misconfigured.

4.9 DNS from first principles

Humans prefer names such as example.com. Networks forward packets using addresses. DNS bridges that gap by mapping names to records. A resolver asks questions. An authoritative server provides answers for a domain. A recursive resolver performs the lookup work on behalf of a client.

4.10 Common DNS record types

  • A: maps a name to an IPv4 address.
  • AAAA: maps a name to an IPv6 address.
  • CNAME: aliases one name to another name.
  • MX: identifies mail exchangers for a domain.
  • TXT: stores text data, often used for domain verification and email-security policies.
  • NS: identifies name servers for a zone.
  • SOA: stores zone authority metadata.
  • PTR: maps an address back to a name in reverse DNS.
  • CAA: indicates which certificate authorities may issue certificates for a domain.

4.11 DNS caching and TTL

TTL means time to live. In DNS, TTL tells resolvers how long an answer may be cached. Caching improves speed and reduces load, but stale or poisoned cache entries can cause security and reliability problems.

4.12 DNSSEC

DNSSEC adds origin authentication and integrity protection to DNS data. It helps a resolver verify that a DNS answer is authentic and has not been modified. It does not provide confidentiality by itself; DNS queries may still be visible unless protected by mechanisms such as encrypted DNS transports.

4.13 DNS privacy

Traditional DNS is often visible to networks on the path. Encrypted DNS mechanisms can improve privacy, but they also change enterprise monitoring and policy design. Security teams must balance privacy, abuse detection, resolver trust, and user expectations.

4.14 DNS abuse

Attackers use DNS for phishing domains, command-and-control, domain generation algorithms, fast-flux hosting, tunneling, exfiltration attempts, and infrastructure hiding. Defenders monitor DNS for unusual domains, newly observed domains, abnormal query volume, high NXDOMAIN rates, and suspicious query patterns.

4.15 Routing from first principles

Routing chooses a next hop for a packet. Your device usually has a routing table. If the destination is local, the packet stays local. If not, the device sends it to a default gateway. Routers repeat forwarding decisions until the packet reaches the destination network or fails.

4.16 Longest-prefix match

When several routes match a destination, routing usually chooses the most specific prefix. A route for 10.1.2.0/24 is more specific than 10.0.0.0/8. This matters because a more-specific route can override a broader route.

4.17 Default route

A default route is used when no more-specific route matches. In home networks, it often points to the router. In cloud networks, it may point to an internet gateway, NAT gateway, firewall appliance, or virtual router.

4.18 BGP and internet routing

Border Gateway Protocol, or BGP, exchanges reachability information between autonomous systems on the internet. It is essential, but historically relies heavily on trust and policy. Misconfigurations or malicious announcements can cause route leaks, outages, or traffic redirection.

4.19 Route hijacking and route leaks

A route hijack occurs when traffic is attracted to an unintended route, often through incorrect or malicious route announcements. A route leak occurs when routing information is propagated beyond intended policy boundaries. Both can affect availability, confidentiality, and integrity at the path level.

4.20 RPKI and route-origin validation

Resource Public Key Infrastructure, or RPKI, helps networks validate whether an autonomous system is authorized to originate a route for an IP prefix. It reduces some routing risks, but deployment, policy, and operational correctness still matter.

4.21 Fragmentation and path MTU

Networks have maximum transmission units. When packets are too large, fragmentation or path MTU discovery may be involved. Fragmentation can complicate firewalls and intrusion detection because the full meaning of traffic may only be clear after reassembly.

4.22 Security principle: control the name, path, and flow

For an important service, defenders should know: which names resolve to it, which resolvers are trusted, which routes lead to it, which sources may connect, which ports and protocols are required, which logs prove behavior, and which alerts indicate misuse.

7. Mathematical Foundations

7.1 Prefix length and address ranges

In IPv4, an address has 32 bits. A CIDR prefix such as /24 means 24 bits identify the network, leaving 8 bits for host addresses.

Formula: approximate address count = 2host bits. For a /24, host bits = 32 - 24 = 8, so 28 = 256 total addresses. Some addresses may be reserved depending on the environment.

7.2 Longest-prefix example

Suppose a router has these routes:

  • 10.0.0.0/8 via Gateway A
  • 10.10.0.0/16 via Gateway B
  • 0.0.0.0/0 via Gateway C

A packet for 10.10.5.7 matches all three, but /16 is most specific, so Gateway B is chosen.

7.3 TCP sequence numbers

TCP uses sequence numbers to track byte positions in a stream. If a sender transmits 100 bytes starting at sequence number 1000, the next expected sequence number is 1100. This is part of how TCP detects missing data and supports retransmission.

7.4 DNS TTL and cache behavior

If a DNS answer has a TTL of 300 seconds, a resolver may reuse that answer for up to about 5 minutes. Shorter TTLs can make changes propagate faster but increase query load. Longer TTLs reduce load but can keep stale data active longer.

7.5 Detection rate intuition

A simple threshold might alert when a host produces more than 50 failed DNS lookups in 10 minutes. This is not proof of compromise. It is a signal requiring context: software updates, typo-heavy users, broken configuration, malware domain-generation behavior, or DNS tunneling could all change the interpretation.

Python example: safe DNS lookup

# Safe DNS lookup example: resolves one documentation domain only.
# This is not a scanner and does not contact random targets.

import socket

def resolve_name(name: str) -> None:
    try:
        results = socket.getaddrinfo(name, 443, type=socket.SOCK_STREAM)
    except socket.gaierror as error:
        print(f"Resolution failed: {error}")
        return

    seen = set()
    for family, socktype, proto, canonname, sockaddr in results:
        address = sockaddr[0]
        if address not in seen:
            seen.add(address)
            print(address)

resolve_name("example.com")

Python example: defensive DNS log classification

# Defensive log-parsing example.
# It classifies toy DNS log lines and does not generate network traffic.

from collections import Counter

logs = [
    "2026-06-29T10:00:00 client=lab1 query=example.com rcode=NOERROR",
    "2026-06-29T10:00:01 client=lab1 query=missing.invalid rcode=NXDOMAIN",
    "2026-06-29T10:00:02 client=lab2 query=docs.python.org rcode=NOERROR",
    "2026-06-29T10:00:03 client=lab1 query=random-name-842.invalid rcode=NXDOMAIN",
]

counts = Counter()
for line in logs:
    if "rcode=NXDOMAIN" in line:
        counts["nxdomain"] += 1
    elif "rcode=NOERROR" in line:
        counts["success"] += 1
    else:
        counts["other"] += 1

print(counts)
if counts["nxdomain"] >= 2:
    print("Review: repeated failed DNS lookups can be normal, misconfigured, or suspicious depending on context.")

Network example

A laptop wants to load https://example.com. It asks a resolver for an address, receives an answer, opens a TCP connection to port 443, sends packets to its default gateway if the destination is not local, and receives encrypted HTTPS data after TLS negotiation.

Web example

If DNS is wrong, the browser may connect to the wrong address. TLS certificate validation helps detect name-to-identity mismatch, but users and systems must not disable certificate validation.

Cloud example

In a cloud virtual network, a private subnet might route outbound internet traffic through a NAT gateway, while inbound traffic to databases is denied. Route tables, security groups, DNS private zones, and flow logs together define and observe network behavior.

9. Code Examples

Local-only TCP server

#!/usr/bin/env python3
"""
Local-only TCP demonstration.
Run the server in one terminal, then the client in another.
The server binds to 127.0.0.1, so it is not exposed to the network.
"""

import socket

HOST = "127.0.0.1"
PORT = 9090

with socket.socket(socket.AF_INET, socket.SOCK_STREAM) as server:
    server.bind((HOST, PORT))
    server.listen(1)
    print(f"Listening safely on {HOST}:{PORT}")
    conn, addr = server.accept()
    with conn:
        print(f"Connection from {addr}")
        data = conn.recv(1024)
        print(f"Received: {data!r}")
        conn.sendall(b"hello from local lab\n")

Line-by-line explanation: the script imports Python's socket library, binds only to 127.0.0.1, listens on port 9090, accepts one local connection, reads a small message, and returns a short reply. Binding to loopback keeps the example local to your machine.

Local-only TCP client

#!/usr/bin/env python3
import socket

HOST = "127.0.0.1"
PORT = 9090

with socket.create_connection((HOST, PORT), timeout=3) as client:
    client.sendall(b"hello local server")
    print(client.recv(1024).decode())

Common mistakes: binding a learning server to 0.0.0.0 exposes it on all interfaces; ignoring timeouts can make scripts hang; printing sensitive network data can leak environment details; using public targets for experiments can violate law or policy.

Secure coding relevance: network code must validate inputs, use timeouts, handle errors, log safely, avoid unnecessary exposure, and use TLS for real sensitive communication. A working socket program is not automatically a secure service.

10. Security Perspective

Common attacks

  • DNS spoofing and cache poisoning.
  • Malicious domain infrastructure for phishing or command-and-control.
  • DNS tunneling or unusual DNS-based exfiltration attempts.
  • Route hijacking and route leaks.
  • TCP SYN floods and other denial-of-service patterns.
  • Fragmentation tricks that confuse inspection or reassembly.
  • Open recursive resolver abuse.
  • Man-in-the-middle attempts where routing, DNS, or certificate validation is weak.

Common vulnerabilities

  • Unmanaged resolvers and inconsistent DNS policy.
  • Missing DNSSEC validation where authenticity matters.
  • Overly permissive egress DNS that bypasses monitoring.
  • Incorrect route tables exposing private systems.
  • Weak BGP filtering by network operators.
  • Firewall rules based only on ports rather than service purpose and ownership.
  • Logging gaps for DNS, routing, and network flows.

Defensive strategies

  • Use managed resolvers with logging, policy, and privacy review.
  • Validate DNSSEC where appropriate.
  • Restrict direct outbound DNS when enterprise policy requires monitored resolvers.
  • Review route tables and default routes in cloud and enterprise networks.
  • Use route filtering, RPKI/ROV awareness, and provider coordination for internet-facing networks.
  • Monitor DNS and flow logs for abnormal behavior.
  • Keep certificate validation enabled; TLS and DNS solve different problems.

Detection methods

  • DNS logs: NXDOMAIN spikes, algorithmic-looking names, newly observed domains, unusual TXT queries.
  • Flow logs: unexpected destinations, new countries/ASNs, unusual ports, beacon-like timing, high-volume traffic.
  • Route monitoring: path changes, prefix origin changes, reachability loss, asymmetric behavior.
  • Firewall logs: denied flows, unexpected allowed flows, policy changes.

Mitigation and best practices

Use layered controls: correct DNS configuration, resolver logging, DNSSEC validation, TLS certificate validation, least-privilege routing, default-deny firewalls, segmentation, change review, and incident-response playbooks. Do not rely on one control to solve all path, name, and flow risks.

11. Industry Perspective

  • Banks: monitor DNS and egress flows for fraud infrastructure, protect payment systems with strict route and firewall policy, and coordinate with providers for DDoS and routing incidents.
  • Governments: secure public services, citizen portals, and interagency networks; DNS and routing failures can affect national-scale availability.
  • Cloud providers: operate massive DNS, routing, load-balancing, private networking, and DDoS-protection systems.
  • Healthcare: depend on reliable routing and name resolution for clinical systems, medical devices, identity services, and emergency operations.
  • Defense: treat network paths, route control, DNS infrastructure, and traffic analysis as mission-critical security concerns.
  • Universities: manage open research networks, student networks, labs, public web services, and sensitive administrative systems.
  • Large enterprises: integrate DNS filtering, private DNS, cloud route tables, VPNs, proxies, firewalls, and flow monitoring across hybrid environments.

12. Research Perspective

Current research studies how internet-scale routing fails, how DNS is abused, how encrypted DNS affects enterprise detection and user privacy, how to measure BGP hijacks, how to deploy RPKI safely, and how to detect malicious behavior using metadata without inspecting sensitive content.

Open problems include privacy-preserving DNS monitoring, reducing false positives in DNS abuse detection, making route-origin validation easier to deploy correctly, detecting short-lived route hijacks, securing interdomain routing beyond origin validation, and building explainable network-detection systems.

Notable research directions include internet measurement, DNS abuse analysis, formal verification of network policy, encrypted traffic analysis, routing security economics, DDoS resilience, and post-quantum implications for signed network infrastructure data.

13. Case Studies

Case Study 1: DNS cache poisoning research and operational risk

What happened: practical DNS cache-poisoning weaknesses showed that attackers could sometimes cause resolvers to accept false answers. Root cause: predictable or insufficiently protected DNS transaction behavior and protocol limitations. Impact: users could be sent to wrong destinations. Lessons: resolver hardening, source-port randomization, DNSSEC validation, and monitoring matter.

Case Study 2: Route leaks and internet outages

What happened: routing mistakes have repeatedly caused traffic to follow unintended paths or become unreachable. Root cause: incorrect route propagation, weak filtering, and complex provider relationships. Impact: outages, degraded performance, or unintended traffic paths. Lessons: routing policy, monitoring, provider coordination, and route validation are security controls.

Case Study 3: DNS abuse in phishing and malware operations

What happened: attackers register or compromise domains for phishing, command-and-control, payload delivery, and evasion. Root cause: the global DNS system is open and flexible, which supports both legitimate use and abuse. Impact: credential theft, malware control, and user redirection. Lessons: DNS logs, domain reputation, takedown coordination, and user protection are part of defense.

Case Study 4: Misconfigured cloud route table

What happened: a fictional enterprise attaches a private database subnet to a route table that sends traffic to an internet gateway. Root cause: copied infrastructure-as-code without review. Impact: increased exposure even before application bugs are considered. Lessons: route tables, security groups, DNS names, and firewall rules must be reviewed together.

14. Common Misconceptions

  • “DNS only matters for convenience.” DNS controls where users and systems try to connect.
  • “An IP address proves identity.” Addresses can be reassigned, translated, spoofed in some contexts, or routed incorrectly.
  • “Port 443 means safe.” It means traffic commonly uses HTTPS. The service, certificate, identity, content, and policy still matter.
  • “DNSSEC encrypts DNS.” DNSSEC authenticates and protects integrity of DNS data; it does not provide confidentiality by itself.
  • “The shortest route is always used.” Routing follows policy and prefix selection, not geographic distance.
  • “BGP problems only affect internet service providers.” Any organization with internet-facing services can be affected by routing incidents.
  • “A route table is just networking plumbing.” Route tables define reachability and therefore directly affect security exposure.
  • “Encrypted DNS means defenders cannot do anything.” Defenders still have endpoint, proxy, firewall, flow, EDR, and policy telemetry, but the architecture must be intentional.

15. Summary

TCP/IP, DNS, and routing are the foundation of network communication. DNS maps names to network information. TCP and UDP carry application conversations over IP. Routing chooses packet paths. Security requires checking all three: the name must resolve correctly, the path must be intended, and the flow must be authorized and monitored. Strong defense combines resolver security, DNSSEC where appropriate, TLS validation, route review, route monitoring, least-privilege firewall policy, and useful telemetry.

16. Glossary

TCP/IP
A family of internet protocols used to move data across networks.
IP
Internet Protocol; provides addressing and packet forwarding.
TCP
Transmission Control Protocol; provides reliable ordered byte-stream delivery.
UDP
User Datagram Protocol; sends datagrams without TCP-style reliability.
Header
Protocol metadata used to process, route, or interpret data.
Payload
The carried data inside a protocol unit.
Encapsulation
Wrapping higher-layer data inside lower-layer protocol headers.
DNS
Domain Name System; maps names to network records.
Resolver
A DNS component that performs or requests name resolution.
Authoritative DNS Server
A DNS server that provides answers for a domain zone.
TTL
Time to live; cache lifetime for DNS answers or hop limit concept in packets.
DNSSEC
DNS Security Extensions; adds origin authentication and integrity to DNS data.
Routing Table
A table that maps destination prefixes to next hops or interfaces.
Default Route
The route used when no more-specific route matches.
Longest-Prefix Match
Choosing the most specific matching route for a destination.
BGP
Border Gateway Protocol; exchanges routing reachability between autonomous systems.
Autonomous System
A network or group of networks under common routing administration.
Route Leak
Unintended propagation of routing information beyond policy boundaries.
Route Hijack
Attracting traffic to an unintended network path through incorrect or malicious routing information.
RPKI
Resource Public Key Infrastructure; supports validation of route-origin authorization.

18. Practice Lab

Lab: Safe TCP/IP, DNS, and Routing Observation

Prerequisites: your own laptop or course VM. Use only your machine, localhost, and known documentation/test domains such as example.com. Do not scan or test third-party networks.

Objectives

  • Identify local addresses and routes.
  • Observe one safe DNS resolution.
  • Run a localhost-only TCP client/server example.
  • Write a short defensive report connecting name, path, and flow.

Part A: Local route inventory

  1. Linux: run ip addr and ip route. Windows: run ipconfig /all and route print.
  2. Identify your local/private address range and default gateway concept. Do not publish sensitive details.
  3. Draw a simple diagram: your device → default gateway → internet.

Part B: DNS observation

  1. Resolve example.com using getent hosts example.com, nslookup example.com, or your OS equivalent.
  2. Record the resolver used if visible and the answer type if shown.
  3. Explain how caching might change repeated results.

Part C: Local TCP test

  1. Save the local-only server script from this lesson as server.py.
  2. Save the client script as client.py.
  3. Run the server, then run the client.
  4. Observe that the server listens on 127.0.0.1, not on all network interfaces.

Part D: Defensive report

Submit a one-page report with: scope, commands used, diagram, DNS observation, routing observation, local TCP observation, security risks avoided, and two hardening recommendations. Expected outcome: you can explain how a name becomes an address, how a packet finds a path, and how a local service should avoid unnecessary exposure.
