Lesson 7.3: Firewalls, Segmentation, and Zero Trust

This lesson teaches how organizations decide which network conversations are allowed, which are blocked, how network zones reduce blast radius, and why Zero Trust is a design philosophy rather than a single product.

Estimated reading time420–620 minutes
Difficulty levelBeginner → MS foundation
Required backgroundLessons 7.1 and 7.2 recommended; concepts are reintroduced
Practice time6–10 hours

1. Introduction

A firewall is a control point that decides whether traffic should be allowed, denied, logged, or sometimes modified. A segment is a part of a network separated from other parts by boundaries and rules. Zero Trust is a security approach that avoids automatic trust based only on location, such as “inside the company network.”

Plain English: firewalls are gates, segmentation creates rooms, and Zero Trust asks for proof before opening the right door. A strong network design uses all three. It does not assume that every device inside the building, campus, VPN, or cloud account is safe.

This topic exists because flat networks fail badly. If every system can talk to every other system, one compromised laptop, web server, or account may become a path to databases, file shares, admin systems, backups, and identity infrastructure. Firewalls and segmentation reduce unnecessary paths. Zero Trust adds stronger identity, device, application, data, and context checks.

Where it is used: banks segment card-processing networks, hospitals isolate clinical devices, cloud providers use virtual network controls, governments separate classified and public systems, universities separate students from administration systems, and large enterprises design zones for users, servers, developers, production workloads, and security tooling.

Safety boundary: do not change production firewall rules or bypass network controls. Labs in this lesson use read-only inspection, fictional policies, localhost examples, and VM-only practice.

2. Learning Outcomes

  • Explain what firewalls, firewall rules, zones, segmentation, and Zero Trust mean from first principles.
  • Distinguish packet filters, stateful firewalls, next-generation firewalls, host firewalls, network firewalls, cloud security groups, and web application firewalls.
  • Explain rule order, deny by default, allowlisting, least privilege, logging, and rule ownership.
  • Design simple segments such as internet, DMZ, application, database, admin, and monitoring zones.
  • Describe north-south and east-west traffic and why both matter to defenders.
  • Explain Zero Trust components such as policy engine, policy administrator, and policy enforcement point.
  • Identify common firewall and segmentation mistakes.
  • Create a safe fictional segmentation model and evaluate toy flows using code.
  • Write a professional network-control recommendation with residual risk.

3. Prerequisites

This lesson reintroduces needed concepts before using them:

  • Packet: a small unit of network data with headers and payload.
  • IP address: a network-layer address used to route packets.
  • Port: a transport-layer number used by TCP or UDP to identify an application service.
  • Flow: a conversation between endpoints, often described by source IP, destination IP, protocol, source port, and destination port.
  • Subnet: a range of IP addresses grouped together.
  • Trust boundary: a place where assumptions change, such as internet to DMZ or user network to database network.
  • Identity: the digital representation of a user, device, service, or workload.

4. Core Concepts

4.1 What problem does a firewall solve?

Networks need communication, but not unlimited communication. A firewall helps enforce the policy: “this source may talk to this destination using this service for this reason.”

Example: students may reach a portal over HTTPS. They should not connect directly to the grade database over PostgreSQL. The firewall does not know whether a student is honest. It only enforces the allowed network paths.

4.3 Deny by default

Deny by default means traffic is blocked unless a rule explicitly permits it. This is safer than allowing everything and trying to block only known bad traffic, because attackers constantly find new paths.

4.4 Stateless vs stateful firewalls

A stateless packet filter evaluates packets mostly in isolation. A stateful firewall tracks whether traffic belongs to an expected connection. For example, if a client opens a TCP connection to a web server, return traffic from that server may be allowed because it belongs to an established flow.

4.5 Types of firewalls

  • Host firewall: runs on an endpoint or server, such as Windows Defender Firewall, Linux nftables, or UFW.
  • Network firewall: sits between networks or zones.
  • Cloud security group: virtual firewall-like rules attached to cloud resources or virtual networks.
  • Next-generation firewall: may add application awareness, user identity, threat intelligence, and deeper inspection.
  • Web application firewall: focuses on HTTP/web-layer traffic and application-layer patterns. It does not replace secure code.

4.6 Segmentation

Segmentation divides a network into smaller zones. Each zone has a purpose and allowed flows. Segmentation reduces blast radius, meaning the amount of damage possible from one compromise.

Common zones include:

  • Internet: untrusted external network.
  • DMZ: public-facing services isolated from sensitive internal systems.
  • User zone: laptops, desktops, and user devices.
  • Application zone: internal services and business logic.
  • Database zone: restricted data stores.
  • Admin zone: privileged management systems.
  • Monitoring zone: logging, SIEM, sensors, and security tooling.

4.7 North-south and east-west traffic

North-south traffic enters or leaves a network boundary. East-west traffic moves internally between systems. Traditional perimeter firewalls often focused on north-south traffic. Modern environments also need east-west controls because attackers frequently move laterally after initial access.

4.8 VLANs, ACLs, and security groups

A VLAN is a way to separate broadcast domains at layer 2. An ACL is a list of allow/deny rules. A security group is a cloud control that behaves like a virtual firewall. These controls are useful, but none of them automatically creates a good design. The design comes from clear assets, flows, identities, and risk decisions.

4.9 Microsegmentation

Microsegmentation applies fine-grained controls between workloads, services, containers, or identities. Instead of one large “server network,” each service may have specific allowed communication. This is powerful but operationally difficult if teams do not understand real application flows.

4.10 Zero Trust

Zero Trust does not mean “trust no one ever.” It means no request receives implicit trust simply because it comes from a familiar network. Access decisions should evaluate identity, device health, application, data sensitivity, policy, and context.

In NIST-style architecture, important logical components include:

  • Policy Engine: makes the access decision.
  • Policy Administrator: establishes or terminates the connection path based on the decision.
  • Policy Enforcement Point: enforces the decision near the resource or access path.
Important: Zero Trust is not a single appliance, VPN replacement, or marketing label. It is an architecture and migration strategy.

4.11 Policy quality

A good network policy is specific, justified, logged where useful, reviewed periodically, and mapped to an owner. A bad policy says “allow any to any” or creates permanent exceptions nobody owns.

7. Mathematical Foundations

7.1 Risk reduction intuition

A simple teaching model is:

Risk = Likelihood × Impact

Segmentation can reduce likelihood by removing unnecessary paths and reduce impact by limiting what a compromised system can reach.

7.2 Attack path count

Suppose a flat network has 5 zones and every zone can talk to every other zone. Directed possible zone-to-zone paths are:

n × (n - 1) = 5 × 4 = 20 possible directed paths

If policy allows only 4 justified flows, then the allowed path count falls from 20 to 4. This does not prove security, but it reduces unnecessary reachable paths by:

Reduction = (20 - 4) / 20 = 0.80 = 80%

7.3 Rule precision

Compare two rules:

  • Bad: allow any source to any destination on any port.
  • Better: allow 10.10.10.0/24 to 10.10.20.10 on TCP port 443.

The better rule has smaller scope. Smaller scope usually means less exposure.

7.4 Availability trade-off

If a firewall is too restrictive, legitimate services may fail. If it is too permissive, attack paths expand. Network security requires balancing confidentiality, integrity, availability, and operational need.

Python example: toy policy evaluator

# Tiny educational firewall-policy evaluator.
# It does not change your real firewall. It only evaluates toy flows.

from dataclasses import dataclass
from ipaddress import ip_address, ip_network

@dataclass
class Rule:
    name: str
    src: str
    dst: str
    port: int
    action: str  # "allow" or "deny"

rules = [
    Rule("students_to_web", "10.10.10.0/24", "10.10.20.10/32", 443, "allow"),
    Rule("admins_to_db", "10.10.30.0/24", "10.10.40.10/32", 5432, "allow"),
]

def matches(rule: Rule, src: str, dst: str, port: int) -> bool:
    return (
        ip_address(src) in ip_network(rule.src)
        and ip_address(dst) in ip_network(rule.dst)
        and port == rule.port
    )

def decide(src: str, dst: str, port: int) -> str:
    for rule in rules:
        if matches(rule, src, dst, port):
            return f"{rule.action.upper()} by {rule.name}"
    return "DENY by default"

for flow in [
    ("10.10.10.25", "10.10.20.10", 443),
    ("10.10.10.25", "10.10.40.10", 5432),
    ("10.10.30.8", "10.10.40.10", 5432),
]:
    print(flow, "=>", decide(*flow))

9. Code Examples

The Python policy evaluator above demonstrates a firewall idea without touching a real firewall. Each rule has source, destination, port, and action. The decide() function checks rules in order. If no rule matches, the flow is denied by default.

Common mistakes in policy code or policy-as-code

  • Using 0.0.0.0/0 when a narrower CIDR is possible.
  • Opening administrative ports to the internet.
  • Forgetting IPv6 rules while securing only IPv4.
  • Creating rules without owners or review dates.
  • Allowing broad east-west traffic because troubleshooting was rushed.
  • Not testing whether a deny rule breaks required business flows.

Why secure coding practices matter here

Infrastructure-as-code and policy-as-code are software. A typo, broad CIDR, missing condition, or wrong rule order can expose production systems. Treat network policy changes like code: review them, test them, log them, and keep evidence.

10. Security Perspective

Common attacks and failure modes

  • Lateral movement: attacker uses one compromised system to reach others.
  • Direct database exposure: sensitive data store reachable from too many sources.
  • Open administrative ports: SSH, RDP, database, or management consoles exposed broadly.
  • Rule sprawl: old exceptions remain after projects end.
  • Shadow IT: teams deploy services outside approved zones.
  • Overtrusted VPN: once connected, users receive broad internal access.
  • Flat cloud networks: workloads can reach each other unnecessarily.
  • Misplaced WAF confidence: teams rely on filtering instead of fixing application flaws.

Common vulnerabilities

  • Any-to-any rules.
  • Management services exposed to untrusted networks.
  • No separation between user devices and servers.
  • No segmentation between application and database tiers.
  • Missing logs for denied or sensitive allowed flows.
  • Stale temporary exceptions.
  • Inconsistent rules across IPv4 and IPv6.
  • Cloud security groups copied without review.

Defensive strategies

  • Use deny by default.
  • Allow only business-justified flows.
  • Separate public, application, database, admin, and monitoring zones.
  • Require stronger controls for privileged access.
  • Log high-risk allowed flows and denied attempts where appropriate.
  • Review rules periodically.
  • Use identity-aware access for sensitive applications.
  • Apply segmentation in cloud, container, and on-premises environments.

Detection methods

  • Firewall deny logs.
  • NetFlow/VPC flow logs.
  • DNS logs.
  • Proxy logs.
  • Endpoint connection telemetry.
  • Configuration-drift alerts.
  • Route and security group change events.
  • Unexpected east-west connections.

Mitigation and best practices

Mitigate by narrowing rules, removing stale exceptions, enforcing privileged access paths, logging high-risk flows, testing changes before deployment, and maintaining an architecture diagram that matches reality.

11. Industry Perspective

  • Banks: segment payment systems, customer databases, trading systems, user networks, and administrative access.
  • Governments: separate public services, internal systems, classified networks, citizen data, and privileged administration.
  • Cloud providers: expose virtual network controls such as security groups, network ACLs, private endpoints, service perimeters, and identity-aware access.
  • Healthcare: isolate electronic health record systems, medical devices, guest networks, and vendor access.
  • Defense: use strict zoning, cross-domain controls, mission system isolation, and strong monitoring.
  • Universities: separate students, faculty, administration, research labs, guest Wi-Fi, and public services.
  • Large enterprises: use layered network architecture, cloud segmentation, privileged access workstations, zero trust access, and continuous policy review.

12. Research Perspective

Current research and engineering work includes:

  • Policy verification: proving that firewall and cloud policies match intended security properties.
  • Intent-based networking: describing desired outcomes and generating device-specific rules safely.
  • Microsegmentation at scale: learning legitimate service flows without blocking production systems.
  • Zero Trust metrics: measuring maturity, effectiveness, and residual risk without relying on marketing claims.
  • Software-defined perimeters: hiding services until identity and policy checks succeed.
  • Service mesh security: enforcing workload identity, mutual TLS, and fine-grained service policies.
  • Cloud network verification: detecting public exposure, unintended paths, and policy drift.
  • Usable policy management: reducing human error in complex rule bases.

Open problems include rule explosion, false confidence from diagrams that do not match deployed reality, handling unmanaged devices, secure access for third parties, policy enforcement across hybrid cloud, and validating Zero Trust claims with evidence.

13. Case Studies

Target breach and third-party access

Public reporting on the Target breach has often highlighted that third-party access and internal movement played major roles. The lesson for network security is not “one firewall would have fixed everything.” The defensible lesson is that third-party access, monitoring, segmentation, and least privilege must be designed together.

WannaCry and flat networks

WannaCry spread rapidly in many organizations through vulnerable systems and reachable network paths. Segmentation, patching, blocking unnecessary SMB exposure, and monitoring could reduce spread and impact.

Cloud security group exposure

Many cloud incidents involve administrative ports, databases, storage services, or internal dashboards exposed too broadly. Cloud firewalls are powerful, but mistakes can be deployed quickly and globally if review is weak.

VPN overtrust

In many traditional designs, VPN access places users “inside” the network. If that access is broad, a stolen account or compromised device can reach internal services unnecessarily. Zero Trust access patterns attempt to limit access to specific resources under policy.

14. Common Misconceptions

  • “A firewall makes a network secure.” Wrong. A firewall enforces rules. Bad rules create bad security.
  • “Inside the network means trusted.” Wrong. Internal devices and accounts can be compromised.
  • “Zero Trust means no one is trusted.” Misleading. It means access is explicitly evaluated and limited.
  • “Segmentation is only for large enterprises.” Wrong. Even small systems benefit from separating public, application, database, and admin paths.
  • “A WAF fixes vulnerable code.” Wrong. It may reduce some attacks, but code defects still need remediation.
  • “More firewall rules always mean better security.” Wrong. More rules can create complexity, conflict, and stale exposure.
  • “VPN equals Zero Trust.” Wrong. A VPN can be part of access, but Zero Trust requires policy, identity, device, data, and resource-level decisions.
  • “Cloud security groups are automatically safe.” Wrong. They are only as safe as their configuration and review process.

15. Summary

Firewalls control network conversations. Segmentation limits unnecessary reachability and reduces blast radius. Zero Trust removes implicit trust from network location and strengthens access decisions using identity, device, resource, data, and context. The most important practical habit is to document allowed flows, deny everything else by default, log meaningful decisions, review exceptions, and validate that deployed controls match the intended architecture.
