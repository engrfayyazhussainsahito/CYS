Lesson 7.4: Traffic Analysis, IDS/IPS, and Detection

This lesson teaches how defenders observe network communication, convert traffic into evidence, and use intrusion detection and prevention systems to find suspicious behavior without pretending that any tool is perfect.

Estimated reading time430–650 minutes
Difficulty levelBeginner → MS foundation
Required backgroundLessons 7.1–7.3 recommended; packet, port, DNS, firewall, and segmentation concepts are reviewed
Practice time6–10 hours

Week 7 Overview

Week 7 studies network security. Earlier lessons explained packets, DNS, routing, firewalls, segmentation, and Zero Trust. This lesson adds the defender’s observation layer: how to watch network activity, reduce raw traffic into useful evidence, detect suspicious behavior, and decide what to do next.

Traffic analysis is not the same as spying on people. In a professional environment it must be authorized, governed, minimized, logged, and aligned with privacy and legal requirements.

1. Introduction

Traffic analysis means studying network communication patterns and metadata to understand what systems are doing. A defender may ask: Who talked to whom? Which protocol was used? Was the amount of data normal? Did the domain name look suspicious? Did a server suddenly begin connecting to many internal systems?

An Intrusion Detection System, or IDS, observes activity and raises alerts when it sees suspicious patterns. An Intrusion Prevention System, or IPS, sits in a position where it may block, drop, reset, or otherwise prevent traffic that violates policy.

Plain English: a network sensor is like a security camera for network behavior. It does not automatically understand every event. It records signals, applies rules or models, and gives analysts clues. The analyst still needs context.

This topic exists because prevention controls fail. A firewall may allow HTTPS because the business needs it. Malware can also use HTTPS. A DNS resolver may be required for normal browsing. Malware can also use DNS. Detection helps defenders identify abuse inside allowed communication.

Who uses it: Security Operations Center analysts, network engineers, incident responders, threat hunters, cloud security teams, malware analysts, university security teams, bank fraud teams, defense networks, healthcare security teams, and managed security providers.

Safety boundary: this lesson only uses local, fictional, or instructor-provided data. Do not capture traffic on networks you do not own or administer. Do not run packet captures, IDS sensors, or scanning tools against public or third-party networks without written authorization.

2. Learning Outcomes

  • Explain traffic analysis, IDS, IPS, network security monitoring, telemetry, events, and alerts from first principles.
  • Distinguish packets, flows, sessions, logs, metadata, signatures, rules, anomalies, indicators, and behaviors.
  • Explain the difference between IDS and IPS, including placement, visibility, and operational risk.
  • Read simple packet, flow, DNS, HTTP, and TLS metadata examples.
  • Explain how signature-based, anomaly-based, protocol-based, reputation-based, and behavior-based detections work.
  • Describe false positives, false negatives, detection tuning, severity, confidence, and alert fatigue.
  • Write safe toy detection logic for local log data.
  • Explain how tools such as Zeek, Suricata, and Snort-style rule engines fit into a monitoring architecture.
  • Design a safe detection plan for a fictional segmented network.
  • Discuss privacy, encryption visibility limits, research challenges, and industry use.

3. Prerequisites

This lesson reintroduces every necessary prerequisite before using it:

  • Packet: a small unit of network data. Packets have headers and payloads.
  • Header: metadata needed to deliver or interpret a packet, such as source address, destination address, protocol, or port.
  • Payload: the data being carried. In encrypted traffic, the payload may not be readable by the sensor.
  • Flow: a summary of a network conversation, commonly represented by source IP, destination IP, source port, destination port, protocol, byte count, packet count, and timestamps.
  • DNS: the naming system that maps names such as example.edu to IP addresses.
  • HTTP/HTTPS: common web protocols. HTTPS is HTTP protected by TLS.
  • Log: a record of an event. Logs may come from firewalls, servers, endpoints, cloud platforms, DNS resolvers, proxies, IDS sensors, and applications.
  • Firewall: a control that permits, denies, and logs traffic according to policy.
  • Segment: a separated part of a network, such as student Wi-Fi, DMZ, application zone, or database zone.

4. Core Concepts

4.1 From raw traffic to security evidence

Raw packets are detailed but heavy. Flow records are smaller but less detailed. DNS logs show names. HTTP logs show web requests when visible. TLS metadata may show certificate or handshake properties, but encrypted content is not visible without endpoint logging or approved decryption architecture.

Data typeWhat it showsStrengthLimitation

Flow logWho talked to whom, when, and how muchScalable behavioral viewUsually no content

4.2 IDS versus IPS

An IDS is usually passive: it observes and alerts. An IPS is usually inline: traffic passes through it, and it may block or modify traffic. Passive monitoring is safer operationally because it cannot accidentally block business traffic. Inline prevention can stop attacks earlier, but bad rules may cause outages.

FeatureIDSIPS
PositionPassive sensor, tap, span port, cloud mirror, host agentInline network path, gateway, host enforcement point
Main actionAlert, log, enrichBlock, drop, reset, quarantine, alert
Operational riskLower risk of business disruptionHigher risk if rules are wrong or device fails
Best useVisibility, investigation, threat huntingHigh-confidence blocking, known bad patterns, policy enforcement

4.3 Sensor placement

Where a sensor is placed determines what it can see. A sensor at the internet edge sees north-south traffic. A sensor inside the data center sees east-west movement. A cloud flow log sees virtual network conversations. A DNS sensor sees name resolution. Endpoint telemetry sees process-level context that network sensors may miss.

4.4 Detection logic

A detection is a structured way to turn observations into a security signal. It has input data, conditions, context, severity, confidence, and a response path.

  • Signature-based detection: matches known byte patterns, protocol fields, strings, or rule conditions.
  • Anomaly-based detection: alerts when behavior deviates from an expected baseline.
  • Protocol-based detection: checks whether traffic violates protocol expectations.
  • Reputation-based detection: uses known bad domains, IPs, hashes, or certificates.
  • Behavior-based detection: looks for meaningful patterns, such as a workstation connecting to many internal servers.

4.5 Events, alerts, incidents, and cases

An event is a record that something happened. An alert is an event that detection logic marked as suspicious. An incident is a confirmed or reasonably suspected security violation requiring response. A case is a collection of evidence, notes, timelines, and actions about an investigation.

4.6 False positives and false negatives

A false positive is an alert that looks suspicious but is not actually malicious. A false negative is malicious activity that was missed. Detection engineering tries to reduce both, but there is usually a trade-off.

Accuracy rule: a detection tool does not prove an incident by itself. It produces evidence that must be validated with context such as asset importance, user identity, endpoint activity, firewall logs, DNS data, and change records.

4.7 Alert severity versus confidence

Severity describes potential impact. Confidence describes how likely the alert is to be true. A high-severity, low-confidence alert may still need triage. A low-severity, high-confidence alert may be automated or batched.

4.8 Encryption and visibility

Encryption protects confidentiality, but it also limits network-content inspection. This is not a flaw in encryption. Defenders compensate with endpoint telemetry, DNS logs, TLS metadata, proxy logs, application logs, identity logs, and behavior analytics. Decryption proxies require explicit governance, privacy controls, exception handling, and legal review.

4.9 Detection tuning

Tuning means improving detection quality. It may include suppressing known benign behavior, adding context, adjusting thresholds, narrowing scope, documenting assumptions, and testing against known-good and known-bad examples. Over-tuning can hide attacks; under-tuning creates alert fatigue.

4.10 Evidence chain

Detection evidence must be reproducible. A good alert record includes timestamp, source, destination, protocol, rule name, rule version, sensor, raw supporting event, enrichment data, analyst notes, and final disposition.

7. Mathematical Foundations

7.1 Rate calculations

Traffic analysis often starts with rates. If a host sends 600 megabytes in 10 minutes, the average rate is:

rate = data / time
rate = 600 MB / 10 min = 60 MB/min

If that same host normally sends 3 MB/min, the activity is 20 times the normal baseline. That does not prove compromise, but it justifies investigation.

7.2 Packets per second

If a sensor records 120,000 packets over 60 seconds:

packets_per_second = 120000 / 60 = 2000 packets/second

This matters for capacity planning. If a sensor cannot process traffic at line rate, it may drop packets and miss evidence.

7.3 Precision and recall

Detection quality can be described using common measurement terms:

  • True Positive (TP): malicious activity correctly alerted.
  • False Positive (FP): benign activity incorrectly alerted.
  • False Negative (FN): malicious activity missed.
  • True Negative (TN): benign activity correctly ignored.
precision = TP / (TP + FP)
recall    = TP / (TP + FN)

Worked example: a detection produced 40 true positives and 60 false positives. It missed 10 malicious events.

precision = 40 / (40 + 60) = 0.40 = 40%
recall    = 40 / (40 + 10) = 0.80 = 80%

This detection catches many malicious events, but it is noisy. Analysts may suffer alert fatigue unless the rule is enriched, scoped, or tuned.

7.4 Baseline and standard deviation intuition

A baseline is a model of normal behavior. If a server usually makes 5 to 10 DNS queries per minute, then 1,000 queries per minute is unusual. A simple anomaly threshold might use a mean and standard deviation:

threshold = mean + 3 * standard_deviation

This is only a starting point. Real networks have business cycles, backups, patch windows, labs, holidays, and cloud autoscaling. A threshold without context produces bad alerts.

7.5 Risk scoring for alerts

A practical triage score can combine severity, confidence, and asset value:

alert_priority = severity × confidence × asset_value

Example: severity 4, confidence 0.8, asset value 5:

4 × 0.8 × 5 = 16

This is not a universal formula. It is a teaching model showing that the same alert matters more on a domain controller, database, payment system, or hospital device than on a disposable lab machine.

8. Practical Examples

8.1 Linux examples

These commands inspect your own local machine. They do not target other systems.

# Show listening TCP and UDP sockets on your own machine
ss -tulpen

# Show recent local system logs on systems using systemd
journalctl --since "1 hour ago" --no-pager

# Capture only loopback traffic for local lab testing, then stop with Ctrl+C
sudo tcpdump -i lo -n

Common mistake: running packet captures on shared, university, workplace, or public networks without authorization. Local loopback and instructor-provided packet captures are safer for learning.

8.2 Windows examples

# Show active connections on your own Windows machine

# Show DNS client cache entries

# Show recent firewall log path setting

8.3 Python example: analyze fictional flow logs

from collections import Counter

# Fictional, safe flow records. These are not real victim systems.
flows = [
    {"src": "student-laptop", "dst": "portal", "port": 443, "bytes": 12000},
    {"src": "student-laptop", "dst": "portal", "port": 443, "bytes": 9000},
    {"src": "lab-pc-01", "dst": "db-01", "port": 5432, "bytes": 300},
    {"src": "lab-pc-01", "dst": "db-02", "port": 5432, "bytes": 350},
    {"src": "lab-pc-01", "dst": "db-03", "port": 5432, "bytes": 400},
]

# Count unique destinations per source host.
destinations_by_source = {}
for flow in flows:
    src = flow["src"]
    destinations_by_source.setdefault(src, set()).add(flow["dst"])

# Simple teaching rule: flag sources talking to three or more destinations.
for src, destinations in destinations_by_source.items():
    if len(destinations) >= 3:
        print(f"Review {src}: contacted {len(destinations)} destinations: {sorted(destinations)}")

This is not a production detector. It teaches the idea of counting behavior. A real detector needs asset context, expected behavior, time windows, exclusions, and testing.

8.4 Network example

A web server normally receives inbound HTTPS from users and sends database queries to one database. If it suddenly makes outbound connections to many unknown IP addresses, defenders may investigate possible compromise, update process, backup job, misconfiguration, or third-party integration.

8.5 Web example

A web application log might show many failed login attempts from one IP address or many usernames from one source. Network telemetry alone may show repeated HTTPS connections; application logs add user and result context.

8.6 Cloud example

Cloud environments often provide virtual private cloud flow logs, load balancer logs, DNS query logs, WAF logs, and identity logs. A useful cloud detection might combine: “new public exposure” + “unusual inbound traffic” + “admin API call from new location.”

8.7 Safe IDS-style rule example

The following toy rule is for understanding syntax. It alerts on a harmless string in HTTP traffic inside a lab. Do not deploy untested rules to production.

alert http any any -> any any (msg:"LOCAL LAB harmless test string observed"; content:"lab-test-marker"; sid:7000001; rev:1;)

Line-by-line intuition:

  • alert: generate an alert instead of silently passing traffic.
  • http: inspect HTTP traffic.
  • any any -> any any: source and destination are broad for teaching only; production rules should be scoped.
  • msg: human-readable alert text.
  • content: harmless lab string to match.
  • sid: local signature identifier.
  • rev: rule revision number.

9. Code Examples

9.1 Safe local log parser

"""
Safe teaching script: parse fictional firewall logs and find denied connections.
It reads a local file you create yourself. It does not scan or connect to anything.
"""
from pathlib import Path

log_file = Path("fictional_firewall.log")

sample = """2026-06-29T09:00:01Z ALLOW src=student-laptop dst=portal proto=tcp dport=443 bytes=12000
2026-06-29T09:01:03Z DENY src=student-laptop dst=database proto=tcp dport=5432 bytes=0
2026-06-29T09:02:44Z DENY src=guest-wifi dst=admin-server proto=tcp dport=22 bytes=0
2026-06-29T09:03:20Z ALLOW src=app-server dst=database proto=tcp dport=5432 bytes=8300
"""

# Create a local sample file if it does not exist.
if not log_file.exists():
    log_file.write_text(sample, encoding="utf-8")

denied = []
for line in log_file.read_text(encoding="utf-8").splitlines():
    parts = line.split()
    if len(parts) < 2:
        continue
    timestamp = parts[0]
    action = parts[1]
    if action == "DENY":
        fields = dict(item.split("=", 1) for item in parts[2:] if "=" in item)
        denied.append((timestamp, fields))

for timestamp, fields in denied:
    print(f"Denied at {timestamp}: {fields.get('src')} -> {fields.get('dst')} port {fields.get('dport')}")

9.2 Explanation of the code

  • Path gives a safe way to work with local file paths.
  • The sample log is fictional and local.
  • The script creates the file only if it does not already exist.
  • Each line is split into fields.
  • Only lines with DENY are selected.
  • Key-value fields such as src=student-laptop are parsed into a dictionary.
  • The output is a short human-readable summary.

9.3 Common mistakes

  • Parsing logs without validating format.
  • Assuming every denied event is malicious.
  • Ignoring timestamps and time zones.
  • Storing sensitive logs in public repositories.
  • Writing detection scripts that connect to real networks without permission.
  • Alerting on broad patterns without asset context.

9.4 Secure coding practices

Detection code can create security risk. Logs may contain secrets, personal data, IP addresses, usernames, and internal hostnames. Good scripts minimize output, avoid exposing logs, handle errors safely, and document assumptions.

10. Security Perspective

10.1 Common attacks observed in traffic

  • Reconnaissance: unusual connection attempts, repeated DNS queries, many failed connection attempts, or broad service discovery inside an authorized environment.
  • Credential attacks: repeated login failures, password spraying patterns, or authentication attempts against many accounts.
  • Command and control: periodic outbound connections, unusual domains, strange user agents, or rare external destinations.
  • Data exfiltration: unusual outbound volume, long sessions, new destinations, or abnormal upload/download ratios.
  • Lateral movement: a workstation connecting to many internal servers or administrative ports.
  • Denial of service: traffic volume or packet rates exceeding normal capacity.

10.2 Common vulnerabilities and weaknesses

  • No network visibility in internal segments.
  • Unmonitored DNS resolvers.
  • Flat networks with no lateral movement detection.
  • IDS sensors overloaded or dropping packets.
  • Rules deployed but never tuned.
  • Alerts without asset ownership or response process.
  • Privacy rules ignored during packet capture.

10.3 Defensive strategies

  • Collect flow, DNS, firewall, proxy, endpoint, cloud, and identity logs.
  • Place sensors near important trust boundaries.
  • Use deny-by-default segmentation and monitor denied traffic.
  • Create detections from threat models, incident history, and known attacker behaviors.
  • Tune rules using real benign baselines and safe test data.
  • Preserve evidence with timestamps, source, rule version, and analyst notes.
  • Document false positives and rule changes.

10.5 Mitigation and response

Detection by itself does not mitigate. Response actions may include confirming asset owner, checking endpoint telemetry, isolating a host, blocking a destination, resetting credentials, tuning firewall rules, preserving evidence, and opening an incident case.

10.6 Best practices

  • Use detection as part of a layered defense, not a replacement for secure design.
  • Start with high-value use cases rather than hundreds of noisy rules.
  • Assign owners to detections.
  • Review detections after incidents and environment changes.
  • Measure alert quality.
  • Protect logs from tampering and unauthorized access.
  • Respect privacy and legal boundaries.

12. Research Perspective

Network detection research has several hard problems:

  • Encrypted traffic analysis: detecting malicious behavior without breaking confidentiality.
  • Adversarial machine learning: attackers may manipulate features to avoid models.
  • Concept drift: normal network behavior changes as organizations adopt cloud, remote work, and new applications.
  • Alert prioritization: ranking signals so analysts handle the most important work first.
  • Host-network correlation: joining network flows with process, user, container, and cloud identity data.
  • High-speed monitoring: capturing and analyzing traffic at 10G, 40G, 100G, and cloud scale without unacceptable packet loss.
  • Privacy-preserving monitoring: balancing legitimate defense with user privacy and data minimization.
  • Explainable detection: helping analysts understand why a model produced an alert.

Notable directions include Zeek-based network security monitoring, Suricata/Snort-style rule systems, behavioral analytics, graph-based detection, encrypted traffic classification, and combined endpoint/network telemetry.

13. Case Studies

13.1 Target breach and network monitoring gaps

Public reporting on the Target breach described attacker movement from a third-party access path toward payment systems. The lesson for this topic is that segmentation, monitoring, and alert handling must work together. A sensor that creates an alert is not enough if the organization does not investigate and respond effectively.

13.2 SolarWinds and trusted traffic

The SolarWinds compromise showed that attackers may operate through trusted software and legitimate-looking network paths. Network detection must therefore look beyond obvious blocking rules and include unusual destinations, identity context, software update behavior, and endpoint evidence.

13.3 DDoS traffic floods

Distributed denial-of-service activity can be visible as abnormal traffic volume, protocol abuse, or service exhaustion. Detection helps trigger rate limiting, scrubbing, upstream filtering, or incident response.

13.4 DNS as a detection surface

Malware often needs to locate command-and-control infrastructure. DNS logs can help defenders find suspicious domains, unusual query volume, rare domains, algorithmically generated names, or unexpected resolver use. DNS evidence must still be validated; unusual does not always mean malicious.

14. Common Misconceptions

“IDS alerts prove an attack.”
No. Alerts are evidence that require validation.
“IPS is always better than IDS.”
No. IPS can block attacks, but it can also break production traffic if rules or placement are wrong.
“Encrypted traffic cannot be monitored at all.”
Encrypted payloads are protected, but metadata, DNS, endpoint, proxy, application, and identity logs can still support detection.
“More rules mean better security.”
More rules can create noise. Quality, context, testing, and ownership matter more.
“False positives are harmless.”
No. False positives consume analyst time and can cause real alerts to be missed.
“A SIEM automatically detects attacks.”
A SIEM stores, searches, correlates, and alerts on telemetry. Detection engineering and response process are still required.
“Packet capture is always acceptable.”
No. Packet capture may collect sensitive personal, business, or regulated data. It requires authorization and governance.
“Detection replaces prevention.”
No. Detection complements prevention, hardening, segmentation, identity controls, and incident response.

15. Summary

Traffic analysis turns network communication into evidence. IDS tools observe and alert; IPS tools can block. Packet captures, flow logs, DNS logs, proxy logs, TLS metadata, firewall logs, and endpoint telemetry each show different parts of the truth. Effective detection requires good sensor placement, clear use cases, rule tuning, context enrichment, evidence handling, and a response process. Detection tools are useful, but they are not magic: they create signals that humans and workflows must validate.

Step 2: Run the safe analyzer

from collections import Counter, defaultdict
from pathlib import Path

path = Path("fictional_network.log")
lines = path.read_text(encoding="utf-8").splitlines()

denies = []
nxdomain_by_host = Counter()
destinations_by_host = defaultdict(set)

for line in lines:
    parts = line.split()
    timestamp = parts[0]
    event_type = parts[1]
    fields = dict(item.split("=", 1) for item in parts[2:] if "=" in item)

    if event_type == "FLOW":
        src = fields.get("src")
        dst = fields.get("dst")
        if src and dst:
            destinations_by_host[src].add(dst)
        if fields.get("action") == "DENY":
            denies.append((timestamp, fields))

    if event_type == "DNS" and fields.get("answer") == "NXDOMAIN":
        nxdomain_by_host[fields.get("src", "unknown")] += 1

print("Denied flows:")
for timestamp, fields in denies:
    print(f"- {timestamp} {fields.get('src')} -> {fields.get('dst')}:{fields.get('dport')}")

print("\nHosts with multiple destinations:")
for host, destinations in destinations_by_host.items():
    if len(destinations) >= 3:
        print(f"- {host}: {len(destinations)} destinations {sorted(destinations)}")

print("\nHosts with repeated NXDOMAIN responses:")
for host, count in nxdomain_by_host.items():
    if count >= 3:
        print(f"- {host}: {count} NXDOMAIN responses")

Step 3: Interpret cautiously

Your findings are not proof of compromise. Write hypotheses:

  • Denied database flows may indicate policy working, misconfigured software, or unauthorized access attempts.
  • Many destinations may indicate lateral movement, inventory software, patching, backups, or lab activity.
  • Repeated NXDOMAIN responses may indicate mistyped names, broken software, or suspicious domain generation behavior.

Step 4: Write a report

Include:

  • Scope and data source.
  • Detection logic used.
  • Findings with timestamps.
  • Possible benign explanations.
  • Recommended next evidence sources.
  • Decision: close, monitor, tune, or escalate.

Expected outcomes

  • A reproducible local analysis script.
  • A short detection report.
  • A list of false-positive considerations.
  • A recommendation for better monitoring placement.
