Lesson 6.2: Sessions, Cookies, and Browser Security

This lesson explains how browsers remember users, how cookies and sessions work, why browser security boundaries exist, and how defenders reduce session theft, cross-site request forgery, client-side data exposure, and unsafe browser behavior.

Estimated reading time320–450 minutes
Difficulty levelBeginner → MS foundation
Required backgroundLesson 6.1 recommended; no previous web security background required
Practice time5–8 hours

1. Introduction

HTTP is mostly stateless: one request does not automatically remember the previous request. Yet real web applications need memory. A bank must remember that the same customer logged in. A university portal must remember which student is using the browser. A shopping site must remember a cart. That memory is called state.

A session is a period of interaction between a client and an application. In many web applications, the browser stores a random session identifier in a cookie. The server uses that identifier to find server-side session data, such as the logged-in user, permissions, expiration time, and risk signals.

This topic exists because the browser is both useful and dangerous. It runs code from websites, stores data, automatically sends cookies, enforces origin boundaries, and connects user identity to remote services. A mistake in session handling or browser security can expose accounts even when passwords and TLS are strong.

Why it matters

  • A stolen session can act like a stolen login.
  • Weak cookie attributes can expose session identifiers to script, network, or cross-site request risks.
  • Browser storage is convenient but can become a credential exposure point.
  • Same-origin policy, CORS, CSP, and related controls are central to modern web security.
  • Defenders need to read browser behavior, HTTP headers, web logs, and session events as evidence.

Where it is used

Sessions, cookies, and browser security appear in banking portals, cloud consoles, healthcare systems, government services, email, learning platforms, enterprise dashboards, single sign-on, mobile-web backends, and software-as-a-service applications.

Who uses it

  • Web developers design login, logout, session renewal, and browser storage behavior.
  • Security engineers review cookies, headers, session lifecycle, CSRF protections, and browser-side controls.
  • SOC analysts investigate suspicious login, token, session, and web-request activity.
  • Cloud and platform engineers configure edge TLS, headers, WAFs, identity providers, and session-aware proxies.
  • Researchers study browser isolation, tracking, privacy, cross-origin data flows, and client-side integrity.
Safety boundary: all demonstrations in this lesson use localhost, fictional logs, and toy code. Do not test session handling, cookies, CSRF behavior, browser storage, or security headers on systems you do not own or have explicit written authorization to assess.

2. Learning Outcomes

  • Explain why HTTP needs added state for login-based applications.
  • Define session, session identifier, cookie, cookie jar, cookie attribute, browser storage, origin, same-site, same-origin, CSRF, XSS relationship to cookies, CSP, SRI, HSTS, and security headers.
  • Describe the lifecycle of a web session from login to logout and expiration.
  • Explain Set-Cookie, Cookie, Secure, HttpOnly, SameSite, Domain, Path, Max-Age, and Expires.
  • Distinguish server-side sessions from client-side tokens at a conceptual level.
  • Explain why cookies are automatically sent by the browser and why that creates CSRF risk.
  • Explain why HttpOnly reduces script access to cookies but does not stop the browser from sending cookies in requests.
  • Use safe local commands to inspect HTTP headers, cookies, and browser storage.
  • Write and explain a safe Python localhost demonstration that sets educational cookies.
  • Recommend defensive session and browser-security controls for a fictional application.

3. Prerequisites

HTTP request and response

A browser sends an HTTP request to a server. The server sends an HTTP response. Headers carry metadata. Bodies carry optional content such as HTML or JSON.

HTTPS and TLS

HTTPS means HTTP protected by Transport Layer Security, or TLS. TLS protects the network connection from casual reading and tampering. It does not fix application logic mistakes.

Authentication and authorization

Authentication asks “who are you?” Authorization asks “what are you allowed to do?” A session usually records authentication state, but every sensitive action still needs authorization checks.

Origin

An origin is the combination of scheme, host, and port. For example, https://portal.example.edu and https://api.example.edu are different origins because their hosts differ.

JavaScript

JavaScript is code that can run in the browser. Browser security controls often decide what JavaScript can read, send, load, or execute.

4. Core Concepts

4.1 State

State means remembered information. A stateless protocol treats each request independently. A stateful application connects many requests into one continuing interaction.

Analogy: a receptionist who forgets everyone after each sentence is stateless. A receptionist who keeps a visitor badge number linked to your appointment is maintaining state.

4.2 Session

A session is application memory about one continuing interaction. It may include user ID, login time, last activity, multi-factor status, selected language, shopping cart, CSRF token, and risk flags.

4.3 Session identifier

A session identifier, often called a session ID, is a random value that points to a session. It should be long, unpredictable, and transmitted only through protected channels.

Important: a session ID is like a temporary badge. Whoever presents it may be treated as the session owner unless the server adds extra checks.

4.4 Cookie

A cookie is a small name-value item stored by the browser and sent back to matching sites. Servers set cookies using the Set-Cookie response header. Browsers return matching cookies using the Cookie request header.

HTTP/1.1 200 OK
Set-Cookie: session_id=abc123; Path=/; Secure; HttpOnly; SameSite=Lax
GET /account HTTP/1.1
Host: portal.example.edu
Cookie: session_id=abc123

4.5 Cookie jar

The cookie jar is the browser's internal storage for cookies. The browser decides which cookies match a request based on domain, path, security, expiration, and same-site rules.

4.6 Cookie attributes

AttributePlain-English purposeSecurity meaning
SecureSend cookie only over HTTPS, except special localhost behavior in modern browsers.Reduces network exposure over plaintext HTTP.
HttpOnlyDo not expose the cookie to JavaScript through document.cookie.Reduces cookie theft through script access, but does not stop all XSS impact.
SameSiteControl when the cookie is sent with cross-site requests.Helps reduce CSRF risk.
PathLimit cookie to URL paths.Reduces accidental sending, but is not a strong isolation boundary.
DomainControl which hostnames receive the cookie.Too-broad domains can expose cookies to subdomain risk.
Max-Age / ExpiresSet cookie lifetime.Controls persistence and cleanup.

4.7 Host-only cookies

If a cookie does not set a Domain attribute, it is usually host-only. Host-only cookies reduce accidental sharing across subdomains.

4.8 Cookie prefixes

Cookie name prefixes such as __Host- and __Secure- let browsers enforce additional rules when supported. __Host- is useful for strong host-bound session cookies because it requires Secure, Path=/, and no Domain attribute.

4.9 Same-site vs same-origin

Same-origin compares scheme, host, and port. Same-site is broader and is based on registrable site plus scheme. Cookie SameSite behavior is about site context, not the full origin definition.

4.10 SameSite values

ValueMeaningTypical use
StrictSend cookie only in same-site contexts.Highly sensitive actions, where cross-site navigation convenience is less important.
LaxSend cookie in many same-site contexts and limited top-level cross-site navigations.Common default for ordinary sessions.
NoneAllow cross-site sending; must be paired with Secure in modern browsers.Embedded third-party use cases that truly need cross-site cookies.

4.11 Session lifecycle

  1. User arrives unauthenticated.
  2. User authenticates.
  3. Server creates or rotates a session ID.
  4. Server sends a cookie containing the session ID.
  5. Browser sends the cookie on later matching requests.
  6. Server checks session validity, expiration, and authorization.
  7. Session may be renewed, reauthenticated, revoked, or expired.
  8. Logout invalidates the server-side session and clears the browser cookie where possible.

4.12 Session fixation

Session fixation occurs when an application accepts a session ID chosen before login and does not rotate it after authentication. Defensive design rotates the session ID on login, privilege change, and other sensitive transitions.

4.13 Session expiration

Sessions need expiration. Idle timeout ends a session after inactivity. Absolute timeout ends a session after a maximum total lifetime. Sensitive applications may require step-up authentication for high-risk actions.

4.14 Logout

Logout should invalidate server-side session state. Merely deleting a browser cookie is not enough if the server would still accept the old session ID.

4.15 Server-side session vs client-side token

In a server-side session, the browser holds a reference and the server holds the session data. In a self-contained client-side token, the browser holds a signed token containing claims. Tokens can be useful, but revocation, storage, audience, lifetime, and leakage risks must be designed carefully.

4.16 Browser storage

Browsers provide storage mechanisms such as cookies, localStorage, sessionStorage, IndexedDB, cache storage, and memory variables. These are not interchangeable. Sensitive session credentials usually should not be placed in JavaScript-readable persistent storage.

4.17 Cross-site scripting and sessions

Cross-site scripting, or XSS, means untrusted script runs in a page's security context. HttpOnly can stop JavaScript from directly reading a session cookie, but XSS can still perform actions as the user while the browser automatically includes cookies.

4.18 Cross-site request forgery

Cross-site request forgery, or CSRF, tricks a browser into sending a request to a site where the user is already logged in. Because browsers automatically include matching cookies, the server may mistake the request as intentional unless it verifies intent.

4.19 CSRF defenses

  • Use anti-CSRF tokens bound to the session and verified by the server.
  • Use SameSite cookies where compatible with the application flow.
  • Validate Origin or Referer headers for sensitive actions where appropriate.
  • Require reauthentication or user interaction for highly sensitive operations.
  • Use safe HTTP method semantics: GET should not perform state-changing actions.

4.20 Same-origin policy

The same-origin policy is a browser boundary that restricts how documents and scripts from one origin can interact with data from another origin. It is one of the browser's core security mechanisms.

4.21 CORS

Cross-Origin Resource Sharing, or CORS, lets a server tell browsers which other origins may read responses through browser JavaScript. CORS is not a replacement for authentication or authorization.

4.22 Content Security Policy

Content Security Policy, or CSP, is a browser-enforced policy that restricts where scripts, images, styles, frames, and other resources can load from. CSP is defense in depth; it reduces impact but does not remove the need to fix XSS bugs.

4.23 Subresource Integrity

Subresource Integrity, or SRI, allows a page to include a cryptographic hash for an externally loaded script or stylesheet. The browser checks that the downloaded file matches the expected hash before using it.

4.24 HSTS

HTTP Strict Transport Security, or HSTS, tells browsers to use HTTPS for future requests to a site. It helps prevent accidental downgrade to plaintext HTTP after the first secure visit or through preload mechanisms.

4.25 Referrer Policy and Permissions Policy

Referrer Policy controls how much URL information is sent in the Referer header. Permissions Policy controls access to selected browser features such as camera, microphone, geolocation, and other powerful APIs.

4.26 Clickjacking and framing controls

Clickjacking tricks users into clicking hidden or disguised interface elements. Defenses include CSP frame-ancestors and, for older compatibility, X-Frame-Options.

4.27 Mixed content

Mixed content occurs when an HTTPS page loads resources over HTTP. This can weaken the protection of the secure page because some content travels without TLS protection.

4.28 Session risk signals

Applications may evaluate risk signals such as impossible travel, device changes, IP reputation, unusual user agent, sudden privilege changes, and high-value actions. These signals should support, not replace, strong session design.

4.29 Logging and privacy

Web logs should help investigate events without leaking secrets. Never log full session IDs, access tokens, passwords, password reset tokens, or sensitive personal data unless there is a legally approved, minimized, protected need.

4.30 Beginner rule

A session is security-critical state. Treat session identifiers and tokens like temporary credentials. Protect creation, storage, transmission, rotation, expiration, revocation, logging, and monitoring.

7. Mathematical Foundations

7.1 Entropy for session identifiers

Entropy measures uncertainty. A session identifier with n random bits has 2^n possible values.

possibilities = 2^n

For a 128-bit random session ID:

2^128 ≈ 3.40 × 10^38 possible values

That is large enough that guessing a correctly generated 128-bit session ID is not practical with current ordinary computing assumptions.

7.2 Guessing probability

If an attacker makes q independent guesses against a space of N equally likely session IDs, the rough success probability is:

P(success) ≈ q / N, when q is much smaller than N

Example: if N = 2^128 and q = 10^12 guesses:

P ≈ 10^12 / 3.40×10^38
P ≈ 2.94×10^-27

The real system can still fail if the random generator is weak, IDs are logged, tokens are exposed to scripts, or sessions are not revoked.

7.3 Birthday collision intuition

If a system creates many random IDs, defenders also care about accidental collisions. A rough birthday-bound approximation is:

P(collision) ≈ k(k - 1) / (2N)

where k is the number of generated IDs and N is the number of possible IDs. Large random ID spaces reduce collision risk.

7.4 Timeout reasoning

Timeouts are risk trade-offs. Short timeouts reduce exposure after an unattended device or stolen cookie, but they may frustrate users. Long timeouts improve convenience but increase the window of misuse.

exposure window ≈ time until session expires or is revoked

High-risk systems usually combine idle timeout, absolute timeout, reauthentication for sensitive actions, and centralized revocation.

Browser: inspect cookies safely

  1. Open a local lesson demo at http://localhost:8000.
  2. Open Developer Tools.
  3. Find Application or Storage.
  4. Look for Cookies, Local Storage, and Session Storage.
  5. Record which items are persistent, which are script-readable, and which are sent automatically.

Network example: cookie in request headers

GET /account HTTP/1.1
Host: localhost:8000
Cookie: theme=dark; demo_session=localdemo

The browser sends matching cookies automatically. That automatic behavior is useful for sessions but is also why CSRF needs server-side intent checks.

Web example: safe cookie attributes

Set-Cookie: __Host-session=RANDOM_VALUE; Path=/; Secure; HttpOnly; SameSite=Lax; Max-Age=1800

This is a good educational pattern for a host-bound session cookie. In production, the value must be generated using a cryptographically secure random generator and stored or validated carefully.

Cloud example: session-aware edge architecture

A cloud-hosted application may terminate TLS at a load balancer, forward requests to an application service, use a managed identity provider for login, store session metadata in a managed cache, and send logs to a SIEM. Each boundary needs clear responsibility.

9. Code Examples

9.1 Safe localhost cookie demonstration in Python

This toy server is for local learning only. It creates a fake session cookie and shows how a server can clear it. It does not implement real authentication.

from http.server import BaseHTTPRequestHandler, HTTPServer
from secrets import token_urlsafe

HOST = "127.0.0.1"
PORT = 8000

class DemoHandler(BaseHTTPRequestHandler):
    def send_html(self, body, extra_headers=None):
        self.send_response(200)
        self.send_header("Content-Type", "text/html; charset=utf-8")
        if extra_headers:
            for name, value in extra_headers:
                self.send_header(name, value)
        self.end_headers()
        self.wfile.write(body.encode("utf-8"))

    def do_GET(self):
        if self.path == "/login-demo":
            session_id = token_urlsafe(32)
            cookie = (
                f"__Host-demo_session={session_id}; "
                "Path=/; HttpOnly; SameSite=Lax; Max-Age=1800"
            )
            self.send_html(
                "<h1>Demo session created</h1>"
                "<p>Open browser developer tools and inspect cookies.</p>",
                [("Set-Cookie", cookie)]
            )
        elif self.path == "/logout-demo":
            expired_cookie = "__Host-demo_session=; Path=/; Max-Age=0; HttpOnly; SameSite=Lax"
            self.send_html("<h1>Demo session cleared</h1>", [("Set-Cookie", expired_cookie)])
        else:
            self.send_html(
                "<h1>Local browser-security demo</h1>"
                "<p>Visit <a href='/login-demo'>/login-demo</a> to set a demo cookie.</p>"
                "<p>Visit <a href='/logout-demo'>/logout-demo</a> to clear it.</p>"
            )

if __name__ == "__main__":
    server = HTTPServer((HOST, PORT), DemoHandler)
    print(f"Serving local demo at http://{HOST}:{PORT}")
    server.serve_forever()

token_urlsafe(32)Generates a random URL-safe value.Use cryptographically strong randomness for session identifiers.

9.2 Safe JavaScript browser-storage example

<button id="save-theme">Save demo theme</button>
<script>
  const button = document.querySelector("#save-theme");
  button.addEventListener("click", () => {
    localStorage.setItem("demo_theme", "dark");
    alert("Saved a harmless preference, not a session token.");
  });
</script>

This example stores a harmless preference. Do not store session IDs, access tokens, password reset links, or sensitive personal data in JavaScript-readable storage unless a careful security design explicitly justifies it.

9.3 Common mistakes

  • Using predictable session IDs such as user IDs or timestamps.
  • Failing to rotate session IDs after login.
  • Thinking HttpOnly stops all XSS damage.
  • Putting long-lived access tokens in localStorage without understanding script exposure.
  • Logging full cookies or tokens.
  • Using CORS as if it were an authorization control.
  • Allowing state-changing actions through GET.

10. Security Perspective

Common attacks

  • Session hijacking: misuse of a stolen session identifier.
  • Session fixation: keeping a pre-login session ID after login.
  • CSRF: unwanted cross-site action using automatically sent cookies.
  • XSS-assisted session abuse: malicious script performs actions or steals non-HttpOnly data.
  • Token leakage: secrets exposed in URLs, logs, storage, browser extensions, or third-party scripts.
  • Clickjacking: deceptive framing causes unintended clicks.
  • Misconfigured CORS: sensitive browser-readable responses exposed to unintended origins.

Common vulnerabilities

  • Weak randomness for session IDs.
  • Missing Secure, HttpOnly, or appropriate SameSite.
  • Overbroad Domain settings.
  • No server-side logout invalidation.
  • No idle or absolute timeout.
  • State-changing GET endpoints.
  • Missing CSRF tokens on cookie-authenticated state-changing requests.
  • Verbose logging of tokens or cookies.
  • Weak CSP or unsafe inline script allowances.

Defensive strategies

  1. Generate session IDs using a cryptographically secure random generator.
  2. Rotate session IDs after login, privilege change, and account recovery.
  3. Use Secure, HttpOnly, and suitable SameSite.
  4. Prefer host-only session cookies for sensitive applications.
  5. Set short enough idle and absolute timeouts for the risk level.
  6. Invalidate server-side sessions on logout and credential reset.
  7. Use anti-CSRF tokens and origin checks for sensitive state-changing actions.
  8. Deploy CSP, SRI, HSTS, framing controls, and careful referrer policy.
  9. Log session events without logging the secret values themselves.
  10. Monitor unusual session behavior and provide user-visible session management.

Detection methods

  • Multiple locations using the same session in implausible timing.
  • Session use after logout or password reset.
  • Repeated CSRF-token validation failures.
  • Sudden user-agent or device change for an active session.
  • High-value actions soon after suspicious login or session creation.
  • Unusual CORS errors or blocked CSP reports.

Mitigation

Immediate mitigation after suspected session compromise may include revoking sessions, forcing reauthentication, rotating secrets, reviewing logs, disabling risky integrations, tightening cookie attributes, and deploying emergency detection rules.

Best practices

Use server-side session management for ordinary web sessions unless tokens are needed for a clear architectural reason. Keep session identifiers secret, short-lived, revocable, logged safely, and bound to server-side authorization decisions.

12. Research Perspective

Current research and engineering work around sessions and browser security includes:

  • Browser isolation: separating sites, processes, documents, and sensitive contexts more strongly.
  • Client-side integrity: improving assurance that JavaScript delivered to users has not changed unexpectedly.
  • Privacy-preserving state: reducing cross-site tracking while preserving legitimate login and embedded-service use cases.
  • Passkeys and phishing-resistant authentication: reducing dependence on passwords while preserving usable sessions.
  • Agentic browsers: studying whether AI-driven browser agents can bypass assumptions built into same-origin policy and user-intent checks.
  • Session anomaly detection: identifying compromised sessions using behavior, device, network, and timing signals without excessive false positives.
  • Formal verification of browser policies: modeling policies such as SOP, CORS, CSP, and sandboxing to find edge-case failures.

Open problems include balancing privacy with fraud detection, preventing token theft in complex single-page applications, making security headers easier to deploy correctly, and maintaining user experience while reducing session risk.

13. Case Studies

Case study 1: Session sidejacking over unencrypted networks

What happened: tools publicly demonstrated that unencrypted session cookies on shared Wi-Fi could be captured and reused. Root cause: sites protected login but did not consistently protect the whole session with HTTPS. Impact: users could lose account control even without password theft. Lesson: protect the entire session with HTTPS and Secure cookies, not only the login page.

Case study 2: CSRF against state-changing actions

What happened: many historical web applications allowed state-changing requests without proving user intent. Root cause: servers trusted the presence of a cookie as enough evidence. Impact: attackers could trick logged-in browsers into sending unintended requests. Lesson: cookie authentication needs CSRF defenses for state-changing operations.

Case study 3: Third-party script compromise and payment data theft

What happened: attackers have abused third-party or injected scripts on commerce pages to steal data entered in browsers. Root cause: client-side code had access to sensitive page context and was not sufficiently constrained or monitored. Impact: payment and personal data exposure. Lesson: browser security includes supply-chain thinking: minimize third-party script power, use CSP/SRI where suitable, monitor changes, and segment sensitive flows.

Case study 4: Overbroad subdomain cookie scope

What happened: organizations sometimes set cookies for a parent domain so many subdomains receive them. Root cause: convenience was prioritized over host isolation. Impact: a weaker or outsourced subdomain can increase cookie exposure risk. Lesson: use host-only or __Host- cookies for sensitive sessions when possible.

14. Common Misconceptions

MisconceptionWhy it is incorrect
“HTTPS means the session is secure.”HTTPS protects transport, but session logic, cookie attributes, storage, CSRF defenses, and authorization can still fail.
“A cookie is always dangerous.”Cookies are normal web state tools. Risk depends on contents, attributes, lifecycle, and server behavior.
“HttpOnly stops XSS.”HttpOnly blocks script access to the cookie, but XSS can still run actions in the user's browser context.
“SameSite fully prevents CSRF.”SameSite helps, but applications still need correct methods, CSRF tokens, origin checks, and sensitive-action protections.
“CORS protects APIs from attackers.”CORS controls browser read access. APIs still need authentication, authorization, validation, and rate limiting.
“localStorage is safer than cookies.”localStorage is readable by JavaScript on the origin, so XSS can expose stored values.
“Logout means the cookie disappeared.”Secure logout requires server-side invalidation; cookie clearing alone is incomplete.
“Long sessions are always bad.”Risk depends on sensitivity, device trust, monitoring, step-up authentication, and revocation. Some systems require balanced usability.

15. Summary

Sessions let stateless HTTP support login-based applications. Cookies are the browser mechanism often used to carry session identifiers. Browser security controls such as same-origin policy, CORS, CSP, SRI, HSTS, and cookie attributes reduce specific risks, but none of them replace secure server-side design. Strong session security requires unpredictable identifiers, secure transport, careful cookie attributes, session rotation, expiration, revocation, CSRF protections, safe storage, safe logging, and monitoring.

A browser mechanism that allows servers to declare which origins may read cross-origin responses.
Cross-site request forgery
An attack pattern where a browser is tricked into sending an unwanted request to a site where the user is already authenticated.
HttpOnly
A cookie attribute that prevents JavaScript access through document.cookie.
Idle timeout
A session expiration rule based on inactivity.
Same-origin policy
A browser security boundary restricting how one origin can interact with another origin's data.
SameSite
A cookie attribute controlling whether a cookie is sent in cross-site contexts.
Secure cookie
A cookie marked to be sent only over HTTPS, subject to browser-specific development exceptions.
Session
Application state representing a continuing interaction between a client and an application.
Session fixation
A weakness where a pre-login session identifier remains valid after authentication.
Session hijacking
Misuse of a stolen or exposed session identifier.
Session identifier
A random value used to refer to session state.
Subresource Integrity
A browser feature that checks loaded external resources against a declared cryptographic hash.

    AnswerBecause HTTP requests are mostly independent, but applications need to remember login and interaction state.

    AnswerCookie.
  4. What does HttpOnly reduce?
    AnswerJavaScript access to the cookie through document.cookie.

    AnswerThe cookie should be sent only over HTTPS, with browser-specific localhost development behavior.
  6. Which cookie attribute helps reduce CSRF risk?
    AnswerSameSite.
  7. Does CORS replace authorization?
    AnswerNo. CORS controls browser read behavior; server authorization is still required.
  8. What is a session ID like in the badge analogy?
    AnswerA temporary badge that points to server-side state.
  9. Why rotate a session ID after login?
    AnswerTo reduce session fixation risk.
  10. What is an idle timeout?
    AnswerExpiration after a period of inactivity.
  11. What is an absolute timeout?
    AnswerMaximum session lifetime regardless of activity.
  12. Why should full session IDs not be logged?
    AnswerLogs could become a source of session theft.
  13. What does CSP restrict?
    AnswerResources and actions the browser allows for a page, such as script sources.
  14. What does SRI verify?
    AnswerThat an external resource matches an expected cryptographic hash.
  15. Can XSS still be harmful with HttpOnly cookies?
    AnswerYes. The script may still perform actions in the user's browser context.

18. Practice Lab

Title

Safe Local Cookie and Browser-Security Review

Prerequisites

  • Python 3 installed.
  • A modern browser with Developer Tools.
  • Terminal or PowerShell.
  • No internet-facing target is required.

Objectives

  • Run a localhost-only server.
  • Inspect Set-Cookie and Cookie headers.
  • Observe which values appear in browser cookie storage.
  • Explain why HttpOnly, SameSite, and Secure matter.
  • Create a defensive session review checklist.

Steps

  1. Create a folder named session-lab.
  2. Copy the Python demo from Section 9.1 into demo_server.py.
  3. Run python3 demo_server.py on Linux/macOS or python demo_server.py on Windows.
  4. Open http://127.0.0.1:8000/login-demo.
  5. Use Developer Tools to inspect the cookie name, value visibility, expiration, path, and SameSite behavior.
  6. Use curl -i http://127.0.0.1:8000/login-demo or PowerShell's Invoke-WebRequest to inspect the response headers.
  7. Visit /logout-demo and verify that the cookie is cleared.
  8. Write a one-page review explaining which production changes would be required before using this pattern in a real system.

Expected outcomes

  • You can explain the difference between response-set cookies and request-sent cookies.
  • You can identify secure cookie attributes.
  • You can explain why local demos differ from production HTTPS deployments.
  • You can write a basic session-security checklist.

Safety rules

  • Use only localhost.
  • Do not copy real cookies from any account.
  • Do not test third-party sites.
  • Do not capture other users' traffic.
  • Do not store or share real session identifiers.
