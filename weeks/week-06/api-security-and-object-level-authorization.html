Week 6 · Web and API Security

Lesson 6.4: API Security and Object-Level Authorization

This lesson teaches APIs from first principles, then focuses on the most common API security failure: an authenticated user can access or modify an object that does not belong to them. You will learn REST-style design, JSON, tokens, scopes, object ownership, BOLA/IDOR, property-level authorization, schemas, rate limits, API gateways, and evidence-based API logging.

Difficulty levelBeginner → MS foundation
Required backgroundLessons 2.4, 6.1, 6.2, and 6.3 recommended; prerequisites are reintroduced

1. Introduction

An Application Programming Interface, usually shortened to API, is a controlled way for one piece of software to ask another piece of software for data or actions. A web page may call an API to load messages. A mobile banking app may call an API to show account balances. A cloud dashboard may call APIs to create servers, read logs, or change permissions.

The plain-English idea is simple: an API is a service desk for software. A client makes a request, the API checks the request, performs a controlled operation, and sends back a response. The hard security problem is deciding whether the caller is allowed to do exactly that action on exactly that object.

This lesson focuses on object-level authorization. An object is a specific record or resource: invoice 913, patient record 42, file abc123, repository project-7, order 5001, or message m-88. Object-level authorization asks: Can this authenticated subject perform this action on this specific object right now?

Why this topic exists

Traditional web pages often hid object identifiers inside server-rendered pages. Modern APIs expose many identifiers directly in URLs, JSON bodies, headers, or GraphQL variables. This makes APIs powerful and scalable, but it also creates a large attack surface if the backend trusts client-supplied IDs without checking ownership, role, relationship, tenant, purpose, and action.

Why it matters

  • An API can be perfectly authenticated and still dangerously unauthorized.
  • Changing an object ID in a request can expose another user's data if object-level checks are missing.
  • APIs are used by mobile apps, single-page applications, partners, cloud automation, microservices, and AI tools.
  • API failures can expose data at scale because APIs are designed for structured, automated access.
  • Object-level authorization belongs in server-side policy enforcement, not in hidden buttons or client-side checks.

Where APIs are used

APIs are used in banks, hospitals, universities, cloud platforms, identity providers, logistics systems, e-commerce, defense systems, research platforms, social networks, mobile apps, internal dashboards, and business-to-business integrations.

Who uses API security

  • Developers design endpoints, schemas, authorization checks, tests, and error handling.
  • Security engineers review API threat models, access control logic, tokens, logs, gateways, and abuse cases.
  • Platform engineers operate API gateways, service meshes, rate limits, observability, and secrets.
  • SOC analysts investigate token misuse, suspicious object access, enumeration, and abnormal API volumes.
  • Researchers study automated API testing, authorization inference, OpenAPI security metadata, GraphQL risks, and microservice policy enforcement.
Safety boundary: all examples in this lesson use fictional APIs and local toy code. Do not test object ID changes, token behavior, rate limits, or API authorization on real websites, apps, university systems, cloud tenants, or third-party APIs without explicit written authorization.

2. Learning Outcomes

  • Define API, endpoint, resource, object, route, method, JSON, request body, response body, status code, bearer token, scope, claim, tenant, object-level authorization, BOLA, IDOR, and object-property authorization.
  • Explain the difference between authentication, authorization, function-level authorization, object-level authorization, and property-level authorization.
  • Describe REST-style API structure using resources, HTTP methods, status codes, and JSON.
  • Explain why hiding a button in the browser does not enforce authorization.
  • Explain why guessing or changing object IDs must not give access to another user's object.
  • Write safe toy pseudocode for checking ownership before returning or modifying an object.
  • Distinguish horizontal privilege failure from vertical privilege failure.
  • Explain API risks including BOLA/IDOR, broken authentication, mass assignment, excessive data exposure, unrestricted resource consumption, SSRF, misconfiguration, inventory gaps, and unsafe third-party consumption.
  • Use schemas and allowlists to validate API input shape.
  • Explain rate limiting, quota, pagination limits, and response-size controls.
  • Identify useful API logs for detection while avoiding token and secret leakage.
  • Complete a safe local object-authorization lab and produce a professional remediation report.

3. Prerequisites

This lesson reintroduces the necessary background. You do not need to be a web developer before starting.

3.1 Client and server

A client asks for something. A server receives the request, decides what to do, and sends a response. A browser, mobile app, script, or another backend service can act as a client.

3.2 HTTP

HTTP is the request-response protocol used by the web. An HTTP request includes a method such as GET, POST, PUT, PATCH, or DELETE, a path such as /api/records/42, headers, and sometimes a body. The response includes a status code, headers, and sometimes a body.

3.3 JSON

JavaScript Object Notation, or JSON, is a common text format for structured data. It represents objects, arrays, strings, numbers, booleans, and null values. APIs often receive and return JSON.

3.4 Identity

A digital subject is the user, service, device, or workload making a request. The system must know which subject is acting before it can decide what the subject is allowed to do.

3.5 Authentication and authorization

Authentication answers, "Who or what is making this request?" Authorization answers, "Is that authenticated subject allowed to do this action on this resource?" Many API breaches happen because authentication exists but authorization is incomplete.

3.6 Database records

A database stores records. In an API, an object identifier may point to one record. The API must not assume that a user is allowed to access a record simply because the user knows or can guess its identifier.

3.7 Trust boundary

A trust boundary is a place where data crosses from a less trusted area to a more trusted area. Every API request crosses a trust boundary. The server must validate the request and enforce policy.

4. Core Concepts

4.1 What an API really does

An API accepts structured requests and returns structured responses. A human may never see the API directly, but software depends on it. A browser page can use JavaScript to call /api/courses. A mobile app can call /api/accounts/123/balance. A cloud tool can call /api/v1/instances.

4.2 Endpoint

An endpoint is a specific API address plus behavior. For example, GET /api/invoices/913 may return invoice 913, while DELETE /api/invoices/913 may delete it. Same path, different method, different security impact.

4.3 Resource and object

A resource is a type of thing the API exposes, such as invoices, messages, files, courses, devices, accounts, or tickets. An object is one specific item within that resource, such as invoice 913 or file abc123.

4.4 REST-style thinking

REST-style APIs often model resources with paths and use HTTP methods to express actions. It is not necessary to master REST theory here. The practical model is enough:

4.5 Status codes

API responses use status codes. 200 usually means success. 201 means created. 400 means bad request. 401 means unauthenticated. 403 means authenticated but not allowed. 404 means not found. 429 means too many requests. 500 means server error.

4.6 Request body and response body

The request body contains data sent to the API, often JSON. The response body contains data returned by the API. Both are security-sensitive because they can contain identifiers, fields, roles, tokens, personal data, or error details.

4.7 Authentication tokens

APIs often use tokens instead of browser sessions. A bearer token means whoever possesses the token can present it to access the API. This is convenient, but dangerous if tokens leak. Tokens should be protected like passwords.

4.8 Claims and scopes

A token may contain or reference claims, such as subject ID, issuer, audience, expiration time, tenant ID, or role. A scope is a permission-like string such as files:read. Scopes are useful, but a scope such as files:read does not automatically mean the subject can read every file. The API still needs object-level checks.

4.9 Authentication is not enough

If Alice logs in, the API knows the caller is Alice. That does not mean Alice can read Bob's invoice. Authentication identifies Alice. Authorization decides whether Alice may access the requested object.

4.10 Function-level authorization

Function-level authorization asks whether the subject can use a feature or endpoint at all. Example: only administrators can call POST /api/users to create users.

4.11 Object-level authorization

Object-level authorization asks whether the subject can access a specific object. Example: a student can read their own transcript, but not another student's transcript. This must be checked every time the API reads, updates, deletes, exports, shares, or acts on an object.

4.12 BOLA and IDOR

Broken Object Level Authorization, or BOLA, is an API flaw where object-level checks are missing or incorrect. Insecure Direct Object Reference, or IDOR, is a related older term often used when changing a direct identifier gives unauthorized access. Modern API guidance commonly uses BOLA because the issue is authorization, not merely identifier format.

4.13 Sequential IDs and UUIDs

Sequential numbers like 41, 42, and 43 are easy to guess. Random-looking identifiers such as UUIDs are harder to guess, but they are not authorization. If a user obtains a UUID through logs, browser history, email, sharing, analytics, or another endpoint, the API must still check permission.

4.14 Horizontal and vertical failures

A horizontal authorization failure happens when one normal user accesses another normal user's object. A vertical authorization failure happens when a lower-privileged user performs a higher-privileged action. APIs need defenses against both.

4.15 Tenant isolation

In multi-tenant systems, one application serves multiple organizations. A tenant is one customer, institution, or organization. Object authorization must often include tenant checks: the object must belong to the caller's tenant, and the caller must have rights inside that tenant.

4.16 Relationship-based access

Access is not always simple ownership. A professor may read assignments for their course. A doctor may access records for assigned patients. A support engineer may access limited data during an approved support case. These are relationship-based policies and require careful design.

4.17 Policy decision and policy enforcement

A Policy Decision Point, or PDP, decides whether access should be allowed. A Policy Enforcement Point, or PEP, enforces the decision at runtime. In a small app, both may be inside the same function. In a large platform, they may be separate services, libraries, gateways, or policy engines.

4.18 Object-property authorization

Sometimes a subject may access the object but not every field. A patient portal may show appointment time but not internal clinical notes. A student API may show course grade to the student but not instructor-only comments. Object-property authorization checks which fields may be read or modified.

4.19 Mass assignment

Mass assignment occurs when an API automatically copies client-supplied JSON fields into server-side objects without allowlisting permitted fields. A user profile update should not let the client send {"role":"admin"} and have it applied. Use explicit field allowlists.

4.20 Excessive data exposure

Excessive data exposure happens when an API returns more fields than the client needs. Even if the user interface hides sensitive fields, the API response may still contain them. Security must be enforced in the API response, not only in the front end.

4.21 Broken authentication in APIs

APIs may fail when tokens are weak, long-lived, accepted for the wrong audience, not checked for expiration, stored insecurely, or not bound to intended contexts. Authentication checks should verify issuer, audience, expiration, signature, token type, and revocation strategy where applicable.

4.22 Unrestricted resource consumption

APIs consume CPU, memory, bandwidth, database queries, third-party billing, and storage. Without limits, attackers or buggy clients can cause cost, latency, or availability problems. Controls include rate limits, quotas, pagination limits, request-size limits, response-size limits, timeouts, and circuit breakers.

4.23 API schemas

An API schema describes the expected shape of requests and responses: fields, types, formats, required values, and examples. Schemas help validation, documentation, testing, client generation, and security review. Schema validation does not replace authorization, but it reduces unexpected input shape.

4.24 OpenAPI

OpenAPI is a common way to describe REST-style APIs. It can document endpoints, methods, parameters, request bodies, response bodies, and authentication schemes. Its security value depends on accuracy and enforcement. A stale API specification can create false confidence.

4.25 API gateway

An API gateway sits in front of services and can centralize routing, authentication integration, rate limiting, logging, TLS termination, request-size limits, and coarse policy. Gateways are useful, but they usually cannot understand every object-level business rule. Object-level authorization still belongs close to the data and business logic.

4.26 Service-to-service APIs

APIs are not only public. Microservices call each other. Internal APIs still need authentication, authorization, encryption in transit, logging, and inventory. "Internal" is not the same as "trusted forever."

4.27 GraphQL and object authorization

GraphQL lets clients ask for specific fields in a structured query. This flexibility is powerful, but it can increase authorization complexity. Field-level authorization, query depth limits, cost limits, schema review, and resolver-level checks become important.

4.28 Webhooks

A webhook is an HTTP callback from one system to another. Webhook security requires signature verification, replay protection, event-type validation, idempotency, and safe logging. A webhook sender should not be trusted merely because the request arrived at the right URL.

4.29 Error handling

API errors should help legitimate clients without leaking secrets, stack traces, database names, internal paths, or policy details. A common pattern is to return a stable error code to the client and log detailed evidence server-side.

4.30 Logging and telemetry

Useful API logs include timestamp, request ID, subject ID, tenant ID, endpoint, method, object type, object ID hash or safe reference, decision, status code, latency, client application, and correlation ID. Avoid logging tokens, passwords, full personal data, or secrets.

4.31 Detection thinking

API abuse often appears as patterns: many object IDs accessed in sequence, many forbidden responses, unusual export volume, token use from new locations, impossible tenant combinations, abnormal request rates, many schema validation failures, or endpoint use outside normal workflows.

4.32 Best beginner model

For every API request, ask: Who is calling? What action? Which object? Which fields? Which tenant? Which context? Where is the decision enforced?

7. Mathematical Foundations

7.1 Authorization as a predicate

An authorization decision can be modeled as a Boolean predicate:

Allow(subject, action, object, context) → true or false

For example:

Allow(alice, read, transcript_42, current_time) = false

This model is useful because it reminds us that authorization is not just a role check. It includes the subject, action, object, and context.

7.2 Object enumeration risk

If object identifiers are sequential and an attacker can make many requests, the number of possible neighboring IDs is small. If a system has records 1 through 10,000, an attacker can guess many possible IDs. Random IDs increase guessing difficulty, but they do not remove the need for authorization.

7.3 Rate limiting formula

A simple fixed-window rate limit can be described as:

requests(subject, window) ≤ limit

If the limit is 100 requests per minute, then request 101 in the same minute is rejected or delayed. More advanced systems use sliding windows, token buckets, adaptive throttling, or per-endpoint cost models.

7.4 Token bucket intuition

A token bucket has capacity C and refill rate r. Each request costs one or more tokens. A request is allowed only if enough tokens exist.

new_tokens = min(C, old_tokens + r × elapsed_time)
allow request if new_tokens ≥ request_cost

Worked example: capacity is 10 tokens, refill is 2 tokens per second, current tokens are 4, and 3 seconds pass. New tokens equal min(10, 4 + 2 × 3) = 10. A request costing 4 tokens is allowed, leaving 6 tokens.

7.5 Response minimization

Let F be all fields in an object and A be the fields allowed for this subject and action. The API should return:

returned_fields = F ∩ A

If F = {name, grade, advisor_note} and A = {name, grade}, then advisor_note must not appear in the response.

7.6 Detection ratios

A simple signal for suspicious object probing is the denial ratio:

denial_ratio = forbidden_or_not_found_object_requests / total_object_requests

A high denial ratio is not proof of attack, but it is a useful triage signal when combined with request volume, endpoint type, user history, and client application.

8. Practical Examples

8.1 Linux examples

Inspect a local JSON API response saved to a file:

cat response.json
python3 -m json.tool response.json
jq '.id, .owner_id, .status' response.json  # if jq is installed

Search local application logs for API denials without exposing tokens:

grep 'status=403' api.log
awk '/status=403/ {print $1, $2, $4, $6}' api.log

8.2 Windows examples

In PowerShell, pretty-print a local JSON file:

Get-Content .\response.json | ConvertFrom-Json | Format-List
Select-String -Path .\api.log -Pattern 'status=403'

8.3 Python examples

Use Python to check a fictional authorization rule:

def can_read_record(user, record):
    return record["tenant_id"] == user["tenant_id"] and (
        record["owner_id"] == user["id"] or "advisor" in user["roles"]
    )

8.4 Network examples

At the network level, API requests are HTTP messages over TLS. On a real system you normally cannot read the encrypted payload without endpoint access. In a local lab, you can inspect request paths, methods, status codes, timing, and logs generated by your own toy service.

8.5 Web examples

A single-page web app may call:

GET /api/profile/me
GET /api/courses/2026-security/assignments

The backend must not rely on hidden buttons. If the browser does not show an admin button, a user may still send a direct API request. The server must enforce policy.

8.6 Cloud examples

Cloud APIs use identity, permissions, resource identifiers, policies, and logs. A cloud control plane must decide whether a subject can read, create, modify, or delete a specific resource. Cloud audit logs are critical evidence for investigating unauthorized API activity.

9. Code Examples

Important: these examples are intentionally small toy examples for local learning. They are not production frameworks, not complete authentication systems, and not permission to test real APIs.

9.1 Unsafe pattern: checking login only

# Toy example: unsafe pattern.
# The function checks that someone is logged in,
# but it does not check whether the record belongs to that user.

records = {
    1: {"id": 1, "owner_id": "alice", "tenant_id": "university-a", "grade": "A"},
    2: {"id": 2, "owner_id": "bob", "tenant_id": "university-a", "grade": "B"},
}

def get_record_unsafe(current_user, record_id):
    if current_user is None:
        return {"status": 401, "error": "authentication required"}

    # Mistake: the API returns the object only because the caller is logged in.
    # It never asks whether this specific object is allowed.
    record = records.get(record_id)
    if record is None:
        return {"status": 404, "error": "not found"}
    return {"status": 200, "data": record}

Line-by-line explanation

  • records is a fictional in-memory database.
  • current_user represents the authenticated subject.
  • The function correctly rejects unauthenticated callers.
  • The function incorrectly returns any existing record to any logged-in user.
  • The missing step is object-level authorization.

9.2 Safer pattern: check subject, action, object, and tenant

records = {
    1: {"id": 1, "owner_id": "alice", "tenant_id": "university-a", "grade": "A", "advisor_note": "On track"},
    2: {"id": 2, "owner_id": "bob", "tenant_id": "university-a", "grade": "B", "advisor_note": "Needs support"},
}

PUBLIC_FIELDS_FOR_STUDENT = {"id", "owner_id", "tenant_id", "grade"}

def can_read_record(current_user, record):
    same_tenant = current_user["tenant_id"] == record["tenant_id"]
    owns_record = current_user["id"] == record["owner_id"]
    is_advisor = "advisor" in current_user.get("roles", [])
    return same_tenant and (owns_record or is_advisor)

def filter_fields_for_user(current_user, record):
    if "advisor" in current_user.get("roles", []):
        return dict(record)
    return {key: value for key, value in record.items() if key in PUBLIC_FIELDS_FOR_STUDENT}

def get_record_safe(current_user, record_id):
    if current_user is None:
        return {"status": 401, "error": "authentication required"}

    record = records.get(record_id)
    if record is None:
        return {"status": 404, "error": "not found"}

    if not can_read_record(current_user, record):
        return {"status": 403, "error": "forbidden"}

    safe_record = filter_fields_for_user(current_user, record)
    return {"status": 200, "data": safe_record}

Line-by-line explanation

  • PUBLIC_FIELDS_FOR_STUDENT is an allowlist of fields normal students may see.
  • can_read_record checks tenant match and relationship to the object.
  • filter_fields_for_user applies object-property authorization.
  • get_record_safe authenticates, loads object metadata, checks authorization, filters fields, and returns only allowed data.

9.3 Safe field updates with an allowlist

ALLOWED_PROFILE_FIELDS = {"display_name", "phone", "preferred_language"}

def update_profile(current_user, profile, requested_changes):
    if current_user["id"] != profile["owner_id"]:
        return {"status": 403, "error": "forbidden"}

    clean_changes = {}
    for field, value in requested_changes.items():
        if field not in ALLOWED_PROFILE_FIELDS:
            return {"status": 400, "error": f"field not allowed: {field}"}
        clean_changes[field] = value

    profile.update(clean_changes)
    return {"status": 200, "data": profile}

This prevents mass assignment because the client can only modify fields explicitly allowed by the server.

9.4 Schema-style validation with Python

def validate_profile_patch(body):
    if not isinstance(body, dict):
        raise ValueError("JSON object required")

    allowed = {"display_name": str, "phone": str, "preferred_language": str}
    for key, value in body.items():
        if key not in allowed:
            raise ValueError(f"unexpected field: {key}")
        if not isinstance(value, allowed[key]):
            raise ValueError(f"wrong type for field: {key}")
        if len(value) > 120:
            raise ValueError(f"value too long for field: {key}")
    return True

9.5 Safe API logging

import hashlib

def safe_object_ref(object_id):
    digest = hashlib.sha256(str(object_id).encode("utf-8")).hexdigest()
    return digest[:12]

def log_api_decision(subject_id, method, path, object_id, decision, status):
    print(
        "api_decision",
        f"subject={subject_id}",
        f"method={method}",
        f"path={path}",
        f"object_ref={safe_object_ref(object_id)}",
        f"decision={decision}",
        f"status={status}",
    )

This toy logger records useful decision evidence but avoids printing tokens or full sensitive object data.

9.6 Common coding mistakes

  • Checking only that a user is logged in.
  • Checking only role but not object ownership or tenant.
  • Trusting object IDs from the client.
  • Applying authorization in the front end only.
  • Returning full database objects to the client.
  • Letting client JSON update all server-side fields.
  • Using UUIDs as a substitute for authorization.
  • Logging bearer tokens or full personal data.
  • Not writing negative authorization tests.

10. Security Perspective

Common attacks

  • BOLA/IDOR: changing an object identifier to access another user's object.
  • Broken function-level authorization: calling an endpoint reserved for higher privileges.
  • Broken object-property authorization: reading or changing fields the user should not access.
  • Mass assignment: submitting unexpected fields such as roles, balances, approval flags, or tenant IDs.
  • Token abuse: using stolen, leaked, over-scoped, expired, or wrong-audience tokens.
  • Enumeration: attempting many object IDs, usernames, invoice numbers, or file references.
  • Unrestricted resource consumption: using expensive API calls to cause cost or availability problems.
  • SSRF through APIs: causing the server to request attacker-chosen URLs when API inputs feed backend fetchers.
  • API misconfiguration: verbose errors, debug endpoints, permissive CORS, missing TLS, or exposed admin APIs.
  • Inventory gaps: forgotten API versions, shadow APIs, undocumented partner endpoints, or old mobile endpoints.

Common vulnerabilities

  • Missing server-side object checks.
  • Policy spread inconsistently across controllers.
  • Trusting client-side filtering.
  • Overly broad roles and scopes.
  • Lack of tenant isolation.
  • Raw database object serialization.
  • Missing request schema validation.
  • No pagination or response-size limits.
  • No useful audit trail for denied and allowed decisions.
  • No regression tests for forbidden object access.

Defensive strategies

  • Centralize authorization patterns while keeping business context close to the data.
  • Check subject + action + object + fields + tenant + context for every object access.
  • Use deny-by-default policy design.
  • Use field allowlists for input and output.
  • Validate request shape with schemas.
  • Apply rate limits, quotas, pagination, and cost controls.
  • Use short-lived, appropriately scoped tokens.
  • Log decisions without secrets.
  • Write negative tests: user A must not access user B's object.
  • Review old API versions and remove unused endpoints.

Detection methods

  • High rate of 403 or 404 responses for object endpoints.
  • Sequential object ID access by one subject.
  • Cross-tenant access attempts.
  • Large export volume from unusual clients.
  • Token use from impossible or unusual locations.
  • Endpoint calls outside normal user interface flow.
  • Schema validation failures clustered by client or route.
  • Unexpected field names in update requests.

Mitigation

Mitigation should fix the root cause, not only hide symptoms. If an API has BOLA, rate limiting alone is insufficient. The API must enforce object-level policy. If an API exposes excessive fields, the response model must be reduced or filtered server-side. If mass assignment exists, the update model must use explicit allowlists.

Best practices

  • Design authorization during API design, not after release.
  • Document object ownership and relationships.
  • Use stable policy helpers or policy engines where appropriate.
  • Test authorization at unit, integration, and end-to-end levels.
  • Maintain an accurate API inventory.
  • Version APIs deliberately and retire old versions.
  • Use least privilege for databases and service accounts.
  • Separate public, partner, internal, and administrative APIs.
  • Review OpenAPI specifications against implementation behavior.

11. Industry Perspective

Banks

Banks use APIs for mobile apps, payment networks, partner integrations, fraud systems, and open banking. Object-level authorization is critical because an authenticated customer must only access their own accounts, cards, statements, and payment instructions unless a delegated access relationship exists.

Governments

Government APIs handle benefits, tax, identity, permits, law-enforcement data, and public services. They must enforce citizen, agency, role, case, and jurisdiction boundaries. Logging and auditability are especially important.

Cloud providers

Cloud control-plane APIs create and modify infrastructure. Authorization must consider account, project, resource, region, action, tag, policy, and service principal. A weak API policy can lead to large-scale resource exposure or destructive changes.

Healthcare

Healthcare APIs must protect patient records, appointments, prescriptions, lab results, and insurance data. Access often depends on patient relationship, care team, consent, emergency context, and regulatory obligations.

Defense

Defense systems use APIs for logistics, communications, intelligence workflows, and operational platforms. Object-level access may be tied to mission, classification, need-to-know, unit assignment, and audit requirements.

Universities

Universities use APIs for learning management systems, grades, transcripts, housing, research data, identity, and library services. Common policy questions include student ownership, instructor course membership, advisor access, department administration, and research-project roles.

Large enterprises

Enterprises operate many internal and external APIs. Their risks include shadow APIs, inconsistent authorization libraries, stale API versions, partner tokens, overbroad service accounts, and incomplete logging across microservices.

12. Research Perspective

API security research is active because APIs combine software engineering, authorization policy, distributed systems, formal models, usability, and automated testing.

Current research directions

  • Authorization inference: automatically learning expected access-control rules from code, traces, schemas, or tests.
  • BOLA detection: finding endpoints where object IDs flow from requests to data access without policy checks.
  • OpenAPI security extensions: describing object-level and field-level policy requirements in machine-readable API specifications.
  • GraphQL authorization: enforcing field-level and resolver-level policies while controlling query depth and cost.
  • Microservice policy consistency: preventing policy drift across many services and versions.
  • Property-based and differential testing: generating tests that compare what different users can see and change.
  • API abuse detection: distinguishing malicious enumeration from legitimate high-volume automation.
  • Zero Trust API access: continuously evaluating identity, device, workload, context, and resource sensitivity.

Open problems

  • How can developers express object-level authorization requirements clearly without excessive complexity?
  • How can tools identify missing authorization with low false positives?
  • How should API schemas represent object ownership, tenant boundaries, and field-level policy?
  • How can large organizations prove that every API endpoint has current authorization tests?
  • How should AI agents safely call APIs on behalf of users without overbroad delegated authority?

Notable paper directions

Recent research explores BOLA under zero-trust principles, empirical taxonomies from bug-bounty reports, and OpenAPI extensions for object-level security metadata. Read these as research directions, not as production recipes. Reproduce only in authorized local environments.

13. Case Studies

Case Study 1: Facebook/Cambridge Analytica platform data access

What happened: third-party application access to social-platform data became a major privacy and governance controversy. Root cause: broad API access patterns, platform policy decisions, and data-sharing controls created exposure beyond what many users expected. Impact: regulatory, legal, reputational, and public-trust consequences. Lessons learned: API authorization is not only technical; it includes consent, data minimization, third-party governance, review, and auditability.

Case Study 2: Twitter API misuse and data exposure patterns

What happened: multiple public incidents over the years have involved API behavior, account metadata, scraping, or token/access-control issues. Root cause: APIs that expose structured data can be abused at scale when access, rate limits, data minimization, or monitoring are insufficient. Impact: user privacy risk and regulatory scrutiny. Lessons learned: rate limiting and authentication are not enough without data-minimizing responses and abuse monitoring.

Case Study 3: Cloud metadata and API credential abuse

What happened: attackers in several cloud incidents have abused application weaknesses to reach metadata services or steal API credentials. Root cause: server-side request paths, over-permissive roles, and insufficient egress/metadata protections. Impact: unauthorized cloud API access, data exposure, or resource manipulation. Lessons learned: API security includes workload identity, metadata protections, least privilege, monitoring, and SSRF-resistant design.

Case Study 4: Partner and mobile APIs

What happened: many real-world API flaws appear in partner, mobile, or legacy endpoints rather than the main website. Root cause: old versions, undocumented endpoints, inconsistent authorization checks, and assumptions that only the official client will call the API. Impact: unauthorized data access and operational risk. Lessons learned: maintain an API inventory and test authorization at the server, independent of any client user interface.

14. Common Misconceptions

"If the user is logged in, the API is secure."
Wrong. Login proves identity; it does not prove permission to access every object.
"UUIDs fix IDOR."
Wrong. UUIDs reduce guessing but do not enforce authorization.
"The button is hidden, so users cannot call the endpoint."
Wrong. API endpoints can be called directly. Server-side policy must enforce access.
"The API gateway handles authorization."
Partly. Gateways can enforce coarse controls, but most object-level business rules require service-level checks.
"Scopes solve object-level authorization."
Not by themselves. A scope may allow reading files generally, but object-level policy decides which files.
"Internal APIs do not need security."
Wrong. Internal systems are still reachable by compromised workloads, insiders, misconfigurations, and lateral movement.
"Returning extra fields is fine because the UI hides them."
Wrong. API clients can inspect raw responses.
"Rate limiting prevents BOLA."
Wrong. Rate limiting reduces scale but does not fix unauthorized access.
"OpenAPI documentation guarantees security."
Wrong. Documentation helps only if accurate and backed by enforcement and tests.
"403 and 404 choices are always obvious."
Not always. Sometimes systems avoid revealing object existence. The right choice depends on product, threat model, and usability.

15. Summary

APIs let software talk to software. They are essential for modern applications, mobile apps, cloud platforms, microservices, and partner integrations. Their security depends on more than authentication. Every object access must be authorized server-side.

The central lesson is: authenticate the caller, validate the request, load the object metadata, authorize the action on that object, filter fields, execute safely, and log the decision without secrets.

BOLA/IDOR occurs when an API exposes object identifiers but fails to check whether the caller is allowed to access the requested object. UUIDs, hidden buttons, mobile-app assumptions, and broad scopes do not replace object-level policy. Strong API security combines design, schemas, least privilege, rate limits, gateways, object and field-level checks, negative tests, observability, and inventory management.

16. Glossary

API
Application Programming Interface; a controlled way for software to request data or actions from other software.
API Gateway
A component that sits in front of APIs and can provide routing, authentication integration, rate limiting, logging, and coarse controls.
Bearer Token
A token that grants access to whoever presents it.
BOLA
Broken Object Level Authorization; an API weakness where object-level permission checks are missing or incorrect.
Claim
A statement about a subject or token, such as user ID, issuer, audience, role, or expiration.
Endpoint
A specific API path and method that performs a defined operation.
Function-Level Authorization
Checking whether a subject may use a feature or endpoint.
GraphQL
An API query language that lets clients request specific fields from a schema.
IDOR
Insecure Direct Object Reference; a weakness where direct object identifiers can be manipulated to access unauthorized objects.
JSON
JavaScript Object Notation; a text format for structured data.
Mass Assignment
A weakness where client-supplied fields are automatically copied into server objects without allowlisting.
Object
A specific resource instance, such as one invoice, file, record, account, or message.
Object-Level Authorization
Checking whether a subject can perform an action on a specific object.
Object-Property Authorization
Checking whether a subject can read or modify specific fields of an object.
OpenAPI
A specification format for describing REST-style APIs.
Policy Decision Point
The component that evaluates policy and decides whether access is allowed.
Policy Enforcement Point
The component that enforces a policy decision at runtime.
Rate Limit
A control that restricts request volume over time.
Resource
A type of thing exposed by an API, such as users, invoices, files, or tickets.
Scope
A permission-like value associated with a token or client authorization.
Tenant
An organization, customer, or logical boundary in a multi-tenant system.
Webhook
An HTTP callback sent from one system to another when an event occurs.
