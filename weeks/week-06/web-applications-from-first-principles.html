Week 6 · Web and API Security

Lesson 6.1: Web Applications from First Principles

Before studying web attacks, students need a precise model of the web itself: browsers, URLs, DNS, HTTP, HTTPS, requests, responses, cookies, sessions, APIs, origins, trust boundaries, logs, and server-side enforcement.

Estimated reading time300–420 minutes
Difficulty levelBeginner → MS foundation
Required backgroundWeeks 0–5 recommended; no web background required
Practice time5–8 hours

1. Introduction

A web application is software that people use through a browser or another HTTP client. Examples include online banking, university portals, shopping sites, healthcare dashboards, cloud consoles, email, learning platforms, and APIs used by mobile apps.

At the beginner level, a web application looks simple: you type an address, click a button, and a page appears. Under the surface, many parts work together: browser, URL, DNS, TCP/IP, TLS, HTTP, server, application code, database, cookies, sessions, JavaScript, APIs, logs, and cloud infrastructure.

This lesson starts from first principles. It explains how the web exists, why it uses request/response communication, why browsers enforce security boundaries, and why most web vulnerabilities are failures of trust, validation, authentication, authorization, state handling, or configuration.

Why the topic exists

The web exists to let clients request resources from servers using common protocols. HTTP became the shared language for retrieving documents, submitting forms, calling APIs, and building distributed applications.

Why it matters

  • Most organizations expose web applications to users, partners, employees, and customers.
  • Web applications often connect directly to sensitive data and business processes.
  • A small mistake in authentication, authorization, validation, session handling, or API design can expose large amounts of data.
  • Modern cloud and mobile systems often depend on web APIs even when users never open a browser.

Where it is used

Web application concepts are used in public websites, internal dashboards, cloud consoles, APIs, mobile backends, payment systems, student systems, patient portals, government services, identity providers, and software-as-a-service platforms.

Who uses it

  • Developers build pages, APIs, authentication flows, and server logic.
  • Security engineers review design, test controls, inspect logs, and harden deployments.
  • SOC analysts monitor web logs, API logs, authentication events, and suspicious traffic.
  • Cloud engineers configure load balancers, TLS, WAFs, identity providers, and runtime platforms.
  • Researchers study browser isolation, web privacy, API authorization, client-side security, and automated vulnerability discovery.
Safety boundary: all examples in this lesson use local files, localhost, browser developer tools, and fictional logs. Do not scan, intercept, test, or attack real websites or APIs unless you have explicit written authorization and scope.

2. Learning Outcomes

  • Define web application, browser, client, server, URL, domain, DNS, HTTP, HTTPS, TLS, request, response, method, path, query string, header, body, status code, cookie, session, token, origin, same-origin policy, CORS, HTML, CSS, JavaScript, API, JSON, REST-style API, backend, frontend, database, cache, reverse proxy, load balancer, CDN, WAF, and log.
  • Explain how a browser loads a web page from URL entry to rendered page.
  • Describe the structure of an HTTP request and response.
  • Explain why HTTP is stateless and how cookies and sessions add state.
  • Distinguish authentication from authorization in web applications.
  • Explain the browser same-origin policy at a beginner level.
  • Identify major web attack categories without performing unauthorized testing.
  • Use safe local examples to inspect HTTP requests, cookies, JSON, forms, and logs.
  • Write basic Python code for a local-only learning web server and explain each line.
  • Create a simple web application architecture diagram and identify trust boundaries.

3. Prerequisites

Network

A network connects computers so they can exchange data. The internet is a global network of networks. Earlier lessons introduced IP addresses, ports, DNS, TCP, and TLS. This lesson reuses those ideas slowly.

Client and server

A client asks for something. A server responds. Your browser is usually the client. A website's application server is usually the server.

File

A file stores data. Web pages often include HTML files, CSS files, JavaScript files, images, fonts, and downloadable documents.

Program

A program is instructions executed by a computer. Web applications usually contain server-side code, client-side code, configuration, templates, and database queries.

Identity

Identity is how a system represents a person, service, device, or workload. Web applications use identities to decide who can log in and what actions are allowed.

Cryptography

HTTPS uses TLS to protect traffic between client and server. You do not need to master cryptography for this lesson, but you should know that TLS provides encryption, integrity protection, and server authentication when configured correctly.

4. Core Concepts

4.1 The web in one sentence

The web is a system where clients request resources from servers using URLs and HTTP, then render or process the responses.

4.2 Browser

A browser is a program that retrieves web content, displays pages, runs JavaScript, stores cookies, enforces security boundaries, and gives users an interface to the web. Examples include Chrome, Firefox, Safari, and Edge.

4.3 URL

A Uniform Resource Locator, or URL, is an address for a resource. Example:

https://portal.example.edu:443/courses?id=123#overview
\___/  \______________/ \_/ \______\/______\________/
scheme       host       port   path   query   fragment

The scheme says which protocol to use. The host names the server. The path identifies the resource. The query string adds parameters. The fragment is handled by the browser and is not normally sent to the server.

4.4 Domain name and DNS

A domain name is a human-friendly name such as example.edu. Domain Name System, or DNS, translates names into IP addresses. The analogy is a phonebook: humans remember names, networks route to addresses.

4.5 HTTP

Hypertext Transfer Protocol, or HTTP, is the application-level protocol used by web clients and servers. HTTP is request/response based: the client sends a request, and the server returns a response.

4.6 HTTPS and TLS

HTTPS is HTTP over TLS. TLS protects the connection against eavesdropping and tampering and helps the browser authenticate the server name. HTTPS does not automatically make an application secure; it protects the transport channel.

4.7 Request

An HTTP request is a message from client to server. It usually contains a method, path, headers, and sometimes a body.

GET /courses?id=123 HTTP/1.1
Host: portal.example.edu
User-Agent: ExampleBrowser/1.0
Accept: text/html
Cookie: session=abc123

4.8 Response

An HTTP response is a message from server to client. It usually contains a status code, headers, and sometimes a body.

HTTP/1.1 200 OK
Content-Type: text/html
Set-Cookie: session=abc123; HttpOnly; Secure; SameSite=Lax

<html><body>Course page</body></html>

4.9 HTTP methods

A method describes the general action requested. Common methods include:

MethodPlain meaningCommon use
GETRetrieveLoad a page or resource
POSTSubmit/processLogin, submit form, create resource
PUTReplaceReplace a resource in APIs
PATCHModifyPartially update a resource
DELETEDeleteDelete a resource in APIs
OPTIONSAsk capabilitiesCORS preflight and API capability checks

4.10 Status codes

A status code tells the client how the request was handled.

  • 2xx: success, such as 200 OK.
  • 3xx: redirection, such as 301 Moved Permanently.
  • 4xx: client-side problem, such as 404 Not Found or 403 Forbidden.
  • 5xx: server-side problem, such as 500 Internal Server Error.

4.11 Headers

Headers are metadata fields in requests and responses. They can carry content type, caching rules, cookies, authentication tokens, security policy, and browser instructions.

4.12 Body

The body is the optional data part of a request or response. A login request may include username and password fields. An API response may include JSON.

4.13 HTML, CSS, and JavaScript

  • HTML gives page structure: headings, paragraphs, links, forms.
  • CSS controls presentation: layout, color, spacing.
  • JavaScript adds behavior: dynamic forms, API calls, client-side interaction.

4.14 Frontend and backend

The frontend is the browser-facing part. The backend is the server-side application and supporting services such as database, cache, message queue, identity provider, and storage.

4.15 Stateless HTTP

HTTP is often described as stateless: each request is independent at the protocol level. The server does not automatically remember previous requests. Web applications add state using cookies, sessions, tokens, databases, and server-side application logic.

4.16 Cookies

A cookie is a small piece of data the server asks the browser to store and send with later matching requests. Cookies are often used to link multiple requests to the same session. Important cookie attributes include HttpOnly, Secure, and SameSite.

4.17 Sessions

A session is server-side state associated with a user or browser interaction. A common design is: the browser stores a random session ID in a cookie; the server stores the actual session data.

4.18 Tokens

A token is a value used to represent authorization, authentication state, or access rights. Tokens must be protected like credentials. A JSON Web Token, or JWT, is one common token format, but using JWT does not automatically make a system secure.

4.19 Authentication and authorization

Authentication asks: who are you? Authorization asks: what are you allowed to do? A user may be correctly logged in but still not authorized to view another student's transcript or another customer's bank account.

4.20 Origin

An origin is the combination of scheme, host, and port. These are different origins:

https://example.edu
http://example.edu
https://example.edu:8443
https://api.example.edu

4.21 Same-origin policy

The same-origin policy is a browser security boundary that restricts how scripts loaded from one origin can interact with data from another origin. It helps prevent one website from freely reading another website's sensitive data in the user's browser.

4.22 CORS

Cross-Origin Resource Sharing, or CORS, is a controlled way for servers to tell browsers which cross-origin requests are allowed. CORS is not an authentication system. Misconfigured CORS can expose data to unintended origins.

4.23 Forms

A web form lets users submit data. Security issues often appear when forms trust input too much, skip CSRF protections, expose sensitive values, or rely only on client-side checks.

4.24 API

An Application Programming Interface, or API, lets software talk to software. Web APIs often use HTTP and JSON. Mobile apps, single-page applications, partner integrations, and cloud automation commonly rely on APIs.

4.25 JSON

JavaScript Object Notation, or JSON, is a text format for structured data. Example:

{
  "course_id": 123,
  "title": "Cybersecurity",
  "enrolled": true
}

4.26 Common web architecture components

  • Reverse proxy: receives client traffic and forwards it to backend servers.
  • Load balancer: distributes traffic across multiple servers.
  • Content Delivery Network, or CDN: caches content near users.
  • Web Application Firewall, or WAF: inspects web traffic for known attack patterns and policy violations.
  • Database: stores application data.
  • Cache: stores temporary data to improve speed.
  • Identity provider: handles login, single sign-on, or token issuance.

4.27 Trust boundary

A trust boundary is a place where data moves between different trust levels. Examples: browser to server, public internet to reverse proxy, application to database, third-party API to internal system.

4.28 Web logs

Web logs record requests, responses, status codes, user agents, IP addresses, timing, and errors. Logs help defenders detect attacks, investigate incidents, debug problems, and measure availability.

4.29 Web security categories

Major categories include broken access control, injection, cross-site scripting, cross-site request forgery, insecure design, authentication failures, cryptographic failures, security misconfiguration, vulnerable dependencies, logging gaps, SSRF, and API authorization errors. Later lessons will cover these in detail.

4.30 Key beginner rule

Never trust input merely because it came from a browser. Browsers can be modified, requests can be scripted, and client-side checks can be bypassed. The server must enforce security decisions.

7. Mathematical Foundations

7.1 Request rate

Web defenders often measure request rate:

requests_per_second = total_requests / seconds

Worked example: if a server receives 12,000 requests in 60 seconds, then 12,000 / 60 = 200 requests per second.

7.2 Error rate

Error rate helps detect outages, abuse, or failed releases:

error_rate = error_responses / total_responses × 100%

Worked example: if 500 of 10,000 responses are 5xx errors, the error rate is 500 / 10,000 × 100% = 5%.

7.3 Availability

Availability can be estimated as:

availability = uptime / total_time × 100%

If a service is up for 29 days and down for 1 day in a 30-day month, availability is 29 / 30 × 100% = 96.67%.

7.4 Token guessing intuition

If a random session token has b bits of entropy, there are 2^b possible values. Larger unpredictable tokens are harder to guess. This assumes the random generator is strong and tokens are not leaked.

7.5 Latency percentiles

Latency is the time a request takes. The 95th percentile means 95% of requests are that fast or faster. Security controls must be designed with performance because very slow controls may be disabled by frustrated teams.

7.6 Risk model for web endpoints

A simple endpoint risk score can be:

risk_score = data_sensitivity + exposure + privilege_required + action_impact - control_strength

This is not a universal formula. It is a teaching aid that reminds students to consider data, exposure, privileges, impact, and controls together.

Python examples: local-only web server

from http.server import BaseHTTPRequestHandler, HTTPServer
from html import escape
from urllib.parse import urlparse, parse_qs

class DemoHandler(BaseHTTPRequestHandler):
    def do_GET(self):
        # Parse the URL path and query string from the request.
        parsed = urlparse(self.path)
        params = parse_qs(parsed.query)

        # Read a name parameter if provided. Use a safe default.
        name = params.get("name", ["student"])[0]

        # Escape output before placing user-controlled text into HTML.
        safe_name = escape(name)

        body = f"<html><body><h1>Hello, {safe_name}</h1></body></html>"
        body_bytes = body.encode("utf-8")

        self.send_response(200)
        self.send_header("Content-Type", "text/html; charset=utf-8")
        self.send_header("Content-Length", str(len(body_bytes)))
        self.end_headers()
        self.wfile.write(body_bytes)

if __name__ == "__main__":
    server = HTTPServer(("127.0.0.1", 8000), DemoHandler)
    print("Local demo server: http://127.0.0.1:8000/?name=Samad")
    server.serve_forever()

Line-by-line explanation:

  • BaseHTTPRequestHandler gives us a basic request-handler class.
  • HTTPServer creates a small local server.
  • escape converts special HTML characters so user input is displayed as text, not interpreted as markup.
  • urlparse separates the path from the query string.
  • parse_qs converts query parameters into a dictionary-like structure.
  • do_GET runs when the server receives a GET request.
  • self.path contains the requested path and query string.
  • params.get safely reads the name parameter or falls back to student.
  • safe_name is output-encoded before being placed in HTML.
  • send_response, send_header, and end_headers build the HTTP response.
  • wfile.write sends the response body.
  • 127.0.0.1 binds to localhost, not the public network.

9. Code Examples

9.1 Unsafe output pattern

# Do not copy this pattern into real applications.
# It places user-controlled text into HTML without output encoding.
name = "<script>alert('demo')</script>"
html = f"<h1>Hello, {name}</h1>"
print(html)

Problem: the variable name is treated as HTML. In a browser, markup-like input may be interpreted as code or structure. The server should encode output for the context.

9.2 Safer output pattern

from html import escape

name = "<script>alert('demo')</script>"
safe_name = escape(name)
html = f"<h1>Hello, {safe_name}</h1>"
print(html)

Why this is safer: escape converts characters such as < and > into text representations. The browser displays the input instead of treating it as markup.

9.3 Simple JSON response pattern

import json

record = {
    "course_id": 123,
    "title": "Cybersecurity",
    "public": True
}

body = json.dumps(record)
print(body)

Line-by-line: json is Python's JSON library. record is a dictionary. json.dumps serializes the dictionary into a JSON string. JSON serialization is safer than manually building JSON with string concatenation.

9.4 Common mistakes

  • Trusting hidden form fields for authorization decisions.
  • Checking authorization only in the user interface, not on the server.
  • Putting secrets in JavaScript files.
  • Returning detailed stack traces to users.
  • Assuming HTTPS prevents SQL injection, XSS, or authorization bugs.
  • Using CORS as if it were authentication.
  • Logging passwords, tokens, or session cookies.
  • Accepting JSON input without validating fields, types, size, and authorization.

9.5 Secure coding practices

  • Validate input on the server.
  • Encode output for the correct context.
  • Use framework security features instead of hand-rolled authentication.
  • Store secrets outside source code.
  • Use parameterized database queries.
  • Set secure cookie attributes.
  • Implement object-level authorization on every sensitive object access.
  • Log security-relevant events without exposing secrets.
  • Return generic errors to users and detailed diagnostics only to protected logs.

10. Security Perspective

Common attacks

  • Broken access control: a user accesses data or actions they should not access.
  • Injection: untrusted input is interpreted as a command, query, or code.
  • Cross-site scripting: untrusted input becomes script or markup in a victim's browser.
  • Cross-site request forgery: a browser is tricked into sending a state-changing request using existing credentials.
  • Authentication attacks: credential stuffing, password spraying, session theft, MFA fatigue, weak recovery flows.
  • Server-side request forgery: a server is tricked into making unintended requests.
  • API authorization failures: an API exposes object identifiers and fails to check whether the caller may access each object.
  • Security misconfiguration: debug mode, default credentials, permissive CORS, weak headers, public storage, verbose errors.

Common vulnerabilities

  • Missing server-side authorization checks.
  • Unsafe string concatenation into SQL, HTML, shell commands, or file paths.
  • Weak session management.
  • Sensitive data in URLs, logs, browser storage, or client-side code.
  • Missing rate limiting.
  • Overly broad CORS rules.
  • Outdated dependencies and vulnerable plugins.
  • Missing security logging and alerting.

Defensive strategies

  • Threat model the application before coding.
  • Use secure framework defaults.
  • Enforce authentication and authorization on the server.
  • Validate input and encode output contextually.
  • Use parameterized database APIs.
  • Use HTTPS everywhere with correct TLS configuration.
  • Set secure cookie attributes.
  • Apply content security policy where appropriate.
  • Use dependency scanning, SAST, DAST, and manual review.
  • Monitor logs for suspicious behavior and abnormal error rates.

Detection methods

  • Spikes in 401, 403, 404, and 5xx status codes.
  • Repeated requests to unusual paths.
  • High request rates from a single source or token.
  • Unexpected user-agent strings or missing headers.
  • Suspicious query strings with encoded characters or command-like patterns.
  • Failed login bursts, password spraying patterns, and impossible travel events.
  • API access to many object IDs in sequence.
  • Server errors triggered by malformed input.

Mitigation

Mitigation combines design controls, implementation controls, runtime controls, and operational controls. A WAF may reduce some known attack patterns, but it does not replace secure design or code fixes.

Best practices

  • Minimize exposed endpoints.
  • Use deny-by-default authorization.
  • Keep sensitive state server-side when possible.
  • Use least privilege for databases and service accounts.
  • Separate development, staging, and production environments.
  • Never expose debug tools in production.
  • Document trust boundaries and data classification.
  • Build incident-response playbooks for web and API events.

11. Industry Perspective

Banks

Banks use web apps for online banking, trading, customer onboarding, fraud review, internal dashboards, and payment APIs. They emphasize strong authentication, transaction authorization, fraud monitoring, audit logs, encryption, and regulatory reporting.

Governments

Government portals support taxes, benefits, identity services, permits, voting information, and public records. They must handle accessibility, privacy, identity proofing, data retention, and resilience against disruption.

Cloud providers

Cloud providers expose management consoles and APIs. Their security depends heavily on identity, API authorization, tenant isolation, rate limits, audit trails, and secure defaults.

Healthcare

Healthcare web systems include patient portals, provider dashboards, billing, telehealth, and lab results. They prioritize confidentiality, integrity, availability, auditability, and patient safety.

Defense

Defense environments use web applications for logistics, intelligence workflows, command support, training, and internal services. They focus on identity assurance, segmentation, monitoring, secure supply chains, and mission continuity.

Universities

Universities run student portals, research systems, learning platforms, identity systems, and public websites. They must support open collaboration while protecting student records, research data, and institutional credentials.

Large enterprises

Large enterprises use thousands of web apps and APIs. Their biggest challenges are inventory, legacy systems, third-party integrations, single sign-on, API governance, vulnerability management, and consistent logging.

12. Research Perspective

Web security remains active research because the platform keeps changing. Browsers add APIs, applications become more client-heavy, organizations expose more APIs, and cloud architectures distribute trust across many services.

Current research areas

  • Browser isolation: improving origin separation, site isolation, sandboxing, and permission models.
  • Web privacy: limiting tracking, fingerprinting, third-party cookies, and cross-site data leakage.
  • API security: automated discovery of API endpoints, object-level authorization testing, schema validation, and runtime anomaly detection.
  • Client-side security: detecting vulnerable JavaScript dependencies, malicious scripts, supply-chain attacks, and insecure browser storage.
  • Formal methods: modeling web protocols, authentication flows, and browser policies.
  • Automated testing: using static analysis, dynamic analysis, fuzzing, symbolic execution, and machine learning to find web vulnerabilities.
  • Usable security: designing authentication, consent, privacy, and warning systems that real users can understand.
  • LLM-enabled applications: studying prompt injection, tool-use boundaries, data exfiltration paths, and web-agent security.

Open problems

  • Finding authorization bugs reliably without knowing business rules.
  • Securing complex single-page applications with many third-party scripts.
  • Balancing privacy protections with fraud detection and abuse prevention.
  • Creating API inventories that stay accurate in fast-moving organizations.
  • Making browser security policies understandable to developers.
  • Preventing supply-chain compromise in frontend packages and build pipelines.

Notable research directions

Students should explore papers and systems on browser sandboxing, site isolation, OAuth/OIDC formal analysis, web tracking measurement, JavaScript supply-chain security, API fuzzing, cross-site leaks, and web cache deception. Do not reproduce copyrighted papers; read abstracts, methodology, assumptions, and limitations, then write your own critical summaries.

13. Case Studies

Equifax breach

What happened: attackers exploited an unpatched web application framework vulnerability in a public-facing system. Root cause themes: patch-management failure, asset visibility issues, vulnerable web component exposure, and monitoring gaps. Impact: sensitive personal information of many individuals was exposed. Lessons: know exposed web assets, patch critical web components quickly, verify remediation, monitor for exploitation, and segment sensitive data.

Capital One cloud/web SSRF incident

What happened: a web-facing component was abused to access cloud metadata and obtain credentials in a cloud environment. Root cause themes: server-side request behavior, cloud IAM permissions, metadata access, and detection challenges. Impact: unauthorized access to customer data. Lessons: web vulnerabilities can become cloud incidents; restrict metadata access, apply least privilege, monitor unusual API calls, and review SSRF risk.

MOVEit Transfer mass exploitation

What happened: attackers exploited vulnerabilities in a file-transfer web application used by many organizations. Root cause themes: exposed internet-facing managed file transfer software and vulnerability exploitation at scale. Impact: data theft across many affected organizations. Lessons: internet-facing web apps that handle files and sensitive data need rapid patching, logging, compromise assessment, and third-party risk management.

GitHub OAuth token exposure incidents

What happened: stolen OAuth tokens and application integrations have been used in multiple incidents to access repositories or related resources. Root cause themes: token handling, third-party app trust, overbroad permissions, and monitoring. Impact: source-code and secret exposure risk. Lessons: treat tokens as credentials, scope them narrowly, rotate compromised tokens, monitor access, and review third-party integrations.

14. Common Misconceptions

“HTTPS means the website is secure.”
Incorrect. HTTPS protects the connection. The application can still have authorization flaws, injection bugs, weak sessions, vulnerable dependencies, or misconfiguration.
“Client-side validation is enough.”
Incorrect. Users can modify requests. The server must validate and authorize every sensitive action.
“A hidden form field is secret.”
Incorrect. Hidden fields are visible in browser developer tools and can be changed.
“CORS is authentication.”
Incorrect. CORS is a browser sharing policy. It does not prove who a user is.
“APIs are different from web security.”
Partly wrong. APIs often use the same HTTP foundations, but they add risks around object-level authorization, schemas, rate limits, tokens, and machine clients.
“A WAF fixes insecure code.”
Incorrect. A WAF can reduce some traffic risk, but root-cause fixes still belong in design and code.
“JWTs are always safer than sessions.”
Incorrect. JWTs and sessions are design choices. Security depends on signing, validation, storage, expiration, revocation, audience, issuer, and authorization checks.
“If a button is hidden, the action is protected.”
Incorrect. Hiding a UI element is not authorization. The server must block unauthorized requests.
“Only public websites need web security.”
Incorrect. Internal web apps can be abused by compromised insiders, stolen credentials, malware, or lateral movement.
“Logs are only for debugging.”
Incorrect. Logs are also security evidence for detection, investigation, accountability, and compliance.

15. Summary

Web applications are distributed systems built around clients, servers, URLs, DNS, HTTP, HTTPS, browsers, cookies, sessions, APIs, databases, and logs. The browser sends requests, the server returns responses, and application logic decides what data and actions are allowed.

The web is powerful because it uses common protocols and reusable components. It is risky because many trust boundaries meet in one place: user input, browser state, authentication, authorization, server logic, database queries, third-party scripts, APIs, and cloud services.

The most important beginner security rule is simple: the server must enforce security decisions. Browsers help with security boundaries, but users and attackers can modify requests. Later Week 6 lessons will build on this foundation to study sessions, web attacks, API security, and secure design in detail.

16. Glossary

API
Application Programming Interface; a way for software systems to communicate.
Backend
The server-side part of an application, including business logic and data access.
Browser
A client program that retrieves, renders, and runs web content.
CDN
Content Delivery Network; distributed infrastructure that caches or serves content near users.
Client
A system that sends requests to a server.
Cookie
Small data stored by the browser and sent with matching requests.
CORS
Cross-Origin Resource Sharing; a browser mechanism for controlled cross-origin access.
DNS
Domain Name System; translates domain names into network addresses.
Frontend
The browser-facing part of a web application.
Header
Metadata field in an HTTP request or response.
HTTP
Hypertext Transfer Protocol; the main request/response protocol of the web.
HTTPS
HTTP over TLS.
JSON
JavaScript Object Notation; a text format for structured data.
Origin
The combination of scheme, host, and port.
Reverse Proxy
A server that receives requests and forwards them to backend services.
Same-Origin Policy
A browser security policy limiting how scripts from one origin interact with another origin.
Server
A system that receives requests and sends responses.
Session
State that links multiple requests to a user or interaction.
Status Code
A numeric HTTP result code such as 200, 403, 404, or 500.
TLS
Transport Layer Security; cryptographic protocol used by HTTPS.
Token
A credential-like value representing authentication or authorization state.
Trust Boundary
A point where data crosses between different trust levels.
URL
Uniform Resource Locator; an address for a web resource.
WAF
Web Application Firewall; a control that inspects web traffic for policy violations or attack patterns.
Web Application
Software accessed through a browser or HTTP client.

18. Practice Lab

Title

Local Web Request, Cookie, and Log Inspection Lab

Prerequisites

  • Linux, macOS, or Windows with Python 3.
  • A browser with developer tools.
  • Basic terminal skills from Week 0.
  • No internet-facing testing.

Objectives

  • Run a localhost-only web server.
  • Inspect HTTP requests and responses.
  • Observe query strings, headers, status codes, and bodies.
  • Practice safe output encoding.
  • Identify trust boundaries in a tiny local web application.
  • Write a short security observation report.

Safety rules

  • Use only 127.0.0.1 or localhost.
  • Do not test public websites.
  • Do not intercept other people's traffic.
  • Do not collect real credentials, tokens, or personal data.
  • Use fictional names and fake data only.

Step 1: Create the local server

cat > local_web_demo.py <<'PY'
from http.server import BaseHTTPRequestHandler, HTTPServer
from html import escape
from urllib.parse import urlparse, parse_qs
from datetime import datetime, timezone

class DemoHandler(BaseHTTPRequestHandler):
    def log_message(self, format, *args):
        timestamp = datetime.now(timezone.utc).isoformat()
        print(f"{timestamp} {self.client_address[0]} {self.command} {self.path} {format % args}")

    def do_GET(self):
        parsed = urlparse(self.path)
        params = parse_qs(parsed.query)
        name = params.get("name", ["student"])[0]
        safe_name = escape(name)

        if parsed.path == "/health":
            body = "OK"
            content_type = "text/plain; charset=utf-8"
        else:
            body = f"""<!doctype html>
<html lang="en"><head><meta charset="utf-8"><title>Local Demo</title></head>
<body>
  <h1>Hello, {safe_name}</h1>
  <p>This is a localhost-only learning page.</p>
</body></html>"""
            content_type = "text/html; charset=utf-8"

        body_bytes = body.encode("utf-8")
        self.send_response(200)
        self.send_header("Content-Type", content_type)
        self.send_header("Content-Length", str(len(body_bytes)))
        self.send_header("Set-Cookie", "demo_session=fake-local-value; HttpOnly; SameSite=Lax")
        self.end_headers()
        self.wfile.write(body_bytes)

if __name__ == "__main__":
    server = HTTPServer(("127.0.0.1", 8000), DemoHandler)
    print("Open http://127.0.0.1:8000/?name=student")
    server.serve_forever()
