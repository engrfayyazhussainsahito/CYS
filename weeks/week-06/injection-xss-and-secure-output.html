Lesson 6.3: Injection, XSS, and Secure Output

This lesson explains injection from first principles: untrusted data crosses a boundary, enters an interpreter, and changes meaning. You will learn SQL injection concepts, cross-site scripting, context-aware output encoding, parameterized queries, safe rendering, and defensive testing using only local toy examples.

Difficulty levelBeginner → MS foundation
Required backgroundLessons 4.3, 6.1, and 6.2 recommended; prerequisites are reintroduced

1. Introduction

Injection happens when data supplied by a user, browser, API client, file, database, or other external source is accidentally treated as instructions by an interpreter. An interpreter is any component that reads structured text and performs actions: a database engine interprets SQL, a browser interprets HTML and JavaScript, a shell interprets commands, and a template engine interprets markup.

The beginner intuition is simple: an application builds a sentence for another system to understand. If the application mixes user words directly into that sentence, the user may change the sentence's meaning. Secure software keeps structure and data separate.

This topic exists because modern applications constantly pass data across boundaries. A search box becomes a database query. A profile name becomes HTML. A URL parameter becomes a log line. A JSON field becomes business logic. Every boundary is an opportunity for confusion unless the software clearly defines what is allowed and how output should be represented.

Why it matters

  • Injection flaws can expose, modify, or destroy data.
  • XSS can run attacker-controlled script in a user's browser context.
  • Injection often bypasses business logic because the interpreter sees a different command than the developer intended.
  • These weaknesses are common enough to appear in major vulnerability taxonomies and web security guidance.
  • Defenses are usually design-level habits: parameterized queries, context-aware encoding, safe frameworks, validation, least privilege, logging, and testing.

Where it is used

Injection prevention is used in web applications, APIs, admin dashboards, search pages, reporting systems, content management systems, cloud consoles, identity portals, payment systems, healthcare portals, and any software that accepts input then generates commands, queries, pages, or serialized output.

Who uses it

  • Developers use parameterized queries, safe templates, encoders, validators, and secure APIs.
  • Security engineers review data flows, dangerous sinks, output contexts, and unsafe dynamic command construction.
  • SOC analysts investigate injection attempts, web application firewall alerts, unusual database errors, and script-like input patterns.
  • Researchers study browser parsing, sanitization, template injection, automatic vulnerability discovery, and exploit mitigation.
  • Cloud and platform teams configure WAFs, logging, database permissions, and CI/CD security tests.
Safety boundary: this lesson does not teach attacking real websites. All examples use toy strings, local SQLite databases, and localhost-only rendering. Do not test injection or XSS against systems without explicit written authorization.

3. Prerequisites

HTTP and web pages

A browser sends a request to a server. The server sends a response. A response may contain HTML, JavaScript, CSS, JSON, images, or headers. The browser interprets these materials to show a page and run scripts.

HTML and JavaScript

HTML describes document structure. JavaScript is code that browsers can run. If untrusted user content is inserted into a page without safe encoding, the browser may interpret it as markup or script instead of harmless text.

Databases and SQL

A database stores structured data. SQL, or Structured Query Language, is a language used to ask relational databases for data or to change data. SQL is an interpreter language: the database engine parses a query and executes it.

Strings and concatenation

A string is text. Concatenation joins strings together. Unsafe injection often appears when developers build commands by concatenating trusted code with untrusted input.

Input validation and output encoding

Input validation checks whether data has the allowed shape, type, length, range, or format. Output encoding changes special characters into a representation that is safe for a specific output context.

Trust boundary

A trust boundary is a point where data moves from a less trusted area to a more trusted area. User input, HTTP headers, cookies, uploaded files, third-party API results, database records, and message-queue events should all be treated as potentially untrusted until handled safely.

4. Core Concepts

4.1 The central rule

Data must not become code. In secure systems, user data remains data even when it is stored, queried, displayed, logged, transformed, or sent to another service.

4.2 Interpreter

An interpreter reads structured input and performs actions based on that structure. SQL engines, browsers, shells, LDAP filters, XPath processors, template engines, and command-line parsers are interpreters. Injection is possible when untrusted data reaches an interpreter without separation or correct representation.

4.3 Source

A source is where data enters the application. Examples include form fields, URL query parameters, HTTP headers, cookies, JSON bodies, file uploads, database records, message queues, environment variables, and third-party API responses.

4.4 Sink

A sink is where data is used in a sensitive context. Examples include SQL execution, HTML rendering, JavaScript generation, shell command execution, file path construction, template evaluation, redirect URLs, log writing, and deserialization.

4.5 Tainted data

Tainted data means data whose trustworthiness is not yet established. Treating data as tainted does not mean the user is malicious; it means the program must prove the data is safe for each use.

4.6 Context

A context is the environment where output appears. The same text may be safe in one context and unsafe in another. For example, plain text displayed between HTML tags, text inside an HTML attribute, a URL value, a JavaScript string, and CSS all have different parsing rules.

4.7 Injection root cause

Injection occurs when the program fails to separate the instructions it intends from the data it receives. The interpreter then parses a combined string and may treat user-controlled characters as syntax.

4.8 SQL injection concept

SQL injection occurs when user-controlled data changes the meaning of a database query. The secure design is to send the SQL structure separately from the parameter values. This is called parameterization or using a prepared statement.

4.9 Prepared statements

A prepared statement defines the SQL code first, then supplies data values separately. The database treats parameters as values, not as SQL syntax. This is the primary defense against SQL injection for ordinary data values.

4.10 Parameterization analogy

Imagine a form where the database query has blank fields. The program says: "Find user where email equals [blank]." The database knows that the blank is only data. Even if the value contains characters that look meaningful in SQL, those characters stay inside the value.

4.11 Escaping is not the same as parameterization

Escaping changes special characters so an interpreter treats them as literal data. Escaping can be necessary in some contexts, but SQL parameterization is safer and less error-prone than manually escaping SQL strings.

4.12 Stored procedures

Stored procedures are database-side routines. They can be safe when they use parameters correctly. They can still be unsafe if they build dynamic SQL by concatenating untrusted values.

4.13 Object-relational mappers

An object-relational mapper, or ORM, maps programming-language objects to database records. ORMs often use parameterization for common operations. They do not eliminate injection risk if developers use raw query features unsafely.

4.14 XSS concept

Cross-site scripting, or XSS, occurs when untrusted data is included in a web page in a way that the browser interprets as active content. XSS is not only about stealing cookies. It can perform actions as the user, read page data, modify the page, abuse browser APIs, or pivot into account workflows.

4.15 Reflected, stored, and DOM-based XSS

4.16 Output encoding

Output encoding converts characters that have special meaning in a context into safe representations. In HTML text, for example, the less-than character must be represented so the browser displays it as text rather than parsing it as a tag.

4.17 Context-specific encoding

There is no universal encoder for all web contexts. HTML text encoding, HTML attribute encoding, URL encoding, JavaScript string encoding, and CSS encoding have different rules. Using the wrong encoder can leave a vulnerability.

4.18 Dangerous contexts

Some contexts are too dangerous for untrusted data, even with attempted encoding. Examples include raw script blocks, event-handler attributes, raw CSS with complex syntax, HTML comments, and places where browser parsing rules are ambiguous. Secure design avoids putting untrusted data in dangerous contexts.

4.19 Sanitization

Sanitization attempts to remove or allow only selected safe parts from rich input, such as user-submitted HTML. Sanitization is hard because browsers parse HTML in complex ways. Prefer plain text when possible. If rich text is required, use a well-maintained sanitizer and a restrictive allowlist.

4.20 Validation

Validation checks whether input is acceptable for the business rule: type, range, length, format, and allowed values. Validation reduces bad data but does not replace output encoding because valid data can still contain characters that need encoding in a particular output context.

4.21 Canonicalization

Canonicalization converts input to a standard form before validation. For example, URLs, Unicode text, file paths, and encoded characters may have multiple representations. Validating one representation and then using another can create bypasses.

4.22 DOM sinks

A DOM sink is a browser-side API that can turn a string into markup, script, URL navigation, or other behavior. Safer APIs set text content as text. Riskier APIs interpret strings as HTML or code.

4.23 Safe template engines

Many modern web frameworks automatically escape variables in templates. This is useful, but developers can often disable escaping or use raw HTML features. Framework auto-escaping is a safety net, not permission to ignore context.

4.24 Content Security Policy as defense in depth

Content Security Policy, or CSP, can limit which scripts run and from where. CSP reduces impact and can help detect policy violations, but it does not fix the root cause of XSS. The root fix is correct output handling and safer DOM usage.

4.25 Trusted Types

Trusted Types is a browser security mechanism that can restrict dangerous DOM injection sinks to values produced by approved policies. It is useful for large client-side applications, but it requires careful migration and policy design.

4.26 Command injection

Command injection occurs when untrusted data changes the meaning of an operating-system command. The beginner defense is simple: avoid shell commands when a library API can do the job. If a command must run, pass arguments as a list, avoid shell interpretation, validate allowlisted options, and apply least privilege.

4.27 LDAP, XPath, NoSQL, and template injection

Injection is a family of problems. LDAP filters, XPath queries, NoSQL query objects, server-side templates, expression languages, and serialization formats can all become injection surfaces. The pattern is the same: untrusted data reaches an interpreter in the wrong form.

4.28 Log injection

Log injection occurs when untrusted data writes misleading lines, fake fields, control characters, or hidden content into logs. Defensive logging uses structured formats, escaping, length limits, and avoids secrets.

4.29 Second-order injection

Second-order injection occurs when input seems harmless at entry, is stored, and later becomes dangerous when used in a different context. Stored XSS is a common example. The lesson is that output must be safe at the point of use, not only at the point of collection.

4.30 Least privilege for damage reduction

Parameterization and encoding prevent many injection flaws. Least privilege limits impact when a bug remains. A web application's database account should not have unnecessary administrative privileges. A rendering service should not have broad network and file access if it only needs to display content.

4.31 Detection thinking

Detection is not just searching for scary characters. Good detection looks for impossible workflows, database errors, unusual query patterns, unexpected script-like output, WAF alerts, CSP violation reports, sudden data access spikes, and application exceptions near input-handling code.

4.32 Best beginner model

For every feature, ask four questions: Where did this data come from? Where will it go? Which interpreter will read it? How do we keep it as data?

7. Mathematical Foundations

7.1 Injection is a language-boundary problem

Think of an interpreter as accepting strings from a formal language. SQL has a grammar. HTML has a grammar. JavaScript has a grammar. Injection happens when untrusted data changes how the string is parsed by that grammar.

7.2 Structure/data separation notation

Let S be trusted structure and D be untrusted data. Unsafe concatenation sends one string:

parse(S + D)

The parser sees a single sequence of characters. Safe parameterization sends two different things:

execute(structure = S, parameters = [D])

The database parser parses S as SQL and treats D as a value.

7.3 Encoding as a mapping

An encoder is a function E_context(x) that maps unsafe characters to safe representations for one context. For HTML text, conceptually:

E_html_text("<") = "&lt;"
E_html_text(">") = "&gt;"
E_html_text("&") = "&amp;"

The important detail is the subscript: E_html_text is not the same as E_url or E_js_string.

7.4 Risk scoring example

A simple internal prioritization score might be:

Risk priority = Exposure × Impact × Exploitability × Data sensitivity

Worked example for a fictional student comments page:

FactorValueReason
Exposure4Publicly reachable feature.
Impact4Could affect authenticated users.
Exploitability3Requires a stored comment path.
Data sensitivity3Student account data may be accessible.
Risk priority = 4 × 4 × 3 × 3 = 144

This is not a universal formula. It is a teaching model showing why context matters. A low-severity-looking output bug can become urgent when exposure and data sensitivity are high.

7.5 False positives and false negatives

Security tests for injection can produce false positives and false negatives. A false positive says a bug exists when it does not. A false negative misses a real bug. Mature programs combine code review, automated testing, runtime protections, and manual reasoning.

The following examples use only local data. They are designed to teach defensive coding, not exploitation.

# Safe local HTML escaping example.
# Run with: python safe_html.py

from html import escape

# In real applications, this value could come from a form or API.
user_display_name = 'Alice & Bob '

# escape() converts special HTML characters into safe text representations.
safe_name = escape(user_display_name)

# The browser would display the text instead of treating it as markup.
html_page = f"<p>Welcome, {safe_name}</p>"

print(html_page)

8.6 Cloud examples

  • Store secrets in a secrets manager, not in web templates or client-side JavaScript.
  • Use CI/CD checks to reject unsafe template patterns, raw SQL concatenation, and dangerous DOM sinks.

9. Code Examples

9.1 Safe SQLite parameterization

# safe_sqlite_demo.py
# Purpose: demonstrate safe parameterized queries using a local SQLite database.
# Run with: python safe_sqlite_demo.py

import sqlite3

# Connect to an in-memory database. Nothing is exposed on the network.
connection = sqlite3.connect(':memory:')

# Create a cursor object. A cursor lets us send SQL statements to the database.
cursor = connection.cursor()

# Create a tiny table for the lesson.
cursor.execute('CREATE TABLE students (id INTEGER PRIMARY KEY, email TEXT, name TEXT)')

# Insert sample data using placeholders. The question marks are parameter slots.
cursor.execute(
    'INSERT INTO students (email, name) VALUES (?, ?)',
    ('alice@example.test', 'Alice')
)

# This value represents data from a user. It must remain data.
submitted_email = 'alice@example.test'

# SAFE: SQL structure and parameter value are separate.
cursor.execute(
    'SELECT id, email, name FROM students WHERE email = ?',
    (submitted_email,)
)

# Fetch and print the result.
row = cursor.fetchone()
print(row)

# Close the connection when finished.
connection.close()

Line-by-line explanation

  • import sqlite3 loads Python's built-in SQLite interface.
  • connect(':memory:') creates a temporary local database in memory.
  • cursor() creates an object for executing SQL.
  • CREATE TABLE defines a small table.
  • VALUES (?, ?) uses placeholders instead of string concatenation.
  • The tuple ('alice@example.test', 'Alice') supplies data values separately.
  • The SELECT statement uses a placeholder for the email value.
  • (submitted_email,) is a one-item tuple; the comma matters in Python.
  • The database treats the submitted email as a value, not SQL syntax.

9.2 Unsafe pattern to recognize, not copy

# Do not use this pattern in real applications.
# It combines SQL structure and user data into one string.
query = "SELECT id, email, name FROM students WHERE email = '" + submitted_email + "'"

This pattern is unsafe because the database receives one combined string. A reviewer should request parameterization instead.

9.3 Safe HTML output

# safe_output_demo.py
# Purpose: show how plain text can be safely represented in HTML.

from html import escape

def render_profile(display_name: str) -> str:
    """Return a tiny HTML fragment with a safely escaped display name."""

    # escape() handles characters such as &, <, >, and quotes when quote=True.
    safe_display_name = escape(display_name, quote=True)

    # The escaped value is inserted into HTML text context.
    return f"<h1>Profile</h1><p>Name: {safe_display_name}</p>"

print(render_profile('Alice & Bob '))

9.4 Safer subprocess use

# subprocess_safe_shape.py
# Purpose: show safer command execution shape for local commands.
# Prefer library APIs when possible. If a command is necessary, avoid shell=True.

import subprocess

filename = 'names.txt'

# SAFE SHAPE: arguments are passed as a list, not a shell-parsed string.
result = subprocess.run(
    ['wc', '-l', filename],
    check=True,
    capture_output=True,
    text=True
)

print(result.stdout)

9.5 Common mistakes

  • Using string concatenation to build SQL.
  • Assuming validation alone prevents XSS.
  • Using HTML encoding inside JavaScript context.
  • Trusting stored database content just because it came from your database.
  • Disabling framework auto-escaping to make formatting easier.
  • Placing tokens or secrets into JavaScript-readable locations.
  • Relying only on a WAF while leaving vulnerable code unchanged.
  • Logging raw request bodies without redaction or size limits.

9.6 Why secure coding matters

Injection prevention is cheaper in design and code review than after deployment. A single unsafe sink can affect many users because web applications reuse templates, APIs, database helpers, and shared components.

10. Security Perspective

Common attacks

  • SQL injection against unsafe query construction.
  • Reflected XSS through unsafe response rendering.
  • Stored XSS through unsafe display of database content.
  • DOM-based XSS through unsafe browser-side sinks.
  • Command injection through shell-parsed strings.
  • Template injection through unsafe template evaluation.
  • Log injection through unescaped newline or control characters.
  • Second-order injection where stored data becomes dangerous later.

Common vulnerabilities

  • Untrusted input reaches an interpreter without parameterization or encoding.
  • Output context is misunderstood.
  • Framework auto-escaping is disabled.
  • Raw SQL or raw HTML features are used without review.
  • Stored data is treated as trusted.
  • Database accounts have excessive privileges.
  • Errors reveal stack traces, SQL fragments, or implementation details.
  • Tests cover normal input but not boundary cases.

Defensive strategies

  • Use prepared statements for database values.
  • Use safe template engines with auto-escaping enabled.
  • Encode output for the exact context.
  • Validate input for business rules.
  • Avoid dangerous output contexts for untrusted data.
  • Use safe DOM APIs such as text-setting APIs instead of HTML-parsing APIs where possible.
  • Apply least privilege to databases, services, and file access.
  • Use CSP as defense in depth.
  • Use SAST, DAST, dependency checks, and manual code review.
  • Add regression tests for fixed bugs.

Detection methods

  • Review application errors near database and template code.
  • Monitor WAF alerts and correlate them with application logs.
  • Collect CSP violation reports where privacy and operational constraints allow.
  • Look for unexpected script execution reports, unusual query shapes, and database error spikes.
  • Track abnormal data access volume after suspicious web requests.
  • Use code scanning to find dangerous sinks and string-built queries.

Mitigation

  • Fix the unsafe code path first.
  • Patch frameworks and libraries.
  • Rotate exposed credentials if logs or XSS may have leaked tokens.
  • Review database permissions and session invalidation if account impact is possible.
  • Add tests and secure coding rules to prevent recurrence.
  • Use WAF or CSP as temporary shielding when code changes need deployment time.

Best practices

  • Keep data and code separate.
  • Prefer allowlists over denylists for validation.
  • Prefer parameterization over manual SQL escaping.
  • Prefer text rendering over rich HTML.
  • Prefer safe framework defaults over custom parsing.
  • Assume stored data may still be untrusted.
  • Document every exception to safe-output rules.

11. Industry Perspective

Banks

Banks use injection prevention in customer portals, payment workflows, internal dashboards, fraud systems, and reporting systems. They emphasize secure SDLC gates, strong logging, database least privilege, transaction monitoring, and rapid patching because injection can affect confidentiality, integrity, and financial operations.

Governments

Government systems often process citizen records, benefits, tax data, immigration information, and public forms. Injection prevention is tied to public trust, legal compliance, accessibility, and secure procurement requirements.

Cloud providers

Cloud providers build consoles, APIs, identity systems, documentation platforms, support systems, and telemetry pipelines. They rely heavily on safe frameworks, code scanning, threat modeling, defense in depth, and compartmentalized services.

Healthcare

Healthcare systems protect patient records, clinical workflows, appointment portals, insurance data, and laboratory results. Stored XSS in clinical notes or unsafe query logic in patient lookup features can create privacy and safety risks.

Defense

Defense environments treat injection as both an application-security and mission-assurance issue. Controls include strict change management, testing, code review, network segmentation, high-assurance logging, and least privilege.

Universities

Universities operate student portals, learning systems, research apps, public websites, and identity services. They must balance openness with security. Injection prevention protects grades, transcripts, research data, and accounts.

Large enterprises

Large enterprises often have many legacy applications. Their challenge is not only knowing the best practice but finding every unsafe sink across many teams, frameworks, languages, and vendors.

12. Research Perspective

Injection remains a research area because the underlying problem is broader than one bug type. It combines programming-language design, browser parsing, developer usability, static analysis, dynamic testing, runtime monitoring, and secure framework design.

Current research directions

  • Context-sensitive analysis: finding whether untrusted data reaches a dangerous sink without the correct context-specific defense.
  • Usable security APIs: making encoders, sanitizers, and safe template APIs hard to misuse.
  • Browser-level mitigations: CSP, Trusted Types, isolation boundaries, and safer defaults.
  • Automatic repair: suggesting safe parameterization or output-encoding patches.
  • Hybrid testing: combining static analysis, fuzzing, browser automation, and runtime instrumentation.
  • Template and client-side framework security: analyzing modern single-page application data flows.
  • Machine-learning-assisted review: helping prioritize suspicious sinks while avoiding high false-positive rates.

Open problems

  • Detecting DOM-based XSS reliably in large JavaScript applications.
  • Sanitizing rich user content without breaking usability or missing obscure parser behavior.
  • Securing legacy applications that mix presentation, database access, and business logic.
  • Providing safe defaults across multiple languages and frameworks.
  • Measuring whether security training changes developer behavior over time.
  • Handling AI-generated code that may reproduce unsafe patterns.

Notable research themes

Useful research themes include taint tracking, type systems for safe output, browser enforcement mechanisms, sanitizer correctness, query language design, secure templates, web measurement studies, and developer-centered API design. Students should read papers for ideas and evidence, not copy attack payloads into unauthorized environments.

13. Case Studies

Case Study 1: TalkTalk breach and SQL injection reporting

What happened: TalkTalk disclosed a major breach in 2015 involving customer data. Public reporting and regulatory discussion associated the incident with SQL injection issues in web-facing systems.

Root cause: The broad lesson is that internet-facing data queries need secure design, patching, validation, parameterization, segmentation, monitoring, and strong incident response.

Impact: The breach affected customers and led to financial, operational, and reputational consequences.

Lessons learned: Common web bugs can become enterprise-level incidents when exposed systems hold sensitive data and controls are weak.

Case Study 2: Samy worm and stored XSS

What happened: The Samy worm spread through MySpace profiles in 2005 using client-side scripting behavior and social-network trust relationships.

Root cause: User-controlled profile content was not sufficiently constrained for the browser context.

Impact: The incident demonstrated how stored script-like content can spread quickly when displayed to many users.

Lessons learned: Stored content must be safely rendered each time it appears, and rich user-controlled HTML needs strict controls.

Case Study 3: British Airways Magecart-style script injection

What happened: British Airways reported a 2018 incident where customer payment data was affected by malicious script activity on web properties.

Root cause: Public reports describe web skimming behavior: client-side script paths can become high-value targets.

Impact: The incident affected customer trust and led to regulatory attention.

Lessons learned: XSS and script integrity are not only technical issues; they affect payment security, privacy, third-party scripts, monitoring, and incident response.

Case Study 4: Stored content in internal tools

What happened: In many organizations, internal dashboards display names, tickets, comments, alerts, or logs received from external sources.

Root cause: Teams often trust internal tools too much and forget that external data can flow into them.

Impact: Unsafe rendering in admin tools can expose high-privilege users.

Lessons learned: Internal does not mean safe. Stored data needs safe output handling everywhere it appears.

14. Common Misconceptions

  • Misconception: Input validation alone prevents XSS.
    Correction: Validation helps, but output must still be encoded for the exact browser context.
  • Misconception: A WAF fixes injection.
    Correction: A WAF may detect or block some traffic, but vulnerable code remains vulnerable.
  • Misconception: Stored database data is trusted.
    Correction: Stored data may originally come from users, APIs, imports, or compromised systems.
  • Misconception: Escaping is the same everywhere.
    Correction: HTML, URL, JavaScript, CSS, SQL, shell, and log contexts have different rules.
  • Misconception: Prepared statements are only for login forms.
    Correction: They should be used for database values throughout the application.
  • Misconception: CSP means XSS no longer matters.
    Correction: CSP is defense in depth; the code still needs correct output handling.
  • Misconception: ORMs automatically prevent all injection.
    Correction: Raw query features, unsafe filters, and string-built clauses can still be risky.
  • Misconception: XSS is only about cookie theft.
    Correction: XSS can perform actions, read page data, modify interfaces, and abuse browser trust.
  • Misconception: Only web forms are sources.
    Correction: Headers, cookies, URLs, APIs, files, logs, queues, and third-party data can also be sources.
  • Misconception: Blocking a few characters is enough.
    Correction: Denylists are fragile because parsers, encodings, and contexts are complex.

15. Summary

Injection is interpreter confusion. The application intends some text to be data, but an interpreter treats it as structure or code. SQL injection is prevented primarily by parameterized queries. XSS is prevented primarily by safe rendering and context-aware output encoding, supported by validation, safe DOM APIs, CSP, and secure framework defaults. The durable principle is simple: keep data and instructions separate at every boundary.

16. Glossary

Canonicalization
Converting data into a standard representation before validation or comparison.
Command injection
A weakness where untrusted data changes the meaning of an operating-system command.
Context
The output environment where data appears, such as HTML text, attribute, URL, JavaScript, CSS, SQL, shell, or log.
Context-aware encoding
Encoding selected for the exact output context.
Cross-site scripting
A vulnerability where untrusted data is interpreted as active browser content.
DOM sink
A browser-side API or operation where data can become markup, script, navigation, or executable behavior.
Escaping
Representing special characters so an interpreter treats them as literal data.
Injection
A class of weakness where untrusted data changes the intended meaning of instructions sent to an interpreter.
Interpreter
A component that parses structured input and performs actions, such as a browser, database, shell, or template engine.
Output encoding
Converting special characters into safe representations for a specific output context.
Parameterized query
A database query where SQL structure and data values are supplied separately.
Prepared statement
A database statement whose structure is prepared before parameter values are bound.
Sanitization
Removing or allowing selected content from input, often used for rich text or HTML.
Second-order injection
An injection pattern where data is stored first and becomes dangerous when used later in a different context.
Sink
A sensitive use of data, such as SQL execution, HTML rendering, command execution, logging, or template evaluation.
Source
A place where data enters the application, such as a form, URL parameter, cookie, API body, file, or third-party service.
SQL injection
An injection weakness where untrusted data changes the meaning of a SQL query.
Tainted data
Data treated as untrusted until validated, encoded, parameterized, or otherwise safely handled for its use.
Template injection
A weakness where untrusted data is interpreted as template syntax or expression language.
Trusted Types
A browser mechanism that restricts certain dangerous DOM sinks to approved value types.
