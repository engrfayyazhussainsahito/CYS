Lesson 6.5: Web Security Testing and Secure Deployment

This lesson teaches how to test web applications safely and deploy them securely. You will learn how to plan authorized testing, collect evidence, use automated tools carefully, verify controls, harden deployment configuration, monitor releases, and write professional security reports.

Estimated reading time360–540 minutes
Difficulty levelBeginner → MS foundation
Required backgroundLessons 6.1–6.4 recommended; prerequisites are reintroduced
Practice time6–10 hours

1. Introduction

Web security testing is the disciplined process of checking whether a web application actually enforces its security requirements. Secure deployment is the disciplined process of releasing that application so its runtime environment, configuration, dependencies, secrets, transport security, logs, and rollback plan support those requirements.

Plain English first: building a secure web app is not finished when the code compiles. You also need to check how it behaves when real requests arrive, how it is configured in the environment where it runs, what happens when something fails, and what evidence proves that the release is safe enough to operate.

This topic exists because many web vulnerabilities are not visible from source code alone. A secure-looking function can be deployed with debug mode enabled. A strong authentication system can be weakened by a missing cookie flag. A patched library can be absent from the production container. A TLS certificate can be expired. A scanner can miss business-logic authorization flaws. Testing and deployment controls close the gap between what developers intended and what the running system actually does.

Why this matters

  • Most users experience software through browsers, mobile apps, and APIs.
  • Web apps sit at the boundary between the internet, identity systems, databases, cloud services, and business logic.
  • Deployment mistakes can turn safe code into unsafe production behavior.
  • Security testing without scope can be illegal and harmful.
  • Testing without evidence produces opinions, not engineering confidence.

Where it is used

Banks test payment flows, session controls, account access, fraud paths, and audit logs. Healthcare organizations test patient portals, privacy controls, file uploads, and consent flows. Cloud providers test consoles, APIs, identity boundaries, multi-tenant isolation, and secure defaults. Universities test learning portals and research systems. Governments and defense organizations test mission systems, procurement portals, identity federation, and citizen services.

Who uses it

  • Developers write security tests, fix vulnerabilities, configure headers, and maintain release pipelines.
  • Application security engineers design testing strategy, review findings, and verify remediation.
  • Penetration testers conduct scoped assessments under rules of engagement.
  • DevOps and platform engineers manage deployment gates, TLS, secrets, container settings, logs, and rollback.
  • SOC analysts monitor production behavior after release.
  • Managers and auditors need evidence that security controls were checked before and after deployment.
Safety boundary: this lesson does not authorize testing any real system. All labs must be performed only against local toy applications or explicit written-scope environments. Do not scan, crawl, fuzz, stress, exploit, or bypass controls on public websites, university systems, employer systems, cloud accounts, or third-party APIs unless you have written permission.

3. Prerequisites

This lesson reintroduces required ideas, but the following concepts help:

HTTP
The request-response protocol used by browsers and many APIs.
HTTPS
HTTP protected by Transport Layer Security so network observers cannot casually read or modify traffic.
Cookie
A small name-value value stored by the browser and sent to matching sites, often used for sessions.
Session
Server-side or token-based state connecting several requests to the same authenticated interaction.
Authorization
The decision about whether a subject may perform an action on an object.
Deployment
The process of making software run in an environment where users or systems can use it.
Evidence
Verifiable information supporting a claim: logs, screenshots, request/response examples, code diffs, test output, hashes, configuration snapshots, and timestamps.

If those terms feel unfamiliar, review Lessons 6.1 through 6.4. You can still proceed, but move slowly and treat the examples as mental models rather than commands to run on real targets.

4. Core Concepts

4.1 Testing vs deployment

Testing asks: does the application behave securely when exercised? Deployment asks: is the application released with secure configuration, runtime controls, secrets, logs, network exposure, and recovery procedures? A bug can be in code, configuration, infrastructure, process, or assumptions between teams.

4.2 Scope

Scope defines what is allowed. It should state targets, dates, accounts, environments, methods, test limits, contact points, data handling rules, and prohibited actions. Scope protects users, systems, and testers.

4.3 Rules of engagement

Rules of engagement are the operating rules for a test. They answer practical questions: Which domains? Which IP ranges? What time window? Are scanners allowed? Are denial-of-service tests prohibited? Are phishing tests prohibited? Who receives urgent findings?

4.4 Test environments

A beginner should use a local lab. A professional team usually uses development, test, staging, and production environments. Staging should resemble production but use test data. Production testing requires stronger safeguards because real users, real data, and real availability are involved.

4.5 Security requirements become tests

A security requirement is useful only if it can be verified. "Only a student can view their transcript" becomes tests such as: Alice can read Alice's transcript; Alice cannot read Bob's transcript; anonymous users receive no transcript; staff can read only within assigned role; logs record denied attempts without exposing secrets.

4.6 Happy-path testing is insufficient

Happy-path tests check expected successful use. Security also needs negative tests: invalid input, forbidden object access, expired sessions, missing CSRF tokens, wrong tenant IDs, oversized uploads, malformed JSON, unexpected methods, and direct requests to hidden endpoints.

4.7 SAST

Static Application Security Testing inspects source code, bytecode, or binaries without needing the application to be running. It can find dangerous functions, missing validation, hardcoded secrets, risky dependencies, and coding patterns. It often needs human review because it may produce false positives.

4.8 DAST

Dynamic Application Security Testing tests a running application from the outside, usually by sending HTTP requests and observing responses. It can find missing headers, error disclosure, authentication problems, injection behavior, and misconfiguration. It may miss code paths it cannot reach.

4.9 IAST

Interactive Application Security Testing combines runtime instrumentation with testing. It observes how the application handles requests internally. It can provide useful evidence but requires controlled environments and careful deployment.

4.10 SCA

Software Composition Analysis inspects third-party dependencies. It helps identify known vulnerable packages, licenses, and transitive dependency risks. SCA does not prove an application is exploitable; it gives information that must be triaged with exposure, version, reachability, and compensating controls.

4.11 Manual testing

Manual testing uses human reasoning to test workflows, roles, business logic, authorization, abuse cases, and chained weaknesses. Many important web issues are logic flaws that scanners cannot reliably understand.

4.12 Test evidence

A finding should include evidence: environment, timestamp, account type, request, response, expected result, actual result, impact, affected asset, reproduction steps in authorized scope, recommended fix, and retest result. Tool output alone is not a complete finding.

4.13 False positives and false negatives

A false positive is a reported issue that is not actually present as claimed. A false negative is a missed issue. Automated testing reduces effort but cannot remove judgment.

4.14 Severity vs priority

Severity describes technical impact. Priority describes how soon to fix it based on exploitability, exposure, asset value, business deadlines, legal obligations, and available mitigations. A medium technical issue on a public authentication path may be urgent; a high issue in a disabled internal feature may be lower priority after validation.

4.15 Deployment configuration

Deployment configuration includes environment variables, secret sources, debug settings, database permissions, file permissions, TLS certificates, allowed origins, cookie settings, headers, logging, rate limits, feature flags, and service accounts. Configuration should be reviewed, versioned where safe, and separated from real secrets.

4.16 Secure headers

HTTP response headers can ask browsers to enforce security rules. Examples include Strict-Transport-Security, Content-Security-Policy, X-Frame-Options or frame controls in CSP, Referrer-Policy, and safe cache-control headers for sensitive pages. Headers are defense in depth, not replacements for server-side fixes.

4.17 HSTS

HTTP Strict Transport Security, or HSTS, tells browsers to access a site over HTTPS for a specified time after a valid HTTPS response with the header. HSTS helps reduce downgrade and TLS-stripping risk, but it must be configured carefully, especially with subdomains and preload choices.

4.18 CSP

Content Security Policy, or CSP, lets a site restrict where scripts, images, styles, frames, and other resources may load from. CSP can reduce the impact of some injection bugs, but unsafe policies such as allowing broad inline script execution weaken it.

4.19 TLS deployment

TLS deployment includes certificate validity, hostname matching, protocol version, cipher choices, automatic renewal, redirect behavior, HSTS strategy, and monitoring for expiration. The user should never be trained to ignore browser certificate warnings.

4.20 Cookie deployment checks

Session cookies should usually use Secure, HttpOnly, and suitable SameSite settings. Sensitive cookies should have narrow domain and path scope where possible. A secure application should not rely on JavaScript-accessible storage for high-value session secrets without a careful threat model.

4.21 Debug mode and error pages

Debug mode is useful during development but dangerous in production. Detailed stack traces may reveal code paths, secrets, database names, filesystem paths, dependency versions, internal IPs, or configuration values. Production errors should be user-safe while detailed diagnostics go to protected logs.

4.22 File upload deployment checks

File upload features need size limits, type validation, storage isolation, malware scanning where appropriate, randomized storage names, access controls, content-disposition decisions, and safe processing. Upload directories should not execute uploaded files as server-side code.

4.23 Rate limiting and resource limits

Web systems need limits on login attempts, password reset requests, API requests, expensive queries, uploads, downloads, and export jobs. Limits protect availability and cost, but they must be designed so legitimate users are not unnecessarily blocked.

4.24 WAF strengths and limits

A Web Application Firewall can detect or block some known attack patterns, protocol anomalies, and abusive traffic. A WAF cannot understand every business rule, cannot fix missing authorization, and can be bypassed or misconfigured. Treat it as one layer.

4.25 Release gates

A release gate is a decision point before deployment. It might require passing tests, dependency checks, secret scans, container scans, approval from service owners, successful staging validation, and a documented rollback plan.

4.26 Canary release

A canary release exposes a small amount of traffic or a small user group to a new version first. Monitoring can reveal errors, latency, security events, or unexpected behavior before full rollout.

4.27 Blue-green deployment

Blue-green deployment maintains two production-like environments. One serves traffic while the other receives the new release. Traffic can switch after validation. This can improve rollback but requires careful data, session, and configuration handling.

4.28 Rollback

Rollback returns to a known good version. A rollback plan should include application version, database migration strategy, configuration version, feature flags, cache behavior, monitoring, and decision authority.

4.29 Monitoring after deployment

Deployment is not finished at release. Monitor authentication failures, authorization denials, 4xx/5xx spikes, latency, unusual user agents, upload errors, CSP violations, suspicious object enumeration, new dependency alerts, certificate expiration, and WAF events.

4.30 Security reporting

A professional report is not a list of scary words. It should be accurate, reproducible in authorized scope, prioritized, and useful to developers. Include executive summary, scope, methodology, findings, evidence, impact, remediation, retest status, and residual risk.

4.31 The core testing question

For each feature, ask: What should happen? What should never happen? How do I safely prove both? What evidence would convince a skeptical engineer?

7. Mathematical Foundations

7.1 Basic risk prioritization

A simple teaching formula is:

Risk score = Likelihood × Impact

If likelihood is 4 out of 5 and impact is 5 out of 5, the score is 20 out of 25. This is not universal truth; it is a simplified model for comparison.

7.2 Adding exposure and confidence

For web testing, a more useful classroom model is:

Priority = Severity × Exposure × Confidence
  • Severity: how bad the impact would be.
  • Exposure: how reachable the issue is.
  • Confidence: how strong the evidence is.

Example: a possible missing header may have medium severity, high exposure, but low confidence if unverified. A confirmed object-level authorization flaw may have high severity, high exposure, and high confidence.

7.3 Availability calculation

If a service is expected to be available 99.9% of a 30-day month:

30 days × 24 hours/day = 720 hours
0.1% allowed downtime = 0.001 × 720 = 0.72 hours
0.72 hours × 60 = 43.2 minutes

Security deployment must consider availability because aggressive controls, broken certificates, bad WAF rules, or failed migrations can create outages.

7.4 Rate-limit reasoning

If a password reset endpoint allows 5 requests per user per hour and 100 requests per IP per hour, then a single user should not receive unlimited reset messages from one client. Rate limits should be designed around user protection, business workflows, and attacker work factor.

7.5 Scanner coverage

Coverage can be described as:

Coverage = tested features / known features

If a scanner reaches 20 endpoints but the OpenAPI specification lists 50 endpoints, raw scanner output covered only 40% of listed endpoints. Coverage does not equal security, but it exposes gaps in testing.

8. Practical Examples

8.1 Linux: check local response headers

# Only use this against your own local lab server.
curl -I http://127.0.0.1:8000/

This asks the local server for response headers only. It is useful for checking whether headers such as Content-Security-Policy or Strict-Transport-Security are present in a lab.

8.2 Windows PowerShell: check a local endpoint

# Only use this against your own local lab server.
$response = Invoke-WebRequest -Uri "http://127.0.0.1:8000/" -Method Head
$response.Headers

8.3 Python: safe local header checker

from urllib.request import Request, urlopen
from urllib.error import URLError

TARGET = "http://127.0.0.1:8000/"  # local lab only

required_headers = [
    "Content-Security-Policy",
    "X-Content-Type-Options",
    "Referrer-Policy",
]

request = Request(TARGET, method="HEAD")

try:
    with urlopen(request, timeout=3) as response:
        headers = response.headers
        for name in required_headers:
            if name in headers:
                print(f"[present] {name}: {headers[name]}")
            else:
                print(f"[missing] {name}")
except URLError as error:
    print(f"Could not reach local lab server: {error}")

8.4 Network example: HTTPS redirect thinking

A deployment should usually redirect public HTTP traffic to HTTPS. In a local lab, you can model this with a simple note or test server. Do not test third-party redirect behavior without permission.

8.5 Web example: negative authorization test

Expected rule:
  Alice may read /transcripts/alice-demo
  Alice may not read /transcripts/bob-demo

Positive test:
  Alice reads Alice's transcript -> 200 OK

Negative test:
  Alice requests Bob's transcript -> 403 Forbidden or safe 404

Evidence:
  Request ID, account used, endpoint, expected result, actual result, timestamp

8.6 Cloud example: deployment review

For a cloud-hosted web app, review public exposure, TLS certificate automation, secret source, service account privileges, object storage permissions, database network access, logs, dependency alerts, backup and restore tests, and rollback procedure. Use only your own training account or an assigned lab environment.

9. Code Examples

9.1 Minimal local web server with safe headers

from http.server import SimpleHTTPRequestHandler, HTTPServer

class SecureLearningHandler(SimpleHTTPRequestHandler):
    def end_headers(self):
        # Prevent some browsers from MIME-sniffing a response as another type.
        self.send_header("X-Content-Type-Options", "nosniff")

        # Reduce how much URL information is sent in the Referer header.
        self.send_header("Referrer-Policy", "no-referrer")

        # Basic frame protection for older deployments.
        self.send_header("X-Frame-Options", "DENY")

        # A restrictive learning CSP. Real apps need policies tuned to their resources.
        self.send_header("Content-Security-Policy", "default-src 'self'; frame-ancestors 'none'")

        # Finish the headers by calling the parent method.
        super().end_headers()

server = HTTPServer(("127.0.0.1", 8000), SecureLearningHandler)
print("Local lab server: http://127.0.0.1:8000")
server.serve_forever()

Line-by-line explanation: the code imports a simple HTTP server, creates a handler class, adds response headers before the response ends, binds only to 127.0.0.1, and serves local files. Binding to localhost keeps the demo off the network.

9.2 Common mistake: binding to all interfaces

# Riskier for a beginner lab: this listens on all network interfaces.
# Do not use this unless you understand your network exposure.
server = HTTPServer(("0.0.0.0", 8000), SecureLearningHandler)

0.0.0.0 means the service may be reachable from other machines depending on firewall and network settings. Beginner labs should prefer 127.0.0.1.

9.3 Simple deployment checklist in Python

checklist = {
    "debug_mode_disabled": True,
    "real_secrets_absent_from_repo": True,
    "dependency_scan_reviewed": True,
    "session_cookie_secure": True,
    "authorization_negative_tests_pass": True,
    "rollback_plan_written": False,
}

for item, passed in checklist.items():
    status = "PASS" if passed else "BLOCK"
    print(f"{status}: {item}")

if not all(checklist.values()):
    raise SystemExit("Release gate failed: fix blocked items before deployment.")

This toy script shows the release-gate idea. Real pipelines should use tested tooling, signed artifacts, clear ownership, and auditable results.

9.4 Safe pytest-style negative test example

def can_read_transcript(user, transcript):
    return transcript["owner"] == user["id"] or user["role"] == "advisor"


def test_student_cannot_read_another_students_transcript():
    alice = {"id": "alice", "role": "student"}
    bobs_transcript = {"owner": "bob", "grade": "A"}
    assert can_read_transcript(alice, bobs_transcript) is False

This is not an attack. It is a local unit test that proves a forbidden action is denied.

9.5 Common secure coding mistakes in deployment

  • Debug mode left enabled in production.
  • Real secrets committed to Git.
  • Session cookies missing Secure or HttpOnly.
  • Error pages exposing stack traces.
  • Authorization tested only through the user interface, not direct API requests.
  • Scanner results accepted without verification.
  • Production release done without rollback plan.

10. Security Perspective

Common attacks and failures

  • Authentication bypass through misconfiguration.
  • Missing object-level authorization.
  • Session cookie theft due to weak flags or XSS.
  • Debug console exposure.
  • Stack trace information disclosure.
  • Injection through unvalidated inputs.
  • Dependency vulnerabilities left untriaged.
  • Secrets exposed in repositories, containers, logs, or client JavaScript.
  • Clickjacking due to missing frame controls.
  • TLS downgrade or expired certificates.
  • Oversized uploads or expensive requests causing availability issues.
  • Open CORS policy exposing sensitive APIs to browser-based abuse.

Defensive strategies

  • Written scope and rules of engagement for testing.
  • Threat models and security requirements before implementation.
  • Unit, integration, API, and negative authorization tests.
  • SAST, DAST, SCA, secret scanning, and manual review.
  • Secure default configuration.
  • Security headers and strong cookie settings.
  • TLS automation and certificate monitoring.
  • Least-privilege service accounts.
  • Central logs, request IDs, and alerting.
  • Canary releases and rollback plans.

Detection methods

  • Many 401/403/404 responses from one subject or client.
  • Sequential object ID access.
  • New 500 error spikes after deployment.
  • Unexpected CSP violation reports.
  • Unusual export volumes.
  • New dependency alerts.
  • Certificate expiration warnings.
  • WAF anomaly spikes.
  • Login failures followed by successful access from unusual locations.

Mitigation and best practices

Fix root causes in application logic, but use layers: headers, TLS, cookie flags, dependency checks, WAF, rate limits, isolation, monitoring, backups, and rollback. The strongest deployment posture combines secure design, secure implementation, reliable verification, and operational observability.

11. Industry Perspective

Banks

Banks need strong testing for session security, transaction authorization, fraud paths, API integrations, customer data, logging, and regulatory evidence. Deployment failure can cause financial loss, privacy harm, and service disruption.

Governments

Government portals require accessibility, privacy, identity assurance, auditability, and secure handling of citizen data. Testing must avoid disrupting public services.

Cloud providers

Cloud providers test consoles and APIs for tenant isolation, identity boundaries, object authorization, rate limits, abuse resistance, and secure defaults. Deployment pipelines must protect huge multi-tenant blast radii.

Healthcare

Healthcare systems test patient portals, appointment systems, lab results, consent flows, privacy controls, audit logs, and file uploads. Secure deployment protects patient safety and confidentiality.

Defense

Defense environments emphasize authorization boundaries, supply-chain assurance, change control, secure deployment evidence, monitoring, and incident readiness.

Universities

Universities manage student portals, research apps, identity federation, learning platforms, and many decentralized systems. Safe testing scope is critical because academic networks contain mixed-sensitivity systems.

Large enterprises

Enterprises use standardized testing gates, application-security programs, vulnerability management, risk acceptance workflows, deployment approvals, and continuous monitoring across many teams.

12. Research Perspective

Research in web security testing and deployment focuses on making verification more accurate, scalable, and usable. Important directions include:

  • Automated detection of logic flaws: scanners are better at known patterns than business rules. Research explores ways to model workflows and authorization expectations.
  • Specification-driven testing: OpenAPI and similar specifications can guide coverage, but they need accurate security metadata.
  • Browser security evolution: researchers study CSP adoption, isolation mechanisms, cross-origin policies, cookie behavior, and secure defaults.
  • Security headers and deployment measurement: internet-scale studies measure how sites deploy HSTS, CSP, cookie flags, and TLS.
  • DevSecOps evidence: research explores how pipelines can produce trustworthy provenance, attestations, and security evidence.
  • AI-assisted testing: language models can help generate tests and review code, but they can also hallucinate findings or miss contextual authorization rules.
  • Usable security for developers: a major open problem is helping ordinary developers apply security controls correctly without requiring every developer to become a specialist.

Open problems remain: reducing false positives, proving object-level authorization coverage, testing multi-step workflows, safely testing production-like systems, and making secure deployment the default rather than a checklist afterthought.

13. Case Studies

Case Study 1: Equifax and patch management failure

The Equifax breach is commonly used to show that known web-framework vulnerabilities, patch management failure, asset visibility gaps, and monitoring weaknesses can combine into major impact. The lesson for this topic is that deployment security includes knowing what is running, patching it, verifying exposure, and monitoring for compromise.

Case Study 2: Capital One cloud breach

The Capital One incident is often discussed in relation to server-side request forgery, cloud metadata access, identity permissions, and detection. The deployment lesson is that web application behavior, cloud IAM, network controls, and logs must be reviewed together rather than as isolated topics.

Case Study 3: Debug and error disclosure incidents

Many organizations have exposed stack traces, debug consoles, admin panels, or environment variables by deploying development settings to production. The root cause is usually process failure: weak release gates, poor configuration separation, and missing production checks.

Case Study 4: Magecart-style web skimming

Web skimming incidents show that browser-side supply chain and script integrity matter. Payment pages can be affected by injected or compromised scripts. Defenses include dependency governance, CSP, script inventory, SRI where appropriate, monitoring, and strict change control.

Case Study 5: Expired certificates and outage risk

Expired or misconfigured TLS certificates can cause user-facing outages and train users to distrust warnings. Certificate monitoring and automatic renewal are operational security controls, not optional maintenance tasks.

14. Common Misconceptions

"A scanner found nothing, so the app is secure."
Wrong. Scanners miss logic flaws, hidden paths, authorization context, and many workflow bugs.
"A WAF means we do not need to fix the code."
Wrong. A WAF is a layer. It cannot reliably enforce business-specific authorization rules.
"Security headers solve XSS."
Wrong. CSP and related headers can reduce impact, but output encoding and safe coding are still required.
"Staging is exactly the same as production."
Often wrong. Data, scale, secrets, network rules, certificates, and integrations may differ.
"Only public pages need security testing."
Wrong. Authenticated areas, admin functions, APIs, internal dashboards, and file handlers are often more sensitive.
"HTTPS means the application is secure."
Wrong. HTTPS protects transport. It does not prove authorization, input handling, or deployment safety.
"Tool output is a report."
Wrong. A report needs verified evidence, impact, remediation, and context.
"Production testing is always forbidden."
Not always, but it requires explicit authorization, strict controls, and careful planning. Beginners should use local labs.

15. Summary

Web security testing verifies that a web application enforces its security requirements. Secure deployment ensures the running environment supports those requirements. A strong program uses written scope, threat models, positive and negative tests, SAST, DAST, SCA, manual review, secure headers, TLS checks, cookie checks, least privilege, logs, monitoring, release gates, canaries, rollback plans, and professional evidence-based reporting. The central rule is simple: test only what you are authorized to test, and deploy only when you have evidence that the release is safe enough for its risk context.

Verifiable information supporting a security claim or finding.
False Negative
A missed issue that exists but is not reported.
False Positive
A reported issue that is not actually present as claimed.
HSTS
HTTP Strict Transport Security; a header telling browsers to use HTTPS for a site for a defined time.
IAST
Interactive Application Security Testing; runtime-instrumented testing of application behavior.
Release Gate
A required approval or automated decision point before deployment.
Rollback
Returning to a known good version after a failed or risky release.
Rules of Engagement
The authorized operating rules for a security test.
SAST
Static Application Security Testing; inspecting code or binaries without running the app.
SCA
Software Composition Analysis; checking dependencies and third-party components.
Security Header
An HTTP response header that asks the browser to enforce a security behavior.
Staging
A production-like environment used to validate releases before production.
WAF
Web Application Firewall; a layer that inspects or blocks some web traffic based on rules or behavior.
WSTG
Web Security Testing Guide; OWASP guidance for web application security testing.

18. Practice Lab

Title

Safe Local Web Deployment Review

Prerequisites

  • Python 3 installed.
  • A terminal or PowerShell.
  • Basic browser use.
  • No external target is required.

Objectives

  • Run a localhost-only web server.
  • Check response headers safely.
  • Create a simple deployment checklist.
  • Write positive and negative security tests.
  • Produce a short security review report.

Safety rules

  • Use only 127.0.0.1 or files on your own machine.
  • Do not scan public websites.
  • Do not use real credentials, real cookies, real tokens, or real personal data.
  • Do not stress test or fuzz production systems.
  • Do not publish claims about real systems from this lab.

Step 1: Create a lab folder

mkdir web-deployment-lab
cd web-deployment-lab
echo "<h1>Local Web Security Lab</h1>" > index.html

Step 2: Create the local server

Create server.py using the secure-header example from Section 9.1.

Step 3: Run the server

python3 server.py

Open http://127.0.0.1:8000 in your browser.

Step 4: Check headers

curl -I http://127.0.0.1:8000/

Record which headers are present. Explain what each one does and what it does not solve.

Step 5: Run the Python header checker

Use the Python checker from Section 8.3. Record which headers are present or missing.

Step 6: Write a deployment checklist

Create a checklist with at least ten items, including debug mode, secrets, dependencies, cookies, headers, TLS plan, access logs, error handling, authorization tests, monitoring, and rollback.

Step 7: Write two negative tests

Write test descriptions for two forbidden behaviors, such as accessing another user's object or uploading an oversized file. You do not need to implement a real app.

Step 8: Write the report

Your report should include scope, environment, tools used, findings, evidence, risk, recommended fixes, limitations, and retest plan.

Expected outcomes

  • You can safely inspect local web headers.
  • You can distinguish testing from deployment hardening.
  • You can explain why scanner output needs verification.
  • You can write a concise, evidence-based security review.
  • You can create a beginner-friendly deployment gate.
