Week 15 · Lesson 15.1

AI Security, LLM Risk, and Model Governance

AI security is not solved by asking whether a model is “safe.” A deployed AI system is software, data, prompts, retrieval sources, integrations, users, policies, vendors, logs, human decisions, and business consequences. This lesson teaches how to govern and secure that whole system.

Estimated reading time95–125 minutes
Difficulty levelAdvanced MS foundation
Required backgroundWeeks 6, 10, 11, 12, 13, and 14 recommended
Lab modeSafe local AI governance triage using fictional JSON data
Boundary: This lesson teaches defensive AI security, LLM application risk management, and governance. It does not provide prompt-injection payloads, model-abuse methods, jailbreak instructions, offensive automation, model extraction procedures, or real-system testing steps. Real AI assessments require authorization, synthetic/safe test data, and named accountable owners.

1. Learning Objectives

By the end of this lesson, you should be able to:

  • Explain AI security as a system and lifecycle problem, not only a model problem.
  • Map an AI or LLM application into data, prompts, retrieval, tools, outputs, humans, vendors, and logs.
  • Use NIST AI RMF language to govern, map, measure, and manage AI risk.
  • Recognize common LLM application risk categories such as prompt injection, insecure output handling, sensitive information disclosure, excessive agency, overreliance, and supply-chain weakness.
  • Separate model governance, application security, data governance, privacy review, and operational monitoring.
  • Define safe controls for retrieval-augmented generation, tool use, human review, logging, versioning, and incident response.
  • Build an AI system inventory record that supports risk decisions and audit evidence.
  • Identify when an AI use case should be blocked, redesigned, piloted, or approved with conditions.

2. Research Basis

This lesson is grounded in current AI risk and security references. NIST AI RMF 1.0 provides a voluntary framework for incorporating trustworthiness considerations into AI design, development, use, and evaluation. NIST AI 600-1 applies the AI RMF to generative AI and calls out risks such as information integrity, data privacy, harmful content, security, and value-chain/component integration. OWASP Top 10 for Large Language Model Applications provides an application-security vocabulary for LLM risks, including prompt injection, insecure output handling, sensitive information disclosure, excessive agency, overreliance, and model theft. NIST AI 100-2 provides adversarial machine-learning taxonomy and terminology. MITRE ATLAS gives defenders a living knowledge base of tactics and techniques against AI-enabled systems. ISO/IEC 42001:2023 gives organizations an AI management-system model for establishing, implementing, maintaining, and improving AI governance.

The practical conclusion is simple: AI adoption must be managed like a socio-technical system that changes software risk, data risk, human decision risk, vendor risk, privacy risk, and incident-response assumptions at the same time.

3. The System View of AI Security

The weakest AI security discussions start with the model and stop there. A stronger analysis starts with the system boundary:

  • Users: employees, customers, developers, analysts, administrators, and attackers pretending to be normal users.
  • Inputs: prompts, files, logs, tickets, documents, database fields, images, audio, code, or telemetry.
  • Context: system prompt, policies, application logic, retrieval results, tool outputs, prior conversation, and user permissions.
  • Model: hosted model, self-hosted model, fine-tuned model, embedding model, classifier, or agent planner.
  • Tools: APIs, email senders, payment systems, workflow engines, ticket writers, code repositories, or search indexes.
  • Outputs: generated text, classifications, decisions, code, summaries, recommendations, tool calls, or business actions.
  • Controls: access control, input validation, output validation, human approval, rate limits, retrieval boundaries, logging, monitoring, and incident response.

The model is only one component. The security question is what the connected workflow can see, decide, and do.

4. Reference Architecture

A typical LLM application can be represented as a controlled pipeline. The exact implementation may vary, but the security logic is stable: classify inputs, constrain context, limit retrieval, guard tool use, validate outputs, and keep humans accountable for high-impact actions.

flowchart TD
  U[User or system input] --> C[Input classification and policy checks]
  C --> P[Prompt and context builder]
  P --> R{Retrieval needed?}
  R -->|yes| K[Approved knowledge sources and access filters]
  R -->|no| M[Model call]
  K --> M[Model call]
  M --> O[Output validation and safety checks]
  O --> H{Material action or sensitive output?}
  H -->|yes| A[Human review and approval]
  H -->|no| D[Deliver response]
  A --> D
  O --> L[Telemetry, versioning, review outcome, incident signals]
  D --> L

This architecture does not guarantee safety. It gives you places to put controls, evidence, owners, and monitoring.

5. Govern, Map, Measure, Manage

NIST AI RMF organizes AI risk management into four functions:

FunctionSecurity meaningEvidence to collect
GovernDefine accountability, policy, roles, risk appetite, escalation, and lifecycle ownership.AI policy, approved use cases, owner records, exception decisions, review cadence.
MapUnderstand context: purpose, users, data, affected parties, model source, architecture, integrations, and impacts.System inventory, data-flow map, model/system card, vendor record, threat model.
MeasureEvaluate behavior, security controls, privacy impact, output quality, misuse resistance, and monitoring signals.Test results, red-team records, monitoring metrics, false-output reports, privacy review.
ManagePrioritize treatment, redesign, approve, restrict, monitor, transfer, or stop AI use.Risk register, remediation tickets, approval conditions, residual-risk acceptance, rollback plan.

The functions should not be treated as a one-time checklist. AI systems change when prompts, tools, retrieval sources, vendors, model versions, user groups, or business uses change.

6. AI System Inventory

An AI inventory should record more than “we use a chatbot.” It should include:

  • System name, business purpose, owner, data owner, security owner, and risk owner.
  • Deployment stage: experiment, pilot, production, retired, or blocked.
  • Model source: public tool, enterprise SaaS, third-party API, self-hosted model, fine-tuned model, or embedded vendor feature.
  • Data inputs, sensitive-data categories, retention, logging, geographic constraints, and data-sharing terms.
  • Retrieval sources, indexes, document owners, freshness, access filters, and citation behavior.
  • Tools/actions available to the model or agent, with permission boundaries and approval rules.
  • Human review requirements and the decision authority of the AI output.
  • Testing, evaluation, validation, monitoring, incident-response, and rollback evidence.

An inventory is not bureaucracy. It is the minimum record needed to know what changed when an AI system creates a privacy event, bad decision, data leak, service failure, or audit question.

7. LLM Application Risk Categories

LLM applications inherit normal web, API, identity, cloud, data, and software-supply-chain risks. They also introduce or amplify patterns that require special attention:

  • Prompt injection: untrusted content manipulates model behavior or the surrounding workflow.
  • Insecure output handling: downstream systems trust generated output as code, commands, queries, configuration, or facts without validation.
  • Sensitive information disclosure: prompts, outputs, logs, retrieval results, or vendor processing expose data that should not be disclosed.
  • Excessive agency: the model or agent can take actions beyond what is appropriate without sufficient limits or approval.
  • Overreliance: humans accept generated claims, summaries, classifications, or code without review.
  • Supply-chain vulnerabilities: models, datasets, plugins, prompts, libraries, agents, or vendors become untrusted dependencies.
  • Model theft or misuse: valuable models, prompts, embeddings, or system behavior are extracted or abused.

The control strategy depends on the business context. A summarizer, coding assistant, fraud classifier, medical triage tool, and payment-refund agent do not have the same risk profile.

8. Prompt Injection Without the Hype

Prompt injection is not magic. It is a control-boundary problem. LLMs treat text as instruction-like context unless the surrounding application constrains what text can influence. The risky pattern is not merely that a user typed something clever; it is that untrusted content can affect data access, tool use, policy decisions, output routing, or user trust.

Defensive design should assume some inputs and retrieved documents are adversarial or low-trust. Practical controls include:

  • Separate system instructions, user content, retrieved content, and tool results in the application design.
  • Do not let retrieved text grant permissions or change policy.
  • Constrain tools by identity, least privilege, and business rule, not by model instruction alone.
  • Require human approval for material external actions.
  • Use allowlisted retrieval sources and access checks.
  • Log enough context to investigate failures without over-collecting sensitive content.

The right goal is not “make injection impossible.” The right goal is to prevent untrusted text from causing unauthorized data access, unsafe actions, false trust, or uncontrolled disclosure.

9. Data Governance for AI

AI systems can concentrate data risk. Prompts, embeddings, training sets, retrieval indexes, vector stores, output logs, review queues, and feedback datasets may all contain sensitive information. Treat each as a data store with ownership, access control, retention, classification, and disposal requirements.

Key questions:

  • Which data is sent to a model provider, stored in logs, used for improvement, or retained in a retrieval index?
  • Is the data necessary for the use case, or is the system collecting more than needed?
  • Can users paste secrets, customer data, regulated data, or source code into the system?
  • Are prompts and outputs monitored for sensitive-data leakage?
  • Are training, tuning, evaluation, and feedback datasets versioned with provenance?
  • Who can approve new datasets, new retrieval sources, or new output destinations?

A model governance process that ignores data governance is incomplete.

10. Retrieval-Augmented Generation Risk

Retrieval-augmented generation can make outputs more useful, but it also creates new trust boundaries. A RAG system may search documents, tickets, code, policies, records, or web content and then blend retrieved text into a response. The security question becomes: can the user access the source, is the source trustworthy, is it fresh, and can malicious or stale text influence the response?

Controls include document-owner approval, retrieval allowlists, per-user access filters, metadata freshness checks, citation requirements, chunk-level provenance, sensitive-document exclusion, and monitoring for retrieval failures. Avoid using RAG to bypass normal access control. The model should not become a side channel into documents the user could not otherwise read.

11. Tool Use and Agentic AI

Tool-using AI is higher risk because generated reasoning can become real-world action. The first design rule is: the model should never be the root authority for permissions. Authorization must come from identity, application policy, data classification, workflow state, and human approval where needed.

CapabilityRiskControl
Read-only lookupUnauthorized disclosure or broad querying.Per-user access checks, query limits, audit logs, sensitive-field filtering.
Draft creationIncorrect, harmful, or misleading output.Human review, output validation, labeling, citation and confidence expectations.
External message sendingUnauthorized disclosure or reputation damage.Approval workflow, templates, recipient checks, rate limits, rollback/recall plan.
Financial or account actionFraud, loss, compliance violation, customer harm.Strong authentication, segregation of duties, explicit approval, thresholds, transaction logging.
Code or infrastructure changeVulnerability, outage, secret exposure, supply-chain compromise.Pull requests, CI/CD policy, static analysis, human approval, environment separation.

Higher autonomy requires stronger constraints. “The model decided” is not a governance answer.

12. Model and AI Supply Chain

AI systems depend on models, datasets, embeddings, prompts, packages, plugins, containers, APIs, evaluation tools, annotation vendors, and hosting services. Supply-chain questions include:

  • Where did the model come from, and what license, terms, limitations, or safety notes apply?
  • Can the vendor use prompts, outputs, feedback, or files for training or service improvement?
  • How are model versions changed, announced, tested, and rolled back?
  • What evidence exists for security, privacy, vulnerability handling, incident notification, and service availability?
  • Are third-party plugins, tools, or extensions reviewed like software dependencies?
  • Can an open-source model or dataset be traced to an approved source with integrity checks?

Model supply chain is not separate from software supply chain. It is an additional layer on top of normal dependency, build, deployment, and vendor assurance practices.

13. Testing, Evaluation, Verification, and Validation

AI testing should include functional behavior and risk behavior. Useful test categories include:

  • Use-case fit: Does the system support the intended task without inventing unsupported claims?
  • Data handling: Does it avoid unnecessary sensitive data and respect retention/access controls?
  • Abuse resistance: Does it behave safely under policy-conflicting, ambiguous, or adversarial inputs?
  • Output reliability: Are citations, summaries, classifications, or code reviewed against expected evidence?
  • Tool safety: Are actions constrained, logged, reversible, and approved?
  • Operational resilience: What happens when the model, vendor, index, or policy service fails?
  • Human factors: Are users trained not to over-trust outputs?

Tests should be authorized, documented, repeatable, and scoped. Use synthetic data when possible, protect evaluation logs, and record pass/fail criteria before testing.

14. Monitoring and Incident Response

AI systems need telemetry that supports both security and governance:

  • System name, model/provider/version, prompt template version, retrieval source identifiers, tool-call identifiers, and output destination.
  • Policy decisions, blocked inputs, failed output checks, approval decisions, user feedback, and reported hallucinations or harmful outputs.
  • Rate-limit events, unusual usage, sensitive-data warnings, tool failure, model/vendor outage, and incident escalation records.
  • Change events: new model version, new retrieval source, new tool, new data category, new user population, or changed output destination.

Do not log everything blindly. Prompt and output logs can contain personal data, secrets, customer data, source code, or incident details. Good telemetry is enough to investigate and govern the system while minimizing secondary data exposure.

15. Human Oversight and Overreliance

Human review is not automatically effective. A tired analyst rubber-stamping AI output is not a control. Oversight must be designed:

  • Reviewers need enough source evidence to challenge the output.
  • High-impact decisions need clear approval authority and escalation paths.
  • Interfaces should show uncertainty, provenance, and limitations where possible.
  • Generated code, legal/security summaries, incident conclusions, and customer-impact decisions should require verification against source material.
  • Metrics should track acceptance/rejection rates, corrected outputs, false confidence, and user-reported failures.

The goal is calibrated trust. Users should neither ignore useful AI assistance nor accept generated output as authority.

16. Model Governance Evidence

A practical model-governance package should include:

  • Approved use case and prohibited use cases.
  • Business owner, data owner, security owner, risk owner, and support owner.
  • Model/provider, version, hosting location, terms, retention, and update/change process.
  • Data-flow diagram and classification of prompts, retrieval sources, outputs, logs, and feedback.
  • Threat model covering prompt injection, data leakage, insecure output handling, tool misuse, supply chain, overreliance, and monitoring gaps.
  • Test plan and results for expected use, misuse, privacy/security checks, human review, and failure modes.
  • Operational controls: access, logging, alerts, incident response, rollback, and periodic review.
  • Residual-risk decision with expiry or review trigger.

For many enterprise uses, the required artifact is not a “model card” alone. It is a full system governance record that connects model behavior to business workflow and risk ownership.

17. Case Study: Refund Agent Proposal

A business team proposes an autonomous AI agent that reads customer history, decides refund eligibility, sends the customer an email, and calls a payment API. They say it will reduce support cost. The system touches customer profile data, purchase history, support notes, and payment token references. It has no human approval, no rollback plan, no privacy review, and no named security owner.

A weak response says, “Use a better model.” A stronger response says: “This proposal changes the system’s decision authority and action authority. It combines sensitive data, external actions, customer impact, payment workflow, and missing ownership. It cannot move to production until action permissions, human approval thresholds, privacy review, monitoring, rollback, abuse testing, incident response, and residual-risk decision are documented.”

18. Common Failure Modes

  • Model-only thinking: teams evaluate the model but ignore retrieval, tools, logs, users, vendors, and downstream systems.
  • Pilot drift: a demo becomes production without risk review, monitoring, support ownership, or rollback.
  • Blind data sharing: prompts, source code, documents, or incident data are sent to tools without understanding retention and training terms.
  • Permission by prompt: teams rely on system instructions instead of application-level access control.
  • Unverified RAG: retrieval sources are stale, overbroad, malicious, or unavailable to the user.
  • No version accountability: a model, prompt, or policy change alters behavior but nobody can reconstruct what happened.
  • Overreliance: users accept summaries, recommendations, or code without reviewing evidence.
  • Unsafe autonomy: AI workflows take material actions without approval, limits, monitoring, or rollback.

19. Practical Control Patterns

  • Maintain an AI system inventory with owners, data, model source, tools, outputs, and review dates.
  • Use default-deny approvals for new data categories, retrieval sources, tools, and external actions.
  • Classify prompts, files, outputs, embeddings, logs, and feedback datasets.
  • Apply least privilege to retrieval and tools; do not grant access through model instruction.
  • Validate outputs before using them in code execution, database queries, workflows, messages, decisions, or reports.
  • Require human review for high-impact decisions and material external actions.
  • Collect vendor evidence for retention, training use, model changes, incident notification, and data handling.
  • Monitor misuse, sensitive-data exposure, hallucination reports, tool failures, and version changes.
  • Build AI incident-response playbooks before production deployment.

Step 2 — Create ai_governance_lab.py

#!/usr/bin/env python3
"""
Safe AI governance triage lab.
This script uses fictional local JSON only. It performs no network activity,
no model calls, no prompt testing against real systems, no vendor access,
no exploitation, and no legal/compliance determination.
"""
import json
from pathlib import Path

HIGH_SENSITIVITY = {"customer_profile", "purchase_history", "payment_token_reference", "source_code", "secrets_risk", "security_alerts", "usernames"}
REQUIRED_ROLES = ["business_owner", "risk_owner", "data_owner", "security_owner"]
MINIMUM_LOGS = {"model_version"}
AGENTIC_TOOL_RISKS = {"payment_refund_api", "email_sender", "crm_reader", "ticket_draft_writer"}


def sensitivity_score(system):
    return sum(1 for item in system.get("sensitive_data", []) if item in HIGH_SENSITIVITY)


def missing_roles(system):
    missing = []
    for role in REQUIRED_ROLES:
        value = str(system.get(role, "")).strip().lower()
        if not value or value == "unassigned":
            missing.append(role)
    return missing


def governance_gaps(system):
    gaps = []
    if missing_roles(system):
        gaps.append("missing named owner(s): " + ", ".join(missing_roles(system)))
    if sensitivity_score(system) and not system.get("privacy_review"):
        gaps.append("sensitive data present without privacy review record")
    if "no model-card review" in system.get("known_gaps", []):
        gaps.append("model/system documentation not reviewed")
    if "no red-team record" in system.get("known_gaps", []):
        gaps.append("no documented adversarial or misuse test record")
    if system.get("deployment_stage") == "production" and system.get("known_gaps"):
        gaps.append("production system has unresolved governance gaps")
    return gaps


def security_gaps(system):
    gaps = []
    controls = set(system.get("controls", []))
    logs = set(system.get("logging", []))
    tools = set(system.get("tools", []))
    if system.get("retrieval") and "retrieval_allowlist" not in controls and "local_inference" not in controls:
        gaps.append("retrieval workflow lacks allowlist/local-boundary control evidence")
    if system.get("external_actions"):
        if not system.get("human_review_required"):
            gaps.append("external action allowed without human approval")
        if "tool permission design" not in " ".join(system.get("known_gaps", [])) and not tools.intersection(AGENTIC_TOOL_RISKS):
            gaps.append("tool permissions need explicit design evidence")
        if "rollback plan" in " ".join(system.get("known_gaps", [])):
            gaps.append("external action workflow lacks rollback plan")
    if not system.get("human_review_required") and sensitivity_score(system):
        gaps.append("sensitive workflow lacks human review")
    if MINIMUM_LOGS.isdisjoint(logs):
        gaps.append("logging lacks model/version traceability")
    if "abuse monitoring" in " ".join(system.get("known_gaps", [])):
        gaps.append("abuse monitoring gap recorded")
    return gaps


def risk_band(system):
    score = 0
    score += sensitivity_score(system) * 2
    if system.get("external_actions"):
        score += 5
    if system.get("deployment_stage") == "production":
        score += 2
    if missing_roles(system):
        score += 3
    score += len(system.get("known_gaps", []))
    if not system.get("human_review_required"):
        score += 2
    if score >= 10:
        return "red", score
    if score >= 5:
        return "amber", score
    return "green", score


def recommended_actions(system):
    actions = []
    band, _ = risk_band(system)
    if band == "red":
        actions.append("do not approve production use until ownership, safety controls, privacy review, monitoring, and rollback decisions are documented")
    if system.get("external_actions"):
        actions.append("constrain tool permissions, require approval for material actions, and define rollback/stop conditions")
    if sensitivity_score(system):
        actions.append("confirm data minimization, retention, access control, privacy review, and output disclosure controls")
    if system.get("retrieval"):
        actions.append("validate retrieval source ownership, document freshness, access boundaries, and citation/evidence behavior")
    if system.get("model_source") in {"third_party_api", "hosted_enterprise_ai_tool"}:
        actions.append("collect vendor evidence for data use, retention, logging, model updates, incident notice, and service boundaries")
    if "no red-team record" in system.get("known_gaps", []):
        actions.append("run authorized misuse/abuse testing with synthetic prompts and documented pass/fail criteria")
    if not actions:
        actions.append("continue monitoring, periodic review, version tracking, and user feedback triage")
    return actions


def analyze(system):
    band, score = risk_band(system)
    return {
        "system_id": system["system_id"],
        "name": system["name"],
        "risk_band": band,
        "risk_score": score,
        "governance_gaps": governance_gaps(system),
        "security_gaps": security_gaps(system),
        "recommended_actions": recommended_actions(system),
        "decision_line": f"{system['name']} is {band.upper()} because it combines stage={system['deployment_stage']}, source={system['model_source']}, external_actions={system['external_actions']}, sensitive_items={len(system.get('sensitive_data', []))}, and open_gaps={len(system.get('known_gaps', []))}.",
    }


def build_report(systems):
    analyzed = [analyze(system) for system in systems]
    summary = {"red": 0, "amber": 0, "green": 0}
    for item in analyzed:
        summary[item["risk_band"]] += 1
    return {
        "notice": "Educational fictional AI governance triage only — validate real AI decisions with legal, privacy, security, risk, data, and business owners.",
        "summary": summary,
        "systems": analyzed,
    }


def main():
    systems = json.loads(Path("ai_systems.json").read_text())
    print(json.dumps(build_report(systems), indent=2))


if __name__ == "__main__":
    main()

import importlib.util
import json

spec = importlib.util.spec_from_file_location("ai_governance_lab", "ai_governance_lab.py")

spec.loader.exec_module(module)

print("AI governance lab tests passed")

Expected learning outcome

The output should show that AI risk is driven by data sensitivity, external action authority, production state, unresolved gaps, ownership, privacy review, logging, and human oversight. A low-risk summarizer and a high-risk autonomous payment agent should not receive the same approval path.

21. AI Security Readiness Checklist

  • Is the AI use case approved, documented, and tied to a business owner?
  • Are the risk owner, data owner, security owner, privacy owner, and support owner known?
  • Are prompts, retrieval sources, outputs, logs, embeddings, and feedback datasets classified?
  • Can the model access only data the user or workflow is authorized to use?
  • Are tools constrained by application policy and least privilege?
  • Are external actions approved, logged, rate-limited, reversible, or blocked where appropriate?
  • Are outputs validated before they affect code, decisions, messages, workflows, or reports?
  • Are model/provider versions, prompt templates, retrieval sources, and policy changes tracked?
  • Are misuse testing, privacy review, monitoring, incident response, and rollback documented?
  • Is residual risk accepted by a named owner with review triggers?
