Lesson 9.4: EDR, Telemetry, and Detection

This lesson teaches endpoint detection from first principles. You will learn what endpoint telemetry is, how endpoint detection and response tools use it, how alerts are created, why false positives and false negatives happen, and how defenders investigate safely.

1. Introduction

Endpoint detection and response, usually shortened to EDR, is a category of security technology and practice focused on observing endpoint activity, detecting suspicious behavior, supporting investigation, and enabling response actions such as isolation or process containment.

Plain English: EDR is like a security camera, event notebook, alarm system, and investigation console for laptops, desktops, servers, and workloads. It does not make endpoints impossible to attack. It helps defenders notice, understand, and respond when suspicious activity happens.

This topic exists because prevention is incomplete. Patches can be missing, users can open risky files, credentials can be stolen, malware can change its appearance, and attackers can abuse legitimate tools. Endpoint telemetry gives defenders a way to see behavior rather than relying only on known malicious file signatures.

The topic matters because many incidents leave traces on endpoints: a new process started, a file was created, a script executed, a registry key changed, a service was installed, a network connection was opened, a credential was used, or a security tool was disabled. Without telemetry, defenders may only discover the incident after damage is already visible.

EDR is used by security operations centers, incident responders, enterprise administrators, malware analysts, threat hunters, managed detection and response providers, cloud security teams, and researchers. Banks use it to watch high-value workstations. Hospitals use it to protect clinical endpoints. Governments use it for fleet-wide detection. Cloud providers use similar telemetry ideas for workloads and infrastructure.

Safety boundary: this lesson is defensive. Labs use fictional logs, normal operating-system commands, and benign examples. Do not run malware, bypass endpoint tools, tamper with security agents, or investigate systems without authorization.

2. Learning Outcomes

After finishing this lesson, you should be able to:

  • Define endpoint telemetry, EDR, alert, detection rule, triage, and response action.
  • Explain the difference between raw events, normalized events, enriched events, detections, alerts, and incidents.
  • Identify major endpoint telemetry sources: process, file, registry, network, authentication, service, driver, script, and security-control events.
  • Explain why process command lines, parent-child relationships, hashes, users, timestamps, and network destinations matter.
  • Describe how EDR differs from traditional antivirus, host logging, SIEM, and XDR.
  • Write simple defensive detection logic over fictional logs.
  • Calculate basic detection metrics such as true positives, false positives, precision, recall, and alert rate.
  • Perform safe triage using evidence: what happened, where, when, under which user, and what changed afterward.
  • Explain common attacks against endpoint visibility, including log clearing, agent tampering, living-off-the-land behavior, and alert fatigue.
  • Recommend defensive strategies for endpoint monitoring, tuning, retention, privacy, and response.

3. Prerequisites

3.1 What is an endpoint?

An endpoint is a device or workload where computing activity happens. Examples include a laptop, desktop, server, virtual machine, cloud instance, container host, point-of-sale terminal, or administrator workstation.

3.2 What is a process?

A process is a running program. If you open a browser, the operating system creates one or more browser processes. Process events are central to endpoint detection because attackers often cause programs, scripts, and tools to run.

3.3 What is a log?

A log is a record of events. A security log might record login attempts. A process log might record when a program started. A firewall log might record blocked network traffic. Logs are evidence, but they are not automatically conclusions.

3.4 What is detection?

Detection is the process of identifying activity that may require attention. A detection can be based on a known indicator, a suspicious behavior pattern, a policy violation, an anomaly, or a combination of evidence.

3.5 What is response?

Response means taking action after suspicious or confirmed malicious activity. Response actions can include isolating a host, collecting evidence, resetting credentials, killing a process, blocking a hash, blocking a domain, opening an incident, or restoring from backup. Response must be authorized and documented.

4. Core Concepts

4.1 Endpoint telemetry

Telemetry means measured event data collected from systems. In endpoint security, telemetry is evidence about endpoint activity. It may include:

  • Process events: program started, command line, parent process, user, path, hash, integrity level.
  • File events: file created, modified, deleted, renamed, quarantined, downloaded, or executed.
  • Registry events: Windows configuration changes, persistence keys, policy changes.
  • Network events: outbound connection, listening socket, remote IP, domain, port, protocol.
  • Authentication events: successful login, failed login, privilege use, remote session.
  • Service events: new service installed, service started, service stopped, startup type changed.
  • Script events: PowerShell, shell, Python, JavaScript, macro, or command interpreter activity.
  • Security-control events: antivirus detection, exploit protection event, firewall change, EDR tamper event.

Intuition: telemetry is like footprints. A single footprint may not prove who entered the building, but a sequence of footprints can build a timeline.

A common beginner mistake is to treat every alert as a confirmed attack. An alert is a question that needs investigation.

4.3 EDR vs antivirus

Traditional antivirus historically focused on preventing or detecting known malicious files. Modern endpoint protection can include behavior-based detection, exploit protection, cloud reputation, attack-surface reduction, and EDR features. EDR emphasizes visibility, investigation, and response over time.

Analogy: antivirus is like a guard checking a list of banned items. EDR is like a camera system that records behavior, helps reconstruct events, and lets security staff take action.

4.4 EDR vs SIEM vs XDR

  • EDR: focuses on endpoint activity and response.
  • SIEM: collects and correlates logs from many systems, not only endpoints.
  • XDR: extends detection and response across multiple domains such as endpoint, identity, email, cloud, and network.
  • MDR: managed detection and response; an external provider helps monitor and respond.

These categories overlap in real products. The exact label matters less than the data collected, detections available, response authority, and operational workflow.

4.5 Endpoint event fields

Useful endpoint fields include:

  • Timestamp: when the event happened.
  • Hostname/device ID: where it happened.
  • User: under which account it happened.
  • Process name: which executable ran.
  • Process path: where the executable was located.
  • Command line: which arguments were passed.
  • Parent process: what launched it.
  • File hash: fingerprint of the file content.
  • Remote IP/domain: network destination.
  • Action/result: allowed, blocked, failed, quarantined, isolated, or remediated.

Example: cmd.exe starting from a user terminal may be normal. cmd.exe spawned by a word processor after opening an email attachment is more suspicious because the parent-child relationship is unusual.

4.6 Parent-child process relationships

A parent process launches a child process. Process trees help defenders understand causality. For example:

explorer.exe
  └── chrome.exe
      └── downloads/report.exe
          └── powershell.exe

This chain suggests a user downloaded and ran a file, and that file launched PowerShell. It is not proof of malware by itself, but it deserves review.

4.7 Living off the land

Living off the land means abusing legitimate tools already present on a system. Examples include shells, scripting engines, system administration utilities, remote management tools, and cloud command-line utilities. This is difficult to detect because the tool itself may be trusted.

The detection question is not only “Is this tool bad?” It is “Is this tool being used in a context, sequence, or pattern that is risky?”

4.9 Detection rule anatomy

A detection rule usually contains:

  • Name: concise title.
  • Data source: process events, authentication logs, network events, etc.
  • Selection: conditions that must match.
  • Filters: conditions that remove known benign events.
  • Severity: triage priority, not guaranteed impact.
  • Rationale: why the pattern matters.
  • False positive notes: expected legitimate causes.
  • Response guidance: what an analyst should check next.

4.10 Alert triage

Triage means quickly sorting alerts into categories such as false positive, benign true positive, suspicious, or confirmed incident. A good first triage answers:

  • What happened?
  • Which endpoint?
  • Which user?
  • When did it start?
  • What process, file, or network activity was involved?
  • Was anything blocked or allowed?
  • Is the activity normal for this user and endpoint?
  • Are there related events before or after?

4.11 Timeline analysis

A timeline orders events by time. Timelines prevent analysts from overfocusing on one event and missing the sequence. A suspicious script execution may be less important if it was blocked immediately, or more important if it was followed by credential access, persistence, and outbound connections.

4.12 Response actions

Common EDR-assisted response actions include:

  • Isolate device: restrict network communication while preserving management connectivity.
  • Collect investigation package: gather logs, file metadata, memory artifacts, or process lists.
  • Terminate process: stop a running process if authorized.
  • Quarantine file: prevent execution or access to a suspicious file.
  • Block indicator: block a hash, certificate, domain, or URL.
  • Run script or query: collect approved information from endpoints.
  • Open incident: escalate to coordinated response.

Response actions can disrupt users or destroy volatile evidence. Use documented authority and incident-response procedures.

4.13 False positives and false negatives

A false positive is an alert on benign activity. A false negative is missed malicious activity. Detection work always balances both. Too many false positives cause alert fatigue. Too many false negatives leave attacks unnoticed.

4.14 Detection coverage

Detection coverage means what behavior your telemetry and rules can realistically observe. If you do not collect command lines, you cannot reliably detect suspicious command-line arguments. If Linux servers do not have endpoint agents, endpoint detections built for Windows will not cover them.

4.15 Privacy and proportionality

Endpoint telemetry can contain sensitive information: usernames, file paths, document names, command lines, URLs, and sometimes fragments of user activity. A mature program limits collection to legitimate security purposes, controls access, protects logs, defines retention, and redacts unnecessary personal data in reports.

4.16 EDR limitations

EDR is valuable but limited. It can be misconfigured, overloaded, evaded, tampered with, or blind to activity outside its collection scope. Attackers may use legitimate tools, memory-only behavior, signed binaries, kernel-level techniques, disabled agents, or stolen credentials. Defenders need layered controls: hardening, patching, identity security, network monitoring, backups, incident response, and user education.

7. Mathematical Foundations

7.1 Confusion matrix

Detection systems can be evaluated with four basic categories:

CategoryMeaning
True positiveMalicious or policy-relevant activity correctly alerted.
False positiveBenign activity incorrectly alerted.
True negativeBenign activity correctly ignored.
False negativeMalicious or policy-relevant activity missed.

7.2 Precision

Precision answers: “When this detection alerts, how often is it right?”

precision = true positives / (true positives + false positives)

If a rule produced 20 true positives and 80 false positives:

precision = 20 / (20 + 80) = 0.20 = 20%

This means only one in five alerts was useful. Analysts may experience alert fatigue.

7.3 Recall

Recall answers: “Of the activity we wanted to catch, how much did we catch?”

recall = true positives / (true positives + false negatives)

If there were 25 real malicious events and the rule caught 20:

recall = 20 / (20 + 5) = 0.80 = 80%

7.4 Alert rate

Alert rate helps estimate operational load.

alert rate = alerts per day / analyst capacity per day

If a rule generates 120 alerts per day and a team can review 60 similar alerts per day, the rate is unsustainable unless automation, tuning, or staffing changes occur.

7.5 Base-rate problem

When malicious activity is rare, even a detection with good accuracy can produce many false positives. Example: if only 1 in 10,000 events is malicious, a broad rule that catches many benign events can overwhelm analysts. This is why context, enrichment, and tuning matter.

7.6 Worked detection-quality example

MetricValue
True positives30
False positives70
False negatives10
precision = 30 / (30 + 70) = 30%
recall = 30 / (30 + 10) = 75%

The rule finds many relevant events but creates significant analyst workload. A defender might add better filters, enrich with asset criticality, or split the detection into high-confidence and low-confidence variants.

8. Practical Examples

All examples are safe, defensive, and local. Do not query or modify systems you do not own or administer.

8.1 Linux examples

# Show recent authentication-related messages on a lab Linux system.
# This reads local logs only.
sudo journalctl -u ssh --since "24 hours ago"

# List running processes with process IDs and parent process IDs.
ps -eo pid,ppid,user,comm,args --sort=ppid | head -30

# Show listening sockets and owning process information.
ss -tulpen

# Look for recently modified files in a lab directory you control.
find ./lab-folder -type f -mtime -1 -ls

# Show failed sudo attempts, depending on your distribution's logging.
sudo journalctl --since "24 hours ago" | grep -i "authentication failure"

8.2 Windows examples

# Show recent process creation events if process auditing is enabled.
# Run only on your own lab machine.
Get-WinEvent -FilterHashtable @{LogName='Security'; Id=4688; StartTime=(Get-Date).AddHours(-24)} -MaxEvents 20

# Show recent Defender operational events.
Get-WinEvent -LogName 'Microsoft-Windows-Windows Defender/Operational' -MaxEvents 20

# Show running processes with parent process IDs.
Get-CimInstance Win32_Process | Select-Object ProcessId, ParentProcessId, Name, CommandLine | Select-Object -First 20

# Show recent service-control events.
Get-WinEvent -FilterHashtable @{LogName='System'; ProviderName='Service Control Manager'} -MaxEvents 20

8.3 Python example: parse fictional endpoint logs

# Defensive training script.
# It reads fictional endpoint events from a CSV file and flags suspicious patterns.

import csv                      # Import the CSV library for reading structured text files.
from pathlib import Path         # Import Path for safer file path handling.

LOG_FILE = Path("fictional_endpoint_events.csv")  # File expected in the current folder.

SUSPICIOUS_PROCESSES = {"powershell.exe", "cmd.exe", "bash", "curl", "wget"}  # Tools that may be normal or risky.
RISKY_PARENTS = {"winword.exe", "excel.exe", "outlook.exe", "chrome.exe"}      # Parent processes that deserve review.

with LOG_FILE.open(newline="", encoding="utf-8") as f:  # Open the CSV safely as text.
    reader = csv.DictReader(f)                           # Read rows as dictionaries by column name.
    for row in reader:                                   # Loop through each event.
        process = row.get("process", "").lower()         # Normalize process name to lowercase.
        parent = row.get("parent", "").lower()           # Normalize parent process name.
        user = row.get("user", "unknown")                # Get the user or use a safe default.
        host = row.get("host", "unknown")                # Get the host or use a safe default.
        command = row.get("command_line", "")             # Get the command line for context.

        if process in SUSPICIOUS_PROCESSES and parent in RISKY_PARENTS:
            print(f"Review needed: {host} {user} {parent} -> {process} :: {command}")

This script does not declare anything malicious. It says “review needed.” That distinction matters: detection logic produces evidence for analysis, not automatic certainty.

8.4 Network example

An endpoint network event might show:

timestamp=2026-06-29T10:30:11Z
host=LAB-WS-01
process=powershell.exe
remote_domain=updates.example.invalid
remote_port=443
action=allowed

The remote port 443 only tells you the connection used HTTPS-like transport. It does not prove the destination was safe. Investigators need process context, destination reputation, user context, timing, and related events.

8.5 Web example

If a browser downloads a file and that file launches a script interpreter, endpoint telemetry can connect the web event to host behavior:

chrome.exe downloads invoice.zip
explorer.exe opens extracted invoice.exe
invoice.exe launches powershell.exe
powershell.exe connects to unknown domain

Detection quality comes from the chain, not one isolated event.

8.6 Cloud example

Cloud workloads can produce endpoint-like telemetry from virtual machines and containers. A cloud detection might combine:

  • Cloud identity event: new access token issued.
  • Endpoint event: shell spawned on a VM.
  • Network event: outbound connection to unusual region.
  • Storage event: many objects downloaded.

This is where EDR ideas meet XDR and cloud detection engineering.

9. Code Examples

9.1 Create a fictional CSV dataset

# This script creates a fictional endpoint log file for practice.
# It does not inspect your system and does not connect to any network.

import csv
from pathlib import Path

rows = [
    {"time": "2026-06-29T09:00:00Z", "host": "LAB-WS-01", "user": "student", "parent": "explorer.exe", "process": "chrome.exe", "command_line": "chrome.exe"},
    {"time": "2026-06-29T09:05:00Z", "host": "LAB-WS-01", "user": "student", "parent": "chrome.exe", "process": "notepad.exe", "command_line": "notepad.exe notes.txt"},
    {"time": "2026-06-29T09:20:00Z", "host": "LAB-WS-01", "user": "student", "parent": "winword.exe", "process": "powershell.exe", "command_line": "powershell.exe -NoProfile -ExecutionPolicy Bypass"},
]

output = Path("fictional_endpoint_events.csv")
with output.open("w", newline="", encoding="utf-8") as f:
    writer = csv.DictWriter(f, fieldnames=rows[0].keys())
    writer.writeheader()
    writer.writerows(rows)

print(f"Wrote {output}")

9.2 Explain every line

  • import csv loads Python’s CSV handling library.
  • from pathlib import Path gives a safer object for file paths.
  • rows = [...] creates fictional endpoint events.
  • output = Path(...) names the file to write.
  • output.open(...) opens the file in text mode with UTF-8 encoding.
  • csv.DictWriter writes dictionary keys as columns.
  • writeheader() writes the first row containing column names.
  • writerows(rows) writes the fictional events.
  • print(...) tells the learner where the file was written.

9.3 Detection rule in Sigma-style pseudocode

title: Office Application Spawning Script Interpreter
status: experimental
logsource:
  category: process_creation
detection:
  selection_parent:
    ParentImage|endswith:
      - '\\winword.exe'
      - '\\excel.exe'
      - '\\powerpnt.exe'
  selection_child:
    Image|endswith:
      - '\\powershell.exe'
      - '\\cmd.exe'
      - '\\wscript.exe'
  condition: selection_parent and selection_child
falsepositives:
  - Some legitimate office automation workflows
level: medium

This is educational pseudocode in a Sigma-like style. Real deployment requires mapping field names to your logging platform, testing false positives, documenting approved automation, and peer review.

9.4 Common coding mistakes

  • Mistake: Hardcoding real hostnames, usernames, or secrets in training scripts. Fix: use fictional data.
  • Mistake: Automatically declaring “malware” from one string match. Fix: output “needs review” unless evidence is conclusive.
  • Mistake: Running response actions from unreviewed scripts. Fix: separate detection from response and require authorization.
  • Mistake: Logging sensitive command lines into public reports. Fix: redact tokens, private paths, personal data, and secrets.
  • Mistake: Ignoring time zones. Fix: normalize timestamps and document the time source.

9.5 Why secure coding matters

Detection engineering is software engineering. Bad detection code can miss attacks, create excessive alerts, leak sensitive data, or trigger unsafe response actions. Secure practices include input validation, careful parsing, least privilege, audit logging, code review, testing with sample data, and change control.

10. Security Perspective

10.1 Common attacks visible through endpoint telemetry

  • Suspicious script execution.
  • Credential dumping tool behavior.
  • New persistence through services, scheduled tasks, startup folders, or registry keys.
  • Security tool tampering.
  • Unusual parent-child process chains.
  • Command-and-control beacons.
  • Unexpected archive creation before data exfiltration.
  • Ransomware-like mass file modification.
  • Remote access tool abuse.
  • Lateral movement using legitimate administration tools.

10.2 Common vulnerabilities and visibility gaps

  • Endpoint agent not installed on all devices.
  • Agent installed but unhealthy or outdated.
  • Command-line logging disabled.
  • Short log retention.
  • No central collection from laptops that are often offline.
  • No coverage for macOS, Linux, containers, or legacy systems.
  • Excessive local administrator access enabling tampering.
  • Weak separation between detection and response authority.
  • No documented triage process.
  • No privacy review for collected telemetry.

10.3 Defensive strategies

  • Define required telemetry per endpoint class.
  • Monitor agent health and coverage.
  • Collect command line, parent process, hash, user, and network context where legally and operationally appropriate.
  • Map detections to known adversary techniques and internal risk scenarios.
  • Use allowlists carefully for known administrative automation.
  • Separate high-confidence blocking from lower-confidence alerting.
  • Protect logs from alteration and enforce retention.
  • Write triage playbooks for common alert categories.
  • Run tabletop exercises using fictional telemetry.
  • Review false positives and false negatives after incidents.

10.4 Detection methods

  • Process-chain detection: suspicious parent-child relationships.
  • Command-line detection: risky flags or encoded commands.
  • Persistence detection: new startup entries, services, or scheduled jobs.
  • Tamper detection: security service stopped, logs cleared, protection disabled.
  • Rarity detection: file hash or command seen on very few endpoints.
  • Sequence detection: download → execution → script → network connection.
  • Threshold detection: many failed logins or many file changes.
  • Behavior clustering: grouping related events into one alert story.

10.5 Mitigation and best practices

Use EDR as part of a layered defense. Prevention reduces the number of events that need detection. Detection reduces time to discovery. Response reduces impact. Recovery returns the organization to a trusted state.

11. Industry Perspective

11.1 Banks

Banks use endpoint telemetry to protect privileged workstations, fraud teams, trading systems, call centers, and developer laptops. They need strong audit trails, rapid triage, and careful response because endpoint compromise can lead to fraud, data leakage, or transaction manipulation.

11.2 Governments

Government environments use EDR to monitor large fleets, detect espionage-style behavior, support incident response, and preserve evidence. They often require formal retention, chain-of-custody practices, and strict access control to sensitive logs.

11.3 Cloud providers

Cloud providers collect telemetry from administrative endpoints, production hosts, build systems, and workloads. They emphasize scale, automation, immutable infrastructure, workload identity, and rapid containment without disrupting customer systems.

11.4 Healthcare

Healthcare endpoints include clinical workstations, nursing stations, imaging systems, and administrative devices. Detection must support patient safety: isolating the wrong device at the wrong time may disrupt care, so response playbooks need clinical context.

11.5 Defense

Defense organizations use endpoint telemetry for high-assurance monitoring, insider-risk investigation, mission-system protection, and threat hunting. Data classification and operational security requirements strongly shape what can be collected and where it can be stored.

11.6 Universities

Universities have open networks, unmanaged student devices, research labs, public computing spaces, administrative systems, and sensitive records. Endpoint detection is strongest on managed devices; unmanaged devices often require network and identity telemetry instead.

11.7 Large enterprises

Large enterprises need mature processes: agent deployment, health dashboards, alert routing, analyst queues, detection engineering, tuning, executive metrics, legal review, privacy governance, and integration with incident response.

12. Research Perspective

Endpoint detection remains an active research area. Important directions include:

  • Behavioral detection: identifying malicious activity from sequences rather than single indicators.
  • Graph-based investigation: modeling processes, users, files, and network connections as graphs.
  • Machine learning: detecting anomalous behavior while controlling false positives and explainability problems.
  • Adversarial machine learning: understanding how attackers evade learned detectors.
  • Privacy-preserving telemetry: collecting useful security evidence without excessive user surveillance.
  • Cross-platform telemetry: normalizing Windows, Linux, macOS, container, and cloud workload events.
  • Automated triage: reducing analyst workload while preserving human accountability.
  • Attack emulation evaluation: measuring how well tools detect realistic adversary behaviors.

Notable research directions include provenance graphs, causal analysis, attack-story reconstruction, endpoint alert correlation, detection-as-code, adversary emulation, and telemetry minimization.

Important research caution: a model with high test accuracy may still fail operationally if the training data is unrealistic, the attacker adapts, the telemetry changes, or analysts cannot understand the reason for an alert.

13. Case Studies

13.1 Living-off-the-land intrusion

In many enterprise intrusions, attackers abuse legitimate administration tools instead of dropping obviously malicious files. The root cause is often stolen credentials, over-permissive administration paths, weak monitoring, or delayed triage. Lesson: endpoint detection should look at behavior and context, not only file reputation.

13.2 Ransomware behavior detection

Ransomware incidents often produce endpoint signals before the final impact: unusual process activity, privilege abuse, disabled security tools, large numbers of file modifications, deletion of backups or shadow copies, and lateral movement. Lesson: early detection and containment can reduce blast radius.

13.3 Alert fatigue in a SOC

A security operations center may receive thousands of low-quality alerts per day. Analysts become overloaded, important alerts are missed, and response slows. Root cause: broad rules without context, poor tuning, duplicate alerts, and unclear playbooks. Lesson: detection quality and workflow design matter as much as tool purchase.

13.4 Endpoint agent coverage gap

An organization may deploy EDR to laptops but forget servers, temporary cloud workloads, lab machines, or privileged administrator jump boxes. Attackers then operate in blind spots. Lesson: coverage, agent health, and asset inventory must be continuously measured.

13.5 Security tool tampering

Attackers sometimes attempt to stop services, disable protection, clear logs, or add exclusions. Root cause can include excessive local privilege and weak tamper protection. Lesson: telemetry about security-control changes is high value and should be investigated quickly.

14. Common Misconceptions

  • Misconception: “EDR prevents all endpoint attacks.” Correction: EDR improves visibility and response, but prevention, hardening, identity security, and recovery are still needed.
  • Misconception: “Every alert is an incident.” Correction: alerts require triage; many are benign or need tuning.
  • Misconception: “No alert means no attack.” Correction: telemetry gaps, evasion, misconfiguration, or missing coverage can hide activity.
  • Misconception: “More telemetry is always better.” Correction: excessive telemetry can increase cost, privacy risk, storage burden, and analyst noise.
  • Misconception: “Machine learning removes the need for analysts.” Correction: analysts still validate context, authorize response, handle exceptions, and improve detections.
  • Misconception: “Blocking is always better than alerting.” Correction: blocking can disrupt operations or destroy evidence if used without confidence and authority.
  • Misconception: “A detection mapped to ATT&CK is automatically good.” Correction: mapping helps organize coverage, but detection quality depends on data, logic, tuning, and operational validation.

15. Summary

Endpoint detection starts with telemetry. Telemetry becomes useful when it is normalized, enriched, searched, correlated, and interpreted. EDR tools help collect and analyze endpoint activity, but the defensive value depends on coverage, data quality, detection logic, triage process, response authority, and continuous improvement.

The central lesson is simple: do not treat the tool as magic. Know what you collect, know what you cannot see, test your detections, measure false positives and false negatives, protect privacy, and document every response action.

18. Practice Lab

Lab title

Fictional Endpoint Telemetry Triage

Prerequisites

  • Python installed on your own lab machine.
  • Basic CSV and command-line familiarity.
  • No real malware, no real organizational logs, and no public-system testing.

Objectives

  • Create fictional endpoint telemetry.
  • Write a safe detection script.
  • Build a small event timeline.
  • Classify alerts as benign, suspicious, or needs escalation.
  • Document evidence without exposing sensitive data.

Dataset

Create a file named fictional_endpoint_events.csv with these rows:

time,host,user,parent,process,command_line,remote_domain,action
2026-06-29T09:00:00Z,LAB-WS-01,student,explorer.exe,chrome.exe,chrome.exe,,allowed
2026-06-29T09:05:00Z,LAB-WS-01,student,chrome.exe,notepad.exe,notepad.exe notes.txt,,allowed
2026-06-29T09:20:00Z,LAB-WS-01,student,winword.exe,powershell.exe,powershell.exe -NoProfile,updates.example.invalid,allowed
2026-06-29T09:22:00Z,LAB-WS-01,student,powershell.exe,curl.exe,curl.exe https://updates.example.invalid/a.txt,updates.example.invalid,allowed
2026-06-29T09:40:00Z,LAB-WS-02,admin,services.exe,backup-agent.exe,backup-agent.exe --run,,allowed

Tasks

  1. Load the CSV with Python.
  2. Identify process events where an office application launched a shell or script interpreter.
  3. Build a timeline for LAB-WS-01.
  4. Write a short alert summary: what, where, user, time, evidence, and next step.
  5. List two possible benign explanations.
  6. List two reasons for escalation.
  7. Explain what extra telemetry would improve confidence.
  8. Calculate precision for a fictional rule that produced 3 useful alerts and 7 benign alerts.
  9. Write one tuning idea that does not hide genuinely risky activity.
  10. Submit a report using fictional data only.

Expected outcome

You should produce a defensive triage note that distinguishes evidence from conclusion. A strong answer says “suspicious chain requiring review,” not “confirmed malware,” unless the evidence supports that conclusion.

Safety rules

  • Do not run malware.
  • Do not disable endpoint protection.
  • Do not collect logs from real users or organizations.
  • Do not publish real hostnames, usernames, hashes, IP addresses, URLs, or screenshots.
  • Do not isolate, kill processes, quarantine files, or block domains on systems without explicit authority.
