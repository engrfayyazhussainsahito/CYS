Lesson 3.4: Public-Key Cryptography, Certificates, and TLS

Public-key cryptography solves a problem symmetric cryptography does not solve by itself: how can two parties communicate securely before they already share a secret? This lesson builds from public/private key pairs to digital signatures, certificates, public key infrastructure, HTTPS, and the TLS 1.3 handshake.

Estimated reading time220–320 minutes
Difficulty levelBeginner → MS foundation
Required backgroundLessons 3.1–3.3 recommended; all prerequisites reintroduced
Practice time5–8 hours

Keys do not prove identity by themselves.

A server can send a public key, but the client must know whether that key really belongs to the intended domain. Certificates, certificate authorities, hostname validation, and TLS combine to answer that question.

1. Key pairPrivate key stays secret; public key is shareable.
2. CertificateAn issuer binds a public key to a subject name.
3. TLSThe client validates identity and derives session keys.

Overview

Topic: asymmetric cryptography, key pairs, signatures, key establishment, certificates, certificate authorities, public key infrastructure, HTTPS, and TLS.

Learning objective: understand how modern systems authenticate servers, protect communication, manage certificate trust, and avoid dangerous TLS mistakes.

Required background: no prior PKI knowledge is assumed. You should know basic encryption, hashing, MACs, and symmetric encryption from Lessons 3.1–3.3, but this lesson re-explains required ideas.

Difficulty: begins with lock-and-signature analogies and ends with graduate-level reasoning about trust paths, certificate lifecycle, TLS termination, mTLS, revocation, transparency, and post-quantum migration.

1. Introduction

Public-key cryptography, also called asymmetric cryptography, uses two mathematically related keys. One key is public and can be shared. The other key is private and must be protected. This is different from symmetric cryptography, where both parties need the same secret key.

The topic exists because the internet connects strangers. Your browser does not already share a secret key with every bank, hospital, university, cloud service, and government website. Public-key cryptography makes it possible to authenticate a party or establish a shared secret across an untrusted network.

But a public key alone does not prove identity. An attacker can generate their own key pair and claim, “this is the bank.” A certificate helps solve that by binding a public key to an identity, such as a domain name. A certificate authority signs that binding. A client accepts it only if it can build a valid chain from the server certificate to a trusted root.

TLS, Transport Layer Security, is the protocol used by HTTPS and many other applications to protect communication from eavesdropping, tampering, and message forgery. In modern TLS, public-key cryptography is mainly used for authentication and key establishment, while symmetric authenticated encryption protects the actual application data.

Who uses this? Web browsers, mobile apps, APIs, banks, governments, cloud providers, hospitals, universities, defense systems, software update systems, container registries, VPNs, service meshes, and researchers working on secure communication protocols.

Safety boundary: All demonstrations are local or defensive. Do not intercept, downgrade, spoof, scan, test, or manipulate TLS traffic for systems you do not own or have explicit written authorization to examine.

3. Prerequisites

This lesson uses these concepts. Each is explained here before advanced use:

  • Secret: information that must not be known by unauthorized people or systems.
  • Key: data used by a cryptographic algorithm to encrypt, decrypt, sign, verify, or establish other keys.
  • Hash: a fixed-size fingerprint of data. A cryptographic hash should be hard to reverse or collide intentionally.
  • Signature: cryptographic evidence that a private key was used to sign specific data.
  • Identity: the name or subject being authenticated, such as a domain, service, device, or organization.
  • Protocol: a defined conversation between computers. TLS is a protocol.
  • Client and server: the client starts a connection; the server responds. In HTTPS, a browser is usually the client.
Prerequisite mini-explanation: authentication vs encryption

Encryption hides content. Authentication answers “who am I talking to?” or “who produced this message?” TLS needs both. A connection can be encrypted to the wrong party if certificate validation is disabled or broken.

4. Core Concepts

4.1 The problem with only symmetric keys

Symmetric encryption is efficient, but both sides need the same secret key. That creates a bootstrapping problem: how do two parties agree on a secret when the network may be observed by attackers?

flowchart LR
  A[Alice wants secure communication] --> P{Do Alice and Bob already share a secret?}
  P -- Yes --> S[Use symmetric authenticated encryption]
  P -- No --> K[Need public-key authentication or key establishment]
  K --> T[TLS-style handshake]
  T --> AEAD[Use symmetric AEAD for application data]

Public-key cryptography helps by separating the shareable part from the secret part.

4.2 Public/private key pairs

A key pair contains a public key and a private key. The private key must remain secret. The public key can be distributed widely. The mathematical relationship lets one key perform an operation that can be checked or paired with the other, depending on the algorithm.

TermPlain meaningSecurity role
Private keyThe secret half of the pairSigns data, decrypts in some schemes, or participates in key agreement
Public keyThe shareable half of the pairVerifies signatures, encrypts in some schemes, or participates in key agreement
Key pairMathematically related keysEnables asymmetric cryptographic operations

4.3 Encryption, signatures, and key establishment are different

Beginners often say “public key encryption” for every public-key operation. That is imprecise. Modern systems use different primitives for different goals.

PrimitiveQuestion answeredExample use
Public-key encryptionCan only the private-key holder read this small encrypted value?Hybrid encryption designs; RSA-OAEP in some applications
Digital signatureDid the private-key holder sign this exact data?Certificates, software updates, signed tokens, documents
Key agreement / key encapsulationCan two sides establish a shared secret over an untrusted network?TLS handshakes, VPNs, secure messaging

4.4 Digital signatures

A digital signature is like a tamper-evident seal, but stronger when implemented correctly. The signer uses a private key to sign data. Anyone with the public key can verify that the signature matches the data and public key.

flowchart LR
  M[Message] --> H[Hash message]
  H --> S[Sign hash with private key]
  SK[(Private key)] --> S
  S --> Sig[Signature]
  M --> V[Verifier]
  Sig --> V
  PK[(Public key)] --> V
  V --> R{Valid?}
  R -- Yes --> OK[Accept signature]
  R -- No --> NO[Reject]

A signature does not automatically mean the signer was honest, authorized, or human. It means the private key corresponding to the public key produced a valid signature over that data. Identity and authorization still require context.

4.5 Certificates

A certificate is a signed data structure that binds a public key to subject information. For a website, the subject is usually a domain name such as example.edu. The certificate includes fields such as subject, issuer, public key, validity dates, serial number, extensions, and the issuer's signature.

An analogy: a student ID card contains a name and photo, but it is trusted because the university issued it. A certificate contains a public key and names, but it is trusted because an issuer signed it and the client trusts the issuer chain.

4.6 Certificate authorities and PKI

A certificate authority is an entity trusted to issue certificates. A public key infrastructure, or PKI, is the larger system of certificate authorities, policies, validation rules, trust stores, issuance workflows, revocation mechanisms, audits, and lifecycle processes.

A typical public TLS certificate chain has:

  1. Server certificate: belongs to the website or service.
  2. Intermediate CA certificate: signs server certificates and limits exposure of root keys.
  3. Root CA certificate: a trust anchor stored in operating systems or browsers.
flowchart TB
              R[Trusted Root CA
installed in browser or OS trust store] --> I[Intermediate CA
validated by root signature]
              I --> S[Server certificate
validated by intermediate signature]
              S --> D[Domain name and server public key]
              D --> C[Client accepts only if chain, dates, names, constraints, and policies validate]

4.7 Certificate validation

Certificate validation is not one check. A client normally checks:

  • The certificate chain signatures are valid.
  • The chain ends at a trusted root.
  • The current time is within each certificate's validity period.
  • The server name matches the certificate's Subject Alternative Name.
  • The certificate is allowed for server authentication.
  • Basic constraints and key usage restrictions are respected.
  • The algorithms and key sizes satisfy policy.
  • Revocation or status information is considered according to client policy.
High-risk mistake: disabling certificate validation does not merely hide a warning. It removes the client's evidence that it is connected to the intended server.

4.8 TLS and HTTPS

TLS protects application communication. HTTPS is HTTP over TLS. TLS aims to prevent network attackers from reading or modifying application data and from impersonating the server when validation works.

HTTPS does not prove that a website is ethical, truthful, safe to buy from, free from malware, or immune to server-side compromise. It mainly proves that the browser built an encrypted and authenticated connection to the named site according to certificate-validation rules.

4.9 TLS 1.3 handshake at a high level

TLS 1.3 simplified the handshake compared with older TLS versions. At a high level:

  1. The client proposes supported cryptographic parameters and sends key-share material.
  2. The server chooses parameters, sends its key-share material, and provides a certificate.
  3. The client validates the certificate chain and hostname.
  4. Both sides derive shared secrets and handshake keys.
  5. The server proves possession of the private key using a signature.
  6. Both sides switch to symmetric authenticated encryption for application data.
sequenceDiagram
  participant C as Client
  participant S as Server
  C->>S: ClientHello + supported versions + key share
  S->>C: ServerHello + selected parameters + key share
  S->>C: Certificate + CertificateVerify + Finished
  Note over C: Validate chain, hostname, dates, policy
  C->>S: Finished
  Note over C,S: Both derive symmetric traffic keys
  C-->>S: Encrypted HTTP request
  S-->>C: Encrypted HTTP response

4.10 Mutual TLS

Ordinary HTTPS usually authenticates the server to the client. Mutual TLS, or mTLS, also authenticates the client using a client certificate. It is common in service-to-service communication, high-security APIs, device fleets, and some enterprise systems.

4.11 Certificate lifecycle

Certificate security is an operational process, not a one-time install. Teams must manage issuance, private-key custody, renewal, inventory, monitoring, revocation, emergency replacement, and decommissioning. Certificate expiration can cause outages. Private-key leakage can cause impersonation. Misissuance can create trust risk.

4.12 Post-quantum awareness

Large cryptographically relevant quantum computers could threaten RSA and elliptic-curve public-key systems. That does not mean TLS is broken today, but organizations need crypto inventory, crypto agility, and migration planning. Modern research and standards work includes post-quantum key-establishment and signature schemes and hybrid TLS experiments.

7. Mathematical Foundations

This section gives intuition, not enough detail to design your own cryptographic algorithm. Do not invent crypto from these examples.

7.1 One-way functions

Public-key cryptography relies on mathematical problems that are easy in one direction and hard in the reverse direction without secret information. A beginner analogy is mixing paint: it is easy to mix colors, hard to recover the original exact amounts. The real mathematics is more precise.

7.2 Modular arithmetic

Modulo means remainder after division. For example, 17 mod 5 = 2 because 17 divided by 5 leaves remainder 2. Modular arithmetic is used in many public-key systems.

Example:

  • 7 × 8 = 56
  • 56 mod 11 = 1
  • So 7 × 8 ≡ 1 mod 11

7.3 Toy Diffie-Hellman example

Diffie-Hellman-style key agreement lets two parties derive the same shared value while an observer sees only public values. Real systems use much larger groups or elliptic curves. This tiny example is only for learning.

  1. Public values: prime p = 23, generator g = 5.
  2. Alice chooses private number a = 6.
  3. Bob chooses private number b = 15.
  4. Alice sends A = g^a mod p = 5^6 mod 23 = 8.
  5. Bob sends B = g^b mod p = 5^15 mod 23 = 19.
  6. Alice computes B^a mod p = 19^6 mod 23 = 2.
  7. Bob computes A^b mod p = 8^15 mod 23 = 2.

Both get 2. An attacker saw 23, 5, 8, and 19, but not the private exponents. Real security depends on using approved large parameters and safe implementations.

7.4 RSA intuition

RSA is based on arithmetic modulo a large number n that is the product of two large primes. Multiplying two large primes is easy for a computer; factoring the product back into the original primes is believed to be hard at sufficient sizes for classical computers. Practical RSA must use secure padding such as OAEP for encryption or PSS for signatures. Raw textbook RSA is unsafe.

7.5 Elliptic-curve intuition

Elliptic-curve cryptography uses points on special curves and operations on those points. The useful property is that repeated point addition is easy, but reversing it—the elliptic-curve discrete logarithm problem—is believed to be hard for classical computers at approved sizes. ECC often provides similar classical security with smaller keys than RSA.

7.6 Certificate path validation as a graph problem

You can model certificates as directed edges: issuer signs subject. A client tries to find a valid path from the server certificate to a trusted root. A path is accepted only when signatures, dates, constraints, names, and policies all pass.

flowchart LR
  S[Server cert] --> I[Intermediate CA]
  I --> R[Root CA]
  R --> T[(Trust store)]
  S -. hostname, dates, usage .-> Checks[Policy checks]
  I -. constraints .-> Checks
  R -. trusted anchor .-> Checks
  Checks --> Decision{Accept?}

8. Practical Examples

8.1 Linux examples

# Safe local public-key and TLS lab.
# This creates demo keys and a self-signed certificate on your own computer.
# It does not scan or test any external host.

mkdir -p lesson-3-4-lab
cd lesson-3-4-lab

# 1) Generate an elliptic-curve private key for local testing.
openssl ecparam -name prime256v1 -genkey -noout -out lab-server.key
chmod 600 lab-server.key

# 2) Create a self-signed certificate for localhost.
openssl req -new -x509 -key lab-server.key -out lab-server.crt -days 7 \
  -subj "/CN=localhost" \
  -addext "subjectAltName=DNS:localhost,IP:127.0.0.1"

# 3) Inspect the certificate fields.
openssl x509 -in lab-server.crt -noout -subject -issuer -dates -fingerprint -sha256 -text | less

# 4) Start a local TLS test server in one terminal.
# Stop with Ctrl+C when finished.
openssl s_server -accept 8443 -cert lab-server.crt -key lab-server.key -www

# 5) In a second terminal, connect to your local server.
openssl s_client -connect 127.0.0.1:8443 -servername localhost -showcerts

8.3 Python examples

# Safe local signature example.
# Requires: python -m pip install cryptography
# Goal: sign a message with a private key and verify it with the public key.

from cryptography.hazmat.primitives.asymmetric import ec
from cryptography.hazmat.primitives import hashes, serialization
from cryptography.exceptions import InvalidSignature

# Generate a local teaching key pair.
private_key = ec.generate_private_key(ec.SECP256R1())
public_key = private_key.public_key()

message = b"course=ms-cybersecurity;lesson=3.4;status=draft"

# Sign the message digest with the private key.
signature = private_key.sign(message, ec.ECDSA(hashes.SHA256()))
print("Signature length:", len(signature))

# Verify with the public key. Verification returns None when valid.
public_key.verify(signature, message, ec.ECDSA(hashes.SHA256()))
print("Valid signature")

# Tampering changes the message and makes verification fail.
try:
    public_key.verify(signature, b"course=changed", ec.ECDSA(hashes.SHA256()))
except InvalidSignature:
    print("Tampered message rejected")

# Export the public key in PEM format so others can verify signatures.
pem = public_key.public_bytes(
    encoding=serialization.Encoding.PEM,
    format=serialization.PublicFormat.SubjectPublicKeyInfo,
)
print(pem.decode())

# This is a defensive checklist, not an attack procedure.

9. Code Examples

9.1 Safe Python signature code

# Safe local signature example.
# Requires: python -m pip install cryptography
# Goal: sign a message with a private key and verify it with the public key.

from cryptography.hazmat.primitives.asymmetric import ec
from cryptography.hazmat.primitives import hashes, serialization
from cryptography.exceptions import InvalidSignature

# Generate a local teaching key pair.
private_key = ec.generate_private_key(ec.SECP256R1())
public_key = private_key.public_key()

message = b"course=ms-cybersecurity;lesson=3.4;status=draft"

# Sign the message digest with the private key.
signature = private_key.sign(message, ec.ECDSA(hashes.SHA256()))
print("Signature length:", len(signature))

# Verify with the public key. Verification returns None when valid.
public_key.verify(signature, message, ec.ECDSA(hashes.SHA256()))
print("Valid signature")

# Tampering changes the message and makes verification fail.
try:
    public_key.verify(signature, b"course=changed", ec.ECDSA(hashes.SHA256()))
except InvalidSignature:
    print("Tampered message rejected")

# Export the public key in PEM format so others can verify signatures.
pem = public_key.public_bytes(
    encoding=serialization.Encoding.PEM,
    format=serialization.PublicFormat.SubjectPublicKeyInfo,
)
print(pem.decode())

Line-by-line explanation:

  • ec.generate_private_key(...) creates a local teaching key pair.
  • public_key() derives the public key from the private key.
  • message is harmless data to sign.
  • private_key.sign(...) signs the exact bytes of the message using ECDSA with SHA-256.
  • public_key.verify(...) checks the signature against the message and public key.
  • The tampering block shows that changing the message invalidates the signature.
  • The PEM export shows a common text format for public keys.

9.2 Safe TLS client configuration

# Safe TLS client configuration pattern in Python.
# This example shows correct certificate verification settings.
# It does not connect unless you call create_connection yourself.

import ssl

context = ssl.create_default_context()
context.minimum_version = ssl.TLSVersion.TLSv1_2
context.check_hostname = True
context.verify_mode = ssl.CERT_REQUIRED

# Bad patterns to avoid:
# context.check_hostname = False
# context.verify_mode = ssl.CERT_NONE
# requests.get("https://example.com", verify=False)

print("TLS context requires certificate validation and hostname checking.")

9.3 Dangerous certificate-validation bypass

# Dangerous pattern: do not copy into real software.
# Problem: certificate validation is disabled, allowing man-in-the-middle attacks.

import requests

response = requests.get("https://example.com", verify=False)
print(response.status_code)

# Correct pattern: use default validation or a trusted CA bundle.
# response = requests.get("https://example.com", timeout=10)
Why this matters: A backend service that disables certificate validation can leak API tokens, session cookies, private data, or transaction details to a man-in-the-middle attacker.

9.4 Common mistakes

  • Using self-signed certificates in public production systems without a trust-distribution plan.
  • Disabling hostname validation to fix a certificate-name mismatch.
  • Storing private keys in Git repositories or container images.
  • Letting certificates expire without monitoring.
  • Using one certificate and private key everywhere, increasing blast radius.
  • Assuming HTTPS means the website is trustworthy in a business sense.
  • Confusing digital signatures with encryption.

10. Security Perspective

10.1 Common attacks

  • Man-in-the-middle: attacker positions between client and server and tries to impersonate one side.
  • Certificate misissuance: a CA incorrectly issues a certificate for a domain.
  • Private-key theft: attacker steals a server, code-signing, or CA private key.
  • Expired certificate outage: legitimate service becomes unavailable or untrusted after expiry.
  • Validation bypass: code disables certificate or hostname checks.
  • Downgrade attack: attacker tries to force weaker protocol versions or cipher suites.
  • Weak algorithm use: old hashes, short keys, or deprecated protocols remain enabled.
  • Improper TLS termination: traffic is protected on the public internet but exposed internally without adequate controls.

10.2 Common vulnerabilities

  • No certificate inventory.
  • No certificate expiry monitoring.
  • Private keys readable by too many users or services.
  • Certificates issued for wrong names.
  • Backend clients using verify=False.
  • Old TLS versions or weak cipher suites enabled for compatibility.
  • Missing incident plan for key compromise.

10.3 Defensive strategies

  • Use modern TLS libraries and default validation.
  • Automate certificate issuance and renewal.
  • Maintain a certificate and key inventory.
  • Restrict private-key access with filesystem permissions, secrets managers, HSMs, or managed certificate services.
  • Monitor Certificate Transparency logs for unexpected public certificates.
  • Disable deprecated protocols and weak cipher suites according to policy.
  • Use mTLS for service-to-service or device authentication where appropriate.
  • Document emergency rotation and revocation processes.

10.4 Detection methods

  • Alert on certificates near expiry.
  • Alert on certificate changes for critical domains.
  • Monitor CT logs for unexpected domains or issuers.
  • Scan internal code for validation-bypass patterns such as verify=False or disabled hostname checks.
  • Log TLS handshake failures and unusual client-certificate failures.
  • Monitor private-key file access where supported.

10.5 Mitigation and best practices

  • Prefer TLS 1.3 where supported and keep TLS libraries patched.
  • Use trusted public CAs for public websites and managed private CAs for internal PKI.
  • Use short-lived certificates with automation instead of long-lived manual certificates.
  • Never commit private keys to source control.
  • Use defense in depth: TLS plus authentication, authorization, input validation, logging, and monitoring.
  • Plan for post-quantum migration by maintaining a crypto inventory.

12. Research Perspective

Research on public-key cryptography and TLS is active because real-world trust systems are large, messy, and constantly changing.

  • Post-quantum TLS: designing and deploying key-establishment and signature systems resistant to future quantum attacks.
  • Hybrid key exchange: combining classical and post-quantum methods during migration.
  • Certificate Transparency: improving public auditability of certificates and detection of misissuance.
  • Revocation: making certificate revocation private, reliable, fast, and deployable.
  • Formal verification: mathematically analyzing protocol designs and implementations to reduce subtle bugs.
  • Implementation security: preventing side-channel leaks, parsing bugs, memory-safety failures, and unsafe API usage.
  • Identity alternatives: studying decentralized identifiers, verifiable credentials, DANE, and private PKI models.
  • Certificate lifecycle automation: reducing outages caused by manual processes and hidden certificate dependencies.

Notable research directions include the original Diffie-Hellman public-key exchange idea, RSA, elliptic-curve cryptography, Certificate Transparency, analyses of TLS certificate validation failures, and modern post-quantum cryptography standardization. Do not reproduce papers; read abstracts, threat models, methodology, and conclusions, then write your own critique.

13. Case Studies

13.1 DigiNotar CA compromise

What happened: Attackers compromised a certificate authority and fraudulent certificates were issued. Some were reportedly used to intercept traffic.

Root cause: CA security and monitoring failures, combined with broad trust placed in public root programs.

Impact: Browser and operating-system vendors distrusted the CA, and the incident became a major example of systemic PKI risk.

Lessons learned: CA compromise can affect many users; transparency, monitoring, segmentation, auditing, and incident response matter.

13.2 Heartbleed and private-key exposure risk

What happened: A memory disclosure vulnerability in OpenSSL allowed attackers to read memory from vulnerable systems.

Root cause: Implementation bug in a widely used TLS library.

Impact: Sensitive memory, potentially including keys and credentials, could be exposed. Many organizations had to patch, rotate keys, and reissue certificates.

Lessons learned: Strong protocols still depend on secure implementations, patching, key rotation, and incident planning.

13.3 Certificate expiration outages

What happened: Organizations have repeatedly suffered outages when certificates expired unexpectedly in public services, internal APIs, or device chains.

Root cause: poor inventory, ownership ambiguity, manual renewal, hidden dependencies, and insufficient monitoring.

Impact: service downtime, failed API calls, customer disruption, and emergency operational work.

Lessons learned: certificate management is an availability and resilience issue, not only a cryptography issue.

13.4 Backend validation bypass

What happened: In many real codebases, developers disable TLS validation to work around local testing or misconfigured certificates.

Root cause: confusing errors, missing local development CA, pressure to ship, and lack of secure defaults.

Impact: backend services can leak secrets or trust attacker-controlled endpoints.

Lessons learned: testing environments need proper local trust configuration; production code should fail closed.

14. Common Misconceptions

MisconceptionWhy it is wrong
HTTPS means the website is safe.HTTPS protects the connection to a named site; it does not prove the business is honest or the application has no vulnerabilities.
A public key proves identity.Anyone can generate a key. Identity requires validation, usually through certificates and trust paths.
Self-signed certificates are always bad.They can be appropriate in controlled labs or private systems with managed trust distribution, but they are not automatically trusted by public clients.
Disabling certificate validation is acceptable for production if traffic is encrypted.Encryption to an attacker is not secure communication with the intended server.
TLS replaces application authentication.TLS protects the channel. Applications still need user, device, or service authentication and authorization.
Certificates only matter for public websites.Internal APIs, service meshes, devices, VPNs, and software update systems also depend on certificates and keys.
Public-key encryption should encrypt large files directly.Practical systems use hybrid encryption: public-key methods establish or protect a small key, then symmetric AEAD encrypts data.
Certificate pinning is always good.It can reduce some MITM risk but can also cause outages and renewal failures if not designed carefully.

15. Summary

Public-key cryptography lets systems authenticate identities, verify signatures, and establish shared secrets across untrusted networks. Certificates bind public keys to names. Certificate authorities and PKI create a trust system around those bindings. TLS uses these mechanisms to authenticate servers and derive symmetric traffic keys, while modern AEAD protects the application data.

The most important practical lesson is that algorithms are not enough. Real security depends on correct validation, private-key protection, certificate lifecycle management, monitoring, safe defaults, and migration planning.

17. Self-Assessment

20 MCQs

1. What problem does public-key cryptography help solve?

Answer: A. Public-key cryptography lets parties use public keys for encryption, signatures, or key establishment without first sharing the same secret key.

2. Which key should remain secret in a public-key pair?

Answer: B. The private key must be protected. The public key is designed to be shared.

3. What does a digital signature primarily provide?

Answer: B. A signature lets verifiers check that a private-key holder signed data and that the signed data was not changed.

4. What is a certificate?

Answer: A. An X.509 certificate binds a public key to subject information and is signed by a certificate authority or issuer.

5. What is a certificate authority?

Answer: B. A CA issues certificates and signs them so clients can build trust paths.

6. What does hostname validation check?

Answer: A. A valid certificate for the wrong name should not authenticate the site you meant to visit.

7. What is TLS used for?

Answer: A. TLS protects network application communication, including HTTPS.

8. Which version is the modern IETF standard that simplified the TLS handshake?

Answer: C. TLS 1.3 is specified in RFC 8446 and modernizes the handshake and cipher suite design.

9. Why is public-key cryptography often combined with symmetric encryption?

Answer: A. Hybrid designs use public-key methods to authenticate or establish keys and symmetric AEAD for data protection.

10. What is a root certificate?

Answer: A. Root certificates are trust anchors used to validate certificate chains.

11. What is a certificate chain?

Answer: A. Clients validate chains by checking signatures, validity periods, names, and constraints.

12. What is revocation intended to handle?

Answer: A. Revocation mechanisms help reject certificates after compromise or misissuance.

13. What is Certificate Transparency?

Answer: A. CT helps detect unexpected or misissued public certificates.

14. What is the main danger of disabling certificate validation?

Answer: A. Without validation, a man-in-the-middle can present an untrusted certificate and intercept traffic.

15. What does mTLS add compared with ordinary server-authenticated TLS?

Answer: A. Mutual TLS authenticates both server and client using certificates.

16. What is a private-key custody problem?

Answer: A. Private keys are high-value secrets and require restricted access, auditing, backup, and rotation planning.

17. Why are self-signed certificates not enough for public websites?

Answer: A. Public clients need a trust path to a known trust anchor.

18. What is ACME commonly used for?

Answer: A. ACME is a protocol used to automate certificate management.

19. What is a post-quantum concern for public-key cryptography?

Answer: A. Post-quantum migration addresses the risk that large quantum computers could break RSA and ECC-style public-key systems.

20. What should an application do when TLS certificate validation fails?

Answer: A. Validation failure means identity or trust cannot be established.

18. Practice Lab

Lab title: Local PKI and TLS Validation Lab

Safety: This lab uses only your local machine and harmless demo keys. Do not connect to, scan, intercept, or test third-party systems.

Prerequisites

  • Linux, macOS, or Windows with OpenSSL or equivalent tools.
  • Basic terminal use from Week 0.
  • Python 3.10+ recommended for the signature exercise.

Objectives

  • Create a local private key and self-signed certificate.
  • Inspect certificate fields.
  • Run a local TLS test server.
  • Connect with a local TLS client and observe certificate validation behavior.
  • Generate and verify a local digital signature.
  • Write a short risk memo about certificate validation bypass.

Steps

  1. Create a lab folder and generate an EC private key.
  2. Create a short-lived self-signed certificate for localhost.
  3. Inspect subject, issuer, validity dates, SAN, public key, and SHA-256 fingerprint.
  4. Start a local TLS server on 127.0.0.1:8443.
  5. Connect with openssl s_client and record protocol, cipher suite, certificate subject, and verification result.
  6. Run the Python signature example and explain why the tampered message fails verification.
  7. Write a one-page memo: what would go wrong if an enterprise backend disabled certificate validation?

Expected outcomes

  • You can identify certificate fields and explain their purpose.
  • You can explain the difference between self-signed lab trust and public CA trust.
  • You can explain why certificate validation failures should not be ignored.
  • You can demonstrate signing and verification locally.

Deliverables

  • Terminal screenshots or copied command output.
  • Certificate field notes.
  • Signature verification notes.
  • One-page risk memo.
