Lesson 3.5: Crypto Misuse, Key Management, and Crypto Agility

Most real cryptographic failures do not come from someone breaking AES with pure mathematics. They come from weak randomness, leaked keys, bad modes, disabled validation, missing metadata, poor rotation, and systems that cannot migrate when algorithms age. This lesson teaches how to recognize and prevent those failures.

Estimated reading time230–340 minutes
Difficulty levelBeginner → MS foundation
Required backgroundLessons 3.1–3.4 recommended; concepts reintroduced
Practice time5–9 hours

Strong algorithms fail when systems are weak.

The practical question is not only “Which algorithm?” It is also “Where are the keys? Who can use them? How are they rotated? How is misuse detected? How will this system migrate when cryptography changes?”

1. Avoid misuseNo custom crypto, no ECB, no disabled validation, no raw password hashes.
2. Manage keysGenerate, store, authorize, rotate, audit, revoke, and destroy keys.
3. Design agilityKeep inventory and metadata so cryptography can change safely.

Overview

Topic: cryptographic misuse, operational key management, cryptographic inventory, and crypto agility.

Learning objective: learn how cryptographic systems fail in practice and how to design systems that can rotate keys, recover from compromise, and migrate to newer algorithms.

Required background: no production cryptography experience is assumed. The lesson reintroduces keys, hashes, MACs, authenticated encryption, certificates, and TLS where needed.

Difficulty: starts with beginner examples and ends with graduate-level reasoning about key lifecycles, KMS/HSM designs, cryptographic bills of materials, migration planning, and post-quantum transition risk.

1. Introduction

Cryptographic misuse means using cryptography in a way that fails to provide the intended security goal. The algorithm may be strong, but the system can still be weak. A door made of steel does not help if the key is taped to the wall.

This topic exists because real software has to manage keys, random numbers, modes, parameters, libraries, certificates, protocols, logs, backups, and migration. A cryptographic primitive is only one part of a larger system.

It matters because cryptography often protects the highest-value assets: passwords, bank transactions, health records, government secrets, cloud credentials, software updates, backups, and long-lived personal data. A single key leak can expose years of data. A single certificate-validation bypass can turn encrypted traffic into attacker-controlled traffic.

It is used anywhere cryptography is used: web applications, mobile apps, APIs, databases, disk encryption, cloud storage, payment systems, identity systems, software signing, messaging, VPNs, backups, medical devices, and industrial systems.

Who uses it? Software engineers, cloud engineers, security architects, auditors, incident responders, compliance teams, penetration testers, cryptographers, product managers, and executives who approve risk.

Safety boundary: this lesson teaches defensive review and local training exercises. Do not search for secrets, scan repositories, test systems, or analyze code that you do not own or have explicit written authorization to review.

2. Learning Outcomes

After completing this lesson, you should be able to:

  • Define cryptographic misuse and recognize common examples.
  • Explain why strong algorithms can fail when keys, randomness, modes, validation, or metadata are mishandled.
  • Distinguish keys, secrets, passwords, certificates, tokens, DEKs, KEKs, KMS, and HSMs.
  • Describe the key-management lifecycle: generation, storage, distribution, use, rotation, backup, revocation, audit, and destruction.
  • Explain cryptoperiods, key separation, least privilege for keys, dual control, and separation of duties.
  • Identify dangerous patterns such as ECB mode, nonce reuse, raw password hashes, disabled TLS verification, hardcoded keys, and custom cryptography.
  • Build a simple local cryptographic inventory for a fictional application.
  • Explain crypto agility and why it matters for post-quantum migration.
  • Design versioned encryption metadata so future key rotation and algorithm migration are possible.
  • Write a practical risk memo explaining cryptographic misuse to technical and non-technical audiences.

3. Prerequisites

This lesson assumes no professional cryptography background. The following concepts are useful:

Plaintext
Readable original data before encryption.
Ciphertext
Encrypted data that should not reveal the plaintext without the correct key.
Key
A secret or private value used by a cryptographic algorithm. The key is usually more sensitive than the algorithm name.
Nonce or IV
A value used once, often to ensure that encrypting similar messages does not produce predictable ciphertext.
MAC or authentication tag
A value used to detect tampering and authenticate data under a secret key.
Certificate
A signed statement that binds a public key to an identity, such as a domain name.

Analogy: a safe, a key, a receipt, and a maintenance log serve different purposes. Encryption is like the safe. The key opens it. The authentication tag is like a tamper-evident seal. The inventory is the maintenance log that tells you which safe, which key, and which policy apply.

4. Core Concepts

4.1 Cryptography is a system, not a spell

Beginners often ask, “Is AES secure?” A more useful question is: “Which AES mode? What key length? How is the key generated? Where is the key stored? Is encryption authenticated? Are nonces unique? Who can decrypt? Are actions logged? Can keys be rotated? What happens after compromise?”

A mathematically strong primitive can fail when system assumptions are wrong. Security depends on the whole chain.

4.2 Common cryptographic misuse patterns

4.3 Key management

Key management is the discipline of controlling cryptographic keys across their full lifecycle. It is not only “where do we store the key?” It includes generation, ownership, authorization, use, rotation, backup, recovery, revocation, destruction, monitoring, and evidence.

Key lifecycle:

  1. Generate: create keys with approved random generation and appropriate strength.
  2. Register: assign key ID, owner, purpose, creation date, algorithm, and cryptoperiod.
  3. Store: protect keys using KMS, HSM, secure enclave, or equivalent controls.
  4. Authorize: allow only specific identities, services, and operations.
  5. Use: perform encryption, decryption, signing, verification, wrapping, or derivation as allowed.
  6. Monitor: log key use, denied operations, policy changes, and anomalies.
  7. Rotate: introduce new keys and phase out old keys safely.
  8. Revoke or disable: stop use after compromise, role change, expiry, or policy violation.
  9. Destroy: remove keys when retention and recovery requirements are satisfied.

4.4 Key hierarchy: master keys, KEKs, and DEKs

A data-encryption key, or DEK, encrypts data. A key-encryption key, or KEK, protects other keys. A higher-level master key may live in a managed KMS or HSM. This layered design is called envelope encryption.

Analogy: a file cabinet contains documents. Each drawer has its own small key. Those drawer keys are kept in a locked key box. The key box is controlled by a security office. The drawer keys are DEKs; the key-box key is a KEK.

4.5 KMS and HSM

A Key Management Service, or KMS, is a system for creating, storing, controlling, rotating, and auditing cryptographic keys. A Hardware Security Module, or HSM, is specialized hardware designed to protect keys and perform cryptographic operations in a controlled environment.

KMS/HSM use does not automatically make a system secure. The access policy, logs, administrative controls, backup procedures, and incident response plan still matter.

4.6 Crypto agility

Crypto agility is the ability to change algorithms, keys, parameters, libraries, or providers without redesigning the entire system. It matters because algorithms age, implementations fail, compliance rules change, libraries are deprecated, and post-quantum migration is now a real planning requirement.

Crypto agility requires engineering discipline:

  • Cryptographic inventory.
  • Versioned ciphertext formats.
  • Algorithm identifiers.
  • Key identifiers.
  • Migration tests.
  • Rollback plans.
  • Supplier visibility.
  • Monitoring and audit logs.

4.7 Cryptographic inventory and CBOM

A cryptographic inventory records where cryptography is used: applications, protocols, libraries, certificates, keys, algorithms, data stores, devices, third-party products, APIs, and backup systems. A Cryptographic Bill of Materials, or CBOM, is a structured way to describe cryptographic components and dependencies.

You cannot rotate, audit, or migrate what you cannot find.

7. Mathematical Foundations

7.1 Bits of security

A key with n independent random bits has 2n possible values. For a uniformly random 128-bit key:

Number of possible keys = 2128

This is why guessing a properly generated 128-bit or 256-bit key is not the usual practical attack. Attackers look for weaker paths: leaked keys, bad random generation, wrong modes, exposed backups, weak access control, or disabled validation.

7.2 Expected exposure from a key compromise

A simple operational model:

Expected loss = probability of compromise × exposed data value × exposure duration factor

Suppose a key has a 2% annual compromise probability, protects data valued at $2,000,000, and poor logging means compromise may expose a full year of data. A rough expected annual loss estimate is:

0.02 × 2,000,000 × 1.0 = $40,000

If better key separation and rotation reduce the exposure duration factor to 0.25, the estimate becomes:

0.02 × 2,000,000 × 0.25 = $10,000

This is not a precise formula for all environments. It is a reasoning tool that shows why key separation, logging, and rotation reduce blast radius.

7.3 Birthday-bound intuition

Collision probability grows faster than beginners expect. For a space of size N, after about √N random samples, collisions become plausible. This is why a 64-bit identifier is not “half as strong” as a 128-bit identifier in collision-heavy settings; it is dramatically weaker for large-scale random assignment.

Approximate collision probability for k samples in a space of size N:

p ≈ k(k − 1) / (2N) when p is small.

This intuition matters for nonces, tokens, random IDs, and collision-resistant design.

Python examples

# Safe local crypto inventory example.
# Goal: inspect a small local training folder for obvious risky crypto patterns.
# Boundary: only run this on files you own or are authorized to review.

from pathlib import Path
import re

LAB_ROOT = Path("lesson-3-5-lab/app")
PATTERNS = {
    "hardcoded secret keyword": re.compile(r"(?i)(password|api[_-]?key|secret|token)\s*[:=]"),
    "disabled TLS verification": re.compile(r"(?i)(verify\s*=\s*false|CERT_NONE|verify=False)"),
    "ECB mode reference": re.compile(r"(?i)\bECB\b"),
    "fast hash for password hint": re.compile(r"(?i)(sha1|md5|sha256).*password"),
}

findings = []
for path in LAB_ROOT.rglob("*"):
    if path.is_file():
        text = path.read_text(errors="ignore")
        for line_no, line in enumerate(text.splitlines(), start=1):
            for name, pattern in PATTERNS.items():
                if pattern.search(line):
                    findings.append((path, line_no, name, line.strip()))

if not findings:
    print("No training findings found.")
else:
    for path, line_no, name, line in findings:
        print(f"{path}:{line_no}: {name}: {line}")

print("\nReminder: this is not a full scanner. It is a teaching aid for local files.")

Network example

Network cryptography misuse often appears as disabled certificate validation, outdated TLS versions, expired certificates, weak cipher suites, or private keys stored on too many hosts. In this beginner course, inspect only local or explicitly authorized lab systems.

# Safe local example: inspect your local TLS test server from Lesson 3.4.
# Start a localhost OpenSSL s_server first, then run:
openssl s_client -connect 127.0.0.1:8443 -servername localhost

# Defensive review questions:
# - Was certificate validation successful or self-signed only for lab use?
# - Which protocol version was negotiated?
# - Which cipher suite was selected?
# - Where is the private key stored?
# - How would renewal and rotation happen?

Web example

A secure web application should never treat encryption as a decoration. Sessions, password reset tokens, CSRF tokens, TLS, cookies, password hashing, and secret storage all need different controls.

Web crypto review checklist:
1. Passwords use slow password hashing, not encryption or raw SHA-256.
2. Session cookies use Secure, HttpOnly, and SameSite where appropriate.
3. Password reset tokens are random, single-use, time-limited, and stored safely.
4. TLS validation is never disabled in backend service calls.
5. API signing secrets are stored outside source code.
6. Encryption uses AEAD with unique nonces.
7. Encrypted records include algorithm, key_id, and version metadata.
8. Key use is logged without logging plaintext or secrets.

Cloud example

Cloud platforms commonly provide KMS, secrets managers, certificate managers, and HSM-backed services. The cloud does not remove your responsibility to configure IAM, logging, rotation, network boundaries, and incident response.

Cloud key-management design questions:
- Which service identity can decrypt which data?
- Is the key single-region or multi-region?
- Who can change key policy?
- Are decrypt operations logged?
- Is automatic rotation enabled where suitable?
- Are backups encrypted under separate keys?
- Are third-party SaaS integrations included in the crypto inventory?
- What is the emergency process if a key is suspected compromised?

9. Code Examples

9.1 Versioned encryption metadata

This example demonstrates a crypto-agile record format. It is still a teaching example, not a complete production design.

# Safe local example: versioned encryption metadata.
# Requires: python -m pip install cryptography
# Why it matters: crypto agility needs records that say which key and algorithm protected the data.

import base64
import json
from secrets import token_bytes
from cryptography.hazmat.primitives.ciphers.aead import AESGCM

# Training key registry. Real systems use KMS/HSM, access control, audit logs, and rotation.
KEYS = {
    "dek-2026-01": AESGCM.generate_key(bit_length=256),
    "dek-2026-06": AESGCM.generate_key(bit_length=256),
}
ACTIVE_KEY_ID = "dek-2026-06"
ALGORITHM = "AES-256-GCM"

def b64(data: bytes) -> str:
    return base64.b64encode(data).decode("ascii")

def ub64(text: str) -> bytes:
    return base64.b64decode(text.encode("ascii"))

def encrypt_record(plaintext: bytes, aad: bytes) -> str:
    key = KEYS[ACTIVE_KEY_ID]
    aesgcm = AESGCM(key)
    nonce = token_bytes(12)  # 96-bit nonce recommended for AES-GCM.
    ciphertext = aesgcm.encrypt(nonce, plaintext, aad)
    envelope = {
        "version": 1,
        "algorithm": ALGORITHM,
        "key_id": ACTIVE_KEY_ID,
        "nonce": b64(nonce),
        "aad": b64(aad),
        "ciphertext": b64(ciphertext),
    }
    return json.dumps(envelope)

def decrypt_record(envelope_json: str) -> bytes:
    envelope = json.loads(envelope_json)
    if envelope["algorithm"] != ALGORITHM:
        raise ValueError("unsupported algorithm")
    key = KEYS[envelope["key_id"]]
    aesgcm = AESGCM(key)
    return aesgcm.decrypt(ub64(envelope["nonce"]), ub64(envelope["ciphertext"]), ub64(envelope["aad"]))

record = encrypt_record(b"student_id=42;grade=A", aad=b"table=grades;row=42")
print(record)
print(decrypt_record(record))

# Common mistake: ciphertext with no algorithm, key_id, nonce, or AAD metadata becomes hard to migrate later.

Line-by-line explanation:

  • AESGCM.generate_key creates demonstration keys. Real systems should use a KMS/HSM or approved key-generation process.
  • ACTIVE_KEY_ID identifies the current encryption key so new records can use it.
  • ALGORITHM stores the algorithm name. This supports future migration.
  • token_bytes(12) creates a fresh nonce for AES-GCM.
  • aad binds external context to the ciphertext so context tampering is detected.
  • The JSON envelope stores metadata needed for future decryption and migration.
  • decrypt_record rejects unsupported algorithms instead of guessing.

9.2 Dangerous patterns

# Dangerous examples. Do not copy into real applications.

# 1) Hardcoded key: anyone with source access can recover it.
KEY = b"sixteen byte key"

# 2) Disabled TLS validation: allows man-in-the-middle interception.
# requests.get("https://example.com", verify=False)

# 3) Fast hash for passwords: too easy to guess offline.
# hashlib.sha256(password.encode()).hexdigest()

# 4) ECB mode: identical plaintext blocks produce identical ciphertext blocks.
# Cipher(algorithms.AES(key), modes.ECB())

# 5) Key reuse across purposes: one compromise breaks multiple security goals.
# same_key_used_for_encryption_and_signing = True
Common mistake: copying a Stack Overflow snippet that disables TLS validation or hardcodes a key can turn strong cryptography into a false sense of security.

10. Security Perspective

Common attacks and failures

  • Secret leakage: keys appear in repositories, logs, tickets, screenshots, build artifacts, mobile apps, or backups.
  • Man-in-the-middle attack: certificate validation is disabled or hostname checks are bypassed.
  • Offline password guessing: password hashes are stolen and were stored with weak parameters.
  • Replay attack: signed or encrypted messages lack freshness checks.
  • Padding oracle: error messages reveal information about plaintext validity.
  • Nonce reuse: repeated nonce/key combinations break security assumptions.
  • Privilege abuse: too many users or services can decrypt data.
  • Algorithm stagnation: old algorithms remain because no one knows where they are used.

Defensive strategies

  • Use approved libraries and standard protocols.
  • Use KMS/HSM or managed secrets infrastructure for high-value keys.
  • Enforce least privilege for decrypt, sign, export, rotate, and policy-update operations.
  • Use AEAD modes where appropriate.
  • Never disable certificate validation to bypass errors.
  • Use slow password hashing for passwords.
  • Maintain cryptographic inventory.
  • Log key usage and failed access attempts without logging secrets.
  • Test rotation and recovery before incidents.

Detection methods

  • Secret scanning in authorized repositories.
  • Static analysis for insecure crypto APIs.
  • Configuration checks for weak TLS settings.
  • KMS/HSM anomaly detection for unusual decrypt or sign volume.
  • Certificate transparency monitoring for unexpected certificates.
  • Log reviews for disabled validation, failed decrypt attempts, or sudden key-policy changes.

Mitigation and best practices

When misuse is found, mitigation depends on exposure. A hardcoded key usually needs immediate secret rotation, code cleanup, repository history review, log review, access review, and possibly customer notification. A bad encryption mode may require new encryption for future data and a controlled migration of old data.

12. Research Perspective

Current research is not limited to inventing new algorithms. Important research directions include:

  • Post-quantum migration: how to replace quantum-vulnerable public-key algorithms in protocols, products, identities, and long-lived data stores.
  • Crypto agility: how systems can safely support multiple algorithms and transition without downgrade attacks or operational outages.
  • Formal verification: proving that cryptographic protocols or implementations satisfy security properties.
  • Side-channel resistance: preventing leakage through timing, cache behavior, power consumption, electromagnetic signals, or error responses.
  • Cryptographic inventory automation: discovering cryptographic dependencies inside source code, binaries, configurations, certificates, containers, and network traffic.
  • Usable security: designing APIs that make dangerous cryptographic choices hard to make.
  • Threshold and distributed key management: reducing reliance on a single key holder or single HSM boundary.

Open problems include reliable CBOM standards, large-scale cryptographic discovery, safe hybrid post-quantum deployment, migration of legacy embedded devices, and reducing developer access to dangerous low-level cryptographic APIs.

13. Case Studies

Debian OpenSSL predictable randomness

A change in Debian's OpenSSL package reduced randomness quality, causing predictable cryptographic keys for affected systems. The root issue was not that public-key cryptography as a field failed; it was that key generation depended on correct entropy handling. Lesson: random generation is part of the security boundary.

WEP wireless encryption

WEP used weak design choices around RC4 and IV handling. Attackers could exploit repeated or weak IV behavior to recover keys. Lesson: modes, nonces, IVs, and protocol design matter as much as selecting an encryption algorithm.

DigiNotar certificate authority compromise

A certificate authority compromise led to fraudulent certificates. Lesson: public-key trust depends on operational security, auditing, revocation, browser trust decisions, and ecosystem governance.

Heartbleed and private-key exposure risk

Heartbleed was a memory disclosure bug in OpenSSL that could expose sensitive memory. One possible risk was private-key exposure, which would require certificate and key replacement. Lesson: implementation bugs can force emergency cryptographic rotation.

Source-code secret leaks

Organizations repeatedly suffer incidents where API keys or credentials are committed to repositories. Lesson: secret scanning, rotation, least privilege, and short-lived credentials reduce damage, but cannot replace disciplined secret handling.

14. Common Misconceptions

  • “AES means secure.” Wrong. AES used with a bad mode, bad nonce handling, exposed keys, or no authentication can still fail.
  • “Base64 is encryption.” Wrong. Base64 is encoding; anyone can decode it.
  • “Hashing a password with SHA-256 is enough.” Wrong. Password storage needs slow password hashing with salts and appropriate parameters.
  • “We can keep algorithms secret.” Wrong assumption. Secure systems should not depend on secret algorithms.
  • “Key rotation fixes all problems.” Incomplete. Rotation must include re-encryption, revocation, dependent systems, logs, access review, and recovery planning.
  • “KMS automatically makes us secure.” Wrong. Bad IAM policy, poor logging, overbroad decrypt permissions, or weak process can still expose data.
  • “Post-quantum migration is only for cryptographers.” Wrong. It affects inventory, procurement, software updates, certificates, protocols, vendors, and operations.
  • “If ciphertext decrypts, the design is fine.” Wrong. Correct decryption does not prove authenticity, nonce safety, key management, or compliance.

15. Summary

Cryptographic security depends on correct algorithms, correct usage, protected keys, reliable randomness, authenticated encryption, certificate validation, careful metadata, and operational discipline. The most common failures are avoidable: do not invent crypto, do not hardcode keys, do not use ECB, do not reuse nonces, do not disable TLS validation, do not store passwords with fast hashes, and do not deploy systems without key lifecycle management.

Crypto agility is the long-term engineering answer. Systems should know where cryptography is used, record algorithm and key metadata, support rotation, and be able to migrate when algorithms, threats, regulations, or research change.

16. Glossary

AEAD
Authenticated Encryption with Associated Data; encryption that also provides integrity/authenticity for ciphertext and optional associated data.
CBOM
Cryptographic Bill of Materials; a structured record of cryptographic components and dependencies.
Crypto agility
The ability to change cryptographic algorithms, keys, parameters, or providers with controlled disruption.
Cryptographic inventory
A record of where cryptography is used across systems, applications, protocols, data stores, devices, and vendors.
Cryptographic misuse
Incorrect use of cryptography that fails to provide the intended security property.
Cryptoperiod
The authorized lifetime or usage period of a cryptographic key.
DEK
Data-encryption key; a key used to encrypt data.
Envelope encryption
A design where data is encrypted with a DEK and the DEK is protected by a KEK or master key.
Hardcoded secret
A key, token, password, or credential embedded directly in source code or configuration.
HSM
Hardware Security Module; specialized hardware for protecting keys and performing cryptographic operations.
KEK
Key-encryption key; a key used to protect other keys.
Key rotation
Replacing a key and transitioning systems to the new key.
Key separation
Using different keys for different purposes or domains.
KMS
Key Management Service; a system for managing cryptographic keys and their use.
Nonce reuse
Using the same nonce with the same key where uniqueness is required.
Secret zero
The initial secret or trust bootstrap problem needed to access other secrets.
Side channel
An indirect information leakage path such as timing, power, cache behavior, or error messages.
Zeroization
Clearing sensitive key material from memory or storage.

17. Self-Assessment

20 MCQs

1. What is cryptographic misuse?

Answer: A. Misuse often happens with strong algorithms used with weak randomness, bad modes, leaked keys, disabled validation, or wrong assumptions.

2. Which item is usually the most sensitive in symmetric encryption?

Answer: B. The key controls access. Standard algorithm names are public.

3. Why is AES-ECB unsafe for most data protection?

Answer: A. ECB encrypts identical plaintext blocks to identical ciphertext blocks under the same key.

4. What is key rotation?

Answer: A. Rotation limits exposure and supports response when keys age, roles change, or compromise is suspected.

5. What is a cryptoperiod?

Answer: A. A cryptoperiod defines how long a key is valid for a specified use.

6. Which is a safer password-storage approach?

Answer: A. Passwords need slow password hashing, not fast general-purpose hashes or reversible storage.

7. What is envelope encryption?

Answer: A. Envelope encryption separates data-encryption keys from higher-level wrapping or master keys.

8. What is a KEK?

Answer: A. A KEK wraps, encrypts, or protects data-encryption keys.

9. What is a DEK?

Answer: A. A DEK protects actual application data.

10. What is the safest place for production master keys in many enterprise designs?

Answer: A. KMS/HSM systems centralize protection, authorization, logging, and rotation controls.

11. What is crypto agility?

Answer: A. Agility helps organizations migrate away from weak or quantum-vulnerable cryptography.

12. What is a cryptographic inventory?

Answer: A. Inventory makes migration, audit, incident response, and vendor management possible.

13. Why is hardcoding a secret in source code dangerous?

Answer: A. Once committed, secrets can be copied widely and are hard to fully recover from.

14. What should happen when a production key is suspected compromised?

Answer: A. Compromise response requires containment, replacement, investigation, and communication.

15. What is key separation?

Answer: A. Key separation limits blast radius and avoids cross-protocol or cross-purpose failures.

16. Which metadata helps future decryption and migration?

Answer: A. Versioned metadata is critical for rotation, migration, and incident response.

17. What is a side-channel risk?

Answer: A. Side channels can reveal secrets even when the algorithm is mathematically strong.

18. What does post-quantum migration require first in most organizations?

Answer: A. You cannot migrate what you cannot find.

19. Which statement about custom cryptography is safest?

Answer: A. Security should not depend on unreviewed secret designs.

20. What is the main purpose of a key-management policy?

Answer: A. Key management is a lifecycle, not a single command.

18. Practice Lab

Lab title

Safe Cryptographic Misuse Review for a Fictional Application

Safety and authorization

Perform this lab only on files created in the lab folder. Do not scan public repositories, company systems, classmates' work, or third-party code.

Prerequisites

  • Linux, macOS, or Windows terminal.
  • Python 3.10+.
  • Optional: Python cryptography package for the versioned encryption example.

Objectives

  • Create a fictional configuration file containing unsafe patterns.
  • Identify hardcoded secrets, disabled validation, and weak algorithm references.
  • Create a small crypto inventory table.
  • Generate a local teaching DEK and record its metadata.
  • Explain a safe remediation plan.

Steps

  1. Create the lab folder and demo configuration file using the Linux or Windows example.
  2. Run the local pattern search.
  3. Create a table with columns: file, line, finding, risk, remediation, owner, due date.
  4. Run the Python inventory script.
  5. Run the versioned encryption metadata example.
  6. Write a short memo answering: What must be rotated immediately? What must be redesigned? What evidence should be preserved?

Expected outcomes

  • You can distinguish algorithm strength from system safety.
  • You can explain why source-code secrets are high risk.
  • You can design basic key metadata for future migration.
  • You can write a practical remediation plan.

Deliverables

  • Local findings table.
  • Crypto inventory table.
  • Versioned encryption output sample.
  • One-page remediation memo.
