Week 3 · Lesson 3.2

Randomness, Keys, Hashes, and MACs

Strong cryptographic algorithms can still fail if randomness is weak, keys are mishandled, hashes are used for the wrong purpose, or message authentication is skipped. This lesson explains the building blocks that make later encryption, signatures, TLS, password storage, and secure APIs possible.

Estimated reading time160–220 minutes
Difficulty levelBeginner → MS foundation
Required backgroundLesson 3.1 helpful; no advanced math required
Lesson typeConceptual + safe local lab

Overview

Topic: randomness, cryptographic keys, hashes, password hashing, message authentication codes, and HMAC.

Learning objective: understand the low-level primitives that many secure systems depend on, and recognize when each primitive is appropriate or dangerous.

Required background: basic files, Python, command-line usage, and Lesson 3.1 concepts are helpful. This lesson re-explains the required ideas before using them.

Difficulty: starts from zero, introduces probability and entropy carefully, and ends with MS-level engineering judgment.

1. Introduction

Cryptography often fails at the edges. The encryption algorithm may be strong, but the system can still be broken if a key is predictable, a reset token is guessable, a secret is committed to GitHub, a password is stored with a fast hash, or an API accepts unsigned messages.

Plain English version: this lesson teaches how systems make and protect hard-to-guess values, how they turn data into fingerprints, and how they prove that a message was not changed by someone who does not know the secret key.

The topic exists because digital systems constantly need answers to practical questions:

  • How do we create a password-reset token that attackers cannot guess?
  • How do we know a downloaded file is exactly the file we expected?
  • How do we store passwords without storing the passwords themselves?
  • How do two servers prove that an API message was not modified in transit?
  • How do we manage cryptographic keys across creation, use, rotation, and destruction?

These ideas are used in login systems, banking APIs, software updates, cloud secrets management, container signing, VPNs, TLS, payment systems, backups, healthcare records, military communications, university identity systems, and research prototypes.

Safety boundary: All examples in this lesson are local, defensive, and educational. Do not generate, test, verify, intercept, brute force, or manipulate cryptographic material belonging to systems you do not own or have written authorization to study.

2. Learning Outcomes

After finishing this lesson, you should be able to:

  • Explain entropy, randomness, pseudo-randomness, and cryptographically secure random generation in plain English.
  • Calculate simple keyspace sizes such as 2128 and explain why uniform randomness matters.
  • Distinguish keys, passwords, tokens, salts, peppers, nonces, initialization vectors, and MAC tags.
  • Explain the difference between general-purpose hashes and password-hashing schemes.
  • Explain preimage resistance, second-preimage resistance, and collision resistance.
  • Compute safe local SHA-256 hashes and HMAC-SHA-256 tags using Python and command-line tools.
  • Use constant-time comparison for MAC verification in Python.
  • Describe a key lifecycle: generation, distribution, storage, use, rotation, revocation, archival, and destruction.
  • Identify common failures such as weak random numbers, reused secrets, hardcoded keys, leaked API tokens, MD5/SHA-1 misuse, and fast password hashing.
  • Translate randomness, hash, and key-management issues into practical risk statements for technical and executive audiences.

3. Prerequisites

This lesson assumes no prior cryptography background. It uses these ideas and explains them before advanced use:

  • Bit: one binary value, either 0 or 1.
  • Byte: eight bits. A 32-byte token contains 256 bits.
  • File: named data stored on a computer. Hash tools often work on files.
  • Function: a rule that maps input to output. Hash functions map messages to digests.
  • Secret: information that must not be known by unauthorized parties.
  • Threat model: a description of what the attacker can do and what you need to protect.
Prerequisite mini-explanation: why guessing is a security problem

If a reset token has only six decimal digits, there are only 1,000,000 possible tokens. That may sound large to a human, but it is small to software if no rate limits exist. If a token has 128 uniformly random bits, there are 2128 possibilities. The security difference is enormous.

4. Core Concepts

4.1 Randomness

Randomness means lack of predictability. In security, we care less about whether a value “looks random” and more about whether an attacker can predict it before it is used.

Analogy: rolling a fair die is unpredictable before the roll. Writing today’s date on paper is not random, even if it looks different every day. A cryptographic token must be more like the die roll and less like the date.

Cybersecurity systems use randomness to generate keys, session IDs, nonces, salts, password-reset links, API tokens, one-time challenges, and cryptographic protocol values.

4.2 Entropy

Entropy is a mathematical way to describe uncertainty. A value with high entropy is hard to guess. A value with low entropy is easier to predict.

For a uniformly random value with N equally likely outcomes, the entropy is:

H = log2(N)

If a value has 2128 equally likely outcomes, it has 128 bits of entropy. If a six-digit code has 106 equally likely outcomes, it has about log2(1,000,000) ≈ 19.93 bits of entropy.

Important: entropy depends on how the value was generated. A 32-character string chosen by a human may have much less entropy than a 32-character string generated by a cryptographically secure random generator.

4.3 Pseudo-Random vs Cryptographically Secure Pseudo-Random

A normal pseudo-random number generator produces values from an internal state. It may be good for simulation, games, sampling, or testing. It is not automatically safe for secrets.

A cryptographically secure pseudo-random number generator, often abbreviated CSPRNG, is designed so that outputs are unpredictable even if an attacker sees many previous outputs. Operating systems provide secure random sources for applications.

4.4 Keys

A cryptographic key is a value used by an algorithm to perform a security operation. The algorithm can be public; the key must be protected according to its role.

Examples:

  • An AES key protects encrypted data.
  • An HMAC key protects message integrity and shared-key authentication.
  • A private signing key proves control of a digital-signature identity.
  • A public verification key may be shared widely.
  • An API key often works like a bearer secret: whoever has it can use it.

Key separation means using different keys for different purposes. Do not use one key for encryption, HMAC, backups, webhooks, and database credentials. Reuse increases blast radius and can violate assumptions of cryptographic constructions.

4.5 Key Lifecycle

Key management is the full lifecycle of cryptographic keys. It is often harder than choosing an algorithm.

  1. Generate: create the key with adequate entropy.
  2. Register or label: assign owner, purpose, creation date, and allowed uses.
  3. Store: protect the key using a secrets manager, hardware security module, key-management service, or protected local store.
  4. Distribute: deliver the key only to authorized systems.
  5. Use: restrict operations according to policy.
  6. Rotate: replace with a new key on schedule or after risk events.
  7. Revoke: stop trusting a key when compromise or retirement occurs.
  8. Archive: keep only when needed for legal or data-recovery reasons.
  9. Destroy: remove keys safely when no longer needed.
flowchart LR
  Generate[Generate key] --> Label[Label owner and purpose]
  Label --> Store[Store securely]
  Store --> Use[Use with policy]
  Use --> Rotate[Rotate]
  Rotate --> Revoke[Revoke old key]
  Revoke --> Destroy[Destroy or archive]

4.6 Hash Functions

A cryptographic hash function maps input data of arbitrary length to a fixed-size output called a digest.

Key properties:

  • Deterministic: the same input gives the same digest.
  • Fixed output length: SHA-256 outputs 256 bits, regardless of input size.
  • Avalanche effect: a small input change should drastically change the digest.
  • Preimage resistance: given a digest, it should be infeasible to find an input that creates it.
  • Second-preimage resistance: given one message, it should be infeasible to find another message with the same digest.
  • Collision resistance: it should be infeasible to find any two different messages with the same digest.

Hash functions are used for file integrity checks, digital signatures, content addressing, deduplication, certificate fingerprints, software distribution, evidence handling, and some protocol internals.

Not secrecy: hashing a secret does not automatically make it safe. If the input is guessable, an attacker can hash guesses and compare results.

4.7 Password Hashing Is Different

Password hashing is a special problem because user passwords are often low entropy. Attackers who steal a password database can perform offline guessing: they try candidate passwords, hash them, and compare results.

For password storage, use purpose-built password-hashing schemes such as Argon2id, bcrypt, scrypt, or PBKDF2 with appropriate parameters. These schemes are intentionally slow or memory-hard to raise the cost of each guess.

TechniqueGood for file integrity?Good for password storage?Why
SHA-256Yes, in appropriate contextsNo, not aloneToo fast for low-entropy passwords
HMAC-SHA-256Yes for keyed integrityNo, not as a standalone password hashDesigned for message authentication, not password stretching
Argon2idNoYesDesigned for password hashing with memory and time cost
bcrypt/scrypt/PBKDF2NoYes when configured correctlyDesigned to slow offline guessing

4.8 Salts and Peppers

A salt is a unique random value stored with each password hash. It does not need to be secret. Its purpose is uniqueness: two users with the same password should have different stored hashes.

A pepper is an additional secret value used by the application and stored separately from the password database, such as in a secrets manager or hardware-backed key store. A pepper is defense in depth, not a replacement for password hashing.

flowchart TD
  Password[User password] --> KDF[Password hashing scheme]
  Salt[Unique public salt] --> KDF
  Pepper[Secret pepper stored outside DB] -. optional .-> KDF
  KDF --> StoredHash[Stored password hash record]

4.9 Message Authentication Codes

A message authentication code, or MAC, is a short value computed from a message and a secret key. The receiver recomputes the MAC and checks whether it matches.

A MAC provides two goals between parties that share the key:

  • Integrity: the message was not changed after the tag was created.
  • Shared-key origin authentication: someone with the secret key created the valid tag.

A MAC does not automatically provide confidentiality. The message may still be readable unless it is also encrypted.

4.10 HMAC

HMAC is a widely used MAC construction based on a cryptographic hash function and a secret key. HMAC-SHA-256 means HMAC using SHA-256 as the underlying hash.

HMAC is used in webhooks, APIs, VPNs, cloud request signing, internal service calls, and protocol designs. The receiver must verify the tag over the exact bytes received. Even a harmless-looking formatting change can alter the tag.

sequenceDiagram
  participant Sender
  participant Receiver
  Note over Sender,Receiver: Both share secret key K
  Sender->>Sender: tag = HMAC(K, message)
  Sender->>Receiver: message + tag
  Receiver->>Receiver: expected = HMAC(K, received message)
  alt tags match
    Receiver->>Receiver: accept message
  else tags differ
    Receiver->>Receiver: reject and log safely
  end

4.11 Constant-Time Comparison

When verifying MAC tags, compare values with a function designed to reduce timing leaks. In Python, use hmac.compare_digest(). A naive comparison may reveal partial information if it exits early when characters differ.

For beginners: this is like checking an answer sheet without accidentally telling the attacker how many starting characters were correct.

4.12 Common Misuse Patterns

  • Using random instead of a cryptographic randomness source for tokens.
  • Using timestamps, user IDs, or counters as “random” values.
  • Hardcoding production keys in source code.
  • Committing API keys, private keys, or tokens to Git repositories.
  • Using SHA-256 alone for password storage.
  • Using MD5 or SHA-1 for collision-sensitive new designs.
  • Reusing one key for multiple unrelated purposes.
  • Accepting unsigned webhooks or API callbacks.
  • Verifying only part of a MAC tag.
  • Logging full secrets, tokens, or keys.

7. Mathematical Foundations

7.1 Keyspace Size

If a key has b uniformly random bits, the number of possible keys is:

Number of keys = 2b

Example: a 128-bit key has 2128 possible values. A 256-bit token has 2256 possible values. These numbers are so large that brute-force guessing is infeasible when values are generated uniformly and rate limits or cryptographic assumptions are appropriate.

7.2 Entropy of Uniform Choices

If there are N equally likely outcomes, entropy is:

H = log2(N)

Worked example: A six-digit code has 106 possible values.

H = log2(1,000,000) ≈ 19.93 bits

That is not enough by itself for an unrestricted online guessing scenario. It may be acceptable only with expiry, rate limiting, lockout/backoff, monitoring, and user-notification controls.

7.3 Guessing Probability

If a token has b uniformly random bits and an attacker gets q independent guesses, an approximate upper bound for success is:

P(success) ≈ q / 2b

Worked example: A 128-bit token and one billion guesses:

P ≈ 109 / 2128

This is still extremely small. The real system still needs rate limits and monitoring because implementation mistakes, token leakage, or server-side bugs can bypass the pure guessing model.

7.4 Birthday Bound for Collisions

For a hash or random identifier with b bits, the approximate collision probability after generating n independent values is:

P(collision) ≈ n2 / 2b+1

This is why collision probability grows faster than beginners expect. You do not need to guess a specific value; any pair colliding is enough.

7.5 MAC Forgery Intuition

If a MAC tag is t bits and the construction is secure, a blind random forgery attempt should succeed with probability about:

P(forgery) ≈ 1 / 2t

This simplified formula assumes the attacker does not know the key and that the MAC construction is used correctly. Real security also depends on key secrecy, correct message canonicalization, tag length, replay protection, and implementation safety.

8. Practical Examples

8.1 Linux Examples

These examples operate only on local files and locally generated random values.

# Create a harmless local message file
mkdir -p crypto-lesson-3-2
cd crypto-lesson-3-2
echo "student=alice;amount=100" > message.txt

# Generate 32 random bytes as hex using OpenSSL
openssl rand -hex 32

# Compute a SHA-256 file digest
sha256sum message.txt

# Modify one character and compute the digest again
echo "student=alice;amount=900" > message.txt
sha256sum message.txt

What to notice: the hash changes completely when the message changes from amount 100 to amount 900. The hash does not hide the file contents; it only gives a digest.

8.2 Windows Examples

# Generate 32 random bytes using .NET cryptographic RNG
$bytes = New-Object byte[] 32
[System.Security.Cryptography.RandomNumberGenerator]::Fill($bytes)
[Convert]::ToHexString($bytes)

# Compute a file hash
"hello from Windows" | Out-File -Encoding utf8 message.txt
Get-FileHash .\message.txt -Algorithm SHA256

What to notice: Windows can compute file hashes with PowerShell and can generate random bytes through cryptographic .NET APIs.

8.3 Python Examples

import secrets
import hashlib
import hmac

# 1. Generate a random API-style token.
# secrets uses operating-system randomness suitable for security tokens.
token = secrets.token_urlsafe(32)
print("Random token:", token)

# 2. Hash a message. A hash is not secret and not reversible encryption.
message = b"pay=100&to=library"
digest = hashlib.sha256(message).hexdigest()
print("SHA-256 digest:", digest)

# 3. Compute an HMAC. The key must remain secret.
key = secrets.token_bytes(32)
tag = hmac.new(key, message, hashlib.sha256).hexdigest()
print("HMAC tag:", tag)

# 4. Verify with constant-time comparison.
received_message = b"pay=100&to=library"
received_tag = tag
expected_tag = hmac.new(key, received_message, hashlib.sha256).hexdigest()

if hmac.compare_digest(received_tag, expected_tag):
    print("Message accepted: HMAC is valid")
else:
    print("Message rejected: HMAC is invalid")

# 5. Tampering demo: one byte changes the expected tag.
tampered_message = b"pay=900&to=library"
tampered_tag = hmac.new(key, tampered_message, hashlib.sha256).hexdigest()
print("Tampered message has same tag?", hmac.compare_digest(tag, tampered_tag))

What to notice: the key is random bytes, the digest is public, the HMAC tag depends on both key and message, and compare_digest is used for verification.

8.4 Network Example: Signed Webhook

A webhook is a message one service sends to another when an event happens. A signed webhook lets the receiver check that the body has not been changed and was produced by a party that knows the shared secret.

# Safe local webhook-signing demonstration. No external network target is used.
# Run the script and inspect how the receiver rejects a tampered body.
import hashlib
import hmac
import secrets

shared_webhook_secret = secrets.token_bytes(32)
body = b'{"event":"assignment_submitted","student":"alice"}'

signature = hmac.new(shared_webhook_secret, body, hashlib.sha256).hexdigest()
print("Sender signature:", signature)

# Receiver recomputes the signature from the exact bytes received.
received_body = body
expected = hmac.new(shared_webhook_secret, received_body, hashlib.sha256).hexdigest()
print("Valid receiver check:", hmac.compare_digest(signature, expected))

# A modified body fails verification.
tampered_body = b'{"event":"grade_changed","student":"alice"}'
expected_for_tampered = hmac.new(shared_webhook_secret, tampered_body, hashlib.sha256).hexdigest()
print("Tampered receiver check:", hmac.compare_digest(signature, expected_for_tampered))

8.5 Web Example

A password-reset link should use a high-entropy token, short expiry, one-time use, rate limiting, and safe logging. A weak design uses predictable token IDs such as /reset?token=100123. A safer design uses a random token and stores only a hash or protected representation of it server-side.

8.6 Cloud Example

Cloud systems often avoid exposing raw keys to applications. Instead, the application calls a key-management service to encrypt, decrypt, sign, verify, or generate data keys. This reduces direct key exposure and creates audit logs, but it does not remove the need for access control, monitoring, and secure application design.

9. Code Examples

9.1 Unsafe Beginner Pattern

# Do NOT copy this pattern into real software.
# Problems:
# - random is not intended for cryptographic secrets.
# - The key is hardcoded in source code.
# - SHA-256 alone is not a password-hashing scheme.
import random
import hashlib

api_key = "secret-key-inside-source-code"
reset_token = str(random.randint(100000, 999999))
password_hash = hashlib.sha256(b"Password123").hexdigest()
print(api_key, reset_token, password_hash)

Line-by-line explanation:

  • import random imports a module designed for simulation, not cryptographic secrets.
  • api_key = ... hardcodes a secret in source code. This can leak through Git, backups, screenshots, logs, and support bundles.
  • random.randint(...) creates a token that may be predictable and too small.
  • hashlib.sha256(...) computes a fast hash. It is not a password-hashing scheme.

9.2 Safer Educational Pattern

# Safer educational pattern for local demonstration.
# Production systems should use a mature framework and a reviewed secrets manager.
import os
import secrets
import hashlib

# Load an application secret from the environment, not from source code.
# In a real deployment, this would usually come from a secrets manager.
app_secret = os.environ.get("APP_SECRET")
if not app_secret:
    raise RuntimeError("APP_SECRET is missing")

# Generate an unpredictable reset token.
reset_token = secrets.token_urlsafe(32)

# Demonstration of PBKDF2 from Python standard library.
# For new applications, prefer a modern password-hashing library supporting Argon2id
# unless policy requires PBKDF2. Always use a unique salt per password.
password = b"correct horse battery staple"
salt = secrets.token_bytes(16)
dk = hashlib.pbkdf2_hmac("sha256", password, salt, 600_000)
print("Reset token:", reset_token)
print("Salt:", salt.hex())
print("Stored derived key:", dk.hex())

Line-by-line explanation:

  • import os allows the program to read environment variables for local demonstration.
  • import secrets provides cryptographically strong random values.
  • import hashlib provides standard hashing and key-derivation functions.
  • os.environ.get("APP_SECRET") avoids putting the secret directly in source code. Production systems should use a secrets manager rather than relying only on environment variables.
  • secrets.token_urlsafe(32) generates a high-entropy token suitable for URLs.
  • secrets.token_bytes(16) generates a unique salt for this password-hashing demonstration.
  • hashlib.pbkdf2_hmac(..., 600_000) derives a password hash using many iterations. In new non-FIPS-constrained applications, Argon2id is usually preferred when available through a reviewed library.

9.3 Common Mistakes

  • Mistake: “The token is long, so it must be safe.” Fix: check how it was generated, not only length.
  • Mistake: “Hashing is encryption.” Fix: hashes are one-way digests; encryption is reversible with a key.
  • Mistake: “A salt must be secret.” Fix: salts are public uniqueness values; peppers are secret defense-in-depth values.
  • Mistake: “If the HMAC matches, the message is secret.” Fix: HMAC authenticates and protects integrity; it does not hide the message.
  • Mistake: “I can use one key for everything.” Fix: use key separation and purpose labels.

10. Security Perspective

Common Attacks

  • Token guessing: attacker guesses password-reset links, invite links, or API tokens.
  • Offline password cracking: attacker steals password hashes and tests guesses locally.
  • Secret leakage: keys appear in Git repositories, logs, tickets, screenshots, container images, or CI/CD variables.
  • Replay: attacker resends a previously valid signed message if timestamps, nonces, or idempotency controls are missing.
  • MAC bypass: application verifies only part of a tag or accepts missing signatures.
  • Hash collision abuse: weak hash algorithms are exploited in collision-sensitive contexts.
  • Key compromise: attacker obtains a key and can decrypt, sign, or authenticate depending on the key purpose.

Defensive Strategies

  • Generate secrets with OS-backed cryptographic randomness.
  • Use reviewed cryptographic libraries and standard constructions.
  • Use unique salts and slow password hashing for passwords.
  • Use a secrets manager or key-management service for production secrets.
  • Label keys by owner, purpose, environment, and rotation policy.
  • Rotate and revoke secrets after exposure, compromise, or role changes.
  • Scan repositories and CI/CD systems for secrets.
  • Verify MAC tags over the exact bytes received using constant-time comparison.
  • Protect against replay with timestamps, nonces, event IDs, and short validity windows.

Detection Methods

  • Repository secret-scanning alerts.
  • Unusual API-key use from unexpected IP addresses, regions, or services.
  • Repeated token-verification failures.
  • High rates of password-reset attempts.
  • Use of retired or revoked keys.
  • Unexpected spikes in HMAC verification failures.
  • Logs showing application errors from missing or malformed secrets.

Mitigation and Best Practices

  • Never log full keys, tokens, private keys, or password hashes in normal application logs.
  • Mask secrets in telemetry and support tools.
  • Do not email production keys or paste them into chat systems.
  • Use separate keys for development, staging, and production.
  • Prefer short-lived credentials where possible.
  • Document key owners and emergency rotation procedures.
  • Treat unknown key exposure as a security incident, not a minor cleanup task.

12. Research Perspective

Research in this area studies both mathematical security and engineering reality. A scheme may be sound in theory but still fail through weak randomness, implementation bugs, side channels, or poor key lifecycle controls.

Current Research Directions

  • Randomness generation and entropy estimation: how to measure, certify, and monitor entropy sources in hardware and software systems.
  • Side-channel-resistant implementations: how timing, power, cache behavior, and faults can leak secrets.
  • Formal verification: proving that cryptographic libraries and protocols implement intended properties.
  • Password hashing economics: choosing memory and time costs that meaningfully increase attacker cost without causing denial of service for legitimate users.
  • Secrets detection: reducing false positives and false negatives in repository and artifact scanning.
  • Key transparency and accountability: making key changes auditable for messaging, software updates, and identity systems.
  • Post-quantum migration: updating key-management systems so algorithms can change without redesigning the whole platform.

Notable Papers and Topics to Explore

  • Bellare, Canetti, and Krawczyk's work on HMAC security foundations.
  • The Password Hashing Competition and Argon2 design literature.
  • Research on Debian OpenSSL weak-key detection and ecosystem impact.
  • Studies on secret leakage in software repositories and CI/CD artifacts.
  • NIST work on random bit generation, key management, and hash standards.

Open Problems

  • How can organizations rotate millions of secrets without outages?
  • How can developers get secure defaults without needing cryptography expertise?
  • How can systems detect weak randomness before production damage occurs?
  • How can secret-scanning tools reduce alert fatigue while still catching real keys?
  • How should long-lived encrypted data be protected during post-quantum migration?

13. Case Studies

13.1 Debian OpenSSL Predictable Random Number Generator

What happened: In 2008, Debian disclosed that a Debian-specific OpenSSL change made the random number generator predictable. Keys generated on affected systems could be guessable.

Root cause: a change intended to quiet analysis warnings removed important entropy-related behavior.

Impact: affected cryptographic key material had to be regenerated and replaced.

Lesson: randomness code is security-critical. Small changes can destroy the security of many higher-level systems.

13.2 Android SecureRandom and Bitcoin Wallets

What happened: In 2013, weak Android random-number generation affected Bitcoin wallet applications. Some wallet-generated cryptographic values became predictable enough to endanger funds.

Root cause: weaknesses in the SecureRandom implementation and its use affected key/signature-related randomness.

Impact: wallet providers advised users to update applications and move funds to addresses generated after fixes.

Lesson: cryptographic randomness errors can directly compromise private keys or signatures.

13.3 Flame Malware and MD5 Collision Abuse

What happened: The Flame malware case involved abuse of MD5 collision weaknesses to help forge a certificate chain used for malicious code-signing behavior.

Root cause: a collision-weak hash algorithm remained usable in a security-sensitive certificate context.

Impact: Microsoft invalidated affected certificates and strengthened update trust mechanisms.

Lesson: legacy hash algorithms can remain dangerous long after their weaknesses are publicly known.

13.4 Secrets in Source Code

What happened: Organizations repeatedly expose API keys, tokens, and private credentials in source repositories and collaboration tools.

Root cause: developers treat secrets as configuration text and store them with code, logs, tickets, or local files.

Impact: attackers can use exposed credentials directly, often without exploiting a software vulnerability.

Lesson: key management is not optional paperwork. It is a core security control.

14. Common Misconceptions

  • “Random-looking means secure.” Incorrect. Predictable generators can produce values that look random to humans.
  • “A longer token is always secure.” Incomplete. Length matters only if the token was generated unpredictably.
  • “Hashing encrypts data.” Incorrect. Hashing is one-way digesting; encryption is reversible with a key.
  • “SHA-256 is always enough.” Incorrect. It is not enough for password storage by itself.
  • “Salts must be hidden.” Incorrect. Salts are usually stored with hashes; uniqueness is their main purpose.
  • “HMAC encrypts the message.” Incorrect. HMAC authenticates and protects integrity but does not hide content.
  • “One master key everywhere is simpler and fine.” Incorrect. It creates large blast radius and violates key-separation principles.
  • “If a key leaked once but nobody noticed abuse, it is safe.” Incorrect. Exposure requires rotation or revocation according to risk.
  • “Passing randomness tests proves security.” Incorrect. Tests can detect some bad patterns but cannot prove attacker unpredictability.
  • “Developers do not need to understand key management if they use a library.” Incorrect. Libraries help, but applications still decide where secrets live, who can use them, and when to rotate them.

15. Summary

Randomness, keys, hashes, and MACs are basic but high-impact cryptographic building blocks. Randomness creates values attackers cannot predict. Keys control cryptographic operations and must be managed throughout their lifecycle. Hashes create fixed-size digests for integrity and identification, but they do not provide secrecy. Password hashing is a separate, slower, salt-based process designed to resist offline guessing. MACs such as HMAC provide keyed integrity and shared-key authentication. Real-world failures often come from weak randomness, leaked secrets, wrong primitive choice, poor verification, or weak key-management processes.

16. Glossary

API Key
A secret-like value used by software to authenticate to an API.
Birthday Bound
An approximation showing how collision probability grows as more values are generated.
CSPRNG
Cryptographically secure pseudo-random number generator designed for unpredictable security-sensitive values.
Digest
The fixed-size output of a hash function.
Entropy
A measure of uncertainty or unpredictability.
Hash Function
A function that maps data to a fixed-size digest.
HMAC
A hash-based message authentication code using a secret key and a hash function.
Key Lifecycle
The stages a key passes through: generation, storage, use, rotation, revocation, archival, and destruction.
Key Rotation
Replacing a key with a new one according to policy or risk.
Key Separation
Using different keys for different cryptographic purposes.
MAC
Message authentication code; a keyed tag for integrity and shared-key authentication.
Nonce
A value used once in a cryptographic operation.
Password Hashing
A slow, salted process for storing password verifiers rather than plaintext passwords.
Pepper
A secret value added as defense in depth to password hashing and stored separately from the password database.
Preimage Resistance
The property that it is infeasible to find an input that produces a given digest.
Salt
A unique public random value added to password hashing to make stored hashes unique.
Secret
Information that must be protected from unauthorized access.
Token
A value used to represent authorization, authentication, invitation, session, or reset capability.

17. Self-Assessment

20 Multiple-Choice Questions

1. Which source should Python code normally use for security tokens?

Answer: B. The secrets module is designed for cryptographically strong random values; random is for simulation, not security.

2. What does entropy measure in this lesson?

Answer: A. Entropy is a way to reason about uncertainty or unpredictability.

3. A 128-bit uniformly random key has how many possible values?

Answer: C. Each bit has two possible values, so 128 independent bits produce 2^128 possibilities.

4. Which statement about a cryptographic hash is correct?

Answer: B. A hash maps data to a fixed-size digest but does not provide secrecy or origin authentication by itself.

5. What is a salt used for in password hashing?

Answer: A. A unique salt prevents identical passwords from producing identical stored hashes and blocks precomputed lookup-table reuse.

6. Why is SHA-256 alone not appropriate for password storage?

Answer: C. Password storage needs slow, salted password-hashing schemes such as Argon2id, bcrypt, scrypt, or PBKDF2.

7. What does HMAC provide when the key remains secret?

Answer: B. HMAC lets parties sharing a secret key detect modification and verify that the tag was created by someone with that key.

8. Which value should usually be public in password storage?

Answer: A. The salt is normally stored with the password hash; secrecy is not its purpose.

9. Which value should be stored separately from the password database?

Answer: B. A pepper is a secret shared application-side value and should be kept outside the password database, such as in a secrets manager.

10. What is the safest way to compare a received HMAC tag with an expected tag?

Answer: B. Constant-time comparison reduces timing side-channel leakage during tag verification.

11. What is key rotation?

Answer: A. Rotation replaces or retires keys according to risk, lifecycle, exposure, and operational policy.

12. Which action is usually dangerous?

Answer: B. Hardcoded and publicly committed secrets can be discovered and abused by attackers.

13. What is the birthday bound mainly used for here?

Answer: A. Collision probability grows roughly with the square of the number of generated values.

14. What does key separation mean?

Answer: A. A key used for one purpose should not be reused for unrelated purposes such as encryption, HMAC, and backups.

15. Which hash family is currently preferred over MD5 and SHA-1 for new general integrity uses?

Answer: A. MD5 and SHA-1 are legacy and collision-weak; SHA-2 and SHA-3 families are appropriate for many modern hash uses.

16. What is a MAC tag?

Answer: A. A MAC tag is verified to detect tampering and prove possession of the shared key.

17. What should happen if an HMAC verification fails?

Answer: B. Failed verification means integrity/authentication cannot be trusted, so the message should be rejected.

18. What is a secure key lifecycle?

Answer: A. Keys need management across their whole life, not only at generation time.

19. What is a common limitation of randomness tests?

Answer: A. Statistical tests can find some defects, but passing them does not prove that future outputs are unpredictable to an attacker.

20. Which is the best beginner rule for cryptographic design?

Answer: B. Most systems fail through misuse; standard libraries and sound key handling reduce avoidable risk.
