Week 3 · Lesson 3.1

Cryptography from First Principles

Cryptography is how modern systems protect secrets, detect tampering, prove identity, and establish secure communication over networks that may be watched or manipulated. This lesson starts from ordinary communication problems and builds toward the language used by security engineers and researchers.

Estimated reading time150–210 minutes
Difficulty levelBeginner → MS foundation
Required backgroundNone required; Weeks 0–2 recommended
Lesson typeConceptual + safe local lab

Week 3 Overview

Topic: Cryptography.

Learning objective: understand what cryptography can protect, what it cannot protect, and how to use standard cryptographic building blocks safely.

Required background: basic files, Python, networking, identity, and access-control concepts are helpful but not assumed.

Difficulty: starts beginner-friendly, then introduces graduate-level precision.

1. Introduction

Cryptography is the discipline of protecting information using mathematical techniques. It exists because people need to communicate, store data, identify each other, and prove that information was not changed even when attackers may observe, copy, delay, replay, or modify messages.

Plain English version: cryptography helps answer questions like these:

  • Can anyone else read this? This is a confidentiality question.
  • Was this changed? This is an integrity question.
  • Who sent this? This is an authentication question.
  • Can two systems agree on a secret over the internet? This is a key-establishment question.
  • Can someone later deny signing this? This is a non-repudiation question in some legal and technical contexts.

Cryptography matters because almost every serious digital system depends on it: HTTPS websites, online banking, messaging apps, password storage, software updates, cloud storage, disk encryption, digital signatures, VPNs, blockchains, identity systems, secure boot, electronic health records, government communications, and military systems.

Who uses cryptography? Software engineers use cryptographic libraries. Security engineers review cryptographic design. Cloud providers use encryption and key-management systems. Banks use cryptographic controls for payment systems. Researchers analyze new algorithms and attack models. Auditors check whether systems meet standards. Incident responders investigate whether keys or certificates were compromised.

Important boundary: This course teaches safe cryptographic use, not how to break real systems. Do not test, intercept, decrypt, attack, or manipulate systems you do not own or have written authorization to study.

2. Learning Outcomes

After finishing this lesson, you should be able to:

  • Explain cryptography in plain English and in technical security language.
  • Distinguish encoding, hashing, encryption, message authentication codes, digital signatures, and key exchange.
  • Explain confidentiality, integrity, authenticity, non-repudiation, and forward secrecy at a beginner level.
  • Explain the roles of plaintext, ciphertext, keys, algorithms, nonces, salts, tags, certificates, and trust anchors.
  • Use safe local commands and Python examples to generate random values, compute hashes, and verify HMACs.
  • Identify common cryptographic misconceptions and dangerous implementation mistakes.
  • Explain why cryptographic key management is often more important than the algorithm name alone.
  • Describe why post-quantum cryptography and crypto agility matter for long-lived systems.

3. Prerequisites

This lesson introduces every cryptographic concept from the beginning. These prior ideas help, but the lesson re-explains them as needed:

  • Bits and bytes: computers store information as binary values. A bit is 0 or 1. A byte is usually 8 bits.
  • Files: a file is named data stored on a system. Cryptographic tools often read files and produce hashes, signatures, or encrypted versions.
  • Networks: network messages travel through systems that may not be trusted. Cryptography protects the message even when the path is not trusted.
  • Identity: authentication proves a subject is who it claims to be. Cryptography is often used to support that proof.
  • Threat model: cryptography must be chosen according to what the attacker can do: read, modify, replay, steal keys, compromise endpoints, or use future computing advances.
Prerequisite mini-explanation: bits, keys, and guessing

A cryptographic key is often just a large random number. A 128-bit key has 2128 possible values. That number is so large that exhaustive guessing is infeasible with normal technology when the key is generated randomly and protected correctly.

4. Core Concepts

4.1 The Basic Communication Problem

Imagine Alice wants to send a private message to Bob. Mallory, an attacker, can observe the network. In a stronger threat model, Mallory can also modify messages, replay old messages, impersonate parties, or steal keys. Cryptography gives Alice and Bob tools, but it does not automatically solve every security problem.

flowchart LR
  Alice[Alice: sender] -->|message travels through network| Network[(Untrusted network)]
  Network --> Bob[Bob: receiver]
  Mallory[Mallory: attacker] -. can observe .-> Network
  Mallory -. may modify/replay .-> Network

4.2 Encoding Is Not Encryption

Encoding changes representation. It makes data fit a format. It does not require a secret key and does not provide secrecy. Base64 is encoding. Anyone can decode it.

Encryption uses a key to transform readable plaintext into unreadable ciphertext. Without the correct key, the ciphertext should reveal no useful information about the plaintext except limited metadata such as length.

TechniquePurposeSecret key?Security goal?
EncodingRepresent data in another formatNoNo secrecy by itself
CompressionReduce sizeNoNo secrecy by itself
HashingProduce a fixed-size digestNoIntegrity checks in some contexts, not secrecy
EncryptionHide plaintextYesConfidentiality
MACDetect changes and authenticate sender with a shared keyYesIntegrity + shared-key authentication
Digital signatureProve message came from holder of private keyPrivate key signs; public key verifiesIntegrity + authenticity + possible non-repudiation

4.3 Plaintext, Ciphertext, Algorithm, and Key

Plaintext is readable original information. Ciphertext is the encrypted result. An algorithm is the public method. A key is the secret or private value that controls the operation.

Modern cryptography follows a critical principle: the algorithm should remain secure even if attackers know how it works. Security should depend on the secrecy of the key, not secrecy of the algorithm.

4.4 Symmetric Cryptography

In symmetric cryptography, the same secret key is used by both sides. If Alice and Bob both know the same key, Alice can encrypt and Bob can decrypt. Symmetric encryption is fast and widely used for bulk data protection.

Analogy: Alice and Bob share the same physical key to a locked box. Anyone with that key can lock or unlock the box. The hard problem is how Alice and Bob got the same key safely in the first place.

sequenceDiagram
  participant A as Alice
  participant B as Bob
  Note over A,B: Both already share secret key K
  A->>B: Encrypt_K(message) = ciphertext
  B->>B: Decrypt_K(ciphertext) = message

4.5 Public-Key Cryptography

Public-key cryptography uses a key pair. One key is public and can be shared. The other key is private and must be protected. Public-key cryptography supports digital signatures, key agreement, certificate systems, and some encryption schemes.

Analogy: Bob publishes a special padlock that anyone can use to lock a box for Bob. Only Bob has the private key that opens it. For signatures, the analogy changes: Bob uses his private signing key to stamp a message, and anyone with Bob's public key can verify the stamp.

4.6 Hash Functions

A cryptographic hash function maps input of any size to a fixed-size output called a digest. Good cryptographic hashes are designed to make it infeasible to find two different inputs with the same digest, or to reconstruct the input from the digest.

Important: a hash is not encryption. There is no decryption key. Hashing is one-way by design.

  • Preimage resistance: given a digest, it should be hard to find any input that produces it.
  • Second-preimage resistance: given one input, it should be hard to find another input with the same digest.
  • Collision resistance: it should be hard to find any two different inputs with the same digest.

4.7 MACs and HMAC

A message authentication code, or MAC, uses a secret key to protect message integrity and authenticate that the message came from someone who knows the shared key. HMAC is a widely used MAC construction built from a hash function.

Without the secret key, an attacker should not be able to create a valid tag for a modified message.

4.8 Digital Signatures

A digital signature uses a private key to sign and a public key to verify. Anyone with the public key can verify the signature, but only the private-key holder should be able to produce it.

Digital signatures are used in software updates, certificates, legal-document workflows, package repositories, secure boot, identity protocols, and code signing.

4.9 Key Exchange and Key Establishment

Key exchange solves this problem: two systems need to agree on a shared secret over a network that attackers can observe. Modern protocols usually combine public-key techniques with symmetric encryption. The public-key step establishes a shared secret; symmetric encryption then protects the actual data efficiently.

4.10 TLS and HTTPS

Transport Layer Security, or TLS, is the protocol behind HTTPS. It helps a browser and server authenticate the server, agree on keys, and protect communication from eavesdropping, tampering, and message forgery when configured correctly.

sequenceDiagram
  participant Browser
  participant Server
  Browser->>Server: ClientHello: supported versions, cipher suites, random values
  Server->>Browser: ServerHello + certificate + key share
  Browser->>Browser: Validate certificate chain and server identity
  Browser->>Server: Key agreement messages
  Note over Browser,Server: Both derive symmetric session keys
  Browser->>Server: Encrypted HTTP request
  Server->>Browser: Encrypted HTTP response

4.11 Keys Are Assets

Many cryptographic failures are not caused by weak algorithms. They are caused by weak key management: predictable keys, hardcoded secrets, keys committed to Git, missing rotation, overbroad access, poor backup, unclear ownership, unmonitored use, or failure to revoke compromised keys.

Treat keys like high-value assets. Ask who can create them, read them, use them, rotate them, back them up, destroy them, and audit their use.

4.12 Cryptography Does Not Fix Everything

Cryptography can protect data in transit or at rest, but it does not automatically fix compromised endpoints, malicious insiders, broken authorization, vulnerable applications, insecure logging, poor backup strategy, or social engineering. A stolen session token can bypass beautiful encryption. A vulnerable web application can leak decrypted data after the server legitimately decrypts it.

7. Mathematical Foundations

7.1 Bits and Key Spaces

A key with n bits has 2n possible values. Each extra bit doubles the number of possible keys.

Formula:

number_of_keys = 2^n

Worked example: a 4-bit key has 24 = 16 possible keys. A 128-bit key has 2128 possible keys. The second number is enormous; the security assumption depends on the key being generated randomly and kept secret.

7.2 XOR Intuition

XOR is a simple binary operation. It outputs 1 when the two input bits differ and 0 when they are the same.

ABA XOR B
000
011
101
110

XOR has a useful property:

(message XOR key) XOR key = message

This helps explain the idea of reversible transformation, but XOR alone is not a complete secure encryption scheme unless used in very specific ways such as a correctly generated one-time pad. Do not design your own cipher from XOR examples.

7.3 Modular Arithmetic

Modular arithmetic means arithmetic with wraparound. Clocks use modular arithmetic: after 12 comes 1. In math notation, 14 mod 12 = 2.

A toy Caesar cipher shifts letters using modular arithmetic:

cipher_index = (plain_index + shift) mod 26

This is useful for learning, but it is not secure. An attacker can test all 25 possible shifts quickly.

7.4 Probability of Guessing a Token

If a random token has N equally likely values and an attacker makes g guesses, the approximate chance of success is:

probability ≈ g / N    when g is much smaller than N

Worked example: if a token has 264 possible values and an attacker can try 1,000,000 guesses, the probability is approximately 1,000,000 / 18,446,744,073,709,551,616. That is very small, but online rate limits and monitoring are still required.

7.5 Birthday Bound for Collisions

For a hash with b output bits, collisions become plausible around 2b/2 random attempts, not 2b. This is called the birthday bound. It is one reason collision resistance needs large digest sizes.

Example: a 256-bit hash has a birthday-bound collision scale around 2128, which remains extremely large.

8. Practical Examples

8.1 Linux Examples

These examples operate on your own local files only.

# Create a harmless local file.
printf "hello cybersecurity\n" > message.txt

# Compute a SHA-256 digest of the file.
sha256sum message.txt

# Generate 32 random bytes and display them in hexadecimal.
openssl rand -hex 32

# Inspect the certificate chain for a public HTTPS site without sending credentials.
# Use this only for normal observation, not scanning or attacking.
openssl s_client -connect example.com:443 -servername example.com < /dev/null

8.2 Windows Examples

# Create a harmless local file.
"hello cybersecurity" | Out-File -Encoding utf8 message.txt

# Compute a SHA-256 hash in PowerShell.
Get-FileHash .\message.txt -Algorithm SHA256

# Generate random bytes using .NET cryptographic randomness.
$bytes = New-Object byte[] 32
[System.Security.Cryptography.RandomNumberGenerator]::Fill($bytes)
[Convert]::ToHexString($bytes)

8.3 Python Examples

Python is useful for learning cryptographic concepts, but production systems should use reviewed, maintained libraries and secure protocols. Do not implement your own encryption scheme.

# Safe local example: compute a SHA-256 digest.
# This does not encrypt data. It produces a fixed-size fingerprint.

from pathlib import Path
import hashlib

path = Path("message.txt")
data = path.read_bytes()              # Read local file as bytes.
digest = hashlib.sha256(data).hexdigest()  # Compute SHA-256 and format as hex.
print(digest)

8.4 Network Example

When a browser connects to an HTTPS site, TLS helps the browser verify the server certificate, agree on session keys, and encrypt application data. The student should focus on understanding the layers, not bypassing them.

Browser --TLS handshake--> Server
Browser validates certificate chain
Browser and server derive session keys
HTTP request travels inside encrypted TLS records

8.5 Web Example

Web applications commonly use cryptography for HTTPS, signed cookies, password hashing, CSRF tokens, session tokens, and software-update verification. A common mistake is hiding secrets in JavaScript sent to the browser. Anything delivered to the browser should be considered visible to the user.

8.6 Cloud Example

Cloud providers often offer key-management services. These systems can generate keys, restrict access, log key use, rotate keys, and integrate with storage encryption. The security question is not merely “is it encrypted?” but also “who can use the key, under what conditions, and where is that use logged?”

9. Code Examples

9.1 Secure Random Tokens in Python

# Generate a cryptographically strong random token.
# Use secrets, not random, for security-sensitive randomness.

import secrets

# token_urlsafe returns a string suitable for URLs, cookies, or reset links.
# 32 bytes gives strong randomness for most application tokens.
token = secrets.token_urlsafe(32)

print(token)

Line-by-line: import secrets loads Python's security-focused randomness module. token_urlsafe(32) asks the operating system for 32 random bytes and encodes them safely for URLs. print displays the token for learning; production systems should avoid printing secrets to logs.

Common mistake: using random.random() for passwords, tokens, or cryptographic keys. The random module is designed for simulation, not security.

9.2 HMAC Verification

# Safe local example: protect message integrity with HMAC.
# HMAC is not encryption. It does not hide the message.

import hmac
import hashlib

secret_key = b"lab-only-demo-key-change-me"  # Shared secret key for this toy lab.
message = b"amount=100&to=student"

# Sender computes a tag over the message.
tag = hmac.new(secret_key, message, hashlib.sha256).hexdigest()
print("message:", message.decode())
print("tag:", tag)

# Receiver recomputes the tag and compares safely.
received_message = b"amount=100&to=student"
received_tag = tag
expected_tag = hmac.new(secret_key, received_message, hashlib.sha256).hexdigest()

if hmac.compare_digest(received_tag, expected_tag):
    print("valid: message was not changed and key matched")
else:
    print("invalid: message or tag does not match")

Line-by-line: hmac implements message authentication codes. hashlib.sha256 is the hash inside HMAC. The secret key must remain secret. The tag is computed over the message. The receiver recomputes it. compare_digest avoids timing-leak style comparison mistakes.

Common mistake: using a plain hash such as sha256(secret + message) instead of HMAC. HMAC exists to avoid construction pitfalls.

9.3 Password Hashing Demonstration with PBKDF2

# Educational password hashing example using the Python standard library.
# For new production systems, use a well-maintained password-hashing library
# with Argon2id where available. This example teaches salt and work factor.

import os
import hashlib
import hmac

password = b"correct horse battery staple"
salt = os.urandom(16)             # Unique random salt for this password record.
iterations = 600_000              # Work factor: higher means slower guessing.

stored_hash = hashlib.pbkdf2_hmac(
    "sha256",                    # Underlying hash function.
    password,                     # Password bytes.
    salt,                         # Unique salt.
    iterations                    # Work factor.
)

# Later, verify a login attempt.
attempt = b"correct horse battery staple"
attempt_hash = hashlib.pbkdf2_hmac("sha256", attempt, salt, iterations)

if hmac.compare_digest(stored_hash, attempt_hash):
    print("password accepted")
else:
    print("password rejected")

Secure coding point: password hashing is not the same as file hashing. A password hash must be salted and intentionally expensive to slow offline guessing. Avoid storing plaintext passwords. Avoid unsalted fast hashes.

9.4 Toy Caesar Cipher: Learning Only

# Toy cipher for learning modular arithmetic only.
# This is not secure and must never be used for real protection.

def caesar_encrypt(text, shift):
    result = []
    for ch in text.lower():
        if "a" <= ch <= "z":
            plain_index = ord(ch) - ord("a")
            cipher_index = (plain_index + shift) % 26
            result.append(chr(cipher_index + ord("a")))
        else:
            result.append(ch)
    return "".join(result)

print(caesar_encrypt("attack at dawn", 3))

Why include an insecure cipher? It teaches transformation, keys, and modular arithmetic. It is deliberately labeled insecure so students do not confuse historical examples with modern cryptography.

10. Security Perspective

Common Attacks

  • Eavesdropping: reading data in transit when encryption is missing or broken.
  • Tampering: modifying data when integrity checks are missing.
  • Replay: resending a valid old message to cause an unauthorized action.
  • Man-in-the-middle: intercepting and modifying communication when authentication or certificate validation fails.
  • Offline password guessing: stealing password hashes and testing guesses locally.
  • Key theft: stealing private keys, API keys, signing keys, or database encryption keys.
  • Side-channel attacks: learning secrets from timing, power use, cache behavior, or error differences.
  • Downgrade attacks: forcing weaker protocol versions or algorithms when negotiation is not protected.

Common Vulnerabilities

  • Hardcoded keys in source code.
  • Secrets committed to Git repositories.
  • Predictable random numbers.
  • Missing certificate validation.
  • Using encryption without authentication.
  • Reusing nonces in modes that require uniqueness.
  • Using obsolete algorithms or protocols.
  • Using fast unsalted hashes for password storage.
  • Keeping old keys active after staff changes or incidents.

Defensive Strategies

  • Use established, maintained cryptographic libraries.
  • Prefer secure protocols such as properly configured TLS rather than building custom protocols.
  • Use authenticated encryption where appropriate.
  • Use secure randomness from the operating system.
  • Store secrets in secret managers or key-management services, not in source code.
  • Log key-use events where supported, without logging secret values.
  • Rotate and revoke keys according to policy and incident evidence.
  • Maintain a cryptographic inventory for crypto agility.

Detection Methods

  • Scan repositories for committed secrets.
  • Monitor certificate expiration and unexpected certificate changes.
  • Alert on unusual key-management events.
  • Detect cleartext protocols where encrypted protocols are required.
  • Watch for repeated authentication failures that may indicate password guessing.
  • Inventory algorithms and protocol versions used by services.

Mitigation and Best Practices

Cryptographic mitigation usually combines correct design, safe libraries, good key management, monitoring, and incident response. Do not treat encryption as a checkbox. Ask exactly which goal the cryptography provides and what assumptions must hold.

12. Research Perspective

Cryptography is both a mature engineering discipline and an active research field. Current research areas include:

  • Post-quantum cryptography: algorithms designed to resist attacks from future large-scale quantum computers.
  • Crypto agility: designing systems so algorithms and keys can be replaced without full redesign.
  • Formal verification: proving that protocols or implementations satisfy security properties under stated assumptions.
  • Side-channel resistance: preventing leakage through timing, power, electromagnetic emissions, cache behavior, or fault injection.
  • Privacy-preserving computation: secure multi-party computation, homomorphic encryption, differential privacy, and zero-knowledge proofs.
  • Usable cryptography: making cryptographic systems safe for normal users and administrators.
  • Verified implementations: reducing bugs in cryptographic libraries through type systems, proofs, testing, and constant-time coding.

Open problems include migrating the internet to post-quantum-safe public-key systems, preventing implementation leaks, managing cryptography at cloud scale, making key ownership visible, and designing protocols that remain secure under realistic deployment failures.

13. Case Studies

13.1 Heartbleed

What happened: Heartbleed was a serious vulnerability in OpenSSL's TLS heartbeat implementation. It was not a failure of the AES algorithm or TLS concept itself; it was an implementation bug that could leak memory from affected systems.

Root cause: missing bounds checking in a security-critical library implementation.

Impact: sensitive memory, potentially including private keys and credentials, could be exposed from vulnerable systems.

Lesson: strong cryptographic design still depends on memory-safe, audited, patched implementations and key rotation after compromise.

13.2 DigiNotar Certificate Authority Compromise

What happened: attackers compromised a certificate authority and issued fraudulent certificates.

Root cause: weakness in CA security and operational controls.

Impact: fraudulent certificates threatened trust in browser-server authentication.

Lesson: public-key infrastructure depends on operational trust, not just mathematical algorithms.

13.3 Debian OpenSSL Randomness Bug

What happened: a change in Debian's OpenSSL package reduced randomness used for key generation.

Root cause: accidental weakening of entropy collection.

Impact: generated keys became predictable enough to require regeneration and replacement.

Lesson: randomness is a foundation. Predictable keys defeat strong algorithms.

13.4 Certificate Expiration Outages

What happened: many organizations have experienced outages because certificates expired unnoticed.

Root cause: poor certificate inventory, ownership, alerting, or automation.

Impact: services became unavailable or clients rejected connections.

Lesson: cryptography is operational. Expiration, rotation, monitoring, and ownership matter.

14. Common Misconceptions

  • “Base64 is encryption.” Incorrect. Base64 is encoding and provides no secrecy.
  • “Hashing encrypts data.” Incorrect. Hashing is one-way and has no decryption key.
  • “If data is encrypted, the system is secure.” Incorrect. Endpoints, keys, authorization, logging, and backups still matter.
  • “Secret algorithms are safer.” Usually incorrect. Modern cryptography relies on public review and secret keys.
  • “Long passwords can be stored with SHA-256.” Incorrect. Password storage requires salted, slow password hashing.
  • “TLS means the website is trustworthy.” Incorrect. TLS can protect communication with a domain, but it does not prove the business is honest or the application is bug-free.
  • “Public keys must be hidden.” Incorrect. Public keys are designed to be shared; private keys must be protected.
  • “Encryption gives integrity automatically.” Not always. Use authenticated encryption or an appropriate integrity mechanism.
  • “Quantum computers already broke all encryption.” Overstated. The risk is serious for future public-key systems, especially long-lived data, but migration is standards-driven and threat-model dependent.
  • “Cryptography removes the need for access control.” Incorrect. Access control decides who can use keys and data.

15. Summary

Cryptography helps protect information by supporting confidentiality, integrity, authenticity, digital signatures, and key establishment. The most important distinction is not the algorithm name alone, but the security goal, threat model, key management, implementation quality, and operational controls around it.

Encoding is not encryption. Hashing is not encryption. Encryption without authentication can be dangerous. Password hashing requires slow, salted schemes. Public-key cryptography enables certificates, signatures, and key establishment. TLS combines several cryptographic ideas to protect network communication. Cryptographic systems must be monitored, inventoried, updated, and designed for change.

16. Glossary

Algorithm
A defined procedure for performing a computation, such as encryption or hashing.
Authenticated Encryption
Encryption that also provides integrity and authenticity protection for ciphertext and sometimes associated data.
Certificate Authority
An entity that issues digital certificates binding identities to public keys.
Ciphertext
Encrypted data that should be unreadable without the correct key.
Confidentiality
The security property of keeping information secret from unauthorized parties.
Cryptographic Hash
A one-way function that maps data to a fixed-size digest.
Cryptography
The discipline of protecting information using mathematical methods.
Digital Signature
A value produced with a private key that can be verified with a public key.
Digest
The fixed-size output of a hash function.
HMAC
A keyed message authentication code construction based on a hash function.
Key
A value that controls a cryptographic operation.
Key Management
The lifecycle discipline for generating, storing, using, rotating, revoking, backing up, and destroying cryptographic keys.
MAC
Message authentication code; a keyed integrity and authentication tag.
Nonce
A value used once in a cryptographic operation.
Plaintext
Readable original information before encryption.
Private Key
The secret part of a public-key pair.
Public Key
The shareable part of a public-key pair.
Salt
A unique random value stored with a password hash to prevent identical passwords from producing identical stored hashes.
TLS
Transport Layer Security, a protocol for protecting client/server communication.
Trust Anchor
A trusted starting point, such as a root certificate, used to validate a certificate chain.

4. What does a cryptographic hash function produce?

Answer: A. A hash maps input data to a fixed-size digest.

5. Why is plain SHA-256 not enough for password storage?

Answer: B. Password storage needs slow, salted password hashing such as Argon2id, bcrypt, scrypt, or PBKDF2.

6. What does encryption primarily provide?

Answer: A. Encryption hides plaintext from parties without the key.

7. What does a MAC such as HMAC primarily provide when the key is secret?

Answer: B. A MAC lets someone with the shared secret key verify that data was produced by someone who also had that key and was not changed.

8. How is a digital signature different from a MAC?

Answer: A. Digital signatures support public verification using a public key.

9. Which item must usually remain secret?

Answer: C. The private key must be protected because it proves control of the identity or decrypts/derives sensitive secrets depending on the scheme.

10. What is a nonce?

Answer: A. Nonce means number used once; many schemes require nonce uniqueness.

11. What is the safest beginner rule about cryptographic algorithms?

Answer: B. Custom cryptography usually fails; use reviewed algorithms and maintained libraries.

12. What does TLS protect?

Answer: B. TLS is designed to protect network communication between applications.

13. What is a certificate authority?

Answer: A. A CA is part of the public-key infrastructure used to bind identities to public keys.

14. Which mistake can break otherwise strong encryption?

Answer: B. Nonce reuse can be catastrophic for some encryption modes.

15. What does authenticated encryption combine?

Answer: A. Authenticated encryption protects both confidentiality and integrity.

16. What is key management?

Answer: B. Key management covers the lifecycle of cryptographic keys.

17. Why is randomness important in cryptography?

Answer: B. Predictable randomness can make keys and tokens guessable.

18. Which statement about Base64 is correct?

Answer: B. Encoding changes representation, not secrecy.

19. What is crypto agility?

Answer: A. Crypto agility matters when algorithms age, standards change, or quantum-resistant migration is needed.

20. Which post-quantum concept is now important for long-lived systems?

Answer: B. Organizations should understand where cryptography is used and prepare for standardized post-quantum migration where appropriate.

18. Practice Lab

Authorization boundary: This lab uses only local files and your own commands. Do not intercept, decrypt, scan, or test real third-party systems.

Lab Title

Local Cryptography Building Blocks Lab

Prerequisites

  • A local Linux, macOS, or Windows environment.
  • Python 3 installed.
  • OpenSSL installed if available. If not, complete the Python portions only.

Objectives

  • Create a harmless local file.
  • Compute and compare file hashes.
  • Generate a secure random token.
  • Verify that HMAC detects tampering.
  • Explain why the lab examples do not replace production cryptographic design.

Steps

  1. Create a folder named crypto-lab.
  2. Create a file named message.txt containing one sentence.
  3. Compute the SHA-256 hash of the file using sha256sum, Get-FileHash, or Python.
  4. Change one character in the file and compute the hash again.
  5. Write down what changed and why.
  6. Run the Python HMAC example from this lesson.
  7. Change the message in the verification section and observe that verification fails.
  8. Generate a random token with Python's secrets module.
  9. Write a short report explaining hash, HMAC, token, key, and safe-use limitations.

Expected Outcomes

  • You can distinguish hashing from encryption.
  • You can explain why tiny input changes alter a hash digest.
  • You can demonstrate keyed integrity checking with HMAC.
  • You can generate a secure random token for a local example.
  • You can identify at least five cryptographic misuse risks.

Deliverable

Submit a one- to two-page lab report containing commands used, outputs with secrets redacted where appropriate, answers to the reflection questions, and a short section titled “What this lab does not prove.”
